## [vassal-litigator] этап 6.3: backfill-global (служебный скилл) -- 2026-05-30

### Added
- `skills/backfill-global/SKILL.md` -- новый служебный скилл для разового переноса накопленной локальной аналитики (`.vassal/hearings/judge-profile.md`, `.vassal/analysis/opponent-{filing,hearing}-*.md`) в глобальную память (`$VASSAL_GLOBAL_DIR/judges/`, `$VASSAL_GLOBAL_DIR/counterparties/`) для дел, которые анализировались до этапа 6.1 (включение двойной записи в глобальную) или иначе не попали в глобальный профиль. Алгоритм:
  - Фаза 0 -- сбор списка дел (один путь / папка-родитель с рекурсивным поиском / список путей; парсинг флага `--no-merge`); skip-критерии (нет `case.yaml`, нет `judge-profile.md` И `opponent-*.md`).
  - Фаза 1 -- сбор локальных источников по каждому делу.
  - Фаза 2 -- резолв целевых глобальных файлов (правила слагов из conventions); обработка теневых дублей `inn-*` / `noinn-*` без редактирования соседнего; проверка идемпотентности по полю `backfilled_cases:`.
  - Фаза 3 -- Opus-субагент: маппинг наблюдений в разделы глобального профиля + семантическая дедупликация с **инвертированным дефолтом «не объединяй»**, 7 запретов на merge (разные контексты источника / категории дел / стадии / стороны / нормы / представители / противоречия), обязательная сертификация каждого merge через `similarity_rationale` + `confidence: high|medium|low`. Low-confidence merges автоматически переводятся в new_bullets ещё до preview. Режим `--no-merge` пропускает дедупликацию -- все bullet'ы идут как new.
  - Фаза 4 -- preview со сквозной нумерацией new/merge, показом rationale, метки `[?]` для medium-confidence; точечное отклонение merge командой `отклонить merge {номера}`.
  - Фаза 5 -- apply с backup `.bak.{timestamp}` (ротация 5) на каждый целевой файл; восстановление из backup при сбое; добавление `shared_profile:` в локальные `judge-profile.md` при отсутствии; запись в `.vassal/history.md`.
  - Фаза 6 -- опциональный хук `sync-notion` для обновления Notion-зеркал `Judges` / `Counterparties`.
  - Повторный запуск на уже бэкфилленном деле -- диалог из 4 вариантов: пропустить / полный повтор (Opus решает) / только новые после даты прошлого backfill / произвольная дата отсечки от Сюзерена.
- `commands/backfill-global.md` -- команда-обёртка с парсингом флага `--no-merge` и примерами запуска.

На Windows `~` резолвится в `C:\Users\{имя}\` -- разные имена пользователей на разных машинах превращали путь в разные физические места без синхронизации. Если задана -- путь к конфигу становится `$VASSAL_CONFIG_DIR/notion-config.yaml`. Типовой сценарий: положить конфиг в OneDrive/Dropbox/Yandex.Disk-папку, на каждой машине установить env var на её локальный путь -- так на двух Windows-машинах с разными именами пользователей работает один и тот же синкаемый файл.

- `scripts/notion-config.example.yaml`: header дополнен инструкцией по env var.
- `shared/conventions.md`: подсказка по `REG_EXPAND_SZ` для Windows -- ключевая деталь: `setx` создаёт `REG_SZ` без раскрытия `%VAR%`, поэтому для кросс-машинной конфигурации не годится; нужен `reg add /t REG_EXPAND_SZ`.

### Note about Windows `reg add` vs `setx`
Изначально в этой же ветке предлагалось использовать `setx VASSAL_*_DIR "C:\Users\{имя}\OneDrive\..."`, что требовало знать имя пользователя на каждой машине. Лучшее решение -- `reg add /t REG_EXPAND_SZ /d "%OneDrive%\Документы\Claude Cowork\.vassal-..."`: значение хранится **с литералом `%OneDrive%`**, а Windows раскрывает его под каждого пользователя при создании нового процесса. Одна команда, два разных результата на двух машинах. См. `scripts/notion-init.md` §3.1 для готовых команд.
