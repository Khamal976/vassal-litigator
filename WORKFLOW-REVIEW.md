# vassal-litigator — проблемы workflow и оценка связности

**Дата:** 2026-04-20
**Версия плагина:** 0.4.0 (см. [CHANGELOG.md](CHANGELOG.md))
**Статус:** наблюдения по итогам анализа архитектуры, 12 SKILL.md и `shared/*`

Файл — заметки к будущим улучшениям. Не план работ, не список багов — перечень мест, где workflow недоопределён или даёт сбои.

---

## Проблемы и разрывы

### 1. `init-case` — команда без скилла

Есть [commands/init-case.md](commands/init-case.md), но нет `skills/init-case/SKILL.md`. Остальные 12 скиллов имеют парные команды, `init-case` выпадает.

Следствие: логика инициализации дела (структура `.vassal/`, заполнение `case.yaml`, создание `index.yaml`, `history.md`) живёт только в тексте команды. Переиспользовать её в других скиллах (например, автосоздание `.vassal/` при `intake` в пустой папке) — нельзя.

### 2. `update-index` — нет триггера

Скилл есть, но неясно **когда его запускать**. Между `intake` и `catalog`? Перед каждым `prepare-hearing`? Нет правила — значит, в реальности либо вызывается вручную, либо не вызывается вовсе, и рассинхронизация `index.yaml` с файловой системой копится.

### 3. Ручное попадание файлов в папки дела

Если пользователь скопирует файл в процессуальную папку мимо `intake` / `add-evidence` / `add-opponent`, md-зеркало не создастся, в `index.yaml` записи не появится. Файл будет физически в деле, но невидим для всех скиллов.

`update-index` это теоретически ловит (сканирует ФС, сравнивает с индексом), но только если его вызвать. **Системный риск целостности состояния.**

### 4. Жёсткая зависимость от `arbitrum-docx`

Четыре скилла (`prepare-hearing`, `analyze-hearing`, `appeal`, `cassation`) требуют внешний плагин для генерации `.docx`. В SKILL.md нет fallback-логики «что если `arbitrum-docx` недоступен».

При этом `draft-judgment` генерит `.docx` напрямую, без `arbitrum-docx`. **Непоследовательно** — либо все через внешний плагин, либо все сами.

### 5. Неявные soft-зависимости

- `build-position` «рекомендует» `legal-review`
- `prepare-hearing` «рекомендует» `build-position`
- `appeal` / `cassation` «рекомендуют» наличие позиций и отчётов о заседаниях

В реальности — **что будет, если рекомендованный предшественник пропущен**? Скилл делает упрощённый анализ? Падает? Спрашивает пользователя? В SKILL.md не формализовано. На практике поведение зависит от импровизации модели в момент запуска.

### 6. Изоляция анализов оппонента: письменный и устный не сводятся

**Уточнение** (после проверки SKILL.md): прямой коллизии перезаписи нет, пути разные:

- `add-opponent` пишет в `.vassal/analysis/opponent-{дата}.md` (экспресс-анализ письменного документа).
- `analyze-hearing` пишет в `.vassal/hearings/{дата}-opponent-analysis.md` (анализ устной позиции в заседании).

Реальная проблема — **семантическая**: оба артефакта называются «анализ оппонента за {дата}», но downstream-скиллы их не сводят.

`prepare-hearing` фаза 1 шаг 3 п.7 явно загружает только из `.vassal/analysis/opponent-*.md`. Анализ устной позиции из `analyze-hearing` (в `hearings/`) при подготовке к следующему заседанию **в контекст не попадает**.

Двойной риск:
- При одинаковой дате двух разных оппонентов или двух разных заседаний — каждое имя само в своей зоне всё-таки коллизит.
- Письменный анализ доступен `prepare-hearing`, устный — нет. Результат: подготовка к следующему заседанию строится на половине данных.

Решения:
- Префиксы: `opponent-filing-{дата}.md` (от `add-opponent`) и `opponent-hearing-{дата}.md` (от `analyze-hearing`) — оба в одной зоне `.vassal/analysis/`.
- Или единый индекс анализов оппонента `.vassal/analysis/opponent-index.md` со ссылками.
- В любом случае — в `prepare-hearing` загружать **обе** зоны, а не только `analysis/opponent-*`.

### 7. Нет скилла «мировое соглашение / отказ от иска»

`prepare-hearing` упоминает мировое соглашение как тип документа (в списке опций), но отдельной ветки workflow нет. Для живого ведения дела — дыра: мировое соглашение это часто отдельный трек со своей логикой (согласование условий, проекты, утверждение судом).

### 8. Нет обработки пропущенного заседания / переноса

Сценарий: заседание отложено, перенесено, не состоялось по техническим причинам. Ни один скилл явно это не обрабатывает. `analyze-hearing` предполагает, что транскрипция есть. `prepare-hearing` предполагает следующее заседание впереди. Промежуточные состояния не покрыты.

### 9. Нет версионирования `index.yaml`

Если `update-index` или `intake` сломают `index.yaml` (ошибка при записи, некорректный YAML), восстановить прошлое состояние нечем. Бэкапов нет. `.vassal/history.md` фиксирует действия, но не содержит снапшот данных.

## Оценка связности

| Критерий | Оценка | Комментарий |
|---|---|---|
| Логичность порядка фаз | ✅ | Фундамент → анализ → ведение → обжалование — корректная последовательность |
| Развязанность скиллов | ✅ | Через shared state (`.vassal/`), каждый скилл самодостаточен |
| Явность контрактов между скиллами | ⚠️ | Нет деклараций input/output, всё подразумевается через shared state |
| Защита целостности состояния | ⚠️ | Зависит от ручного `update-index`, нет автосверки |
| Покрытие реальных сценариев | ⚠️ | Нет веток: мировое соглашение, пропуск заседания, смена юрисдикции, третье лицо вступает в дело |
| Внешние зависимости | ⚠️ | `arbitrum-docx` — single point of failure для 4 скиллов |
| Reversibility / rollback | ❌ | Нет снапшотов `index.yaml`, сломанный sync откатить нечем |
| Наблюдаемость | ⚠️ | `history.md` ведётся, но неравномерно (не все скиллы пишут туда) |

---

## Резюме

Архитектура **разумно декомпозирована** на 12 скиллов + shared state, и каждый скилл в отдельности решает свою задачу. Но между скиллами **много неявных допущений**, которые становятся проблемами при реальном использовании:

- Нет формализованных контрактов (что скилл гарантированно производит, что гарантированно ожидает на входе).
- Нет защиты целостности shared state (целостность зависит от дисциплины пользователя).
- Есть сценарии живого ведения дела, которые не покрыты ни одним скиллом (мировое, пропуск заседания, третьи лица).
- Есть внешняя зависимость без fallback.

Если дорабатывать — приоритетно:
1. Превратить `init-case` в скилл + добавить автотриггер `update-index` после каждого модифицирующего скилла (пункты 1, 2, 3, 9).
2. Развести `.vassal/analysis/opponent-*.md` по префиксам (пункт 6).
3. Либо унифицировать генерацию `.docx` через `arbitrum-docx` во всех скиллах, либо сделать её опциональной с fallback на прямую генерацию (пункт 4).
4. Формализовать soft-зависимости: явно описывать в SKILL.md что делать, если предшественник не запускался (пункт 5).
5. Дорисовать отсутствующие ветки workflow (пункты 7, 8).
