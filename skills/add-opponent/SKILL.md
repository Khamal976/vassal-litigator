# add-opponent -- Прием документов оппонента

Прием, систематизация и первичный анализ документов от процессуального оппонента. Работает по принципу **preview -> apply**. Ключевое отличие от add-evidence: документы маркируются `source: opponent`, раскладываются в процессуальные папки (не в «Материалы от клиента/»), и агент сразу фиксирует ключевые аргументы оппонента.

## Предварительные условия

- Дело инициализировано: `.vassal/case.yaml` существует.
- В `case.yaml` определены стороны (чтобы знать, кто оппонент).
- Папка `Входящие документы/` существует.

## Алгоритм

### Фаза 1 -- Сохранение и обработка

1. Прочитай `.vassal/case.yaml` -- определи, кто оппонент (ответчик, истец, третье лицо). Возьми его `inn`, `ogrn`, `name`, `short_name` из `parties[]` -- это понадобится в фазе 4 для двойной записи в глобальную память.
2. Прочитай `.vassal/index.yaml` -- контекст существующих документов.
2.1. **Загрузи глобальный профиль оппонента** (если есть) -- по правилу из [shared/conventions.md](../../shared/conventions.md) → раздел «Глобальная память (кросс-дельная)»:
   - Разрезолви `$VASSAL_GLOBAL_DIR` (по умолчанию `~/.vassal-global/`).
   - Собери **оба** возможных имени файла:
     - `primary_path` = `counterparties/inn-{ИНН}.md`, если `inn` оппонента известен из `case.yaml`; иначе `null`.
     - `fallback_path` = `counterparties/noinn-{slug-name}.md` (slug по правилам conventions на основе `short_name`, иначе `name`).
   - Проверь существование обоих:
     - **Если существуют оба** -- ⚠️ это «теневой дубль» (см. раздел «Дедуп оппонента: inn-/noinn- теневой дубль» ниже). Выведи Сюзерену warning по шаблону оттуда. Рабочий профиль -- `primary_path`; но **загрузи оба** -- наблюдения из fallback не теряются, их можно учесть при сжатии для глобального профиля.
     - **Если существует только `primary_path`** -- загрузи его. Это типовой кейс «оппонент с ИНН, уже встречался».
     - **Если существует только `fallback_path`** -- загрузи его как рабочий. Если в текущем `case.yaml` `inn` теперь известен (а в прошлый раз был null) -- зафиксируй в шаге 19 факт: новый exp-анализ ляжет в `primary_path`, на `fallback_path` останется warning «теневой дубль появится после apply».
     - **Если ни один не существует** -- работай без глобального профиля, не падай. Это первый раз, когда этот оппонент встречается в твоей практике (или глобальная память ещё пуста).
   - Что даёт глобальный профиль: фон «у этого оппонента в других делах уже наблюдалось» **до** экспресс-анализа в фазе 4. Особенно ценно: типовые доводы (любит ли ст. 333 ГК, ходатайствует ли о снижении неустойки регулярно), известные представители, тактика по аналогичным категориям споров.
3. **Источник файлов -- `Входящие документы/`**. Если папка пуста -- сообщи и завершись.
4. Скопируй все файлы в `.vassal/raw/opponent-{ГГГГ-ММ-ДД}/`.
5. **OCR pipeline** -- как в intake:
   - PDF -> извлечение текста / OCR
   - DOCX -> Read tool / python-docx
   - Изображения -> tesseract / LLM fallback
6. **Классификация документов оппонента:**
   - Основной процессуальный документ (отзыв, ходатайство, пояснения, возражение, встречный иск)
   - Приложения к нему (доказательства оппонента)
   - Определить: это бандл (основной + приложения) или отдельные документы?

### Фаза 2 -- Preview

7. Определи целевое расположение. Документы оппонента идут в процессуальную папку:
   ```
   {ГГГГ-ММ-ДД} {Сторона оппонента} - {тип документа}/
   ```
   Например: `2026-04-10 Ответчик - отзыв на иск/`

8. Если есть приложения -- оформи как бандл:
   - Основной документ -- anchor бандла
   - Приложения -- members бандла

9. Покажи Сюзерену план:

```
Пакет: opponent-2026-04-10 (4 файла)

2026-04-10 Ответчик - отзыв на иск/
  отзыв.pdf -> 2026-04-10 ООО АСнова Отзыв на исковое заявление.pdf  [основной]
  прил_1.pdf -> 2025-12-15 ООО АСнова Акт приема-передачи.pdf  [приложение]
  прил_2.pdf -> 2026-01-20 ООО АСнова Платёжное поручение №45.pdf  [приложение]

ТРЕБУЮТ ПРОВЕРКИ:
  скан.jpg -> 0000-00-00 Неизвестный Документ.jpg

Ключевые аргументы оппонента (предварительно):
  - Оспаривает сумму: ссылается на частичную оплату 200 000 руб.
  - Заявляет о ненадлежащем качестве товара
  - Просит снизить неустойку по ст. 333 ГК
```

10. Дождись подтверждения.

### Фаза 3 -- Apply

11. Создай процессуальную папку в корне дела.
12. Скопируй файлы с новыми именами.
13. Создай md-зеркала (Haiku-субагенты, model: "haiku").
14. Обнови `.vassal/index.yaml`:
    - Все записи с `source: opponent`
    - Оформи бандл (если есть приложения): `bundle_id`, `anchor`, `members`
    - Обнови `next_id`
15. Добавь запись в `.vassal/history.md`. Если в фазе 4 был выполнен экспресс-анализ -- укажи: создан локальный `opponent-filing-*.md` и обновлён глобальный `$VASSAL_GLOBAL_DIR/counterparties/{slug}.md` (или: глобальная память недоступна -- {причина}).
16. **Очистка «Входящие документы/»** -> «На удаление/» (как в intake).

### Фаза 4 -- Экспресс-анализ (опционально)

17. Если основной документ оппонента -- процессуальный (отзыв, возражение, пояснения), предложи Сюзерену:
    «Сделать экспресс-анализ аргументов оппонента?»
18. Если да -- через Opus-субагент (model: "opus"):
    - Извлеки ключевые доводы оппонента
    - Сопоставь с нашей позицией (если есть файлы `*позиция.md` в корне дела)
    - Сопоставь с глобальным профилем оппонента (загружен в фазе 1 шаг 2.1, если есть): какие доводы повторяются из дела в дело, что нового именно в этом отзыве, какие представители подписали (новые vs известные).
    - Кратко: что нового, что опасно, что легко парируется
    - Сохрани в `.vassal/analysis/opponent-filing-{ГГГГ-ММ-ДД}-{doc-id}.md` (префикс `opponent-filing-` -- анализ **письменного** процдокумента оппонента; зеркальный префикс `opponent-hearing-` использует `analyze-hearing` для анализа устной позиции; doc-id -- идентификатор основного процессуального документа оппонента из `index.yaml`).
    - Покажи краткую сводку Сюзерену в чате
19. **Двойная запись в глобальный профиль оппонента** -- по правилу из [shared/conventions.md](../../shared/conventions.md) → «Глобальная память (кросс-дельная)»:
    - Разрезолви `$VASSAL_GLOBAL_DIR` (по умолчанию `~/.vassal-global/`).
    - Создай папку `$VASSAL_GLOBAL_DIR/counterparties/` если её нет (`mkdir -p`).
    - **Целевое имя файла -- по приоритету:**
      - Если `inn` известен в `case.yaml` → `inn-{ИНН}.md` (канонический путь).
      - Иначе → `noinn-{slug-name}.md` (slug по правилам conventions из `short_name`/`name`).
    - **Проверь теневой дубль перед записью:** ещё раз пройдись по обоим путям (`inn-...md` и `noinn-...md`) -- они уже были собраны в фазе 1 шаг 2.1. Если **существует не-целевой** файл (например, цель = `inn-7707083893.md`, и при этом найден `noinn-OOO-Romashka.md`) -- после успешной записи в целевой выведи Сюзерену warning по разделу «Дедуп оппонента: inn-/noinn- теневой дубль» ниже. Запись в целевой выполняется в любом случае -- агент **не редактирует и не удаляет** соседний файл (аппенд-only принцип распространяется и на «соседей»; мёрдж -- решение Сюзерена).
    - **Если целевой файл существует** -- открой и аппенд-only дополни наблюдениями из текущего экспресс-анализа. Каждое наблюдение помечается маркером `(дело {case.number}, документ "{краткое имя процдокумента}", {ГГГГ-ММ-ДД})`. Старые наблюдения не редактировать; противоречие -- добавлять как «уточнение».
    - **Если целевого файла нет** -- создай новый со фронтматтером (см. раздел «Формат глобального профиля оппонента» ниже) и наполнением из текущего экспресс-анализа. В `cases:` -- единственный текущий номер дела.
    - В обоих случаях: инкрементируй `filings_analyzed` на 1; добавь `case.number` в `cases:` (без дублей); если в подписантах оппонента есть новые ФИО представителей -- добавь в `representatives:`; обнови `last_updated` и `last_case`.
    - **Сбой записи** (нет прав, ФС недоступна): не падай -- зафиксируй локальный `opponent-filing-*.md`, предупреди Сюзерена шаблоном из conventions, отметь факт в `.vassal/history.md` (см. шаг 15).

**Зачем разведение префиксов:** `prepare-hearing` подтягивает аналитику оппонента glob-ом `.vassal/analysis/opponent-*.md`. Без префикса `filing` / `hearing` устная и письменная аналитика конфликтовали бы по имени файла; с префиксом -- обе попадают в контекст одновременно и не теряются между подготовками.

**Зачем двойная запись:** локальный `opponent-filing-*.md` живёт только в этом деле и насыщен doc-id'ами текущего индекса. Глобальный `counterparties/...md` накапливает паттерны оппонента **сквозь все дела Сюзерена** (типовые доводы, тактика, представители) и читается до анализа -- это ровно то, что в одном деле всегда «слепое пятно», а к третьему делу с тем же оппонентом становится ценным фоном.

## Формат глобального профиля оппонента (кросс-дельный)

Файл `$VASSAL_GLOBAL_DIR/counterparties/inn-{ИНН}.md` (или `noinn-{slug-name}.md` при отсутствии ИНН) -- **аппенд-only, агрегируется по всем делам Сюзерена с этим оппонентом**. Управляется по правилам из [shared/conventions.md](../../shared/conventions.md) → «Глобальная память (кросс-дельная)».

```yaml
---
type: counterparty-profile-global
display_name: "ООО «Ромашка»"
inn: "7707083893"                       # null если неизвестен (тогда noinn-{slug}.md)
ogrn: "1027700132195"                   # опционально
slug: "inn-7707083893"                  # совпадает с именем файла без .md
representatives:                         # ФИО представителей, накапливаются
  - "Иванов И. И. (ЗАО „Юр.компания Х"), 2024-2026"
  - "Петров П. П. (без указания организации), 2025"
cases:                                   # список номеров дел, без дублей
  - "А33-12345/2026"
  - "А40-99999/2025"
filings_analyzed: 5                      # экспресс-анализы письменных процдокументов
hearings_analyzed: 0                     # пополняется analyze-hearing (фаза будущая)
last_updated: 2026-05-10
last_case: "А33-12345/2026"
backfilled_cases:                        # опциональное; пишется ТОЛЬКО `backfill-global` (этап 6.3).
  - case: "А33-12345/2026"               # дела, чьи наблюдения попали в профиль не органически
    date: 2026-04-20                     # (через add-opponent/analyze-hearing), а через бэкфилл
                                         # накопленной локальной памяти. Используется для идемпотентности
                                         # повторных запусков backfill-global; другие скиллы это поле игнорируют.
---
```

### Структура (5 разделов с аннотациями источника)

```markdown
## 1. Типовые доводы в отзывах / возражениях
- Регулярно ссылается на ст. 333 ГК для снижения неустойки даже при очевидной несоразмерности (дело А33-12345/2026, отзыв от 2026-04-10; дело А40-99999/2025, отзыв от 2025-08-20).
- В делах о поставке выдвигает довод о ненадлежащем качестве товара без ссылки на акт приёмки (дело А33-12345/2026, отзыв от 2026-04-10).

## 2. Тактика и процессуальные паттерны
- Подаёт встречные иски при сумме > 1 млн руб. (дело А40-99999/2025, встречный от 2025-09-15).
- Ходатайствует об отложении в первом заседании, ссылаясь на болезнь представителя (дело А33-12345/2026, заседание 2026-03-25).
...

## 3. Представители оппонента
- Иванов И. И. -- основной с 2024 г., работает в связке с ЗАО „Юр.компания Х", агрессивный стиль выступлений, сильно опирается на технические возражения по процессу.
- Петров П. П. -- «второй номер», обычно подписывает приложения и формальные ходатайства.

## 4. Доказательственная стратегия
- Часто прилагает акты сверки, подписанные оппонентом в одностороннем порядке (то есть не доказательственные).
- Игнорирует требования суда о представлении первичных документов до 2-3 заседания (дело А33-12345/2026; дело А40-99999/2025).

## 5. Что работало против него
- В деле А40-99999/2025 удалось снять довод по ст. 333 ГК через детальный расчёт фактических убытков клиента (дополнение к позиции от 2025-10-12).
- В деле А33-12345/2026 ходатайство об экспертизе по подписям заставило оппонента отозвать спорный документ (отзыв на ходатайство от 2026-04-25).
```

### Правило накопления

При двойной записи (фаза 4 шаг 19):
- В соответствующий раздел дописываются новые наблюдения с маркером `(дело {номер}, документ "{краткое имя}", {ГГГГ-ММ-ДД})`.
- Старые наблюдения **не удаляются и не редактируются**; противоречие -- добавлять как «уточнение» с новой датой.
- `cases:` пополняется без дублей; `representatives:` -- если новый ФИО представителя, добавлять с указанием годов активности.
- `filings_analyzed` инкрементируется на 1; `last_updated` и `last_case` обновляются.
- `hearings_analyzed` -- пополняется через `analyze-hearing` при анализе устной позиции этого же оппонента (см. соответствующий скилл; механизм симметричен).

## Дедуп оппонента: inn-/noinn- теневой дубль

**Как возникает:** в одном деле `case.yaml.parties[].inn` оппонента не заполнен (например, забыли при init-case) → `add-opponent` создаёт `noinn-{slug}.md`. В другом деле тот же оппонент уже с ИНН → создаётся `inn-{ИНН}.md`. На один реальный субъект -- два файла в `$VASSAL_GLOBAL_DIR/counterparties/`, наблюдения раздроблены.

**Что делает скилл (фаза 1 шаг 2.1, фаза 4 шаг 19):** обнаруживает оба файла перед чтением/записью и предупреждает Сюзерена, **не редактирует** и **не удаляет** ни один из них -- мёрдж это решение человека (риск потерять наблюдения слишком велик, чтобы автоматизировать).

**Шаблон warning Сюзерену:**

```
⚠️ Теневой дубль профиля оппонента в глобальной памяти

Оппонент: {display_name из case.yaml}
Найдено две записи в $VASSAL_GLOBAL_DIR/counterparties/:
  • {primary_path}   (по ИНН -- канонический, рабочий)
  • {fallback_path}  (без ИНН -- образовался, когда в каком-то деле ИНН не был указан)

Текущий экспресс-анализ записан в: {фактический целевой путь}.
Соседний файл не тронут.

Рекомендация: смёрджить вручную.
  1. Открыть оба файла, сравнить разделы «1. Типовые доводы», «2. Тактика», «3. Представители», «4. Доказательственная стратегия», «5. Что работало против него».
  2. Скопировать уникальные наблюдения из {fallback_path} в {primary_path} (с сохранением маркеров источника).
  3. Обновить во frontmatter {primary_path}: `cases:` -- объединение списков; `filings_analyzed` и `hearings_analyzed` -- сумма; `representatives:` -- объединение; `last_updated` -- max.
  4. Удалить {fallback_path} вручную (или переместить в «На удаление/» на уровне глобальной папки -- если такая практика принята).

Превенция повторения: убедись, что в `case.yaml` всех новых дел этого оппонента ИНН заполнен (см. чек в legal-review фазы 3).
```

**Когда warning не выводится:** если найден только canonical (`inn-*.md`) или только fallback (`noinn-*.md`) -- это нормальная работа, тишина.

## Формат имени файла

```
ГГГГ-ММ-ДД Отправитель Описание.расширение
```

Отправитель для документов оппонента -- название организации оппонента (без кавычек).

## Формат процессуальной папки

```
{ГГГГ-ММ-ДД} {Сторона} - {действие}
```

Сторона: «Истец», «Ответчик», «Третье лицо» (или конкретное название, если сторон несколько одного вида).

## Граничные случаи

- Документы от нескольких оппонентов в одном пакете -> создай отдельную папку для каждого.
- Документ от суда (определение, протокол) попал сюда по ошибке -> сообщи, предложи переквалифицировать.
- Оппонент прислал копии наших же документов -> пометь как дубликат, не создавай отдельную запись.
