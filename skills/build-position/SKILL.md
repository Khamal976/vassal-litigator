# Build-Position — Выработка правовой позиции

Скилл анализирует материалы дела и формирует структурированную правовую позицию: основания требований/возражений, сильные и слабые стороны, оценку рисков, стратегию усиления. Работает с Opus. Каждый вывод — в аудируемом формате.

## Предварительные условия

- Дело инициализировано: `.vassal/case.yaml` существует.
- Документы приняты и проиндексированы.
- **Рекомендуется:** завершить `legal-review` (для наличия файлов в `.vassal/analysis/`).
- Если `legal-review` не выполнен — скилл работает, но предупредит: «Правовой анализ не проведён, рекомендуется запустить legal-review для более точной позиции».

## Входные данные

1. **case.yaml** — стороны, предмет, суд.
2. **index.yaml** — реестр документов.
3. **analysis/** — все файлы правового анализа (если есть).
4. **md-зеркала** — ключевые документы (договор, претензия, иск, отзыв, позиции сторон).
5. **Предыдущие позиции** — все файлы `*позиция.md` из корня дела (если есть, для обновления).
6. **Вводная Сюзерена** — за кого работаем (истец/ответчик), какие требования, акценты.

## Алгоритм

### Фаза 0 — Проверка консистентности ФС ↔ индекс

Перед загрузкой контекста выполни Phase 0 по правилу из [shared/conventions.md](../../shared/conventions.md) → раздел «Проверка консистентности ФС ↔ индекс (Phase 0)». При обнаружении рассинхрона предложи Сюзерену запустить `update-index`. Если Сюзерен выбрал «продолжить» -- зафиксируй в выходном артефакте, что позиция строится на устаревшем индексе.

### Фаза 1 — Загрузка контекста

1. Прочитай `.vassal/case.yaml` — определи: кто наш клиент, какая роль (истец/ответчик), предмет спора.
2. Прочитай `.vassal/index.yaml`.
3. Загрузи все файлы из `.vassal/analysis/` (если существуют).
4. Загрузи md-зеркала ключевых документов:
   - Договор + допсоглашения
   - Претензия / ответ на претензию
   - Акты, накладные, платёжные документы
   - Переписка сторон
   - Процессуальные документы обеих сторон (если дело уже в суде)
5. Загрузи предыдущие позиции из корня дела — все файлы по glob `{корень}/*позиция.md` (если есть — для обновления, а не создания с нуля). Сортируй по дате в имени, последняя — текущая. Папка `.vassal/positions/` **не используется** (исторический артефакт, в актуальной версии плагина все позиции живут только в корне дела).
6. Если Сюзерен дал вводную — прочитай и зафиксируй.

### Фаза 2 — Анализ (Opus)

Каждый вывод — в аудируемом формате (см. `shared/conventions.md`).

#### a) Предмет и основания требований

**Для истца:**
- Какие требования заявляем (основной долг, неустойка, проценты, убытки, иное)
- Фактическое основание каждого требования (какие факты подтверждают)
- Правовое основание (нормы ГК, специальных законов)
- Расчёт суммы требования (если возможно из документов)

**Для ответчика:**
- Какие требования предъявлены к нам
- Какие возражения можно заявить по каждому
- Встречные требования (если основания есть)

#### b) Доказательственная база

Для каждого факта, который нужно доказать:

| Факт | Бремя доказывания | Имеющиеся доказательства | Достаточность | Что усилит |
|------|-------------------|-------------------------|---------------|------------|
| Заключение договора | истец | Договор поставки №1045 от 28.11.2025 | достаточно | -- |
| Поставка товара | истец | УПД ЦБ-3823 + ТН ЦБ-574, УПД ЦБ-3848 + ТН ЦБ-577 | достаточно | акт сверки |
| Просрочка оплаты | истец | расчёт задолженности, гарантийное письмо | достаточно | акт сверки |

**ВАЖНО:** В источниках указывать названия документов с датами, а не doc-ID. Документ читает человек, а не агент.

#### c) Сильные стороны позиции

Для каждой:
- В чём сила аргумента
- На какие документы опирается
- Какую норму подкрепляет
- Судебная практика (если известна из контекста)

#### d) Слабые стороны и уязвимости

Для каждой:
- В чём уязвимость
- Как может использовать оппонент
- Можно ли нейтрализовать (и как)
- Насколько критично ([!] критичная | [*] существенная | [ ] управляемая)

#### e) Прогноз аргументов оппонента

- Какие возражения / требования вероятно заявит оппонент
- На чём они будут основаны
- Как на них отвечать (контраргументы)

#### f) Оценка рисков

Для каждого требования / позиции:

| Требование | Вероятность удовлетворения | Обоснование | Ключевой риск |
|-----------|---------------------------|-------------|---------------|
| Основной долг | высокая (80%) | документы в порядке | пропуск давности |
| Неустойка | средняя (50%) | могут снизить по 333 ГК | несоразмерность |

Общая оценка: перспективы дела с позиции нашего клиента.

#### g) Стратегия усиления позиции

Конкретные рекомендации:
- Какие доказательства получить
- Какие аргументы развить
- Какие ходатайства заявить
- Какие экспертизы / запросы инициировать
- Стоит ли предлагать мировое (и на каких условиях)

### Фаза 3 — Preview

7. Покажи Сюзерену краткую сводку:

```
ПРАВОВАЯ ПОЗИЦИЯ: {название дела}
Роль: {истец / ответчик}

Требования:
  1. Основной долг — {сумма} (перспектива: {оценка})
  2. Неустойка — {сумма} (перспектива: {оценка})
  ...

Сильные стороны: {N} аргументов
Слабые стороны: {M} уязвимостей ({X} критичных)

Общая оценка: {оценка перспектив}

Рекомендации: {K} пунктов по усилению

Сохранить?
```

8. Дождись подтверждения. Если Сюзерен корректирует — внеси правки.

### Фаза 4 — Apply

**Два уровня сохранения** (паттерн из [shared/conventions.md](../../shared/conventions.md) → «Двухуровневое сохранение для аналитических скиллов»):

**a) Единый документ для Сюзерена** — case root folder: `{ГГГГ-ММ-ДД} {тип} позиция.md`

Файл с полной позицией (формат — см. ниже). Тип:
- `Начальная` — первичная позиция
- `Обновленная` — обновлённая после новых документов
- `Ответ` — ответ на конкретный документ оппонента

**b) Отдельные файлы для агента** — `.vassal/analysis/position-{ГГГГ-ММ-ДД}-{секция}.md`

Параллельно сохрани каждый аналитический блок отдельным файлом. Это нужно `prepare-hearing`, `appeal`, `cassation` для прямого чтения структурированных секций без парсинга «человечного» сводного `.md`. Состав файлов:

| Файл | Содержание (из фазы 2) |
|---|---|
| `position-{дата}-evidence-base.md` | Доказательственная база -- таблица «Факт / Бремя / Доказательства / Достаточность / Что усилит» (из блока b) |
| `position-{дата}-strengths.md` | Сильные стороны позиции (из блока c) |
| `position-{дата}-weaknesses.md` | Слабые стороны и уязвимости с критичностью (из блока d) |
| `position-{дата}-opponent-forecast.md` | Стратегический прогноз аргументов оппонента (из блока e) -- общерамочный, не привязан к конкретному заседанию (тактический -- зона `prepare-hearing`) |
| `position-{дата}-risks.md` | Оценка рисков по требованиям (из блока f) |
| `position-{дата}-strategy.md` | Стратегия усиления, ACTION ITEMS (из блока g) |

Каждый файл начинается с frontmatter:
```yaml
---
type: position-section
section: evidence-base | strengths | weaknesses | opponent-forecast | risks | strategy
date: ГГГГ-ММ-ДД
case: "{номер дела}"
position_summary: "{ГГГГ-ММ-ДД} {тип} позиция.md"
supersedes: null | "{предыдущий файл этой секции из .vassal/analysis/}"
refs: ["doc-NNN", "doc-MMM", ...]   # doc-id всех источников, упомянутых в файле
---
```

Дальше — содержимое блока в аудируемом формате (как в сводном). Источники указывать **в формате двойных ссылок** `Название документа от ДД.ММ.ГГГГ [doc-NNN]` (см. [shared/conventions.md](../../shared/conventions.md) → «Двойные ссылки»).

Если по позиции какой-то блок отсутствует (например, для ответчика без встречного иска -- `evidence-base.md` неприменим в полном виде) — файл всё равно создаётся, но с явной пометкой `not_applicable: <причина>` во frontmatter и одной строкой в теле, чтобы цепочка ссылок не ломалась.

**c) Обновление карточки и истории**

9. Обнови `case.yaml` — поля только в зоне ownership `build-position` (см. [shared/conventions.md](../../shared/conventions.md) → «Ownership полей case.yaml»: `case.qualification.applicable_norms`, `case.amounts.{claim, court_fee}`).
10. Добавь запись в `.vassal/history.md`:
    ```
    ### {ЧЧ:ММ} — Правовая позиция (build-position)
    - Тип: {initial / updated / response-to-...}
    - Сводный файл: {ГГГГ-ММ-ДД} {тип} позиция.md
    - Файлы для агента: position-{дата}-{evidence-base, strengths, weaknesses, opponent-forecast, risks, strategy}.md в .vassal/analysis/
    - Требования: {перечень с оценками}
    - Общая оценка: {перспективы}
    - Критичные риски: {перечень}
    - Рекомендации: {кол-во} пунктов
    ```

11. Спроси Сюзерена: «Сформировать .docx?» Если да — используй скилл `arbitrum-docx` для оформления.

| Файл | Назначение | Для кого |
|---|---|---|
| `{ГГГГ-ММ-ДД} {тип} позиция.md` (корень дела) | Сводная позиция | Сюзерен |
| `.vassal/analysis/position-{дата}-evidence-base.md` | Доказательственная база | агент |
| `.vassal/analysis/position-{дата}-strengths.md` | Сильные стороны | агент |
| `.vassal/analysis/position-{дата}-weaknesses.md` | Слабые стороны и уязвимости | агент |
| `.vassal/analysis/position-{дата}-opponent-forecast.md` | Стратегический прогноз оппонента | агент |
| `.vassal/analysis/position-{дата}-risks.md` | Оценка рисков | агент |
| `.vassal/analysis/position-{дата}-strategy.md` | Стратегия усиления + ACTION ITEMS | агент |

## Формат файла позиции

```markdown
---
type: position
subtype: initial | updated | response-to-{документ}
date: ГГГГ-ММ-ДД
role: plaintiff | defendant
case: "{номер дела}"
supersedes: null | "{путь к предыдущей позиции}"
---

# Правовая позиция по делу {номер}

## 1. Предмет и требования
...

## 2. Доказательственная база
...

## 3. Сильные стороны
...

## 4. Слабые стороны и уязвимости
...

## 5. Прогноз аргументов оппонента
...

## 6. Оценка рисков
...

## 7. Стратегия усиления
...
```

## Обновление позиции

При повторном запуске (после новых документов или документов оппонента):

1. Загрузи предыдущую сводную позицию из case root folder + предыдущий комплект `.vassal/analysis/position-{предыдущая-дата}-*.md`.
2. Загрузи новые документы (по дате добавления в index.yaml).
3. **Покажи diff:** что изменилось в оценке по сравнению с предыдущей позицией -- по каждой секции (evidence-base, strengths, weaknesses, opponent-forecast, risks, strategy).
4. Сохрани новую версию:
   - Сводный `.md` в корне дела с `supersedes: "{путь к предыдущей сводной}"` в frontmatter (обновлённая дата в имени файла).
   - Комплект `.vassal/analysis/position-{новая-дата}-*.md` -- в каждом файле frontmatter `supersedes: "{путь к предыдущему файлу этой секции}"`.
5. Предыдущий комплект **не удаляется** -- это история эволюции позиции (см. «Идемпотентность»).

## Идемпотентность

Каждый запуск создаёт новый файл с датой. Предыдущие позиции не удаляются — это история развития позиции. Аналогично -- предыдущие комплекты `.vassal/analysis/position-{дата}-*.md` сохраняются, новые лишь ссылаются на них через `supersedes`.

## Постусловия

Перед завершением `apply` Opus-субагент сверяется с чек-листом из [shared/conventions.md](../../shared/conventions.md) → «Постусловия скиллов» категория 2 (аналитические). Минимум для `build-position`:

- Сводный `{ГГГГ-ММ-ДД} {тип} позиция.md` в корне дела создан, валидный YAML frontmatter.
- Все 6 файлов `position-{дата}-{секция}.md` созданы в `.vassal/analysis/` (или явно помечены `not_applicable: <причина>`).
- Сводный файл и детальные файлы согласованы: каждое утверждение в сводном имеет корреспондирующий блок в одном из детальных, и наоборот; источники в обоих местах -- двойные ссылки.
- `supersedes`-цепочка корректна: у первой позиции в деле -- `null`, у каждой последующей -- путь к файлу предшественника той же секции.
- `case.yaml` остался валидным; обновлены только поля в зоне ownership (см. фазу 4 п.9).
- Запись в `.vassal/history.md` зафиксирована.

## Граничные случаи

- Нет analysis/ → работай, но предупреди о рекомендации legal-review.
- Нет документов оппонента → строй позицию только на своих, пометь: «Позиция оппонента неизвестна».
- Спор неденежный (признание, обязание) → адаптируй структуру (вместо «сумма» — «предмет требования»).
- Множество требований → группируй и оценивай каждое отдельно.
- Встречный иск → включи анализ и требований, и встречных требований.

## Маршрутизация моделей

- **Opus**: весь анализ (фаза 2) — стратегическая юридическая работа.
- **Sonnet**: preview, форматирование, сохранение.
- **Haiku**: не используется.
