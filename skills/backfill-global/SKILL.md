# backfill-global -- Перенос локальной аналитики в глобальную память

Служебный скилл. Разово переносит накопленные в `.vassal/hearings/judge-profile.md` и `.vassal/analysis/opponent-{filing,hearing}-*.md` наблюдения в глобальную память (`$VASSAL_GLOBAL_DIR/judges/`, `$VASSAL_GLOBAL_DIR/counterparties/`) для дел, которые анализировались **до** того, как `analyze-hearing`/`draft-judgment`/`add-opponent` обзавелись фазой двойной записи в глобальную (этап 6.1, апрель 2026).

Без этого скилла наблюдения по старым делам остаются заперты в `.vassal/` конкретного дела -- глобальный профиль судьи/оппонента не видит их, и плагин подходит к каждому такому судье как к «новому», даже если у Сюзерена с ним уже было 3-5 дел.

## Когда нужен

- Активные или архивные дела с накопленной локальной аналитикой (`judge-profile.md` с заседаниями или `opponent-*.md` с экспресс-анализами), которые **не прошли** через `analyze-hearing` / `draft-judgment` / `add-opponent` после включения двойной записи в глобальную (этап 6.1).
- Партия дел у одного судьи (3+ заседаний у этого судьи в локальной памяти, но в глобальной -- пусто).
- Несколько дел с одним и тем же оппонентом, которые наполняли локальный профиль до этапа 6.1.

Если все твои свежие дела уже органически наполняют глобальную память через обычный workflow -- этот скилл не нужен. Запуск -- один-два раза на «дочерней» партии накопленных дел.

## Предварительные условия

- `$VASSAL_GLOBAL_DIR` разрезолвлен (по умолчанию `~/.vassal-global/`); папка пишется и читается.
- В каждом обрабатываемом деле -- валидный `.vassal/case.yaml` с заполненными `case.number`, `case.judge`, `case.court`, `case.parties[]`. Без них дело идёт в skip.

## Алгоритм

### Фаза 0 -- Сбор списка дел

1. Разрезолви вход от Сюзерена:
   - Аргументы команды -- путь(и) к корням дел и опционально флаги.
   - **Флаг `--no-merge`**: если присутствует -- выключи Opus-этап семантической дедупликации в фазе 3 (см. фазу 3 → «Режим `--no-merge`»). По умолчанию (без флага) -- дедупликация включена.
   - Если позиционный аргумент -- путь, содержащий `.vassal/case.yaml` → одно дело.
   - Если путь не содержит `.vassal/case.yaml` → папка-родитель: рекурсивно ищи `**/.vassal/case.yaml` с лимитом глубины 4 уровня (защита от глубоких симлинков и архивов).
   - Несколько позиционных путей → список корней дел.
   - Пустой ввод (нет ни одного позиционного аргумента) → спроси Сюзерена: «Передай путь к корню дела или к папке-родителю».
   - Дедуп по `case.number` (если два пути ведут в одно дело -- оставить один; warning в preview).

2. Для каждого кандидата прочитай `.vassal/case.yaml`. Собери:
   - `case_root` (абсолютный путь к корню дела).
   - `case.number`, `case.judge`, `case.court`.
   - `case.parties[]` где роль ≠ роли нашего клиента → оппонент(ы). Для каждого -- `inn`, `ogrn`, `short_name`/`name`.

3. Skip-критерии (с явной причиной в preview, без падения):
   - `.vassal/case.yaml` невалиден или отсутствует.
   - `case.judge` пуст И нет ни одного `opponent-*` файла → нечего бэкфиллить по делу.
   - Папка `.vassal/hearings/` отсутствует И в `.vassal/analysis/` нет `opponent-*` файлов → нечего бэкфиллить.

### Фаза 1 -- Сбор локальных источников

Для каждого не-skip дела собрать списки источников:
- `local_judge_profile` = `case_root/.vassal/hearings/judge-profile.md` (если есть).
- `local_opponent_filings` = `case_root/.vassal/analysis/opponent-filing-*.md` (все по glob).
- `local_opponent_hearings` = `case_root/.vassal/analysis/opponent-hearing-*.md` (все по glob).

### Фаза 2 -- Резолв целевых глобальных файлов

1. Разрезолви `$VASSAL_GLOBAL_DIR`; создай `judges/` и `counterparties/`, если не существуют.

2. **Для судьи** -- целевой путь: `judges/{ФИО-slug}--{court-slug}.md` по правилам слагов из [shared/conventions.md](../../shared/conventions.md) → «Глобальная память (кросс-дельная)» → «Правила слагов».

3. **Для каждого оппонента** дела:
   - Если `inn` известен в `case.parties` → канонический путь `counterparties/inn-{ИНН}.md`.
   - Иначе → fallback `counterparties/noinn-{slug-name}.md` (slug от `short_name`, иначе от `name`).
   - **Проверь теневой дубль** (оба пути одновременно существуют) -- по правилу из [skills/add-opponent/SKILL.md](../add-opponent/SKILL.md) → «Дедуп оппонента: inn-/noinn- теневой дубль». При обнаружении -- warning в preview по тому же шаблону; запись пойдёт **только в канонический**, соседний файл не редактировать.

4. Прочитай существующие глобальные файлы (если есть): фронтматтер (`cases:`, `backfilled_cases:`, счётчики) + тело по разделам -- это будет фоном для Opus-сравнителя в фазе 3.

5. **Идемпотентность**: если `case.number` уже присутствует в `backfilled_cases:` целевого глобального файла → действие зависит от выбора Сюзерена, см. раздел «Повторный запуск на уже бэкфилленном деле» ниже.

### Фаза 3 -- Opus-субагент: маппинг наблюдений с семантическим дедупом

Для каждой пары (локальный источник × целевой глобальный файл) делегируй Opus-субагенту (model: "opus").

**Вход субагенту:**
- Содержимое локального источника целиком.
- Содержимое существующего глобального файла (если есть; иначе -- пометка «целевой файл будет создан»).
- Метаданные: `case.number`, тип источника (`judge` / `opponent-filing` / `opponent-hearing`), целевой путь.
- Правило маппинга разделов для оппонентов: `opponent-filing-*` → разделы 1 «Типовые доводы», 4 «Доказательственная стратегия», 5 «Что работало против него»; `opponent-hearing-*` → разделы 2 «Тактика и процессуальные паттерны», 3 «Представители оппонента». Для судьи -- все 7 разделов в зеркальной структуре.

**Задача субагента:**
1. Извлеки bullet-наблюдения из локального файла с уже имеющимися маркерами `(заседание ГГГГ-ММ-ДД)` / `(решение ГГГГ-ММ-ДД)` / `(документ "..." от ГГГГ-ММ-ДД)`.
2. Преобразуй маркеры в формат глобального: добавь `дело {case.number}, ...` к существующему маркеру. Если маркера даты в локальном bullet'е нет → оставь только `(дело {case.number})` и пометь bullet как `marker_missing: true`.
3. **Семантическая дедупликация -- default «не объединяй»**. Сравни каждый bullet с существующими наблюдениями в соответствующем разделе глобального файла. Основной режим работы -- **каждый bullet идёт как новый**. Объединение -- только при ВЫСОКОЙ уверенности, что это та же самая ситуация, иначе сформулированная. При любом сомнении -- оставляй как два отдельных bullet'а. Ложный merge безвозвратно искажает историю наблюдений (теряется один из двух текстов); лишний new bullet легко свернуть руками потом.

   **Запреты на объединение (если хотя бы один пункт срабатывает -- bullet НОВЫЙ, не merge):**
   - Разные **контексты источника**: устный (заседание) vs письменный (решение, отзыв, документ оппонента).
   - Разные **категории дел**, если в одном из bullet'ов есть привязка (например, «в делах о поставке» / «в банкротстве» / «в подрядных спорах») и она не совпадает.
   - Разные **процессуальные стадии**: первая инстанция / апелляция / кассация / надзор.
   - Разные **стороны**, к которым относится наблюдение (как ведёт себя с истцом vs с ответчиком; нашёл слабость у клиента vs у оппонента).
   - Разные **нормы**, упомянутые в bullet'е (наблюдение про ст. 333 ГК ≠ наблюдение про ст. 395 ГК, даже если оба про «расчёт»; ст. 61.2 ФЗ-127 ≠ ст. 61.3 ФЗ-127).
   - Разные **представители оппонента** (раздел 3 counterparty) -- никогда не объединять между разными ФИО.
   - **Противоречия в деталях** -- не объединять, оставлять как два наблюдения (соответствует общему правилу из conventions.md «Глобальная память» → «Правило накопления»: противоречия добавляются как уточнение, разрешать -- задача Сюзерена при чтении).

   **Каждый merged-кандидат обязан быть сертифицирован**:
   - `similarity_rationale` -- 1-2 предложения: какие признаки совпали, почему ни один из запретов выше не сработал.
   - `confidence: high | medium | low`.
     - `low` -- автоматически переводится в `new_bullet` ещё **до** preview (Сюзерену не показывается); запись в `warnings` -- «N кандидатов на merge отклонены автоматически как low-confidence».
     - `medium` -- помечается в preview как сомнительный; **по умолчанию идёт как new**, объединяется только при явном выборе Сюзерена.
     - `high` -- показывается в preview как merge с rationale; Сюзерен может точечно отклонить (фаза 4).

4. Для opponent-файлов -- распредели наблюдения по разделам 1-5 по правилу маппинга. Bullet может попасть в более чем один раздел, если содержит несколько различных наблюдений -- разбей на отдельные bullet'ы по одному в раздел.

5. Верни структуру (YAML):
   ```yaml
   target_file: "judges/Иванов-И-И--АС-Красноярского-края.md"
   new_bullets:
     - id: 1                                    # сквозная нумерация для preview
       section: "1. Общий стиль"
       text: "Активный, прерывает представителей."
       marker: "(дело А33-12345/2026, заседание 2026-04-10)"
       marker_missing: false
   merged_bullets:
     - id: 1                                    # сквозная нумерация для точечного отклонения в preview
       confidence: high                         # high | medium (low уже отфильтрован выше)
       existing_text_with_marker: "Активный судья, перебивает (дело А40-99999/2025, заседание 2025-08-10)."
       new_marker: "дело А33-12345/2026, заседание 2026-04-10"
       combined_text: "Активный судья, перебивает (дело А40-99999/2025, заседание 2025-08-10; дело А33-12345/2026, заседание 2026-04-10)."
       similarity_rationale: "Оба наблюдения говорят об активности судьи в заседаниях (один источник-тип, устный); оба относятся к стилю общения; противоречий в деталях нет."
   warnings:
     - "3 bullet'а без даты заседания -- помечены только (дело А33-67890/2026)"
     - "2 кандидата на merge отклонены автоматически как low-confidence -- идут как new_bullets"
   ```

**Режим `--no-merge`** (если флаг передан Сюзереном в аргументах команды): Opus-этап семантической дедупликации **пропускается**. Все локальные bullet'ы идут как `new_bullets` без сравнения с существующими наблюдениями в глобальном. `merged_bullets` всегда пустой. Полезно для первого прогона на большой партии дел -- сначала собрать всё в глобальный, потом руками свернуть. В preview явно отмечается режим «no-merge: все наблюдения идут как new».

Параллелить субагентов по делам (несколько Opus-задач одновременно по разным глобальным файлам).

### Фаза 4 -- Preview

Покажи Сюзерену сводный отчёт по шаблону (см. раздел «Формат preview» ниже). В preview обязательно:
- Нумеровать каждый `new_bullet` и каждый `merged_bullet` (сквозная нумерация в рамках одного глобального файла) -- чтобы Сюзерен мог точечно отклонять по номерам.
- Для каждого `merged_bullet` показывать `similarity_rationale`, `confidence` и diff «было / стало».
- `merged_bullet` с `confidence: medium` -- помечать визуально (например, префиксом `[?]` или жёлтой пометкой); они по умолчанию **остаются как merge**, но Сюзерен может отклонить (см. ниже).
- Отдельно отметить, если включён режим `--no-merge` -- preview явно сообщает «семантическая дедупликация выключена, все bullet'ы идут как new».

Дождись подтверждения:
- `yes` / `да` / `apply` → переходи к фазе 5 со всеми изменениями.
- `отклонить merge {номера}` (например, `отклонить merge 3, 7, 12`) → переведи указанные `merged_bullets` в `new_bullets` (с тем же текстом, что был исходным в локальном источнике, и маркером `(дело {номер}, ...)`); пересобери план и покажи обновлённый preview; жди следующего подтверждения. Можно повторять любое количество раз.
- `выборочно` (по делам или глобальным файлам) → попроси Сюзерена указать, что исключить из apply целиком; пересобери план и покажи новый preview; жди `yes`.
- `нет` / `отмена` → завершиться без записи; ничего не менять.

### Фаза 5 -- Apply

Для каждого целевого глобального файла, получившего изменения от Opus-субагента в фазе 3:

1. **Backup** -- перед записью создай `{целевой путь}.bak.{timestamp}` (ротация 5 последних бэкапов на файл, чтобы не разрастаться).

2. **Если файл не существует**:
   - Создай с фронтматтером по формату из [skills/analyze-hearing/SKILL.md](../analyze-hearing/SKILL.md) → «Формат глобального профиля судьи» (для judges) или [skills/add-opponent/SKILL.md](../add-opponent/SKILL.md) → «Формат глобального профиля оппонента» (для counterparties).
   - Заполни тело новыми bullet'ами по соответствующим разделам.
   - Добавь поле `backfilled_cases:` (опциональное, см. раздел «Поле backfilled_cases» ниже) -- запиши `{case: "{номер}", date: "{сегодня}"}`.

3. **Если файл существует**:
   - **Фронтматтер**: `cases:` пополнить без дублей; счётчики (`hearings_total` / `written_decisions_total` / `filings_analyzed` / `hearings_analyzed`) инкрементировать на число `new_bullets` соответствующего типа (merged НЕ инкрементируют); `last_updated` = сегодня; `last_case` = последний обработанный `case.number`; `backfilled_cases:` пополнить (см. раздел ниже).
   - **Тело**: каждый `new_bullet` -- аппенд в соответствующий раздел; каждый `merged_bullet` -- заменить `existing_text_with_marker` на `combined_text`. Никакие старые наблюдения, не вошедшие ни в `new_bullets`, ни в `merged_bullets`, не редактировать и не удалять.

4. **Локальный `judge-profile.md`** обработанного дела -- если во фронтматтере отсутствует `shared_profile:`, добавь его (одна правка во фронтматтер по правилу из conventions). Это обеспечит, что при следующем `analyze-hearing` на этом деле скилл сразу найдёт глобальный путь. Если фронтматтер совсем отсутствует (совсем старый формат) -- создать его с минимальным набором полей по образцу [skills/analyze-hearing/SKILL.md](../analyze-hearing/SKILL.md) → «Формат профиля судьи (локальный, на дело)».

5. **`.vassal/history.md`** каждого обработанного дела -- запись:
   ```
   ГГГГ-ММ-ДД ЧЧ:ММ  backfill-global
     judges/{slug}.md: +N new, M merged
     counterparties/inn-{ИНН}.md: +K new, L merged (разделы: 1, 3, 4)
     warnings: {если были}
   ```

6. **Удаление backup** -- если запись прошла успешно, удалить `{целевой путь}.bak.{timestamp}` (оставить ротированные предыдущие, если есть). При сбое записи -- восстановить из backup, файл пометить как «не обновлён» в сводном отчёте, продолжить с остальными.

7. **Сводный итог Сюзерену** в конце:
   - Обработано дел: X
   - Пропущено: Y (с причинами)
   - Обновлено глобальных файлов: Z (создано -- A, дополнено -- B)
   - Добавлено наблюдений: всего N (new) + M (merged)
   - Warning'и: список (теневые дубли, наблюдения без даты, сбои записи если были)

### Фаза 6 -- Опциональный хук: sync-notion

Если `$VASSAL_CONFIG_DIR/notion-config.yaml` существует -- предложи Сюзерену:
> «Глобальная память обновлена. Прогнать `/vassal-litigator:sync-notion` по обработанным {X} делам, чтобы Notion-зеркала `Judges` / `Counterparties` обновились? [yes / нет]»

Не блокирующий. Если конфига нет -- не предлагай. См. также [skills/notion-sync/SKILL.md](../notion-sync/SKILL.md) → раздел «Авто-триггеры».

## Повторный запуск на уже бэкфилленном деле

При обнаружении `case.number` в `backfilled_cases:` целевого глобального файла (любого из судьи/оппонента):

```
⚠️ Дело {case.number} уже бэкфилилось {date} в:
  - judges/{slug}.md
  - counterparties/{slug}.md
  - ...

С тех пор могли быть новые заседания или документы в локальной памяти этого дела.

Варианты:
  1. Пропустить дело целиком (по умолчанию)
  2. Перезапустить полностью -- Opus сам решит, что новое vs уже бэкфилленное, по семантическому сравнению с текущим состоянием глобального
  3. Только новые наблюдения с маркером даты > {date предыдущего backfill}
     (работает только если все новые bullet'ы в локальном имеют даты заседаний/решений; bullet'ы с marker_missing будут проигнорированы с warning'ом)
  4. Указать дату отсечки вручную
     Введи дату в формате ГГГГ-ММ-ДД -- скилл возьмёт только наблюдения с маркером даты > указанной.
     Bullet'ы с marker_missing будут проигнорированы (warning).
```

Подожди выбор. По умолчанию (нет ответа / `пропустить`) -- вариант 1.

Если выбран **вариант 2** -- Opus-сравнитель опирается на семантический дедуп: то, что уже есть, не задвоит; новое -- добавит. Это безопаснее, чем «полная перезапись», но требует доверия к семантическому сравнителю (можно ошибиться в обе стороны; страховки фазы 3 шаг 3 «запреты на объединение» работают).

Если выбран **вариант 3** -- фильтрация локальных bullet'ов по дате **в маркере наблюдения** (заседание / решение / документ) до Opus-фазы; нефильтрованные bullet'ы и `marker_missing: true` -- в warning «исключено N bullet'ов без даты».

Если выбран **вариант 4** -- то же, что вариант 3, но с произвольной датой от Сюзерена (не `date` предыдущего backfill, а указанная). Зачем нужен: Сюзерен может выбрать дату отсечки **между** датой предыдущего backfill и текущим моментом (например, «после моего отпуска 2026-04-25»). Фильтр работает строго **по дате в маркере наблюдения**, не по `mtime` локального файла -- наблюдение, добавленное в локальный профиль вчера, но относящееся к заседанию 2026-03-10, при отсечке `> 2026-04-25` **не пройдёт**. Если хочешь подтянуть такое наблюдение -- выбирай вариант 2 с Opus-дедупом.

## Поле `backfilled_cases:` во фронтматтере глобального

Опциональное поле, **пишется только этим скиллом**. Назначение:
- Идемпотентность повторных запусков (см. раздел выше).
- Аудит: отличать дела, чьи наблюдения пришли через backfill, от дел в общем workflow (для которых `analyze-hearing` / `draft-judgment` / `add-opponent` наполняли глобальный профиль органически в момент анализа).

Формат:
```yaml
backfilled_cases:
  - case: "А33-12345/2026"
    date: 2026-04-20
  - case: "А40-99999/2025"
    date: 2026-04-20
```

Никакая логика других скиллов на это поле не завязана -- они его игнорируют. Если поле отсутствует -- `backfill-global` обрабатывает дело как «никогда не бэкфилилось». Если ты вручную удалишь это поле -- скилл при следующем запуске сделает повторный backfill (по сценарию выше).

## Граничные случаи

- Локальный `judge-profile.md` старого формата (без `shared_profile:` во фронтматтере) → добавить `shared_profile:` в фазе 5 шаг 4. Никакие другие поля не трогать.
- Локальный `judge-profile.md` без какого-либо фронтматтера (совсем старый формат) → создать фронтматтер с минимальным набором (`type: judge-profile`, `judge:`, `court:`, `shared_profile:`, `hearings_analyzed: {из подсчёта}`, `last_updated:`); warning в preview «локальный profile без frontmatter -- добавлен полный frontmatter».
- Случай «дело инициализировано, заседания были, но `judge-profile.md` отсутствует» → нечего бэкфиллить по судье; если есть `opponent-*.md` -- работаем только с оппонентом.
- Несколько оппонентов в одном деле → по каждому свой целевой глобальный файл; preview и apply -- параллельно по всем.
- Opus-субагент не нашёл семантических дублей → все bullet'ы идут как `new` -- это нормально, не warning.
- Один и тот же оппонент в нескольких обрабатываемых делах → объединять обработку нельзя (нарушит маркеры); каждое дело даёт свой «проход» с маркером `(дело {номер})`. Семантический дедуп между делами сработает на этапе Opus-сравнения.
- Сюзерен выбрал «выборочно по делам» в фазе 4 → собрать список исключённых, повторно показать preview без них, дождаться `yes`.
- Сбой записи в один из глобальных файлов (нет прав, ФС недоступна, OneDrive conflict copy) → не откатывать все апдейты; восстановить только пострадавший файл из backup, отметить в сводном отчёте, продолжить с остальными.

## Постусловия

После apply:
- Каждый целевой глобальный файл, получивший изменения, валиден (фронтматтер парсится как YAML; тело -- валидный markdown).
- `backfilled_cases:` обновлено во всех затронутых глобальных файлах; `cases:` без дублей; счётчики инкрементированы ровно на число `new_bullets` соответствующего типа (`merged_bullets` не инкрементируют).
- **Аппенд-only не нарушен**: ни один существующий bullet не удалён; merged-bullets имеют объединённый маркер (старый текст не потерян, только дополнен новым делом через `;`).
- Локальный `judge-profile.md` каждого обработанного дела имеет `shared_profile:` во фронтматтере.
- `.vassal/history.md` каждого обработанного дела обновлён с записью «backfill-global» и счётчиками.
- При теневом дубле -- соседний (не-канонический) файл НЕ редактировался; warning остался для ручного мёрджа.
- При сбое записи -- backup восстановлен; сводный отчёт явно перечисляет, какие файлы не обновились и почему; `.vassal/history.md` обработанных до сбоя дел уже обновлён.

## Опции команды

- `--no-merge` -- выключить Opus-этап семантической дедупликации (фаза 3). Все локальные bullet'ы идут как `new_bullets` без сравнения с существующими наблюдениями в глобальном. Полезно для первого прогона на большой партии дел: сначала собрать всё, потом свернуть руками или повторным запуском без флага.

## Зависимости

- `$VASSAL_GLOBAL_DIR` (env var; fallback `~/.vassal-global/`).
- Opus-субагент (для семантического сравнения наблюдений). При `--no-merge` -- субагент используется только для распределения наблюдений оппонента по разделам 1-5 и преобразования маркеров; сравнения с глобальным нет.
- Внешние плагины не задействованы (нет `.docx`-генерации, нет OCR).

## Что НЕ делает (осознанные ограничения)

- Не мигрирует формат локальных файлов сверх минимума (только добавляет `shared_profile:` в `judge-profile.md`, если отсутствует; и создаёт фронтматтер, если совсем нет).
- Не сжимает / не упорядочивает существующие глобальные наблюдения (только добавляет `new_bullets` и обновляет маркеры `merged_bullets`).
- Не работает с делами без `.vassal/case.yaml`.
- Не вызывает `notion-sync` автоматически (только предлагает в фазе 6).
- Не извлекает наблюдения из транскрипций или решений -- работает строго с уже сделанной локальной аналитикой в `.vassal/`.
- Не удаляет дубли `inn-*` / `noinn-*` (мёрдж теневого дубля -- ручная операция Сюзерена по инструкции из `add-opponent/SKILL.md`).
- Не объединяет наблюдения по нескольким делам в один проход (каждое дело -- отдельный проход с маркером дела; семантический дедуп между делами работает через Opus-сравнение).
