# init-case -- Инициализация дела

Этот скилл создаёт скелет нового дела в текущей рабочей директории: метаданные (`.vassal/case.yaml`), пустой реестр документов (`.vassal/index.yaml`), журнал действий (`.vassal/history.md`), стандартный набор папок и опционально предлагает синхронизацию с Notion. Дальнейшие шаги -- `intake` (приём материалов клиента), затем `legal-review`, `build-position` и т.д.

## Предварительные условия

- Текущая рабочая директория -- это корень будущего дела. Если Сюзерен запускает скилл «не там», на это нужно указать **до** создания файлов (см. фаза 0, граничные случаи).
- Никаких внешних зависимостей `init-case` не требует -- он не читает файлы извне и не извлекает текст. OCR-пайплайн нужен только последующему `intake`.

## Алгоритм

### Фаза 0 -- Pre-flight

1. Проверить, существует ли `.vassal/case.yaml` в текущей CWD.
   - **Если существует** -- НЕ перезаписывать. Сообщить Сюзерену:
     > «Дело уже инициализировано (`.vassal/case.yaml` найден). Что делать: (а) дополнить недостающие поля карточки (скажи, какие -- `init-case` точечно дозаполнит, чужих полей не трогает); (б) создать заново (удаление `.vassal/` -- ручная операция Сюзерена в его файловой системе, агент сам не удаляет); (в) отменить.»
     Дождаться выбора. По умолчанию -- отмена.
   - **Если не существует** -- идти дальше.

2. Проверить очевидную «не ту» CWD. Сигналы: корень диска, домашняя папка, папка плагина (`vassal-litigator/`), существующая папка другого дела (есть `.vassal/` у соседа). При срабатывании -- переспросить, прежде чем создавать структуру.

3. Проверить минимальную достаточность входных данных. Если Сюзерен не дал **ни сторон, ни сути спора**, остановиться и объяснить:
   > «Для минимальной карточки нужно хотя бы: (а) кто против кого (стороны), (б) о чём спор (1 предложение), (в) кто наш клиент. Остальное (номер дела, судья, даты) можно дополнить позже.»

### Фаза 1 -- Сбор информации о деле

Спроси Сюзерена в свободной форме. Нужны:

- **Номер дела** (например, А33-12345/2026). Если дело ещё не возбуждено -- оставь пустым.
- **Суд** (полное название, например «Арбитражный суд Красноярского края»).
- **Стороны:** кто истец, кто ответчик; для юрлиц -- название и (если есть) ИНН/ОГРН.
  - **ИНН оппонента-юрлица -- особенно важен**: это ключ дедупликации в глобальной памяти (`$VASSAL_GLOBAL_DIR/counterparties/`, см. `shared/conventions.md` → «Глобальная память»). Если ИНН пока неизвестен -- допусти `null`, но `legal-review` либо `add-opponent` должны его дополнить из материалов; без ИНН точность кросс-дельного матчинга падает (омонимы, разные правописания).
- **Кто наш клиент** -- какая из сторон.
- **Суть спора** (1 предложение: «взыскание задолженности по договору поставки»).
- **Судья** (если известен).
- **Дата подачи иска**, **дата следующего заседания** (если известны).
- **Данные представителя** (ФИО, доверенность, контакты).

Не требуй всё сразу. Заполни что дано, остальное -- `null`. Если Сюзерен прислал текстовое описание дела -- извлеки из него всё, что можешь, и переспроси только по обязательному минимуму (стороны + суть + клиент).

### Фаза 2 -- Создание структуры папок

В текущей рабочей директории создай:

```
.vassal/
├── case.yaml
├── index.yaml
├── raw/
├── mirrors/
├── analysis/
├── hearings/
└── history.md

Входящие документы/        ← сюда Сюзерен кладёт файлы для обработки intake
На удаление/               ← корзина (rm не работает в Cowork)
Материалы от клиента/
```

Создание директорий:
```bash
mkdir -p .vassal/{raw,mirrors,analysis,hearings}
mkdir -p "Материалы от клиента"
mkdir -p "Входящие документы"
mkdir -p "На удаление"
```

Папка `.vassal/positions/` **не создаётся** -- позиции по делу хранятся в корне дела (`{ГГГГ-ММ-ДД} {тип} позиция.md`), см. `skills/build-position/SKILL.md`. В старых делах эта папка может существовать как артефакт ранних версий плагина; `build-position` её игнорирует.

### Фаза 3 -- Заполнение `case.yaml`

Прочитай формат из `shared/case-schema.yaml`. Заполни все известные поля; неизвестные оставь как `null`.

Правила:
- **Версия (`version:`)** -- бери актуальное значение из шапки `shared/case-schema.yaml`. Не хардкодь число в этом скилле -- читай схему. Это защищает от рассогласования при будущих миграциях схемы.
- Каждой стороне присвой стабильный `party_id` в формате `{role}-NN`: `plaintiff-01`, `defendant-01`, `third-party-01`. Эти id -- ссылочные ключи для других структур (`our_client.party_id` и т.д.). При нескольких соистцах / соответчиках -- инкрементируй (`plaintiff-01`, `plaintiff-02`).
- Поле `our_client.party_id` ссылается на `party_id` нашего клиента.
- Оценочные поля (`qualification`, `jurisdiction`, `pretrial`) НЕ заполняй -- их пишет `legal-review` после анализа материалов.
- Поля блоков `appeal.*` / `cassation.*` НЕ заполняй -- у них свои owner'ы (`appeal`, `cassation`).
- **Если есть `case.amounts.claim` (сумма иска):** определи `case.production_type` (приказное / упрощённое / общеисковое) по алгоритму из [legal-review/references/production-types.md §6](../legal-review/references/production-types.md). Для приказного -- задать уточняющий вопрос Сюзерену: «Ожидается ли возражение должника? Если да, имеет смысл сразу подавать в упрощённом / общеисковом, чтобы избежать цикла отмены приказа». Для упрощённого -- информировать: «Дело подпадает под упрощённое производство, без вызова сторон, срок апелляции 15 дней; подтвердить?». Зафиксировать обоснование в `case.production_type_basis`. Если поля `production_type` нет в текущей `case-schema.yaml` -- зафиксировать в `case.timeline` и поставить флаг в `case.notes` (поля могут быть добавлены в схему отдельно).

Полная матрица writers/readers -- `shared/conventions.md` → «Ownership полей case.yaml».

### Фаза 4 -- Создание пустого `index.yaml`

```yaml
version: <актуальная version из shared/index-schema.yaml>
last_updated: <ISO datetime>

documents: []
bundles: []

next_id: 1
next_bundle_id: 1
```

Версию -- так же читай из шапки схемы, не хардкодь. Структуру записей не задавай -- её формирует `intake` при первом приёме материалов.

### Фаза 5 -- Создание `history.md`

```markdown
# История действий

## {ГГГГ-ММ-ДД}

### {ЧЧ:ММ} -- Инициализация дела
- Создана карточка дела {номер дела или "без номера"}
- Стороны: {истец} (истец) vs {ответчик} (ответчик)
- Наш клиент: {название} ({роль})
- Представитель: {ФИО представителя или "не указан"}
```

### Фаза 6 -- Сводка для Сюзерена

Выведи краткую сводку:
- Номер дела и суд.
- Стороны и наш клиент.
- **Квалифицированный порядок производства** (`case.production_type` -- приказное / упрощённое / общеисковое), если определён в фазе 3; с обоснованием по [production-types.md](../legal-review/references/production-types.md). Например: «Дело квалифицировано как **упрощённое** (ст. 227 ч. 1 п. 1 АПК -- цена иска 850 000 руб. для юрлица, в пределах порога 1 200 000 руб. в ред. ФЗ № 667-ФЗ от 25.12.2023). Срок апелляции -- 15 дней.»
- Какие поля остались пустыми (чтобы Сюзерен знал, что дополнить и когда -- сразу коротким диалогом или позже через `legal-review` / `add-opponent`).
- Следующий шаг:
  > «Положи материалы клиента в папку **Входящие документы/** -- обработаю их через `/vassal-litigator:intake`.»

### Фаза 7 -- Опциональный хук: синхронизация с Notion

Если Notion-конфиг существует (путь резолвится по правилам `shared/conventions.md` → «Notion-слой» → «Конфигурация»: `$VASSAL_CONFIG_DIR/notion-config.yaml`, по умолчанию `~/.vassal/notion-config.yaml`) -- предложи Сюзерену:
> «Notion-конфиг настроен. Создать запись о деле в базе `Cases` сразу? [`/vassal-litigator:sync-notion`]»

Если конфиг не настроен -- **не предлагать** (не навязывать Notion-интеграцию пользователям, которые её не используют). См. `skills/notion-sync/SKILL.md` → «Авто-триггеры».

Хук **не блокирующий**: `init-case` завершается успешно даже если Сюзерен отказался от sync или sync упал. Факт (выполнен / отказан / сбой) фиксируется в `.vassal/history.md`.

## Граничные случаи

- **Дело уже инициализировано** (`.vassal/case.yaml` существует) -- см. фаза 0, диалог. Без явного выбора Сюзерена -- отмена.
- **Минимальной информации нет** -- см. фаза 0, отказ с пояснением минимума.
- **CWD очевидно «не та»** -- предупредить и переспросить.
- **Стороны с одинаковой ролью** (соистцы, соответчики) -- допустимо; `party_id` инкрементируется.
- **Дело без номера** (иск ещё не подан) -- `case.number: null`; `intake` и аналитика работают; `notion-sync` использует временный ключ `temp-{slug}` до появления номера (см. `notion-sync/SKILL.md` → «Идемпотентность и dedup»).
- **Несколько наших клиентов** в деле -- редко; зафиксировать одного как `our_client`, остальных пометить ролями (соистцы), уточнить с Сюзереном тактику отображения.

## Ownership полей `case.yaml`

`init-case` -- основной owner стартовых метаданных. По матрице `shared/conventions.md` → «Ownership полей case.yaml», скилл пишет:

- `case.{number, court, judge, category, subject, our_client, representative}`
- `case.parties` (структура; уточнение реквизитов -- зона `legal-review`)
- `case.amounts.{claim, court_fee}` (совместно с `build-position`)
- `case.status` (совместно с `appeal`, `cassation`, `analyze-hearing`)
- `case.timeline` (append-only; пополняют также `legal-review`, `analyze-hearing`, `appeal`, `cassation`)
- `case.filing_date`
- `case.next_hearing`, `case.next_hearing_source` (совместно с `analyze-hearing`)

Остальные поля (`qualification`, `jurisdiction`, `pretrial`, `appeal.*`, `cassation.*`) при инициализации **не трогать** -- у них свои owner'ы. В режиме «дополнить недостающие поля» (фаза 0 вариант «а») это правило тоже действует: дозаполняется только своя зона ownership.

## Постусловия

После завершения `init-case` должно быть правдой:

- `.vassal/case.yaml` -- валидный YAML, соответствует `shared/case-schema.yaml` текущей версии; обязательные поля заполнены или явно `null`; `party_id` стабильны (`{role}-NN`); `our_client.party_id` ссылается на существующий `party_id`.
- `.vassal/index.yaml` -- валидный YAML, соответствует `shared/index-schema.yaml`; `documents: []`, `bundles: []`, `next_id: 1`, `next_bundle_id: 1`.
- `.vassal/history.md` существует, первая запись -- «Инициализация дела» (дата, номер, стороны, клиент, представитель).
- Все обязательные папки созданы: `.vassal/{raw, mirrors, analysis, hearings}`, `Входящие документы/`, `Материалы от клиента/`, `На удаление/`.
- Если выполнен Notion-хук -- применяются постусловия `notion-sync/SKILL.md`. Отказ или сбой хука НЕ валит `init-case` -- но факт фиксируется в `.vassal/history.md`.
- При выборе в фазе 0 «отмена» / «дополнить» -- существующий `case.yaml` НЕ перезаписан; никакие чужие зоны ownership не затронуты.
- Оригиналы документов не трогались (их `init-case` вообще не видит).
