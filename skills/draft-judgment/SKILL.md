# draft-judgment -- Проект судебного акта

> **Операционный справочник:** [legal-review/references/judgment-standards.md](../legal-review/references/judgment-standards.md) -- стандарты судебного решения (ППВС № 23 от 19.12.2003 в ред. 09.12.2025 + ст. 170 АПК + ст. 198 ГПК). §6 -- чек-лист качества прогноза: если черновик не отвечает стандартам, прогноз неточен (реальный суд так не напишет, а если напишет -- решение будет отменено в апелляции). Используется в фазе 3 этого скилла (системный промпт Opus-субагента) и фазе 4 (Preview -- явный прогон чек-листа).

Составление проекта решения суда по делу. Учитывает полный контекст дела, позиции обеих сторон, доказательственную базу. При наличии примеров решений конкретного судьи -- адаптирует стиль, структуру и формулировки.

> Источник идеи: алгоритм Мошкина С.А. -- "цифровой профиль судьи" на основе 3-5 решений по аналогичным делам (runetlex.academy)

## Зачем нужен проект решения

Практика подготовки проектов судебных решений широко распространена (особенно в регионах). Хорошо составленный проект:
- Снижает риск правок и ошибок в мотивировочной части
- Повышает устойчивость к обжалованию (когда суд использует аргументацию из проекта)
- Демонстрирует профессионализм и подготовленность стороны
- Позволяет юристу заранее увидеть, как решение будет выглядеть "глазами суда"

## Предварительные условия

- Дело инициализировано, есть `.vassal/case.yaml`.
- Есть хотя бы одна позиция в корне дела.
- Желательно: отчеты о заседаниях (`.vassal/hearings/`), профиль судьи.
- Опционально: 3-5 решений этого судьи по аналогичным делам (для анализа стиля).

## Входные данные от Сюзерена

Перед началом уточни:

1. **Режим проекта решения** (это **первый разветвляющий вопрос** -- от него меняется системный промпт Opus-субагента, имя выходного файла и стиль изложения; см. раздел «Две ветки: проект для судьи vs нейтральный прогноз» ниже):
   - **«В пользу клиента» (проект для судьи)** -- проект, который при практике подбрасывания судье можно использовать как заготовку. Аргументация **односторонняя в нашу пользу**, выборка практики -- в нашу сторону, мотивировочная усиливает резолютивную линию клиента.
   - **«Нейтральный прогноз» (для аналитики)** -- симуляция того, как вероятно решит суд. Взвешенная аргументация, учёт позиции оппонента, оценка рисков. Назначение: предупредить клиента, подготовиться к апелляции заранее.
   - **Если Сюзерен не уверен** -- выяснить use-case: заготовка для судьи (региональная практика) или прогноз для клиента / подготовка к апелляции (стандартный аналитический инструмент).
2. **Есть ли решения этого судьи?** (если есть -- загрузить для анализа стиля; для нейтрального прогноза стиль судьи менее критичен, для проекта в пользу клиента -- желателен).
3. **Какой исход?** (полное удовлетворение, частичное, отказ -- если Сюзерен знает; для проекта в пользу клиента -- обязательный вопрос; для нейтрального прогноза -- факультативный, но влияет на акценты).
4. **Особые указания?** (акценты, конкретные суммы, распределение расходов)

Если Сюзерен дал указания в аргументах -- работай по ним.

## Алгоритм

### Фаза 0 -- Проверка консистентности ФС ↔ индекс

Перед загрузкой контекста выполни Phase 0 по правилу из [shared/conventions.md](../../shared/conventions.md) → раздел «Проверка консистентности ФС ↔ индекс (Phase 0)». Проект решения суда строится по полной картине дела -- если новые документы не учтены в индексе, проект будет резолютивно неполным.

### Фаза 1 -- Загрузка контекста

1. Прочитай `.vassal/case.yaml` -- стороны, суд, предмет, хронология.
2. Прочитай `.vassal/index.yaml` -- полный реестр документов.
3. Загрузи все позиции из корня дела.
4. Загрузи отчеты о заседаниях из `.vassal/hearings/` (если есть).
5. Загрузи **локальный** профиль судьи из `.vassal/hearings/judge-profile.md` (если есть).
5.1. **Загрузи глобальный профиль судьи** (если есть) -- по правилу из [shared/conventions.md](../../shared/conventions.md) → раздел «Глобальная память (кросс-дельная)»:
   - Разрезолви `$VASSAL_GLOBAL_DIR` (по умолчанию `~/.vassal-global/`).
   - По `case.judge` и `case.court` собери имя файла: `judges/{ФИО-slug}--{court-slug}.md`.
   - Если файл существует -- загрузи. Это даёт письменный стиль и устные паттерны судьи **из всех твоих прошлых дел с ним**, а не только из текущего. Особенно ценно для проекта «в пользу клиента» -- стиль адаптации работает тем точнее, чем больше наблюдений.
   - Если нет -- работай с локальным профилем (или совсем без профиля), не падай.
6. Загрузи md-зеркала ключевых документов обеих сторон.
7. Загрузи аналитику: `.vassal/analysis/`.

### Фаза 2 -- Анализ стиля судьи (если есть примеры)

Если Сюзерен загрузил решения судьи по аналогичным делам:

8. Проанализируй каждое решение по отдельности:
   - Структура (вводная, описательная, мотивировочная, резолютивная)
   - Уровень детализации каждой части
   - Характерные формулировки и обороты
   - Стиль цитирования норм права
   - Как ссылается на доказательства
   - Как оценивает доводы сторон
   - Подход к распределению судебных расходов

9. Сделай общий вывод -- цифровой профиль судьи:
   - Структурные особенности (подробная/лаконичная мотивировочная часть)
   - Стилистические особенности (формальный/разговорный тон)
   - Подход к применению норм и аргументации
   - Сохрани результаты в **единый накопительный файл** `.vassal/hearings/judge-profile.md` (его же ведёт `analyze-hearing`). Дополни существующие разделы или добавь раздел «Письменный стиль решений» (см. формат в `skills/analyze-hearing/SKILL.md` -- раздел «Формат профиля судьи (локальный, на дело)»). Старые наблюдения не перезаписывай -- аппенд-only. Инкрементируй `written_decisions_analyzed` в frontmatter, обнови `last_updated`.
   - **Двойная запись в глобальный профиль судьи** -- по правилу из [shared/conventions.md](../../shared/conventions.md) → «Глобальная память (кросс-дельная)»:
     - Разрезолви `$VASSAL_GLOBAL_DIR` (по умолчанию `~/.vassal-global/`).
     - Создай папку `$VASSAL_GLOBAL_DIR/judges/` если её нет (`mkdir -p`).
     - Имя файла: `{ФИО-slug}--{court-slug}.md` -- то же, что у `analyze-hearing` (формат описан в `skills/analyze-hearing/SKILL.md` → «Формат глобального профиля судьи»).
     - **Если файл существует** -- открой и аппенд-only дополни наблюдениями этого запуска в раздел 7 «Письменный стиль решений». Каждое наблюдение помечай маркером `(дело {case.number}, решение {ГГГГ-ММ-ДД})` -- дата решения, не дата запуска скилла.
     - **Если файла нет** -- создай новый со фронтматтером и наполни разделом 7.
     - В обоих случаях: инкрементируй `written_decisions_total` на число проанализированных решений (шаг 8); добавь `case.number` в `cases:` (без дублей); обнови `last_updated` и `last_case`.
     - **Сбой записи**: не падай -- зафиксируй локальный профиль, предупреди Сюзерена шаблоном из conventions, отметь факт в `.vassal/history.md` (фаза 6 шаг 19).

### Фаза 3 -- Генерация проекта (Opus-субагент с разведёнными системными промптами)

Делегируй Opus-субагенту (model: "opus") с **системным промптом, зависящим от режима из вопроса 1 вводных данных** (см. раздел «Две ветки: проект для судьи vs нейтральный прогноз» ниже):

**Если режим «В пользу клиента» (проект для судьи)** -- системный промпт:
> «Ты помощник судьи, составляющий проект решения в пользу стороны X (нашего клиента). Позиция клиента -- ведущая, она задаёт направление мотивировочной части. Аргументы оппонента в мотивировочной отвергаются с указанием конкретной правовой причины. Сомнения толкуются в пользу клиента. Твоя задача -- написать такой проект, который судье технически удобно использовать как заготовку: ссылки на нормы и доказательства подобраны однозначно в одну сторону, резолютивная даёт клиенту максимум, мотивировочная защищает резолютивную от отмены в апелляции. **Не делай вид объективности** -- это не аналитика, это заготовка стороны.»

**Если режим «Нейтральный прогноз» (для аналитики)** -- системный промпт:
> «Ты коллегия (или единоличный судья), оценивающая баланс позиций сторон с равной строгостью. Выработай **вероятный** исход дела, а не желаемый одной из сторон. Учитывай весь представленный массив доказательств, мотивы первой инстанции по аналогичным делам этого судьи (если есть профиль), типичные подходы к делам данной категории. Если позиция клиента слабая -- так и напиши, с указанием конкретных слабых мест. Резолютивная должна отражать **наиболее вероятный** исход, а не оптимальный для клиента. **Не подыгрывай** -- цель прогноза в подготовке к настоящему решению, а не в его желании.»

10. Составить проект решения со следующей структурой. **Опираться на чек-лист стандартов из [legal-review/references/judgment-standards.md §6](../legal-review/references/judgment-standards.md):** мотивировочная должна содержать все 4 элемента п. 4 ППВС № 23 / ч. 4 ст. 170 АПК; по каждому существенному доводу стороны -- явный ответ с мотивом; каждое ключевое доказательство -- с оценкой; резолютивная не противоречит мотивировочной.

**Вводная часть:**
- Наименование суда, дата, номер дела
- Состав суда (ФИО судьи)
- Стороны (истец, ответчик, третьи лица)
- Предмет спора

**Описательная часть:**
- Позиция истца (требования + основания)
- Позиция ответчика (возражения)
- Позиции третьих лиц (если есть)

**Мотивировочная часть:**
- Установленные обстоятельства дела (со ссылками на доказательства через двойные ссылки `Название [doc-NNN]`, см. [shared/conventions.md](../../shared/conventions.md) → «Двойные ссылки»)
- Оценка доводов сторон
- Правовое обоснование (нормы закона + судебная практика)
- Выводы суда по каждому требованию

**Резолютивная часть** (по чек-листу из раздела «Чек-лист резолютивной части» ниже):
- Решение по каждому требованию (удовлетворить/отказать, суммы)
- Распределение судебных расходов
- Порядок и срок обжалования

11. Если есть профиль стиля судьи -- адаптировать формулировки, структуру, уровень детализации (особенно важно в режиме «в пользу клиента» -- стиль судьи + наша линия = максимум шансов на принятие).

12. **Прогон чек-листа резолютивной части** (см. раздел ниже). Opus-субагент сверяет резолютивную с чек-листом и явно фиксирует «Чек-лист пройден» / «Пропущенные пункты: ...».

12.1. **Прогон чек-листа стандартов мотивировочной** ([judgment-standards.md §6](../legal-review/references/judgment-standards.md)). Opus-субагент дополнительно сверяет мотивировочную с 10 пунктами чек-листа стандартов (все 4 элемента п. 4 ППВС № 23; ответы по каждому доводу; оценка каждого ключевого доказательства; внутренняя согласованность; применимость норм; цитирование разъяснений ВС РФ; исполнимость резолютивной). Результат фиксируется отдельной строкой в Preview наряду с чек-листом резолютивной.

### Фаза 5 -- Финализация

15. Сохрани проект как md в корне дела. **Имя зависит от режима** (вопрос 1 вводных данных):
    - Режим «в пользу клиента» → `{ГГГГ-ММ-ДД} Проект решения суда (в пользу клиента).md`
    - Режим «нейтральный прогноз» → `{ГГГГ-ММ-ДД} Проект решения суда (нейтральный прогноз).md`

    Различие в имени -- ключевая защита от смешивания: Сюзерен (или будущий запуск `appeal`) однозначно понимает, какой это артефакт. **Перезапись запрещена** -- при коллизии имени в один день добавляется суффикс `-{ЧЧММ}` по общему правилу из [shared/conventions.md](../../shared/conventions.md) → «Версионирование продуктов с пересчётом».

    Во **frontmatter** проекта обязательно зафиксировать:
    ```yaml
    ---
    type: draft-judgment
    mode: client_favor | neutral_forecast
    generated_at: ГГГГ-ММ-ДД
    case_number: <номер дела из case.yaml>
    judge: <ФИО из case.yaml>
    expected_outcome:
      - requirement: <текст требования>
        decision: granted | denied | partially_granted
        amount: <сумма, если применимо>
        notes: <комментарий по требованию>
    depends_on: [<doc-NNN, ...>]
    ---
    ```
    Этот блок читает `appeal` после вынесения реального решения для сравнения «прогноз vs факт».

17. Если да -- **прямая генерация .docx** через `python-docx` (НЕ через arbitrum-docx; обоснование -- см. раздел «Обоснование прямой генерации .docx» ниже).

    **Feature detection** (паттерн из [shared/conventions.md](../../shared/conventions.md) → «Внешние зависимости и fallback»):
    - Проверь доступность `python-docx` (попытка `import docx` или эквивалент).
    - **Если доступен** -- сгенерируй `.docx`:
      - Шрифт: Times New Roman, 12 кегль
      - Межстрочный интервал: 1.5
      - Каждая классическая часть решения выделена заголовком (жирный, по центру):
        - ВВОДНАЯ ЧАСТЬ
        - ОПИСАТЕЛЬНАЯ ЧАСТЬ
        - МОТИВИРОВОЧНАЯ ЧАСТЬ
        - РЕЗОЛЮТИВНАЯ ЧАСТЬ
      - **Имя файла зеркально md-варианту:**
        - Режим «в пользу клиента» → `{ГГГГ-ММ-ДД} Проект решения суда (в пользу клиента).docx`
        - Режим «нейтральный прогноз» → `{ГГГГ-ММ-ДД} Проект решения суда (нейтральный прогноз).docx`
      - Без «Клиент» -- проект составляется не от клиента, см. [shared/conventions.md](../../shared/conventions.md) → «Продукты плагина».
    - **Если недоступен** -- graceful degradation:
      - `.md` уже сохранён в корне дела на шаге 15 (это и есть fallback-артефакт).
      - **НЕ** импровизировать через альтернативные библиотеки и **НЕ** обращаться к `arbitrum-docx` (его шаблоны не подходят для судебного акта).
      - Явно предупреди Сюзерена по шаблону:

        ```
        Что нужно от Сюзерена: оформите .docx вручную по своему шаблону ИЛИ установите python-docx и перезапустите фазу 5
        ```

19. Обнови `.vassal/history.md`. Зафиксируй: режим проекта (`mode: client_favor | neutral_forecast`), результат прохождения чек-листа резолютивной части, путь к выходному файлу. Если в фазе 2 проанализированы решения судьи и обновлён `judge-profile.md` -- укажи: обновлены локальный `.vassal/hearings/judge-profile.md` и глобальный `$VASSAL_GLOBAL_DIR/judges/{slug}.md` (или: глобальная память недоступна -- {причина}). ## Две ветки: проект для судьи vs нейтральный прогноз

Это **методологически разные задачи**, и они должны делаться разными системными промптами Opus-субагента, иначе оба режима получаются посредственно: «в пользу клиента» -- скатывается в нейтральность (модель тренируется на объективности), «нейтральный» -- скатывается в линию клиента (потому что весь загруженный контекст -- от клиента).

| Аспект | Режим «В пользу клиента» (заготовка для судьи) | Режим «Нейтральный прогноз» (аналитика) |
|---|---|---|
| Системный промпт Opus-субагента | «Ты помощник судьи, склоняющегося к позиции стороны X. Сомнения -- в пользу X.» | «Ты коллегия, оценивающая баланс позиций. Вырабатываешь вероятный исход.» |
| Имя выходного файла | `{дата} Проект решения суда (в пользу клиента).md/.docx` | `{дата} Проект решения суда (нейтральный прогноз).md/.docx` |
| Frontmatter `mode:` | `client_favor` | `neutral_forecast` |
| Аргументы оппонента в мотивировочной | Отвергаются с конкретной правовой причиной | Анализируются с равной строгостью |
| Сомнения в фактах / нормах | Толкуются в пользу клиента | Толкуются объективно (с указанием неопределённости) |
| Резолютивная часть | Максимум для клиента | Наиболее вероятный исход |
| Профиль судьи (если есть) | Используется для адаптации стиля | Используется для оценки тенденций («этот судья обычно отказывает в таких случаях») |
| Когда использовать | Региональная практика «подбрасывания» проекта; внутренняя заготовка для усиления позиции | Прогноз для клиента; подготовка к апелляции; принятие решения о мировом соглашении |
| Use-case для `appeal` | -- (до решения) | После вынесения реального решения `appeal` сравнивает прогноз с фактом → расхождения = кандидаты в основания обжалования |

### Запрет на смешивание

- В одной сессии скилла -- **только один режим**. Если Сюзерен хочет оба варианта -- запустить скилл дважды с разными ответами на вопрос 1.
- Имена файлов в один день различаются суффиксом режима, поэтому коллизии не будет.
- В Preview (фаза 4) **обязательное напоминание** Сюзерену, какой режим выбран -- это предотвращает «не туда смотрел».

### Если Сюзерен передумал в середине

Запрос «теперь сделай нейтральный прогноз вместо проекта в пользу клиента» -- это **новая задача**, не корректировка. Скилл начинает фазу 1 заново с другим режимом. Старый файл (в пользу клиента) сохраняется -- он остаётся валидным артефактом для своей задачи.

## Чек-лист резолютивной части проекта решения

Резолютивная часть -- **самая важная** для последующего использования: она определяет исполнимость, основу для расчётов, объект анализа при обжаловании. Чек-лист **обязателен** к прогону Opus-субагентом перед apply (зеркальный к чек-листу просительной части в [skills/prepare-hearing/SKILL.md](../prepare-hearing/SKILL.md) -- раздел «Чек-лист просительной части процессуального документа»).

### По каждому требованию из позиции клиента / иска -- явный ответ

1. **Удовлетворить / удовлетворить частично / отказать** -- по каждому требованию **отдельно**. Не пропускать ни одно требование, заявленное в первой инстанции.
2. **При удовлетворении взыскания:**
   - **С кого** взыскать (точное наименование должника с ИНН/ОГРН для юрлиц, ФИО + дата рождения для физлиц).
   - **В пользу кого** (точное наименование взыскателя).
   - **Какую сумму** -- с разложением:
     - Основной долг -- точная сумма + валюта.
     - Проценты по ст. 395 ГК / договорные -- сумма на дату вынесения **+ формула «по день фактической оплаты»** (база, ставка, период).
     - Неустойка -- сумма с расчётом.
     - Госпошлина -- сумма (рассчитывается отдельно, не включается в общий итог взыскания).
   - **При нескольких должниках:** солидарно (со ссылкой на основание -- ст. 322 ГК для договорной, ст. 1080 ГК для деликтной) / в долях (с указанием долей) / субсидиарно (с указанием основного должника).
3. **При обязании совершить действие:**
   - Конкретное действие (что должен сделать должник).
   - Срок (в течение N дней с даты вступления в силу).
   - Последствия неисполнения (если предусмотрено: судебная неустойка по ст. 308.3 ГК).
4. **При признании права / факта:**
   - Точная формулировка признаваемого права / факта.
   - Без рассуждений в резолютивной -- только финальная констатация.

### Момент исполнения

5. **Когда исполнять:**
   - **«Решение подлежит немедленному исполнению»** -- если применимо (ст. 182 АПК / ст. 211 ГПК -- алименты, восстановление на работе и т.п.).
   - **«Решение вступает в силу по истечении срока на апелляционное обжалование»** -- стандартное правило.
6. **Если исполнение в натуре** -- срок (в течение N дней).

### Распределение государственной пошлины

7. **Точная сумма пошлины:**
   - При полном удовлетворении -- взыскать с ответчика в пользу истца.
   - При отказе -- оставить за истцом.
   - **При частичном удовлетворении** -- пропорционально удовлетворённой части (ст. 110 АПК / ст. 98 ГПК); расчёт показать прямо: «иск удовлетворён на X%, госпошлина X% × Y руб = Z руб взыскивается с ответчика».
   - **Возврат из бюджета** -- если пошлина была уплачена излишне (например, иск меньше, чем заявлено в платёжке).
8. **Перераспределение пошлины первой инстанции при обжаловании** -- неприменимо для проекта в первой инстанции (только для апелляции/кассации).

### Судебные расходы

9. **Расходы на представителя** -- если заявлены отдельно:
   - Сумма (с обоснованием разумности по п. 11-13 Постановления Пленума ВС РФ от 21.01.2016 № 1).
   - Источник (договор + платёжные документы).
10. **Расходы на эксперта / переводчика / нотариуса** -- если применимо.

### Обеспечительные меры

11. **При удовлетворении иска и наличии обеспечительных мер:**
    - Сохранить до вступления решения в силу / отменить с даты вступления в силу.
12. **При отказе в иске:**
    - Отменить обеспечительные меры с даты вступления решения в силу.

### Обращение к исполнению

13. **Исполнительный лист:**
    - «Выдать по заявлению взыскателя» (стандарт) ИЛИ
    - «Направить взыскателю по почте» (если применимо).

### Порядок и срок обжалования

14. **Срок:** «решение может быть обжаловано в апелляционном порядке в N-й арбитражный апелляционный суд через X (наименование суда первой инстанции) в течение одного месяца со дня его принятия в полном объёме» (ст. 181, 259 АПК / ст. 320, 321 ГПК).
15. **Для упрощённого производства:** срок 15 дней + только в порядке упрощёнки (ст. 229 АПК).

### Универсальные требования

16. **Числа прописью + цифрами** для сумм > 1000 рублей.
17. **Полное наименование сторон** с реквизитами.
18. **Никаких «по справедливости» / «на усмотрение»** -- резолютивная должна быть исполнима без интерпретации.

### Что Opus-субагент делает с чек-листом

После генерации -- сверяет резолютивную с чек-листом и в Preview явно фиксирует:

```
ЧЕК-ЛИСТ РЕЗОЛЮТИВНОЙ ЧАСТИ:
- Требование 1 (взыскание основного долга): ✓ удовлетворить, сумма + солидарность + проценты на дату + формула по день фактической
- Требование 2 (взыскание неустойки): ✗ ПРОПУЩЕН расчёт периода
- Госпошлина: ✓ распределена пропорционально (60% / 40%)
- Расходы на представителя: ⚠ заявлены, но обоснование разумности слабое
- Срок и порядок обжалования: ✓
- ...

Итого: чек-лист пройден частично. Требуется добавить: расчёт периода неустойки, усилить обоснование разумности расходов.
```

Сюзерен видит результат и решает: дорабатывать ли (возврат к фазе 3) или принять как есть.

## Обоснование прямой генерации .docx

В отличие от `appeal`, `cassation`, `prepare-hearing`, `analyze-hearing`, скилл `draft-judgment` **не использует** `arbitrum-docx`. Причина -- архитектурная, не временная:

**Шаблоны `arbitrum-docx` заточены под процессуальные документы сторон:**
- Тип 1 (процессуальный с шапкой) -- «В арбитражный суд X от стороны Y», подпись представителя.
- Тип 2 (аналитический без шапки) -- отчёт стороны клиенту.

**Проект судебного акта структурно иной:**
- Вместо шапки «от стороны» -- состав суда (наименование суда + ФИО судьи).
- Четырёхчастная структура (вводная / описательная / мотивировочная / резолютивная) вместо линейного процессуального документа.
- Без подписи представителя стороны -- акт не исходит от стороны.

**Путь миграции (когда `arbitrum-docx` поддержит «Тип 3: судебный акт»):**
- Удалить этот раздел.
- Заменить шаг 17 на стандартный feature-detection arbitrum-docx с Тип 3.
- Зафиксировать в [ARCHITECTURE.md §10](../../ARCHITECTURE.md) переход.

До тех пор -- прямая генерация остаётся обоснованным исключением, явно отмеченным в архитектуре.

## Граничные случаи

- **Нет решений судьи для анализа стиля** -> составить в стандартном стиле арбитражных решений; в Preview предупредить «адаптация стиля невозможна, используется обобщённый стиль арбитражных актов».
- **Дело с несколькими требованиями** -> отдельный вывод по каждому требованию в резолютивной части (см. чек-лист резолютивной). Не пропускать ни одно требование.
- **Частичное удовлетворение** -> четко указать что удовлетворено, что отказано, пропорциональное распределение расходов с явным расчётом.
- **Решение о прекращении производства** (мировое, отказ от иска, неподведомственность) -> отдельная структура -- мотивировочная сокращённая, резолютивная: «производство по делу прекратить со ссылкой на основание ст. 150 АПК / ст. 220 ГПК». Чек-лист резолютивной адаптируется.
- **Оставление без рассмотрения** (несоблюдение досудебного порядка, иск подан не уполномоченным лицом) -> «исковое заявление оставить без рассмотрения» со ссылкой на ст. 148 АПК / ст. 222 ГПК. Резолютивная без рассмотрения по существу.
- **Заочное решение** (ответчик не явился без уважительной причины, ст. 233-244 ГПК) -> особенности мотивировки + указание на возможность подачи ответчиком заявления об отмене заочного решения в 7-дневный срок.
- **Решение по делу об оспаривании ненормативного акта** (госорган как ответчик, гл. 24 АПК) -> иная структура резолютивной: «признать незаконным / отменить» вместо «взыскать».
- **Банкротное обособленное производство** -> своя специфика мотивировки и резолютивной (ФЗ «О банкротстве»). Если дело банкротное (`case.category == "банкротство"`), скилл предупреждает Сюзерена об ограниченности стандартного шаблона и предлагает скорректировать вручную.
- **Признание права / обязание совершить действие** -> резолютивная содержит не суммы, а действия и сроки (см. чек-лист п. 3-4).
