# draft-judgment -- Проект судебного акта

Составление проекта решения суда по делу. Учитывает полный контекст дела, позиции обеих сторон, доказательственную базу. При наличии примеров решений конкретного судьи -- адаптирует стиль, структуру и формулировки.

> Источник идеи: алгоритм Мошкина С.А. -- "цифровой профиль судьи" на основе 3-5 решений по аналогичным делам (runetlex.academy)

## Зачем нужен проект решения

Практика подготовки проектов судебных решений широко распространена (особенно в регионах). Хорошо составленный проект:
- Снижает риск правок и ошибок в мотивировочной части
- Повышает устойчивость к обжалованию (когда суд использует аргументацию из проекта)
- Демонстрирует профессионализм и подготовленность стороны
- Позволяет юристу заранее увидеть, как решение будет выглядеть "глазами суда"

## Предварительные условия

- Дело инициализировано, есть `.vassal/case.yaml`.
- Есть хотя бы одна позиция в корне дела.
- Желательно: отчеты о заседаниях (`.vassal/hearings/`), профиль судьи.
- Опционально: 3-5 решений этого судьи по аналогичным делам (для анализа стиля).

## Входные данные от Сюзерена

Перед началом уточни:

1. **В чью пользу составляем проект?** (нашего клиента или нейтральный прогноз)
2. **Есть ли решения этого судьи?** (если есть -- загрузить для анализа стиля)
3. **Какой исход?** (полное удовлетворение, частичное, отказ -- если Сюзерен знает)
4. **Особые указания?** (акценты, конкретные суммы, распределение расходов)

Если Сюзерен дал указания в аргументах -- работай по ним.

## Алгоритм

### Фаза 0 -- Проверка консистентности ФС ↔ индекс

Перед загрузкой контекста выполни Phase 0 по правилу из [shared/conventions.md](../../shared/conventions.md) → раздел «Проверка консистентности ФС ↔ индекс (Phase 0)». Проект решения суда строится по полной картине дела -- если новые документы не учтены в индексе, проект будет резолютивно неполным.

### Фаза 1 -- Загрузка контекста

1. Прочитай `.vassal/case.yaml` -- стороны, суд, предмет, хронология.
2. Прочитай `.vassal/index.yaml` -- полный реестр документов.
3. Загрузи все позиции из корня дела.
4. Загрузи отчеты о заседаниях из `.vassal/hearings/` (если есть).
5. Загрузи профиль судьи из `.vassal/hearings/judge-profile.md` (если есть).
6. Загрузи md-зеркала ключевых документов обеих сторон.
7. Загрузи аналитику: `.vassal/analysis/`.

### Фаза 2 -- Анализ стиля судьи (если есть примеры)

Если Сюзерен загрузил решения судьи по аналогичным делам:

8. Проанализируй каждое решение по отдельности:
   - Структура (вводная, описательная, мотивировочная, резолютивная)
   - Уровень детализации каждой части
   - Характерные формулировки и обороты
   - Стиль цитирования норм права
   - Как ссылается на доказательства
   - Как оценивает доводы сторон
   - Подход к распределению судебных расходов

9. Сделай общий вывод -- цифровой профиль судьи:
   - Структурные особенности (подробная/лаконичная мотивировочная часть)
   - Стилистические особенности (формальный/разговорный тон)
   - Подход к применению норм и аргументации
   - Сохрани результаты в **единый накопительный файл** `.vassal/hearings/judge-profile.md` (его же ведёт `analyze-hearing`). Дополни существующие разделы или добавь раздел «Письменный стиль решений» (см. формат в `skills/analyze-hearing/SKILL.md` -- раздел «Формат профиля судьи»). Старые наблюдения не перезаписывай -- аппенд-only. Инкрементируй `written_decisions_analyzed` в frontmatter, обнови `last_updated`.

### Фаза 3 -- Генерация проекта (Opus-субагент)

Делегируй Opus-субагенту (model: "opus"):

10. Составить проект решения со следующей структурой:

**Вводная часть:**
- Наименование суда, дата, номер дела
- Состав суда (ФИО судьи)
- Стороны (истец, ответчик, третьи лица)
- Предмет спора

**Описательная часть:**
- Позиция истца (требования + основания)
- Позиция ответчика (возражения)
- Позиции третьих лиц (если есть)

**Мотивировочная часть:**
- Установленные обстоятельства дела (со ссылками на доказательства)
- Оценка доводов сторон
- Правовое обоснование (нормы закона + судебная практика)
- Выводы суда по каждому требованию

**Резолютивная часть:**
- Решение по каждому требованию (удовлетворить/отказать, суммы)
- Распределение судебных расходов
- Порядок обжалования

11. Если есть профиль стиля судьи -- адаптировать формулировки, структуру, уровень детализации.

### Фаза 4 -- Preview

12. Покажи Сюзерену:
    - Резолютивную часть (кратко: что решено)
    - Ключевые мотивы (2-3 основных довода суда)
    - Оценку рисков: какие части проекта суд может изменить
    - Если были примеры решений судьи: насколько удалось адаптировать стиль

13. Дождись подтверждения. Если Сюзерен корректирует -- вернись к фазе 3.

### Фаза 5 -- Финализация

14. Сохрани проект как md в корне дела:
    ```
    {ГГГГ-ММ-ДД} Проект решения суда.md
    ```

15. Спроси: "Оформить в .docx?"

16. Если да -- **прямая генерация .docx** через `python-docx` (НЕ через arbitrum-docx; обоснование -- см. раздел «Обоснование прямой генерации .docx» ниже).

    **Feature detection** (паттерн из [shared/conventions.md](../../shared/conventions.md) → «Внешние зависимости и fallback»):
    - Проверь доступность `python-docx` (попытка `import docx` или эквивалент).
    - **Если доступен** -- сгенерируй `.docx`:
      - Шрифт: Times New Roman, 12 кегль
      - Межстрочный интервал: 1.5
      - Каждая классическая часть решения выделена заголовком (жирный, по центру):
        - ВВОДНАЯ ЧАСТЬ
        - ОПИСАТЕЛЬНАЯ ЧАСТЬ
        - МОТИВИРОВОЧНАЯ ЧАСТЬ
        - РЕЗОЛЮТИВНАЯ ЧАСТЬ
      - Имя файла: `ГГГГ-ММ-ДД Проект решения суда.docx` (без «Клиент» -- проект составляется не от клиента, см. [shared/conventions.md](../../shared/conventions.md) → «Продукты плагина»)
    - **Если недоступен** -- graceful degradation:
      - `.md` уже сохранён в корне дела на шаге 14 (это и есть fallback-артефакт).
      - **НЕ** импровизировать через альтернативные библиотеки и **НЕ** обращаться к `arbitrum-docx` (его шаблоны не подходят для судебного акта).
      - Явно предупреди Сюзерена по шаблону:
        ```
        ВНИМАНИЕ. Внешняя зависимость недоступна: python-docx
        Сделано: сохранён только .md в корне дела ({ГГГГ-ММ-ДД} Проект решения суда.md)
        Что нужно от Сюзерена: оформите .docx вручную по своему шаблону ИЛИ установите python-docx и перезапустите фазу 5
        ```

17. Помести `.docx` в корень дела (если оформлен).

### Фаза 6 -- Обновление состояния

18. Обнови `.vassal/history.md`. Если `.docx` создан в fallback-режиме (python-docx недоступен) -- зафиксируй явно: «.docx не создан, доступен только `.md` -- python-docx недоступен».

## Обоснование прямой генерации .docx

В отличие от `appeal`, `cassation`, `prepare-hearing`, `analyze-hearing`, скилл `draft-judgment` **не использует** `arbitrum-docx`. Причина -- архитектурная, не временная:

**Шаблоны `arbitrum-docx` заточены под процессуальные документы сторон:**
- Тип 1 (процессуальный с шапкой) -- «В арбитражный суд X от стороны Y», подпись представителя.
- Тип 2 (аналитический без шапки) -- отчёт стороны клиенту.

**Проект судебного акта структурно иной:**
- Вместо шапки «от стороны» -- состав суда (наименование суда + ФИО судьи).
- Четырёхчастная структура (вводная / описательная / мотивировочная / резолютивная) вместо линейного процессуального документа.
- Без подписи представителя стороны -- акт не исходит от стороны.

**Путь миграции (когда `arbitrum-docx` поддержит «Тип 3: судебный акт»):**
- Удалить этот раздел.
- Заменить шаг 16 на стандартный feature-detection arbitrum-docx с Тип 3.
- Зафиксировать в [ARCHITECTURE.md §10](../../ARCHITECTURE.md) переход.

До тех пор -- прямая генерация остаётся обоснованным исключением, явно отмеченным в архитектуре.

## Граничные случаи

- Нет решений судьи для анализа стиля -> составить в стандартном стиле арбитражных решений
- Сюзерен просит проект не в нашу пользу (прогноз проигрыша) -> составить объективно, указать где позиция клиента слабая
- Дело с несколькими требованиями -> отдельный вывод по каждому требованию в резолютивной части
- Частичное удовлетворение -> четко указать что удовлетворено, что отказано, пропорциональное распределение расходов
