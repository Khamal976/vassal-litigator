# Сводный отчёт по ревизии vassal-litigator

**Дата:** 2026-04-21
**Версия плагина:** 0.4.0
**Объём проверенного:** 12 скиллов, 12 парных команд, 4 shared-файла, ARCHITECTURE.md, CHANGELOG.md.

## 1. Что было сделано

### Созданные артефакты

| Файл | Назначение |
|---|---|
| [WORKFLOW-REVIEW.md](WORKFLOW-REVIEW.md) | Архитектурные проблемы workflow между скиллами (9 пунктов) |
| [NOTION-INTEGRATION-PROPOSAL.md](NOTION-INTEGRATION-PROPOSAL.md) | Предложение Notion-слоя как practice-management над плагином |
| [reviews/*.md](reviews/) | 12 детальных ревизий по одному файлу на скилл |

### Методология

1. Для каждого скилла читались: `skills/{имя}/SKILL.md`, `commands/{имя}.md`, все `references/` при наличии, перекрёстные ссылки (`shared/*`, `ARCHITECTURE.md`, соседние скиллы).
2. Каждая ревизия построена по единому шаблону: назначение → контракт → промпт-инженерия → противоречия и баги → edge-cases → реальные проблемы → приоритизированные рекомендации → резюме.
3. После первого прогона все ревизии были **перепроверены вручную** на:
   - Точность цитат из SKILL.md (номера строк).
   - Актуальность ссылок на схемы (case-schema.yaml, index-schema.yaml).
   - **Корректность правовых ссылок** (АПК, ГПК, НК).
   - Согласованность с ARCHITECTURE.md и conventions.md.

### Рамка качества

Из 12 ревизий 4 написаны вручную (intake, legal-review, prepare-hearing, build-position), 8 — LLM-агентами в параллель с последующей проверкой.

В ходе верификации были обнаружены и исправлены:
- **1 галлюцинация** про правовую норму (срок замечаний на протокол в АПК — агент утверждал 3 дня, реально 5 по ст. 155 ч. 7).
- **3 правовые правки**: 2 неточности в `appeal` (неверная статья ГПК 322 → 321; обжалование мирового неверно описано) + 1 устаревание в `cassation` (ставки госпошлины до ФЗ 259-ФЗ от 08.08.2024).
- **~15 мелких неточностей** в номерах строк, списках скриптов, перефразировках цитат.
- **1 собственная ошибка** в WORKFLOW-REVIEW.md про коллизию путей — на самом деле пути разные, проблема в изоляции данных.

**Вывод:** архитектурные и процессные наблюдения в ревизиях надёжны. Правовые ссылки и конкретные числа — требуют сверки с актуальной редакцией НПА перед использованием.

## 3. Сквозные проблемы плагина

Проблемы, которые встречаются в 3+ скиллах — системные, решаются **один раз на уровне плагина**, а не в каждом скилле отдельно.

### 3.1. Отсутствие постусловий и валидации — 12 из 12

**Проблема:** ни один SKILL.md не формализует «что должно быть правдой после успешного завершения». Нет программной проверки целостности state после apply. Модель не может убедиться, что всё прошло корректно.

**Последствия:** молчаливая порча состояния — записи в индексе без файлов, файлы без зеркал, next_id рассогласован, YAML невалиден.

**Решение:** добавить в `shared/conventions.md` раздел «Постусловия скиллов», с чек-листами по типу скилла (ingest, analytic, document-producing). Каждому SKILL.md — обязательная секция «Post-conditions» перед завершением apply.

### 3.2. Эфемерность дорогой Opus-работы — 5 скиллов

**Скиллы:** `prepare-hearing`, `analyze-hearing`, `draft-judgment`, `appeal`, `cassation`.

**Проблема:** Opus-субагенты в этих скиллах создают ценные промежуточные артефакты (red team, стресс-тест, сравнительный разбор, профиль судьи, ранжированные основания), которые **не сохраняются**. Существуют только в тексте финального документа, откуда их сложно переиспользовать.

**Последствия:**
- При повторном запуске — вся аналитика делается заново.
- При сбое сессии — работа потеряна.
- Нет накопления исторической аналитики в деле.

**Решение:** ввести `.vassal/hearings/{дата}-prep.md` (от `prepare-hearing`), `.vassal/analysis/cassation-prep-{дата}.md` (от `cassation`), и т.д. — со структурированным frontmatter + разделами.

### 3.3. Поля `case.yaml` в SKILL.md не существуют в схеме — 4 скилла

**Скиллы:** `legal-review`, `build-position`, `appeal`, `cassation` (для двух последних — рекомендация по симметричному расширению).

**Примеры:**
- `legal-review` обновляет «квалификация» и «подсудность» — полей нет в [case-schema.yaml](shared/case-schema.yaml).
- `build-position` пишет «оценочные поля» — какие именно, в SKILL.md не указано, в схеме их тоже нет.
- `appeal` должен бы записывать срок подачи, но поля `appeal.judgment_issued_date`, `appeal.deadline` отсутствуют. По симметрии — те же поля нужны для `cassation`.
- Статус `pending_appeal` / `pending_cassation` не предусмотрен в `status: active | suspended | completed | archived`.

**Решение:** расширить `case-schema.yaml`:
```yaml
qualification:
  relation_type: null
  applicable_norms: []
jurisdiction:
  competent_court: null
  territorial_basis: null
  prorogation_clause: null
appeal:
  judgment_issued_date: null
  deadline: null
  filed_date: null
  result: null
cassation:
  (аналогично)
status: active | suspended | pending_appeal | pending_cassation | completed | archived
```

### 3.4. Конвенция имён продуктов плагина — 8 скиллов

**Скиллы, нарушающие [conventions.md §«Именование файлов»](shared/conventions.md) «Дата Отправитель Описание»:**

| Скилл | Формат имени | Отклонение |
|---|---|---|
| `legal-review` | `{дата} Предварительный анализ документов.md` | Нет «Отправителя» |
| `build-position` | `{дата} {тип} позиция.md` | «Отправителя» нет, `{тип}` вместо него |
| `prepare-hearing` | `{дата} {тип документа}.md` | Аналогично |
| `catalog` | `Таблица документов.xlsx` | Даты вообще нет |
| `analyze-hearing` | `{дата}-internal.md`, `{дата}-client-brief.md` | Нет «Отправителя», дефис вместо пробела |
| `draft-judgment` | `{дата} Клиент Проект решения суда.docx` | «Клиент» нелогично — проект не от клиента |
| `appeal` | `{дата} Апелляционная жалоба.md` (+ `.docx` с «Клиент») | Нет «Отправителя» в `.md`; «Клиент» в `.docx` корректнее (это документ клиента в суд), но `.md`-вариант выпадает |
| `cassation` | `{дата} Кассационная жалоба.md` (+ `.docx` с «Клиент») | Аналогично |

**Решение:** добавить в `shared/conventions.md` раздел:
> **Продукты плагина** (анализ, позиции, процдокументы, отчёты) именуются по per-skill шаблонам. Общее правило `Отправитель Описание` касается только документов сторон (client / opponent / court / public).
>
> Список per-skill исключений:
> - `{дата} Предварительный анализ документов.md` (legal-review)
> - `{дата} {тип} позиция.md` (build-position)
> - `{дата} {тип документа}.md` (prepare-hearing)
> - `Таблица документов.xlsx` (catalog; без даты — обновляемый артефакт)
> - `.vassal/hearings/{дата}-{subtype}.md` (analyze-hearing; subtype: internal / client-brief / judge-profile / opponent-analysis)
> - `{дата} Проект решения суда.md` (draft-judgment; **убрать «Клиент»**)
> - `{дата} Апелляционная жалоба.md`, `{дата} Кассационная жалоба.md`

### 3.5. Жёсткая зависимость от `arbitrum-docx` без fallback — 4 скилла

**Скиллы:** `prepare-hearing`, `analyze-hearing`, `appeal`, `cassation` (+ `catalog` аналогично для `openpyxl`/скилла `xlsx`).

**Проблема:** если внешний плагин недоступен / не установлен / неправильной версии — скилл падает или импровизирует. Нет feature detection. Нет fallback на `.md`.

**Особенно критично:**
- `appeal`/`cassation` — жёсткие процессуальные сроки (1-2 месяца). Сбой в последний день = катастрофа.
- `analyze-hearing` — замечания на протокол имеют срок 5 дней.

**Решение:** единый паттерн в `shared/conventions.md`:
```
Правило внешних плагинов:
1. Перед использованием — feature detection (проверить доступность SKILL.md).
2. При недоступности — сохранить `.md` в целевом пути, предупредить Сюзерена.
3. Явно указать пользователю: «оформите вручную по своему шаблону».
```

### 3.6. `update-index` не триггерится автоматически — все аналитические скиллы

**Скиллы:** все аналитические (`legal-review`, `build-position`, `prepare-hearing`, `analyze-hearing`, `draft-judgment`, `appeal`, `cassation`) читают `index.yaml`, но не проверяют, что он синхронизирован с ФС.

**Проблема:** если юрист положил файл в папку дела мимо intake/add-*, скиллы его не увидят. Анализ делается на устаревших данных.

**Решение:** в каждом аналитическом скилле — шаг Phase 0 «проверить консистентность ФС↔индекс»; при рассинхроне — автотриггер `update-index` или предупреждение.

### 3.7. Ссылки на документы неоднозначны — 4 скилла

**Проблема:** ревизии показали конфликт:
- В audit-формате (legal-review) ссылки — **по имени**, без doc-id. Хорошо для человека, плохо для агентов.
- В preview и таблицах (catalog) — **по doc-id** или пути. Плохо для человека.

**Последствия:** последующие скиллы (`build-position`, `appeal`, `cassation`) парсят «человечные» ссылки fuzzy-матчем → ломаются при переименовании или дубликатах.

**Решение:** **двойная ссылка** в audit-формате:
```
**Источники:** Договор поставки №47 от 01.06.2025 [doc-042]
```
Человек читает имя, агент — `[doc-042]`. Зафиксировать в `shared/conventions.md`.

### 3.8. ARCHITECTURE.md обещает несуществующие скрипты — ≥3 места

**Проблема:** [ARCHITECTURE.md:93-97](ARCHITECTURE.md) обещает `scripts/`:
- `setup.sh` ✅ существует
- `extract_text.py` ✅ существует
- `create_mirror.py` ❌ **отсутствует**
- `update_index.py` ❌ **отсутствует**
- `generate_table.py` ❌ **отсутствует**

Из 5 скриптов реально есть 2. Логика работает через LLM-прогон — медленнее, дороже, недетерминированно.

**Решение:** либо реализовать 3 отсутствующих скрипта, либо явно удалить их из ARCHITECTURE.md и обозначить как «работает через LLM». Сейчас документация вводит в заблуждение.

### 3.9. Red team / стресс-тест в той же сессии той же модели — 3 скилла

**Скиллы:** `prepare-hearing`, `appeal`, `cassation`.

**Проблема:** «атакующий» и «защищающий» анализ — в одной сессии одного Opus. Известный consistency bias: модель, построив blue team, подсознательно смягчает red team.

**Решение:** делегировать red team **отдельному субагенту** с системным промптом «Ты юрист оппонента. Разбей эту позицию». Второй субагент работает в вакууме, без «нашей» аналитики в контексте.

### 3.10. Изоляция кросс-дельной памяти — 3 скилла

**Скиллы:** `analyze-hearing` (профиль судьи), `draft-judgment` (`judge-style.md`), `add-opponent` (паттерны оппонентов).

**Проблема:** ценные наблюдения живут внутри одного дела, между делами не переиспользуются. У юриста 5 дел у судьи Иванова → 5 отдельных `judge-profile.md`, никто не знает про других.

**Решение:** глобальное хранилище `~/.vassal-global/`:
- `judges/{ФИО}-{суд}.md` — профиль судьи (агрегируется по всем делам)
- `counterparties/{ИНН}.md` — паттерны оппонентов
- `templates/{категория}/{номер}.md` — библиотека переиспользуемых доводов

Это ровно то, что [NOTION-INTEGRATION-PROPOSAL.md](NOTION-INTEGRATION-PROPOSAL.md) предлагает как Notion-слой, но можно сделать и локально.

### 3.11. Противоречия команды и скилла по источнику файлов / аргументам — ≥3 скилла

**Скиллы:** `intake`, `prepare-hearing`, `draft-judgment`.

**Проблема:** команда разрешает `$ARGUMENTS` как путь/параметр, SKILL.md предполагает жёсткий источник («Входящие документы/»). При вызове с аргументом — undefined behavior.

**Решение:** согласованное правило «источник файлов — всегда Входящие; `$ARGUMENTS` — только указания по стратегии/типу/акцентам, не пути». Зафиксировать в conventions.

### 3.12. Нет политики повторного запуска в один день — ≥3 скилла

**Скиллы:** `build-position`, `add-evidence`, `draft-judgment`.

**Проблема:** если скилл запустить дважды в один день (откорректировать вводную, добавить новый файл) — имена файлов совпадут. Не определено: перезапись? суффикс? время?

**Решение:** единое правило — при коллизии имени добавлять `{ГГГГ-ММ-ДД-ЧЧММ}` или суффикс `(2)`. Зафиксировать в conventions.

## 5. Архитектурные паттерны для внедрения

Эти паттерны — обобщение множества частных правок. Внедрение каждого закрывает сразу несколько проблем.

### 5.1. Стандартные постусловия и валидация

Добавить в `shared/conventions.md` секцию «Пост-условия скиллов» с чек-листами:
- **Ingest-скиллы:** все файлы из «Входящих» обработаны ИЛИ оставлены с причиной; каждая новая запись ссылается на существующие file/mirror; next_id инкрементирован ровно; YAML валиден.
- **Аналитические скиллы:** все заявленные файлы в `.vassal/analysis/` созданы; сводный файл соответствует сумме; case.yaml остался валидным YAML.
- **Document-producing скиллы:** .md и .docx существуют; приложения перечислены; индекс обновлён; history зафиксирована.

### 5.2. Content-hash как единый ключ дедупликации

Одно поле в схеме (`content_hash: "sha256:..."`), вычисляется при каждом ingest, используется для:
- Проверки «этот файл уже обрабатывался?» (intake, add-evidence, add-opponent).
- Детекции «файл изменился?» (update-index).
- Связывания переименованных файлов.
- Предупреждения о дубликатах копий (оппонент прислал наш же договор).

Закрывает 4 проблемы одним изменением схемы.

### 5.3. Двухуровневое сохранение (для человека + для агента)

Шаблон:
- **Для Сюзерена** — сводный «человечный» `.md` в корне дела.
- **Для агента** — структурированные файлы в `.vassal/{categorie}/` с frontmatter (type, date, subtype, refs).

Уже сделано в `legal-review` (сводный + 7 файлов в `.vassal/analysis/`). **Не сделано** в `build-position`, `prepare-hearing`, `draft-judgment`, `appeal`, `cassation`. Применить единообразно.

### 5.4. Глобальное хранилище памяти

Путь: `~/.vassal-global/`:
- `judges/{ФИО}-{суд}.md` — накопительный профиль судьи (читается `analyze-hearing`, `draft-judgment`, дополняется каждым делом).
- `counterparties/{ИНН}.md` — паттерны оппонентов (`add-opponent`).
- `templates/{категория}/` — переиспользуемые доводы.

Плюс frontmatter `shared_profile: "~/.vassal-global/judges/..."` в локальных `judge-profile.md` — триггер двойной записи.

Если подключить **Notion-слой** ([NOTION-INTEGRATION-PROPOSAL.md](NOTION-INTEGRATION-PROPOSAL.md)) — база `Judges` и `Counterparties` закрывают ту же задачу с лучшим UX.

### 5.5. Feature detection для внешних зависимостей

Паттерн:
1. Перед использованием `arbitrum-docx` / `openpyxl` / `xlsx`-скилла — проверить доступность.
2. При недоступности — graceful degradation: сохранить минимум (`.md`/`csv`/`markdown-table`), предупредить пользователя.
3. Явно сказать «оформите вручную по своему шаблону, срок: X дней».

### 5.6. Session state + recovery

Для долгих скиллов (prepare-hearing, cassation, draft-judgment):
- В конце каждой фазы — snapshot в `.vassal/{скилл}-wip.md`.
- При повторном запуске — предложить «Обнаружен незавершённый запуск от {время}. Продолжить?».

### 5.7. Чек-листы для критичных секций

Обязательные чек-листы в SKILL.md:
- **Просительная часть** (prepare-hearing) — суммы, солидарность, проценты по день факт.оплаты, расходы, обеспечительные.
- **Резолютивная часть** (draft-judgment) — зеркально просительной.
- **Приложения к апелляции/кассации** — состав по АПК, доказательства направления.

### 5.8. Red team через отдельного субагента

Для всех скиллов со стресс-тестом — red team делегируется **отдельному Opus-субагенту** с системным промптом «Ты юрист оппонента». Получает только финальный документ, без нашей аналитики.

### 5.9. TTL и детект устаревания

Во frontmatter аналитических файлов:
```yaml
generated_at: 2026-04-01
expires_warning_after_days: 60
depends_on:
  - doc-042
  - doc-043
```

Последующие скиллы проверяют: (а) истёк ли срок, (б) не изменились ли зависимости. При проблеме — предупредить.

### 5.10. Ownership полей `case.yaml`

Отдельный документ `shared/case-schema-ownership.md` или комментарии в `case-schema.yaml`:
```yaml
qualification:     # writes: legal-review, build-position
jurisdiction:      # writes: legal-review
amounts.claim:     # writes: init-case, build-position
pretrial:          # writes: legal-review
status:            # writes: init-case, appeal, cassation, analyze-hearing
timeline:          # appends: analyze-hearing, appeal, cassation
```

## 8. Важное примечание о верификации

> **Правовые утверждения в этих ревизиях** — особенно конкретные сроки, номера статей, суммы пошлин, формулировки норм — **были частично написаны LLM-агентами**. В ходе перепроверки найдена 1 галлюцинация (срок замечаний на протокол в АПК), 2 правовые неточности (в appeal) и 1 устаревание (в cassation). Исправлены.
>
> **Перед любым реальным использованием** правовых утверждений из ревизий (в том числе для правок SKILL.md плагина) — **проверить по действующей редакции НПА**. Особенно:
> - Конкретные сроки (могут быть пересмотрены — ст. 232 ГПК была переработана в 2019, ФЗ 259-ФЗ 2024 изменил госпошлины, и т.д.).
> - Размеры ставок и пошлин (НК меняется часто).
> - Ссылки на определения ВС РФ и Постановления Пленума (могут быть отменены или заменены).
>
> Архитектурные наблюдения (отсутствие постусловий, эфемерность Opus-работы, ошибки идемпотентности, расхождения SKILL.md ↔ ARCHITECTURE.md) — **верифицированы** и не требуют дополнительной сверки.

## 9. Итог

Плагин vassal-litigator — **концептуально сильный**: хорошая декомпозиция на 12 скиллов, осмысленная фазовая организация, опора на оформленные методики (Мошкин С.А.), три качественных справочника в `legal-review`, принцип preview→apply.

Главные системные слабости — **между скиллами**:
- Нет явных контрактов и постусловий.
- Дорогая LLM-работа эфемерна.
- Кросс-дельная память отсутствует (профили судей, паттерны оппонентов — утрачиваются).
- Shared state без ownership → рассогласованность полей `case.yaml`.
- Жёсткие зависимости без fallback.

**Самые критичные правки** закрываются небольшим числом системных изменений:
1. Расширение схем + конвенций (этапы 1-2 плана работ).
2. Паттерн постусловий + backup + feature detection (этапы 3-4).
3. Двухуровневое сохранение + глобальная память (этапы 5-6).

Остальное — поступательные улучшения по мере реального использования.

**Общий объём рекомендаций:** ~150 P0-P3 пунктов по всем ревизиям. Из них P0 — ~30, P1 — ~50, остальное P2-P3. На полный рефакторинг — неделя-две активной работы. На минимально достаточный — 3-4 дня (этапы 1-3 + критические из этапа 4).
