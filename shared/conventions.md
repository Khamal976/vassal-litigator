# Конвенции vassal-litigator

## Расположение документов

**Правило:** Все выходные документы, предназначенные для Сюзерена (юриста), сохраняются в корневой папке дела (CASE ROOT). Папка `.vassal/` зарезервирована исключительно для внутренних агентских файлов и не должна быть доступна пользователю.

Все файлы и папки в корне дела начинаются с даты `ГГГГ-ММ-ДД`.

Примеры:
- Предварительный анализ: `{case-root}/2026-03-26 Предварительный анализ документов.md`
- Позиция по делу: `{case-root}/2026-03-26 Начальная позиция.md`
- Процессуальные документы: `{case-root}/2026-04-01 Истец - исковое заявление/`

Внутренние агентские файлы (которые агент читает, но не показывает Сюзерену):
- `.vassal/case.yaml` — карточка дела
- `.vassal/index.yaml` — реестр документов
- `.vassal/mirrors/` — md-зеркала документов
- `.vassal/analysis/{qualification.md, deadlines.md, ...}` — аналитические блоки (для переиспользования между скиллами)
- `.vassal/history.md` — история действий агента
- `.vassal/raw/` — оригиналы загруженных документов

## Именование файлов

Формат: `ГГГГ-ММ-ДД Отправитель Описание.расширение`

**Область применения:** правило касается **документов сторон** — материалов, поступивших от клиента, оппонента, суда или из публичных источников. Имена внутренних продуктов плагина (анализ, позиции, проекты процессуальных документов) формируются по отдельным шаблонам — см. раздел «Продукты плагина» ниже.

Правила:
- Дата — дата документа (не дата создания файла), формат строго `ГГГГ-ММ-ДД`
- Отправитель — организация или лицо, без кавычек: `ООО «Ромашка»` → `ООО Ромашка`
- Описание — тип документа и/или суть. Номер документа, если есть
- Расширение — никогда не менять
- Кодировка — UTF-8, кириллица
- Если дата неизвестна → `0000-00-00`
- Если отправитель неизвестен → `Неизвестный`
- Дубликат имени → суффикс `(2)`, `(3)` и т.д.

## Именование процессуальных папок

Формат: `{ГГГГ-ММ-ДД} {Сторона} - {действие}`

Примеры:
- `2026-03-01 Истец - исковое заявление`
- `2026-03-15 Ответчик - отзыв на иск`
- `2026-04-10 к судебному заседанию` (без стороны — общее)

## Папка «Входящие документы/»

Единая точка входа для всех новых файлов в деле. Создаётся при `init-case`.

Правила:
- Пользователь кладёт сюда любые файлы, которые нужно обработать
- Команды `intake`, `add-evidence`, `add-opponent` берут файлы **только** отсюда
- После успешной обработки файлы из «Входящие документы/» **перемещаются в папку «На удаление/»** (оригиналы хранятся в `.vassal/raw/`)
- Если папка пуста — команда обработки сообщает об этом и завершается

## Папка «На удаление/»

Корзина для файлов, подлежащих удалению. Создаётся при `init-case`.

**Зачем:** файловая система Cowork не поддерживает операцию `rm`. Вместо удаления все ненужные файлы перемещаются (`cp` + обнуление исходника) в эту папку.

Правила:
- Все команды плагина, которым нужно «удалить» файл, **перемещают его в `На удаление/`** вместо удаления
- Структура внутри — плоская, при совпадении имён добавляется суффикс `(2)`, `(3)`
- Пользователь периодически очищает эту папку вручную (на своей стороне, вне Cowork)
- Агент **не удаляет** файлы из «На удаление/» — только кладёт туда

Техническая реализация «перемещения»:
```bash
# Скопировать в «На удаление/»
cp "Входящие документы/файл.pdf" "На удаление/файл.pdf"
# Обнулить исходник (имитация удаления)
: > "Входящие документы/файл.pdf"
```
Обнуление (`: >`) необходимо, потому что `rm` не работает на маунте Cowork.

## Подпапки «Материалы от клиента»

Типовые:
- `Договоры/`
- `Переписка/`
- `Платёжные документы/`
- `Учредительные/`
- `Иное/`

Можно добавлять кастомные по ситуации.

## Таксономия типов документов

Значения поля `type` в index.yaml:

**Договорные:** договор, дополнительное-соглашение, приложение-к-договору, спецификация, акт, накладная, счёт, счёт-фактура

**Переписка:** претензия, ответ-на-претензию, уведомление, письмо, требование

**Процессуальные:** исковое-заявление, отзыв-на-иск, ходатайство, заявление, возражение, пояснения, дополнение-к-позиции, апелляционная-жалоба, кассационная-жалоба, мировое-соглашение, ходатайство-об-утверждении-мирового, заявление-об-отказе-от-иска, заявление-о-признании-иска

**Судебные:** определение, решение, постановление, протокол-заседания

**Платёжные:** платёжное-поручение, выписка, справка-из-банка

**Прочие:** доверенность, выписка-из-егрюл, устав, свидетельство, заключение-эксперта, иной-документ

## Источники документов

Значения поля `source` в index.yaml:
- `client` — получено от нашего клиента
- `opponent` — получено от процессуального оппонента
- `court` — получено из суда
- `public` — из публичных источников

## Маршрутизация моделей

| Задача | Модель | Параметр Agent tool |
|--------|--------|---------------------|
| Извлечение текста, md-зеркала, саммари | Haiku | `model: "haiku"` |
| Систематизация, таблицы, структурирование | Sonnet | основной поток |
| Правовой анализ, позиции, жалобы | Opus | `model: "opus"` |

## Принцип Preview → Apply

Все команды, изменяющие файлы или данные дела, работают в два шага:
1. **Preview** — показать план изменений Сюзерену
2. **Apply** — выполнить только после подтверждения

Касается: intake, add-evidence, add-opponent, update-index (при удалении), обновлений case.yaml.

## Аудируемость выводов

Каждый юридический вывод агента оформляется:
```
**Вывод:** [заключение]
**Фактическое основание:** [на чём основано]
**Правовое основание:** [нормы, статьи]
**Источники:** [Название документа от ДД.ММ.ГГГГ [doc-NNN]; см. раздел «Двойные ссылки»]
**Уверенность:** высокая | средняя | низкая
**Недостающие данные:** [чего не хватает]
```

## Продукты плагина

**Продукты плагина** — внутренние артефакты, создаваемые скиллами (анализ, позиции, проекты процессуальных документов, отчёты). Они не являются документами от сторон, поэтому общее правило `Дата Отправитель Описание` к ним не применяется.

**Общее правило именования продуктов:** `ГГГГ-ММ-ДД {Тип продукта}.{ext}` — без поля «Отправитель».

**Версионирование продуктов с пересчётом:** для аналитических продуктов (`legal-review`, `build-position`) при повторном запуске создаётся **новый файл с новой датой**, старый сохраняется. Это даёт хронологию мысли по делу. Перезапись запрещена.

**Коллизия имён в один день:** при двух запусках одного скилла в одну дату — суффикс `-{ЧЧММ}` после даты: `2026-04-25-1430 Обновлённая позиция.md`. Альтернатива (если время не важно) — суффикс `(2)`, `(3)` по общим правилам.

**Кросс-дельные накопительные файлы (judge-profile и т.п.)** именуются без даты — они аппенд-only по своей природе (см. раздел «Постусловия» для аналитических скиллов).

**Глобальная (кросс-дельная) память.** Часть скиллов (`analyze-hearing`, `draft-judgment`, `add-opponent`, `prepare-hearing`, `build-position`, `appeal`, `cassation`) дополнительно работает с глобальной памятью в `$VASSAL_GLOBAL_DIR/` (профиль судьи, профиль оппонента -- агрегируются по всем делам). Это **отдельное хранилище** вне текущего дела, со своим набором правил именования, накопления и постусловий -- см. раздел «Глобальная память (кросс-дельная)» ниже.

## Двухуровневое сохранение для аналитических скиллов

**Принцип:** дорогая Opus-аналитика (правовой разбор, позиции, blue/red team, стресс-тест, подбор практики ВС РФ) сохраняется в **двух представлениях одновременно**: «человечном» сводном файле для Сюзерена и наборе машинно-читаемых файлов для других скиллов. Это устраняет два антипаттерна: (а) эфемерность дорогой работы (потеря между запусками); (б) парсинг «человечного» сводного `.md` соседним скиллом.

**Уровень a) Для Сюзерена -- сводный `.md` в корне дела.**

Имя -- по шаблону скилла из таблицы «Продукты плагина» (`{ГГГГ-ММ-ДД} Предварительный анализ документов.md` и т.п.). Содержит весь анализ в человечной форме: разделы, таблицы, аудируемые выводы. Юрист читает только этот файл -- он самодостаточен.

**Уровень b) Для агента -- машинно-читаемые файлы в `.vassal/{категория}/`.**

Каждый аналитический блок сохраняется отдельным файлом со стандартным frontmatter:

```yaml
---
type: <тип артефакта, напр. position-section / prep-analysis / cassation-prep>
section: <название блока, если применимо>   # для многосекционных скиллов
date: ГГГГ-ММ-ДД
case: "{номер дела}"
related_summary: "{путь к сводному файлу для Сюзерена}"   # обратная ссылка
supersedes: null | "{предыдущий файл этой же секции}"
refs: ["doc-NNN", ...]   # doc-id всех источников
not_applicable: false | "<причина>"   # если блок не имеет смысла в данном контексте
---
```

Содержимое -- блок из сводного, в аудируемом формате, с двойными ссылками на источники.

**Когда какой паттерн применять:**

| Скилл | Сводный файл | Файлы для агента | Расположение |
|---|---|---|---|
| `legal-review` | `{дата} Предварительный анализ документов.md` | `qualification.md`, `deadlines.md`, `pretrial.md`, `jurisdiction.md`, `completeness.md`, `questions-to-client.md`, `answers.md` | `.vassal/analysis/` |
| `build-position` | `{дата} {тип} позиция.md` | `position-{дата}-{evidence-base, strengths, weaknesses, opponent-forecast, risks, strategy}.md` | `.vassal/analysis/` |
| `prepare-hearing` | `{дата} {тип процдокумента}.md` (текст документа) | `{дата}-prep.md` (Blue/Red team, парирование, стресс-тест, финальная структура) | `.vassal/hearings/` |
| `cassation` | `{дата} Кассационная жалоба.md` (текст жалобы) | `cassation-prep-{дата}.md` (сравнительный разбор актов, кандидаты в основания, фильтр непереоценки, практика ВС, ранжирование, стресс-тест) | `.vassal/analysis/` |
| `settlement` | `{дата} Мировое соглашение — анализ и проект.md` (или варианты для `withdrawal` / `admission`) + комплект `.docx` через `arbitrum-docx` | `settlement-{дата}-{batna, zone-of-agreement, opponent-forecast, risks, terms-draft}.md` | `.vassal/analysis/` |

**Правила консистентности:**

1. **Содержательная согласованность.** Каждое утверждение в сводном `.md` имеет корреспондирующий блок хотя бы в одном из машинно-читаемых файлов; и наоборот, ничего значимого не существует только в служебной зоне.
2. **Двойные ссылки везде.** В обоих представлениях источники -- по правилу «Двойные ссылки» (`Название документа от ДД.ММ.ГГГГ [doc-NNN]`). Юрист в сводном файле читает имя; соседний скилл в машинно-читаемом -- ходит по `[doc-NNN]`.
3. **Цепочка `supersedes`.** При повторном запуске:
   - Сводный получает `supersedes: <путь к предыдущему сводному>` во frontmatter.
   - Каждый машинно-читаемый файл получает `supersedes: <путь к предыдущему файлу той же секции>`.
   - Старые файлы **не удаляются** -- это история мысли по делу.
4. **`not_applicable` вместо пропуска.** Если по контексту дела какой-то блок неприменим (например, у ответчика без встречного иска evidence-base почти пустой) -- файл всё равно создаётся со фронтматтером `not_applicable: <причина>` и одной строкой в теле. Это сохраняет цепочку и даёт явный аудиторский след «почему пусто».
5. **Постусловия.** Категория 2 «Аналитические скиллы» в разделе «Постусловия скиллов» ниже формализует, что должно быть правдой после `apply` -- в том числе обязательное создание сводного и всех заявленных машинно-читаемых файлов.

**Зачем именно так:**
- `prepare-hearing` напрямую читает `.vassal/analysis/position-{последняя-дата}-{секция}.md` без парсинга человечной позиции -- структура у него под контролем frontmatter.
- При повторной кассации (СКЭС) -- читается `cassation-prep-*.md` целиком, фазы 2-3 не повторяются.
- `analyze-hearing` после заседания читает `{дата}-prep.md` для сравнения «что готовили» vs «что фактически произошло».
- При сбое сессии -- работа Opus сохранена и доступна следующему запуску.

## Глобальная память (кросс-дельная)

**Принцип:** некоторые наблюдения ценны не только для текущего дела, но и для всех будущих дел того же юриста: профиль судьи (его стиль, предпочтения, триггеры), паттерны конкретного оппонента (типовые доводы в отзывах, тактика, представители). Эти наблюдения накапливаются в **глобальной памяти**, единой для всех дел Сюзерена, и читаются скиллами **до** анализа -- модель сразу знает, что у этого судьи / оппонента уже наблюдалось в прошлых делах.

Без глобальной памяти возникает антипаттерн «изоляция кросс-дельной ценности»: 5 дел у судьи Иванова → 5 пустых стартовых профилей; тот же оппонент в третьем деле подряд → модель снова с нуля анализирует «нового» оппонента, паттерны которого мы уже видели дважды.

### Расположение

**Базовый путь:** `~/.vassal-global/` (по умолчанию).

**Переопределение:** переменная окружения `VASSAL_GLOBAL_DIR`. Если задана -- используется её значение, иначе -- `~/.vassal-global/`.

Зачем переопределение: типичные сценарии -- хранение глобальной памяти на синхронизируемом диске (Яндекс.Диск / Dropbox / OneDrive) для кросс-машинного синка между рабочей и поездочной машинами; либо изоляция глобальной памяти разных юристов команды на одной рабочей станции.

**Структура папки:**

```
$VASSAL_GLOBAL_DIR/                              # по умолчанию ~/.vassal-global/
├── judges/
│   ├── {ФИО-slug}--{court-slug}.md             # один файл на пару (судья × суд)
│   └── ...
├── counterparties/
│   ├── inn-{ИНН}.md                            # для юрлиц с ИНН -- ИНН как ключ
│   ├── noinn-{slug-name}.md                    # fallback для физлиц / без ИНН
│   └── ...
└── README.md                                    # опционально, заметки Сюзерена
```

**Важно:** глобальная папка живёт **вне** репозитория плагина и **вне** папки конкретного дела. Скиллы создают её при первом обращении (`mkdir -p`), если не существует.

`templates/{категория}/` -- предусмотрено архитектурно, но в этап 6 не входит (наполняется вручную, см. план в FINAL-REPORT.md этап 7).

### Правила слагов

Имена файлов формируются по детерминированному правилу:

1. Кириллица сохраняется (без транслитерации) -- `Иванов`, `Арбитражный`.
2. Пробелы → дефис: `Арбитражный суд` → `Арбитражный-суд`.
3. Точки удаляются: `И. И.` → `И-И`.
4. Кавычки `«»"'` удаляются: `ООО «Ромашка»` → `ООО-Ромашка`.
5. Запятые, слэши, спецсимволы → дефис.
6. Подряд идущие дефисы схлопываются в один.
7. Между двумя слаг-блоками в имени файла -- двойной дефис `--` (визуальный разделитель).

Примеры:
- Судья: `Иванов И. И.` × `Арбитражный суд Красноярского края` → `Иванов-И-И--Арбитражный-суд-Красноярского-края.md`
- Оппонент с ИНН: `inn-7707083893.md` (короткое имя в frontmatter).
- Оппонент без ИНН: `noinn-Иванов-Иван-Иванович.md`.

### Двойная запись и обратные ссылки

Когда скилл наблюдает паттерн (по судье или оппоненту), он пишет **в два места одновременно**:

1. **Локальный файл** -- в `.vassal/` текущего дела (`judge-profile.md`, `opponent-filing-*.md`, `opponent-hearing-*.md`). Содержит наблюдения только из этого дела, привязан к конкретным заседаниям/документам с двойными ссылками `[doc-NNN]`.
2. **Глобальный файл** -- в `$VASSAL_GLOBAL_DIR/`. Содержит **агрегированные** наблюдения по всем делам, каждое наблюдение помечено источником `(дело {номер}, {дата})`.

В локальном файле во frontmatter -- обратная ссылка на глобальный:

```yaml
---
type: judge-profile
judge: "Иванов И. И."
court: "Арбитражный суд Красноярского края"
shared_profile: "judges/Иванов-И-И--Арбитражный-суд-Красноярского-края.md"
# путь относительно $VASSAL_GLOBAL_DIR -- скилл резолвит при чтении
hearings_analyzed: 2          # в этом деле
last_updated: 2026-05-10
---
```

В глобальном файле -- список дел, на которых он построен:

```yaml
---
type: judge-profile-global
judge: "Иванов И. И."
court: "Арбитражный суд Красноярского края"
slug: "Иванов-И-И--Арбитражный-суд-Красноярского-края"
cases:
  - "А33-12345/2026"
  - "А33-67890/2026"
  - "А33-11111/2025"
hearings_total: 8                       # суммарно по всем делам
written_decisions_total: 3              # суммарно
last_updated: 2026-05-10
last_case: "А33-67890/2026"
---
```

### Правило накопления

При каждом обновлении (двойная запись после `analyze-hearing`, `draft-judgment`, `add-opponent`):

- В **локальный** файл -- наблюдения этого запуска, с двойными ссылками на документы текущего дела.
- В **глобальный** файл:
  - Каждое новое наблюдение дописывается **аппенд-only** в соответствующий раздел с обязательным маркером `(дело {номер}, заседание {дата})` или `(дело {номер}, документ {краткое имя})`.
  - Старые наблюдения **никогда не удаляются и не редактируются** (даже если они противоречат новым -- противоречие добавляется как «уточнение» с новой датой; разрешать конфликты -- задача Сюзерена при чтении, не агента при записи).
  - Список `cases:` пополняется (без дублей по номеру дела).
  - Счётчики `hearings_total`, `written_decisions_total`, `filings_analyzed`, `hearings_analyzed` инкрементируются ровно на число добавленных в этом запуске.
  - `last_updated` и `last_case` обновляются.

**Бэкфилл накопленной локальной памяти.** Для дел, чья локальная аналитика (`.vassal/hearings/judge-profile.md`, `.vassal/analysis/opponent-*.md`) накапливалась до этапа 6.1 (двойная запись в глобальную) или иначе не попала в глобальный профиль -- разовый перенос выполняется через служебный скилл `/vassal-litigator:backfill-global` ([skills/backfill-global/SKILL.md](../skills/backfill-global/SKILL.md)). Скилл с Opus-семантической дедупликацией добавляет новые наблюдения в глобальные файлы с маркерами `(дело {номер}, ...)` и пишет опциональное поле `backfilled_cases:` во frontmatter -- его читает только сам `backfill-global` для идемпотентности повторных запусков; другие скиллы это поле игнорируют.

### Чтение глобальной памяти

Скиллы, у которых есть зона релевантности глобальной памяти (`prepare-hearing`, `build-position`, `appeal`, `cassation`, `analyze-hearing`, `draft-judgment`):

1. **В фазе загрузки контекста** -- разрезолвить `$VASSAL_GLOBAL_DIR` (с fallback на `~/.vassal-global/`).
2. По `case.judge` собрать ожидаемое имя файла судьи в `judges/` (по правилам слагов выше). Если файл существует -- загрузить **до** локального `judge-profile.md`. Если нет -- работать без глобального профиля, не падать.
3. По `case.parties` для оппонента (роль ≠ роли нашего клиента) -- собрать имя файла в `counterparties/`. Сначала пробовать `inn-{ИНН}.md` (если ИНН есть в `case.parties[].inn`), потом `noinn-{slug-name}.md`. Если файл существует -- загрузить.
4. Глобальные файлы имеют **приоритет в смысле «ранее увиденного»** -- модель учитывает их паттерны как фон, локальные наблюдения этого дела -- как foreground.

### Конфиденциальность

Глобальная память -- **производная аналитика**, не копия документов:

- ✅ Можно: паттерны судьи, типовые формулировки оппонента, статистика «в N делах из M ходатайствовал по ст. 333 ГК», ФИО представителей, ИНН/ОГРН (это публичные данные).
- ❌ Нельзя: тексты документов клиентов, конкретные суммы по конкретным делам клиентов, персональные данные третьих лиц без юридической необходимости (контрагенты клиента -- да, физлица -- осторожно).

Глобальная папка, как и `.vassal/` дела, лежит **локально на машине Сюзерена**. Если используется кросс-машинный синк через облако -- ответственность за приватность канала на Сюзерене.

### Постусловия (для скиллов, пишущих в глобальную память)

После `apply` скилл обязан проверить (в дополнение к локальным постусловиям своей категории):

- Глобальный файл создан или обновлён по правильному пути (резолв `$VASSAL_GLOBAL_DIR` выполнен).
- Папка `judges/` или `counterparties/` создана при необходимости (`mkdir -p`).
- Содержимое **аппенд-only**: счётчики увеличены ровно на число добавленных наблюдений; список `cases:` -- без дублей; `last_updated` обновлён.
- Локальный файл (если есть) содержит во frontmatter `shared_profile:` -- путь к глобальному относительно `$VASSAL_GLOBAL_DIR`.
- При сбое записи в глобальный файл (ФС недоступна, нет прав) -- скилл **не падает целиком**, а: (а) фиксирует локальный файл; (б) предупреждает Сюзерена по шаблону «Глобальная память недоступна: {причина}; локальный профиль сохранён; глобальный синк выполнить позже»; (в) пишет факт в `.vassal/history.md`.

## Notion-слой (опциональная одностороння синхронизация)

**Принцип:** Notion -- **зеркало**, не источник правды. Скилл `notion-sync` ([skills/notion-sync/SKILL.md](../skills/notion-sync/SKILL.md)) выполняет односторонний push метаданных дела (`case.yaml`), глобального профиля судьи (`$VASSAL_GLOBAL_DIR/judges/`) и (опционально, этап 6.2b) глобальных профилей оппонентов (`$VASSAL_GLOBAL_DIR/counterparties/`) в Notion-базы `Cases`, `Judges`, `Counterparties`. Это даёт юристу единый дашборд по всем делам и кросс-дельную видимость (которая локально лежит в глобальной памяти, но в Notion удобнее просматривать).

Notion-слой -- **опциональный**. Плагин полностью работоспособен без него. Если конфиг `~/.vassal/notion-config.yaml` отсутствует -- авто-хуки в `init-case`, `analyze-hearing`, `appeal`, `cassation` не предлагают синк, ручная команда `/vassal-litigator:sync-notion` явно сообщает о ненастроенности.

### Что синхронизируется

| Данные | В Notion? | Почему |
|---|---|---|
| Метаданные дела (номер, стороны, суд, наш клиент, стадия, следующее заседание) | ✅ | Дашборд |
| Профиль судьи (стиль, паттерны, склонности, кол-во дел) | ✅ | Кросс-дельная память + дашборд |
| Профили оппонентов (`$VASSAL_GLOBAL_DIR/counterparties/`) -- сжатые поля «Типовые доводы», «Тактика», «Представители» | ✅ (опционально, этап 6.2b) | Кросс-дельная память + дашборд; активируется, если `notion.databases.counterparties` задана в конфиге |
| Тексты документов / md-зеркала | ❌ | Конфиденциальность + объём |
| Позиции (`*позиция.md`) | ❌ | Стратегия -- самое чувствительное |
| `.vassal/analysis/*` | ❌ | Оценки рисков, слабостей |
| Транскрипции заседаний | ❌ | Чувствительно |

Полная матрица -- в [NOTION-INTEGRATION-PROPOSAL.md](../NOTION-INTEGRATION-PROPOSAL.md) §3.

### Направление синхронизации

**Push (автоматически по запросу):**
- Скилл `notion-sync` ИЛИ опциональные хуки в `init-case`, `analyze-hearing`, `appeal`, `cassation` (хуки **не блокирующие** -- скилл-инициатор завершается успешно даже если Сюзерен отказался от синка или синк упал).
- Конфликты разрешаются просто: при upsert поля из локальной правды **перезаписывают** Notion-запись, **кроме** `fields_manual_only` из конфига (например, `Статус: won/lost` -- ручное поле юриста).

**Pull (вручную, по запросу) -- зона расширения, не MVP:**
- При следующем `prepare-hearing` плагин может предложить: «По судье Иванову в Notion есть свежие заметки от 15.04 -- подтянуть?». В MVP не реализовано.

**Обратный авто-синк не делается** -- конфликтует с local-first архитектурой. См. [NOTION-INTEGRATION-PROPOSAL.md](../NOTION-INTEGRATION-PROPOSAL.md) §4.

### Конфигурация

Файл: `$VASSAL_CONFIG_DIR/notion-config.yaml` (один на пользователя, не на дело).

**Расположение:**
- По умолчанию: `~/.vassal/notion-config.yaml`.
- Переопределение: переменная окружения `VASSAL_CONFIG_DIR`. Если задана -- используется её значение, иначе -- `~/.vassal/`.

Сценарии переопределения -- те же, что у `$VASSAL_GLOBAL_DIR` (см. раздел «Глобальная память»):
- Хранение в облачном диске для кросс-машинного синка между рабочей и поездочной машинами (важно: на разных Windows-машинах `~` резолвится в разные пути из-за разных имён пользователя; env var с явным OneDrive/Dropbox-путём решает это).
- Изоляция конфигов разных юристов команды.

Логически `$VASSAL_CONFIG_DIR` и `$VASSAL_GLOBAL_DIR` парные -- обычно ставятся в **одну OneDrive-/Dropbox-папку**, как два соседних подкаталога. Можно и порознь, если нужно (например, конфиг -- локально, а память -- на облаке).

**На Windows для кросс-машинного синка** рекомендуется `REG_EXPAND_SZ` со встроенной переменной `%OneDrive%` -- одно значение работает на всех машинах одного юзера, Windows подставит профиль. Точная команда -- в [scripts/notion-init.md](../scripts/notion-init.md) → §3.1. Это ключевая деталь: `setx` создаёт `REG_SZ` без раскрытия `%VAR%`, поэтому `setx` для кросс-машинной конфигурации не подходит -- нужен `reg add /t REG_EXPAND_SZ`.

Шаблон конфига -- [scripts/notion-config.example.yaml](../scripts/notion-config.example.yaml). Bootstrap (создание Notion-баз и заполнение конфига) -- разовая операция по инструкции [scripts/notion-init.md](../scripts/notion-init.md).

Токен Notion -- через MCP-auth, в плагине не хранится.

### Идемпотентность и dedup

- **Cases**: dedup по `case.number`. Дело без номера -- временный ключ `temp-{slug}` (миграция при появлении номера -- зона расширения).
- **Judges**: dedup по `slug` глобального профиля (или `ФИО+Суд` если глобального профиля нет).
- **Counterparties** (если активна): dedup по `slug` (`inn-{ИНН}` или `noinn-{slug-name}` -- тот же ключ, что в файле локальной памяти). Fallback'и: ИНН, TITLE `Организация`. Теневой дубль (`inn-*` + `noinn-*` одновременно в локальной памяти) синкается **только в каноническую запись**, см. [skills/add-opponent/SKILL.md](../skills/add-opponent/SKILL.md) → «Дедуп оппонента: inn-/noinn- теневой дубль».
- При двух последовательных запусках на одно дело -- update, не дубль.

### Опция `--private-case`

Если в `case.yaml` есть поле `case.private: true` (зарезервировано на будущее) -- `notion-sync` **не синкает** это дело и явно сообщает Сюзерену. В MVP поле не используется, но скилл его уважает. Сценарий: дела особо чувствительных клиентов, где даже метаданные не должны уходить в облако.

### Постусловия

- `~/.vassal/notion-config.yaml` прочитан и валиден ИЛИ скилл вышел с явным предупреждением (без изменений).
- Notion MCP проверен на доступность (через короткий probe-запрос) ИЛИ зафиксировано в `.vassal/notion-sync.log` как недоступный.
- При успешном синке: запись в `Cases` создана/обновлена ровно одна; запись в `Judges` -- ровно одна (если `case.judge` не null); записи в `Counterparties` -- по одной на каждого оппонента из `case.parties` (если `notion.databases.counterparties` активна и оппоненты есть); relation `Cases.Судья → Judges` и `Cases.Оппонент → Counterparties` установлены; поля из `fields_manual_only` не перезаписаны.
- `.vassal/notion-sync.log` обновлён (timestamp + что синкалось + статус OK/PARTIAL/FAIL).
- `.vassal/history.md` обновлён.

## Постусловия скиллов

После успешного `apply` каждый скилл должен убедиться, что его выходные артефакты соответствуют контракту. Без явных постусловий скилл не может проверить себя — возможна молчаливая порча состояния (записи в индексе без файлов, файлы без зеркал, рассогласованный `next_id`, невалидный YAML).

В каждом `SKILL.md` обязательна секция **«Постусловия»** перед завершением `apply`. Чек-лист зависит от категории скилла.

### Категория 1: ingest-скиллы (`intake`, `add-evidence`, `add-opponent`)

- `index.yaml` — валидный YAML, соответствует [shared/index-schema.yaml](index-schema.yaml) v3.
- Для каждой новой записи:
  - физический файл по пути `file:` существует;
  - md-зеркало по пути `mirror:` существует и валидно (frontmatter + текст);
  - `content_hash` вычислен и записан;
  - `source` соответствует скиллу (intake/add-evidence → `client`, add-opponent → `opponent`).
- `next_id` инкрементирован ровно на число добавленных документов; `next_bundle_id` — на число добавленных бандлов.
- `.vassal/raw/{скилл}-{ГГГГ-ММ-ДД}/` содержит копии всех обработанных оригиналов (бит-в-бит).
- Все обработанные файлы из «Входящие документы/» перемещены в «На удаление/» (через `cp` + обнуление, см. раздел «Папка „На удаление/"»).
- Если в `Входящих` остались файлы — каждому в отчёте указана причина (нечитаемый, защищён паролем, требует ручного решения).
- Запись в `.vassal/history.md` зафиксирована.
- Если есть бандлы: `bundles[].main_doc` и `members` ссылаются на существующие `doc-NNN`; в каждом `doc-NNN` бандла заполнены `bundle_id`, `parent_id`, `role_in_bundle`.

### Категория 2: аналитические скиллы (`legal-review`, `build-position`, `prepare-hearing`, `analyze-hearing`, `settlement`)

> `settlement` -- гибрид категорий 2 и 3: на нём действуют постусловия обеих категорий. По категории 2 -- блок «зона компромисса» и стресс-тест трёх рисков сохраняются машинно-читаемыми файлами; по категории 3 -- финальные проекты документов (мировое / ходатайство об утверждении / заявление об отказе от иска / заявление о признании иска) создаются как `.md` + `.docx` через `arbitrum-docx`. Подробности -- ниже в категории 3 и в `skills/settlement/SKILL.md` → «Постусловия».

Для `appeal` и `draft-judgment` -- по аналогичному паттерну (если внедрено в SKILL.md).
- Запись в `.vassal/history.md` зафиксирована, в ней указаны имена `.md`, `.docx` (или fallback), prep-артефакта и состояние внешних зависимостей.

### Категория 4: служебные (`update-index`, `catalog`, `backfill-global`)

- **`update-index`:** перед apply снят backup `.vassal/index.yaml.bak.{timestamp}` (ротация 5). После apply — все ingest-постусловия (см. категорию 1) + проверено обратное отношение «зеркало → запись» (нет orphan mirrors).
- **`catalog`:** `Таблица документов.xlsx` создана с колонкой `ID` (doc-id) для двойной ссылки. Все записи `index.yaml` присутствуют в таблице.
- **`backfill-global`:** перед записью каждого целевого глобального файла снят backup `{глобальный путь}.bak.{timestamp}` (ротация 5); при успехе backup удалён, при сбое -- восстановлен. После apply -- все постусловия раздела «Глобальная память» (для пишущих в глобальную): `cases:` без дублей; счётчики инкрементированы ровно на число `new_bullets` (merged не инкрементируют); `backfilled_cases:` обновлено; аппенд-only не нарушен (ни один существующий bullet не удалён, merged-bullets имеют объединённый маркер). Локальный `judge-profile.md` обработанных дел имеет `shared_profile:` во фронтматтере. Соседний (не-канонический) файл при теневом дубле НЕ редактировался. `.vassal/history.md` каждого обработанного дела дополнен записью «backfill-global» со счётчиками.

## Двойные ссылки

**Принцип:** ссылка на документ дела должна быть читаема **и** для человека (юриста), **и** для агента (других скиллов).

**Формат:** `Название документа от ДД.ММ.ГГГГ [doc-NNN]`

Примеры:
- `Договор поставки №47 от 01.06.2025 [doc-042]`
- `Претензия ООО «Ромашка» от 15.08.2025 [doc-088]`
- `Решение Арбитражного суда Красноярского края от 20.03.2026 [doc-201]`

**Где обязательны:**
- Аудируемые выводы (поле `**Источники:**`).
- Сводные аналитические файлы для Сюзерена (`Предварительный анализ документов.md`, позиции).
- Файлы в `.vassal/analysis/*.md`.
- Колонка `ID` в `Таблица документов.xlsx` (от `catalog`).

**Где не нужно:**
- Имена файлов на ФС (там и так есть имя документа, doc-id хранится в индексе).
- Md-зеркала (`.vassal/mirrors/*.md`) — они и есть документ, frontmatter содержит `id`.

**Зачем:**
- Юрист читает «Договор поставки №47 от 01.06.2025», игнорируя `[doc-042]` визуально.
- Агенты других скиллов парсят `[doc-NNN]` и идут точно по индексу — без fuzzy-матчинга по имени, который ломается при переименованиях, дубликатах с суффиксом `(2)`, опечатках.

**Миграция старых артефактов:** существующие аналитические файлы (без doc-id в источниках) остаются как есть — переписывать историю не нужно. Новые версии при перегенерации добавят `[doc-NNN]` автоматически.

## Дедупликация по `content_hash`

**Поле:** `content_hash: "sha256:<hex>"` в каждой записи `index.yaml` (см. [index-schema.yaml](index-schema.yaml)).

**Когда вычисляется:** при ingest файла (intake / add-evidence / add-opponent), до создания зеркала. Хэш считается от **исходного бинарного содержимого** оригинала в `.vassal/raw/`, не от md-зеркала.

**Проверка дублей перед добавлением:**
1. Скилл вычисляет `content_hash` для каждого файла в пакете.
2. Сверяет с существующими `content_hash` в `index.yaml`.
3. При совпадении — **не создавать новую запись `doc-NNN`**, а:
   - Для `intake`/`add-evidence` (наш клиент): пропустить файл, отметить в preview как «дубль уже обработанного `doc-NNN` — пропускаем».
   - Для `add-opponent`: добавить копию в существующую запись через поле `duplicate_copies` (см. схему `index-schema.yaml`). Это юридически значимо — отмечается, что оппонент представил тот же документ.

**Использование при детекции изменений (`update-index`):**
1. Скилл считает `content_hash` каждого файла на диске.
2. Сравнивает с `content_hash` в индексе.
3. Если хэши не совпадают → файл изменился → пересоздать зеркало, обновить `last_verified` и `mirror_stale: false`, новый хэш записать.

**Использование при автосвязке переименований:**
- Если детектирована «осиротевшая запись» (файл по `file:` отсутствует) **и** «новый файл без записи» с **тем же `content_hash`** — это переименование. Связать автоматически (обновить `file:` в существующей записи), без вопроса Сюзерену.

**Хэши не зависят от имени, времени, метаданных файла — только от содержимого.** Это даёт стабильность относительно переименований, повторных пересылок, копирований.

## Ownership полей `case.yaml`

Чтобы скиллы не перезаписывали друг другу значения и не возникала «гонка обновлений», каждое поле `case.yaml` имеет **owner-скиллы** — те, кому разрешено его писать. Все остальные читают, но не пишут.

Матрица фиксируется в [case-schema.yaml](case-schema.yaml) v3 в виде комментариев `# writes: <скиллы>` и `# reads: <скиллы>` рядом с каждым полем (или блоком).

Сводно по основным группам:

| Блок | Writes | Reads |
|---|---|---|
| `case.{number, court, judge, category, subject, our_client, representative}` | `init-case` | все |
| `case.judge`, `case.next_hearing`, `case.next_hearing_source` | `init-case`, `analyze-hearing` | все |
| `case.parties` (структура) | `init-case` | все |
| `case.parties` (уточнение реквизитов) | `legal-review` | — |
| `case.amounts.{claim, court_fee}` | `init-case`, `build-position` | `prepare-hearing`, `appeal`, `cassation` |
| `case.qualification` | `legal-review`, `build-position` (только `applicable_norms`) | `build-position`, `prepare-hearing`, `appeal`, `cassation` |
| `case.jurisdiction` | `legal-review` | `build-position`, `prepare-hearing`, `appeal`, `cassation` |
| `case.pretrial` | `legal-review` | `prepare-hearing`, `appeal` |
| `case.settlement.*` | `settlement` (основные поля), `analyze-hearing` (`confirmed_date`, если суд утвердил/принял прямо в заседании) | `prepare-hearing` (хук в фазе 2 п.13), `cassation` (если `confirmed_date != null` и обжалуется по ст. 141 АПК) |
| `case.appeal.*` | `appeal`, `analyze-hearing` (`hearing_date`, `ruling_date`) | `cassation` |
| `case.cassation.*` | `cassation`, `analyze-hearing` (`hearing_date`, `ruling_date`) | — |
| `case.status` | `init-case`, `appeal`, `cassation`, `analyze-hearing`, `settlement` | все |
| `case.timeline` (append-only) | `init-case`, `legal-review`, `analyze-hearing`, `appeal`, `cassation`, `settlement` | все |
| `case.filing_date` | `init-case` | все |

**Правило:** если скилл не указан в `writes:` для поля — он **не имеет права это поле обновлять**. Если по логике задачи нужно — это сигнал к расширению матрицы (правка conventions + comments в case-schema.yaml), не к молчаливой перезаписи.

## Внешние зависимости и fallback

Часть скиллов опирается на внешние плагины и системные утилиты:

| Зависимость | Кто использует | Назначение |
|---|---|---|
| `arbitrum-docx` (внешний плагин) | `prepare-hearing`, `analyze-hearing`, `draft-judgment`, `appeal`, `cassation`, `settlement` | Оформление `.docx` процессуальных документов по шаблонам (для `settlement` -- мировое соглашение, ходатайство об утверждении, заявления об отказе от иска / о признании иска) |
| `xlsx`-скилл / `openpyxl` | `catalog` | Генерация `Таблица документов.xlsx` |
| `tesseract`, `ocrmypdf` (system) | `intake`, `add-evidence`, `add-opponent`, `update-index` | OCR сканированных PDF |
| `pymupdf`, `python-docx` (Python) | `scripts/extract_text.py` | Извлечение текста из PDF/DOCX |
| Notion MCP (`notion-search`, `notion-create-pages`, `notion-update-page`, `notion-create-database`, `notion-update-data-source`) | `notion-sync` | Push метаданных дела, профиля судьи и (опционально, этап 6.2b) профилей оппонентов в Notion-базы Cases / Judges / Counterparties (опционально, см. раздел «Notion-слой») |
| `$VASSAL_CONFIG_DIR/notion-config.yaml` (по умолчанию `~/.vassal/notion-config.yaml`) | `notion-sync` и хуки в `init-case` / `analyze-hearing` / `appeal` / `cassation` | ID Notion-баз и mapping полей. При отсутствии файла Notion-функционал безопасно отключается (хуки не предлагают синк, ручная команда явно сообщает о ненастроенности). Путь конфигурируется -- см. раздел «Notion-слой» → «Конфигурация» |

Внешние зависимости могут оказаться недоступны: плагин не установлен, версия несовместима, упал процесс. Скилл **не имеет права тихо упасть** — это особенно опасно для `appeal`/`cassation` (жёсткие процессуальные сроки) и `analyze-hearing` (5 дней на замечания на протокол).

### Единый паттерн «feature detection + fallback»

Применяется во всех скиллах, использующих внешние зависимости.

**1. Feature detection в начале скилла:**
- Для плагина: проверить, что соответствующий `SKILL.md` доступен (например, `arbitrum-docx/SKILL.md`).
- Для Python-зависимости: попытка импорта в pre-flight check.
- Для системной утилиты: `which tesseract` / `tesseract --version` с обработкой ненулевого exit code.

**2. При недоступности — graceful degradation:**
- Сохранить **минимально достаточный** артефакт в целевом пути:
  - Вместо `.docx` через arbitrum-docx → создать только `.md` в том же месте.
  - Вместо `.xlsx` через xlsx-скилл → создать markdown-таблицу `Таблица документов.md`.
  - Вместо OCR через tesseract → пометить файл `needs_manual_review: true` и отложить в отдельный отчёт.

**3. Явное предупреждение Сюзерену:**

Шаблон сообщения:
```
ВНИМАНИЕ. Внешняя зависимость недоступна: {имя зависимости}
Сделано: {что именно создано в fallback-режиме, путь}
Что нужно от Сюзерена: {оформите вручную по своему шаблону / установите зависимость и перезапустите / ...}
Срок (если процессуальный): осталось N дней до {дедлайн}
```

**4. Запись в `.vassal/history.md`:** обязательно зафиксировать, что артефакт создан в fallback-режиме (для последующего аудита).

**5. Что НЕ делать:**
- Тихо пропускать создание `.docx` и считать задачу выполненной.
- Импровизировать («попробую через python-docx сам») — fallback должен быть единственным предсказуемым.
- Перепрыгивать на другой формат без предупреждения (например, сохранить `.txt` вместо `.docx`).

## Проверка консистентности ФС ↔ индекс (Phase 0)

**Зачем:** аналитические скиллы (`legal-review`, `build-position`, `prepare-hearing`, `analyze-hearing`, `draft-judgment`, `appeal`, `cassation`) читают `index.yaml` и md-зеркала, но между запусками между ФС и индексом мог накопиться рассинхрон: юрист положил файл мимо `intake`, переименовал зеркало вручную, переложил документ. Без проверки -- скилл анализирует устаревшие данные и формулирует выводы на неполной картине дела.

**Кто обязан выполнять:** все аналитические скиллы, перед началом своей Phase 1 (загрузка контекста).

### Алгоритм Phase 0

1. **Быстрая сверка ФС ↔ индекс** (без полного скана через `update-index`):
   - Прочитай `.vassal/index.yaml` -- получи список путей `file:` для всех записей.
   - Просканируй ключевые папки дела: `Материалы от клиента/` (рекурсивно), все `{ГГГГ-ММ-ДД} *` (рекурсивно), корень дела.
   - Игнорируй: `.vassal/`, `На удаление/`, `Входящие документы/`, `*.bak.*`, `Таблица документов.xlsx`, `.DS_Store`, `Thumbs.db`.
   - Сверь множества: `files_on_disk` vs `files_in_index`.

2. **Классификация рассинхрона:**
   - `new_files`: файлы на диске без записи в `index.yaml`.
   - `orphan_records`: записи `index.yaml` без файла на диске.
   - (Устаревшие зеркала отдельно не проверяются здесь -- это работа `update-index`.)

3. **Решение:**
   - **Если рассинхрона нет** (`new_files == 0 and orphan_records == 0`) -- молча перейти к Phase 1. Никаких сообщений Сюзерену.
   - **Если рассинхрон есть** -- предупредить Сюзерена:
     ```
     ВНИМАНИЕ. Рассинхрон ФС ↔ индекс:
       Новые файлы на диске (нет в индексе): N -- {краткий список первых 3-5}
       Осиротевшие записи (нет файла): M -- {краткий список первых 3-5}

     Рекомендация: запустить update-index перед анализом, чтобы анализ работал на актуальных данных.

     Варианты:
     1. Запустить update-index сейчас (рекомендуется)
     2. Продолжить анализ на текущем индексе (риск: новые файлы не учтутся, ссылки на осиротевшие записи будут битыми)
     3. Прервать
     ```
   - Дождаться выбора Сюзерена.
   - Если выбран п.1 -- передать управление `update-index` (либо инструкцией Сюзерену запустить `/vassal-litigator:update-index`, либо явным вызовом скилла), после завершения -- вернуться в текущий скилл и заново начать Phase 0 (она должна пройти чисто).
   - Если выбран п.2 -- зафиксировать в `.vassal/history.md` и в выходном артефакте: «Анализ выполнен на устаревшем индексе ({N} новых файлов, {M} осиротевших не учтены)». Это часть аудируемости.
   - Если выбран п.3 -- завершить.

### Что НЕ делает Phase 0

- Не пересоздаёт зеркала, не вычисляет хэши -- это полная работа `update-index`.
- Не пишет в `index.yaml` ни в каком случае -- только читает.
- Не сравнивает `mtime` / `content_hash` для устаревших зеркал -- этим занимается `update-index`.

Phase 0 -- быстрая проверка «есть ли смысл доверять текущему индексу как полному снимку дела». Полная сверка -- задача `update-index`.
