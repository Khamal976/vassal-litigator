# Notion-слой для vassal-litigator — предложение

**Дата:** 2026-04-20
**Статус:** draft, на проработку
**Контекст обсуждения:** сессия анализа плагина, обсуждение потенциальных интеграций (Notion, NotebookLM). NotebookLM отклонён: нет API, дублирует локальный Q&A плагина, конфиденциальность. Notion — перспективен как слой над плагином.

---

## 1. Принцип

Notion — **не копия** `.vassal/`, а то, чего плагин сейчас не делает в принципе: **кросс-дело память и practice-management**.

- Источник правды остаётся локальным (`.vassal/`).
- Notion получает метаданные одним направлением (push).
- Чувствительные данные (тексты документов, позиции, стратегия) в Notion не уходят.

Правило разделения: **Notion — про «что», `.vassal/` — про «как» и «почему».**

---

## 2. Структура: 6 баз

### 2.1. `Cases` — реестр всех дел

| Поле | Тип | Источник |
|---|---|---|
| Номер дела | title | `case.yaml: case_number` |
| Суд | select | `case.yaml: court` |
| Наш клиент | text | `case.yaml: our_client` |
| Истец / Ответчик | text | `case.yaml: parties[]` |
| Наша роль | select (истец/ответчик/третье лицо) | `case.yaml` |
| Стадия | select (досудебная → 1 инст → апелляция → кассация → исполнение → закрыто) | `case.yaml` + эвристика из `history.md` |
| Следующее заседание | date | `case.yaml: next_hearing` |
| Судья | relation → `Judges` | `case.yaml: judge` |
| Оппонент | relation → `Counterparties` | `case.yaml: parties[]` с `role != our_client` |
| Статус | select (active / paused / closed / won / lost) | вручную или из `history.md` |
| Путь к папке | url (`file:///...`) | абсолютный путь монтирования |
| Last sync | date | timestamp sync'а |

### 2.2. `Hearings` — заседания (relation → Cases)

| Поле | Источник |
|---|---|
| Дата | `.vassal/hearings/{дата}-*.md` имя файла |
| Дело | relation |
| Тип | первая строка internal.md (подготовительное / основное / отложение) |
| Результат | из `analyze-hearing` → internal.md (заголовок «Итог») |
| Судья | relation → Judges |
| Сводка | первый абзац client-brief.md (НЕ весь отчёт) |

### 2.3. `Judges` — профильная база (**главное value-add Notion**)

| Поле | Источник |
|---|---|
| ФИО | title |
| Суд | text |
| Стиль | text, накапливается из всех дел с этим судьёй |
| Паттерны | text |
| Склонности | text (какие доводы заходят, какие нет) |
| Дел с ним | rollup count → Cases |
| Win-rate | rollup (won / total) |

**Ключевое:** сейчас `judge-profile.md` живёт в одном деле и не знает про того же судью в соседнем. Notion агрегирует этот опыт.

### 2.4. `Counterparties` — оппоненты и их юристы

| Поле | Источник |
|---|---|
| Организация | title |
| ИНН/ОГРН | text |
| Представители | text (накапливаются) |
| Типовые аргументы | text — что этот оппонент обычно пишет в отзывах |
| Связанные дела | relation → Cases |

Тоже кросс-дело: если оппонент встретится в другом деле, у тебя уже есть его паттерны.

### 2.5. `Deadlines` — сроки и напоминания

| Поле | Источник |
|---|---|
| Что | text |
| Когда | date |
| Дело | relation |
| Тип | select (подача жалобы / срок давности / госпошлина / заседание / ...) |
| Статус | select (не начато / в работе / сделано / пропущено) |

Авто-наполнение:
- Из `case.yaml: next_hearing` → дедлайн «Подготовка к заседанию» (−3 дня)
- Из `.vassal/analysis/deadlines.md` (`legal-review`) → сроки давности, досудебный порядок
- Из решений суда → сроки на апелляцию/кассацию (вычисляются автоматически)

### 2.6. `Templates` — библиотека переиспользуемых доводов

| Поле | Назначение |
|---|---|
| Название | title |
| Категория | select (отзывной довод / ходатайство / процессуальное возражение / мотивировочный блок) |
| Отрасль | select (поставка / подряд / банкротство / ...) |
| Текст | rich text |
| Использовано в | relation → Cases |

**Эта база — Notion-native**, плагин её не генерирует сам. Но когда пишется жалоба, можно передать relevant templates в контекст (`cassation` читает Templates по отрасли).

## 3. Матрица: что синхронизируется, что нет

| Данные | В Notion? | Почему |
|---|---|---|
| Метаданные дела (номер, стороны, суд) | ✅ | Для дашборда |
| Заголовки заседаний, результаты | ✅ | Для хронологии |
| Профиль судьи | ✅ | Ценен кросс-дело |
| Сроки | ✅ | Для напоминаний |
| **Тексты документов (PDF/docx)** | ❌ | Конфиденциальность + объём |
| **md-зеркала** | ❌ | Содержат полный текст материалов |
| **Позиции (`*позиция.md`)** | ❌ | Стратегия — самое чувствительное |
| **`.vassal/analysis/*`** | ❌ | Оценки рисков, слабостей |
| **Транскрипции заседаний** | ❌ | Чувствительно |
| Саммари заседания (2-3 строки) | ✅ | Ок для дашборда |

## 4. Направление синхронизации

```
┌─────────────────┐                 ┌──────────────┐
│   .vassal/      │ ──── push ────► │    Notion    │
│ (source of      │                 │  (dashboard, │
│    truth)       │ ◄── manual ──── │   memory)    │
└─────────────────┘                 └──────────────┘
```

**Push (автоматически):**
- Из плагина → в Notion, при изменении `case.yaml`, появлении нового hearing, обновлении судейского профиля.
- Конфликты решаются просто: Notion-запись всегда перезаписывается, поля, помеченные как `manual_only` (например, «Статус» = won/lost) — не трогаются.

**Pull (вручную, по запросу):**
- Обновления по судье (ты вручную дописал в Notion наблюдение) → при следующем `prepare-hearing` в этом деле плагин может спросить: «По судье Иванову в Notion есть свежие заметки от 15.04 — подтянуть?»
- Templates → при `appeal`/`cassation` по запросу: «Подгрузить из Notion шаблоны доводов по категории «подряд»?».

Обратный автосинк **не делаем** — конфликтует с local-first архитектурой.

## 5. Реализация в плагине

### 5.1. Новый скилл `notion-sync`

Путь: `skills/notion-sync/SKILL.md`

Делает:
- Читает `.vassal/case.yaml`, `hearings/*`, `history.md`
- Через Notion MCP (`notion-search` → `notion-update-page` / `notion-create-pages`) идентифицирует дело по `case_number` и апдейтит
- Сохраняет `.vassal/notion-sync.log` с timestamp и статусом

### 5.2. Новая команда `/vassal-litigator:sync-notion`

Ручной триггер.

### 5.3. Авто-триггеры в существующих скиллах (hooks в конце)

- `init-case` → создать запись в `Cases`
- `analyze-hearing` → создать запись в `Hearings`, обновить `Judges`
- `build-position` → обновить `Cases.Стадия`
- `appeal` / `cassation` → обновить `Cases.Стадия` + сроки в `Deadlines`

### 5.4. Конфиг (один раз на пользователя, не на дело)

Путь: `~/.vassal/notion-config.yaml`

```yaml
notion:
  workspace_id: "..."
  databases:
    cases: "db-id-1"
    hearings: "db-id-2"
    judges: "db-id-3"
    counterparties: "db-id-4"
    deadlines: "db-id-5"
    templates: "db-id-6"
  fields_manual_only:
    cases: ["Статус"]
```

Токен Notion — через MCP-auth, не хранится в плагине.

### 5.5. Bootstrap-скрипт

Путь: `scripts/notion-init.sh`

Создаёт все 6 баз с правильной схемой в указанном workspace (через MCP), чтобы не собирать их руками.

## 6. Риски и сложности

| Риск | Митигация |
|---|---|
| **Дубли дел в Notion** (повторный sync создаёт вторую запись) | Dedup по `case_number` обязательно |
| **Partial failure** (sync упал на 3-й базе из 6) | Идемпотентный sync, флаги в `notion-sync.log`, retry |
| **Расхождения**, когда юрист правит в Notion поле, которое плагин перезаписывает | Список `fields_manual_only` в конфиге |
| **Команда / несколько юристов** работают с одной Notion workspace | Отдельная колонка «Owner» в Cases, sync только своих дел |
| **Rate limits Notion API** при массовом первом sync | Пакетирование, задержки |
| **Конфиденциальность**: даже метаданные (номера дел клиентов) могут быть sensitive | Опция `--private-case` — такие дела не попадают в Notion вообще |

## 7. Ценность для юриста

Что получит:
1. **Единый дашборд** всех дел (сейчас — N папок, в каждой `.vassal/case.yaml`, ни одного обзора).
2. **Календарь заседаний** — view в Notion по `Hearings.Дата`.
3. **Память по судьям** — накопленные паттерны через все дела с этим судьёй.
4. **Напоминания о сроках** — Notion умеет слать push/email по date-полям.
5. **Шаблоны доводов** — собственная база, растущая с каждым делом.

Что не получит:
- Не заменит `.vassal/` — зеркала документов, позиции и анализ остаются локально.
- Не ускорит генерацию документов — это делают плагинные скиллы.

## 8. Открытые вопросы

- **MVP scope.** Стартовать с `Cases` + `Judges` (дашборд + кросс-дело память) или с `Cases` + `Deadlines` (дашборд + напоминания)?
- **Команда vs одиночный юрист.** Если плагин развивается как продукт — мультипользовательская модель в Notion (owner-колонка, sharing) нужна с самого начала или отложить?
- **Токен Notion.** Только через MCP-auth или дать опцию локального хранения (для тех, кто не использует Cowork)?
- **Initial backfill.** Как наливать Notion существующими делами юриста, если их уже много локально? Массовый скрипт `notion-backfill`?
- **Номер дела как ключ dedup.** Что если дело без номера (досудебная стадия)? Временный `temp-{uuid}` и миграция при появлении реального номера?
- **Отдельная БД `Events`** (ходатайства поданы, возражения получены и т.п.) — или это избыточно и timeline в `history.md` достаточно?
