> Плагин для помощи юристу в ведении судебных дел: от приёма документов клиента до кассационной жалобы.

**Версия архитектуры:** 0.6.0

## 1. Общая концепция

### 1.1. Роли

- **Сюзерен** — юрист-пользователь, который ведёт дело и принимает решения.
- **Вассал** — агент (Claude), который выполняет рутинные и аналитические задачи по указаниям Сюзерена.

### 1.2. Принципы

1. **Агент не принимает решений за юриста.** Вассал анализирует, систематизирует, предлагает — но каждое юридически значимое действие утверждает Сюзерен.
2. **Preview → Apply.** Все команды, изменяющие файловую структуру или карточку дела, работают в два шага: сначала показывают план изменений (preview), затем выполняют только после подтверждения Сюзерена (apply). Это касается: `intake`, `add-evidence`, `add-opponent`, `update-index` (при удалении записей), обновлений `case.yaml` из транскрипций.
3. **Неизменяемость исходников.** Оригинальные файлы, полученные от клиента или оппонента, сохраняются нетронутыми в `.vassal/raw/`. Все переименования и перемещения выполняются как копирование в новое место. Оригинал никогда не удаляется и не перезаписывается.
4. **Экономия токенов через md-зеркала.** Каждый документ дела при обработке получает полнотекстовое md-представление (канонизированный текст). Агент по умолчанию работает только с md-зеркалами, к оригиналам обращается в исключительных случаях (верификация цитат, проверка визуальных реквизитов). Md-зеркало — это кэш извлечённого текста, а не замена юридически значимого оригинала.
5. **Единый индекс дела.** Все скиллы и команды читают и пишут в один индекс — `.vassal/index.yaml`. Это «память» агента по делу.
6. **Маршрутизация моделей.** Дешёвые операции (саммаризация, классификация) делегируются Haiku через субагентов. Аналитическая работа — Sonnet. Глубокий правовой анализ — Opus. OCR выделен в отдельный pipeline (программный OCR → LLM для интерпретации).
7. **Идемпотентность.** Повторный запуск команды не дублирует файлы и записи в индексе; агент проверяет, что уже сделано.
8. **Аудируемость правовых выводов.** Каждый юридический вывод агента (сроки, подсудность, позиция) сопровождается структурой: заключение, фактическое основание, правовое основание, источники, степень уверенности, недостающие данные.

## 3. Структура папок дела

При инициализации нового дела (`/vassal-litigator:init-case`) создаётся следующая структура:

```
/путь/к/делу/
│
├── .vassal/                              ← Скрытая директория агента (только для агента)
│   ├── case.yaml                         ← Карточка дела (метаданные)
│   ├── index.yaml                        ← Полный индекс документов
│   ├── raw/                              ← Неизменяемые оригиналы (как получены)
│   │   ├── intake-2026-03-26/            ← Пакет приёма с датой
│   │   │   ├── файл_как_прислал_клиент.zip
│   │   │   ├── скан001.jpg
│   │   │   └── ...
│   │   └── intake-2026-04-10/            ← Следующий пакет
│   │       └── ...
│   ├── mirrors/                          ← MD-зеркала всех документов
│   │   ├── doc-001.md
│   │   ├── doc-002.md
│   │   └── ...
│   ├── analysis/                         ← Аналитические блоки (для переиспользования)

│   │   ├── deadlines.md                  ← Реестр пресекательных сроков
│   │   ├── questions-to-client.md        ← Вопросы клиенту
│   │   ├── completeness.md               ← Оценка комплектности
│   │   ├── pretrial.md                   ← Досудебный порядок
│   │   └── jurisdiction.md               ← Подсудность
│   ├── hearings/                         ← Отчёты о заседаниях (для агента)
│   │   └── ...
│   └── history.md                        ← Лог действий агента
│
├── Входящие документы/                   ← Точка входа для всех новых файлов

├── Материалы от клиента/                 ← Исходные документы клиента
│   └── (подпапки по решению Сюзерена)
│
├── {ГГГГ-ММ-ДД} {Сторона} - {действие}/ ← Процессуальные папки

│
├── {ГГГГ-ММ-ДД} Начальная позиция.md     ← Правовые позиции (хронологически)
```

### 3.1. Правила именования процессуальных папок

Шаблон: `{ГГГГ-ММ-ДД} {Сторона} - {название процессуального документа или действия}`

Примеры:
- `2026-03-01 Истец - исковое заявление`
- `2026-03-15 Ответчик - отзыв на иск`
- `2026-04-01 Истец - пояснения по делу`
- `2026-04-10 к судебному заседанию` (если неясно, к какому документу)

Если составление процессуального документа не планируется, файлы перемещаются в `Материалы от клиента/`.

## 5. Формат индекса документов (index.yaml)

Индекс — это единый реестр всех документов дела. Каждый документ получает уникальный ID (`doc-NNN`), который используется для связи с md-зеркалом.

```yaml
# .vassal/index.yaml
version: 2
last_updated: 2026-03-26T14:30:00

documents:
  - id: doc-001
    file: "Материалы от клиента/Договоры/2025-06-01 ООО Ромашка Договор поставки №47.pdf"
    mirror: "mirrors/doc-001.md"
    type: договор
    title: "Договор поставки №47"
    date: 2025-06-01
    parties:
      - "ООО «Ромашка»"
      - "ООО «Лютик»"
    summary: "Договор поставки строительных материалов. Общая сумма 1 500 000 руб. Срок поставки — 30 дней с момента оплаты."
    seal: true
    signature: true
    completeness: full       # full | partial | fragment
    quality: high             # high | medium | low (качество извлечения текста)
    needs_manual_review: false  # true, если OCR низкого качества
    pages: 12
    source: client            # client | opponent | court | public
    added: 2026-03-26
    processed_by: haiku       # модель, которая обработала документ

    # Происхождение файла
    origin:
      name: "договор поставки.pdf"         # исходное имя файла, как получено
      received: 2026-03-26                   # дата получения
      intake_batch: "intake-2026-03-26"      # пакет приёма (папка в raw/)

    # Целостность зеркала
    mirror_stale: false       # true, если оригинал изменился после создания зеркала
    last_verified: 2026-03-26

    # Бандлы (группы связанных документов)
    bundle_id: null            # ID бандла, если документ входит в группу
    parent_id: null            # ID родительского документа (для приложений)
    role_in_bundle: null       # main | appendix | exhibit

    # Процессуальный статус
    filing_status: null        # null | draft | submitted | filed
    filing_folder: null        # путь к процессуальной папке, если подан

    tags:
      - основной-договор
      - поставка

  - id: doc-002
    file: "Материалы от клиента/Переписка/2025-09-15 ООО Ромашка Претензия.pdf"
    mirror: "mirrors/doc-002.md"
    type: претензия
    title: "Претензия о нарушении сроков поставки"
    date: 2025-09-15
    parties:
      - from: "ООО «Ромашка»"
        to: "ООО «Лютик»"
    summary: "Претензия Ромашки о нарушении сроков поставки по Договору №47. Требование: поставить товар в 10-дневный срок."
    seal: true
    signature: true
    completeness: full
    quality: medium
    needs_manual_review: false
    pages: 2
    source: client
    added: 2026-03-26
    processed_by: haiku
    origin:
      name: "претензия скан.pdf"
      received: 2026-03-26
      intake_batch: "intake-2026-03-26"
    mirror_stale: false
    last_verified: 2026-03-26
    bundle_id: null
    parent_id: null
    role_in_bundle: null
    filing_status: null
    filing_folder: null
    tags:
      - досудебный-порядок
      - претензия

# Бандлы — группы связанных документов (документ + приложения)
bundles:
  # Пример:
  # - id: bundle-001
  #   title: "Исковое заявление с приложениями"
  #   main_doc: doc-010
  #   members: [doc-010, doc-011, doc-012]

# Счётчик для генерации следующего ID
next_id: 3
next_bundle_id: 1
```

### 5.1. Типы документов (taxonomy)

Поле `type` — из фиксированного перечня:

**Договорные:** `договор`, `дополнительное-соглашение`, `приложение-к-договору`, `спецификация`, `акт`, `накладная`, `счёт`, `счёт-фактура`

**Переписка:** `претензия`, `ответ-на-претензию`, `уведомление`, `письмо`, `требование`

**Процессуальные:** `исковое-заявление`, `отзыв-на-иск`, `ходатайство`, `заявление`, `возражение`, `пояснения`, `дополнение-к-позиции`, `апелляционная-жалоба`, `кассационная-жалоба`

**Судебные:** `определение`, `решение`, `постановление`, `протокол-заседания`

**Платёжные:** `платёжное-поручение`, `выписка`, `справка-из-банка`

**Прочие:** `доверенность`, `выписка-из-егрюл`, `устав`, `свидетельство`, `заключение-эксперта`, `иной-документ`

## 6. Формат MD-зеркала

Каждый документ дела получает полнотекстовое md-зеркало (канонизированное текстовое представление) в `.vassal/mirrors/`. Имя файла — `{id}.md` (например, `doc-001.md`). MD-зеркало — это кэш извлечённого текста для экономии токенов, а не замена юридически значимого оригинала.

```markdown
---
id: doc-001
source: "Материалы от клиента/Договоры/2025-06-01 ООО Ромашка Договор поставки №47.pdf"
type: договор
title: "Договор поставки №47"
date: 2025-06-01
pages: 12
extraction_method: pdf-text    # pdf-text | ocr | docx-parse | image-read | manual
extraction_model: haiku
extraction_date: 2026-03-26
confidence: high               # high | medium | low (качество извлечения)
---

# Договор поставки №47

## от 01 июня 2025 года

**ООО «Ромашка»**, именуемое в дальнейшем «Покупатель», в лице генерального директора Петрова И.И., действующего на основании Устава, с одной стороны, и **ООО «Лютик»**, именуемое в дальнейшем «Поставщик», в лице генерального директора Сидорова А.А., действующего на основании Устава, с другой стороны, заключили настоящий Договор о нижеследующем:

## 1. Предмет договора
```

### 6.1. Правила создания md-зеркал

1. **Полный текст** — весь текст документа переносится в md, с сохранением структуры (заголовки, пункты, таблицы).
2. **Метаданные в frontmatter** — id, источник, метод извлечения, дата, уверенность.
3. **HTML-комментарии** — только технические пометки о качестве извлечения (проблемные страницы, неточности OCR). Юридические выводы и аналитика агента хранятся в `.vassal/analysis/` и в позициях `{корень дела}/*позиция.md`, а не в зеркале.
4. **Таблицы** — конвертируются в markdown-таблицы.
5. **Изображения** — не переносятся; вместо них ставится `[Изображение: описание]`.
6. **Нечитаемые фрагменты** — помечаются `[Нечитаемо: описание проблемы]`.
7. **Нечитаемые документы** — если OCR даёт низкое качество, документ всё равно получает зеркало с пометкой `confidence: low` и `needs_manual_review: true` в индексе. Агент должен предупредить Сюзерена о таких документах.

## 7. Маршрутизация: OCR pipeline и модели

### 7.1. OCR и извлечение текста (отдельный pipeline)

OCR концептуально отделён от LLM. Сначала программное извлечение, потом LLM для интерпретации.

**Уровень 1 — Программный OCR (бесплатно, детерминированно):**
- PDF с текстовым слоем → `pdftotext` / `pymupdf`
- DOCX → `python-docx` парсинг
- Изображения / сканы без текста → `tesseract` + `ocrmypdf`
- Реализация: `scripts/extract_text.py` + `scripts/setup.sh` (установка tesseract раз за сессию)

**Уровень 2 — MCP-сервер OCR (целевое решение):**
- Локальный MCP-сервер с tesseract, работающий на машине юриста (вне VM Cowork)
- Инструменты: `ocr_file(path) → text`, `extract_pdf_text(path) → text`, `convert_image_to_pdf(path) → path`
- Преимущества: не нужна переустановка каждую сессию, возможность подключить облачный OCR для сложных сканов
- Описывается в `.mcp.json` как опциональный коннектор

**Уровень 3 — LLM fallback:**
- Если программный OCR даёт `confidence: low` — отправить страницу Haiku-субагенту для дочистки
- Для изображений, которые tesseract не берёт (рукописный текст, плохое качество) — Haiku vision

### 7.2. Маршрутизация моделей

Дорогие модели используются только там, где нужна глубина анализа.

| Задача | Модель | Обоснование |
|--------|--------|-------------|
| Создание md-зеркала | Haiku (субагент) | Форматирование извлечённого текста |
| Саммаризация для индекса | Haiku (субагент) | Краткое описание, не юридический анализ |
| Переименование файлов | Haiku (субагент) | Анализ имён и начала документов |
| Систематизация по папкам | Sonnet | Требует понимания контекста дела |
| Составление таблицы | Sonnet | Структурирование данных |
| Первичный правовой анализ | Opus | Глубокий юридический анализ |
| Проверка сроков | Opus | Критически важно — ошибка = потеря права |
| Выработка позиции | Opus | Стратегическое мышление |
| Подготовка к заседанию | Opus | Аргументация, red team/blue team |
| Анализ транскрипции (структурирование) | Sonnet (субагент) | Разбивка, очистка от артефактов |
| Анализ транскрипции (глубокий) | Opus (субагент) | Юридический анализ позиций и рекомендации |
| Генерация процессуальных документов | Opus | Юридическое письмо |

### 7.3. Реализация маршрутизации

В SKILL.md каждого скилла указывается, какие задачи делегировать субагентам:

```
Делегируй Haiku-субагенту (model: "haiku"):
- Извлечение текста из файлов
- Создание md-зеркал
- Саммаризацию для summary в индексе

Делегируй Opus-субагенту (model: "opus"):
- Правовой анализ
- Оценку рисков
- Формулирование аргументов
```

## 8. Описание скиллов и команд

### 8.0. init-case — Инициализация дела

**Тип:** только команда (`/vassal-litigator:init-case`)

2. Создать структуру `.vassal/`.
3. Заполнить `case.yaml`.
4. Создать пустой `index.yaml`.
5. Создать `history.md` с записью об инициализации.

### 8.1. intake — Приём и систематизация материалов клиента

**Тип:** скилл + команда (`/vassal-litigator:intake`)
**Модель:** Haiku (скрейпинг) + Sonnet (систематизация)
**Вход:** Файлы в `Входящие документы/`
**Выход:** Переименованные файлы в структурированных подпапках `Материалы от клиента/`, оригиналы в `.vassal/raw/`, обработанные файлы из `Входящие документы/` → `На удаление/`

**Зависимости:** `init-case` (должна существовать `.vassal/case.yaml`)
**Использует:** Логику file-renamer (шаблон переименования), но адаптированную для юридического контекста

**Алгоритм:**

**Фаза 1 — Сохранение оригиналов:**
1. Прочитать `case.yaml` для контекста (стороны, предмет спора).
2. Забрать файлы из `Входящие документы/`. Если пуста — стоп.
3. Скопировать все файлы «как есть» в `.vassal/raw/intake-{ГГГГ-ММ-ДД}/`.
4. Зафиксировать `origin_name` и `received` для каждого файла.

**Фаза 2 — Извлечение и анализ (скрытая работа):**
4. Обзор файлов (рекурсивно).
5. Распаковка архивов.
6. Преобразование изображений в PDF (если это сканы документов, а не фото).
7. OCR pipeline: сначала программный OCR (`scripts/extract_text.py`), при низком качестве — Haiku-субагент для дочистки.
8. Анализ имён файлов — категоризация на 3 группы (как в file-renamer).
9. Для файлов с недостаточной информацией в имени — чтение начала через Haiku-субагент.

**Фаза 3 — Preview (план на утверждение Сюзерену):**
10. Сформировать и показать Сюзерену:
    - План переименования по шаблону `ГГГГ-ММ-ДД Отправитель Описание.расш`.
    - Предложение структуры подпапок: типовые (`Договоры/`, `Переписка/`, `Платёжные документы/`, `Учредительные/`, `Иное/`) или кастомные.
    - Список файлов с низким качеством OCR (`needs_manual_review`).
    - Уведомление об очистке: «Файлы будут перемещены из Входящие документы/ в На удаление/ (оригиналы → .vassal/raw/)».
11. Дождаться подтверждения Сюзерена.

**Фаза 4 — Apply (выполнение после подтверждения):**
12. Создать `Материалы от клиента/` с подпапками, скопировать и переименовать файлы.
13. Для каждого файла — создать md-зеркало в `.vassal/mirrors/` (делегировать Haiku-субагенту).
14. Обновить `index.yaml` (с полями origin, mirror_stale, needs_manual_review).

**Фаза 5 — Очистка:**
15. Переместить из `Входящие документы/` все успешно обработанные файлы в `На удаление/` (через `cp` + `: >`, т.к. `rm` не работает в Cowork). Оригиналы уже в `.vassal/raw/`. Необработанные файлы остаются.

**Отличия от standalone file-renamer:**
- Источник — выделенная папка `Входящие документы/`, а не произвольная директория.
- Добавляется сохранение оригиналов в `.vassal/raw/`.
- Добавляется двухфазный preview → apply.
- После apply — очистка `Входящие документы/`.
- Добавляется этап систематизации по папкам (file-renamer не перемещает файлы).
- Добавляется создание md-зеркал.
- Добавляется обновление index.yaml.
- Контекст из case.yaml помогает точнее определять стороны.
- OCR через программный pipeline, а не напрямую через LLM.

### 8.2. catalog — Каталогизация документов

**Тип:** скилл + команда (`/vassal-litigator:catalog`)
**Модель:** Haiku (саммаризация) + Sonnet (таблица)
**Вход:** Обработанные документы в `.vassal/mirrors/` и `index.yaml`
**Выход:** `Таблица документов.xlsx` + обновлённый `index.yaml`

**Зависимости:** `intake` (должны существовать файлы и начальный индекс)

**Алгоритм:**
1. Прочитать `index.yaml` и `case.yaml`.
2. Для каждого документа, где summary ещё пуст или недостаточен — прочитать md-зеркало через Haiku-субагент, дополнить:
   - Краткое описание содержания (1-3 предложения)
   - Наличие печати/подписи (из метаданных зеркала)
   - Полнота документа
   - Качество (читаемость)
3. Учесть, что папка может объединять один документ или документ с приложениями — в таблице показывать как группу.
4. Сгенерировать `Таблица документов.xlsx` с колонками:

| № | Дата | Тип документа | Отправитель → Адресат | Название | Краткое содержание | Печать/Подпись | Полнота | Качество | Файл |
|---|------|--------------|----------------------|----------|-------------------|---------------|---------|----------|------|

5. Обновить `index.yaml` дополненными данными.

### 8.3. legal-review — Первичный правовой анализ

**Тип:** скилл + команда (`/vassal-litigator:legal-review`)
**Модель:** Opus (основной анализ)
**Вход:** `case.yaml`, `index.yaml`, вводная от клиента/Сюзерена, md-зеркала
**Выход:**
- Единый документ для Сюзерена: `{case-root}/Правовое заключение.md`
- Аналитические блоки для агента: `.vassal/analysis/{qualification.md, deadlines.md, pretrial.md, jurisdiction.md, completeness.md, questions-to-client.md}`

**Зависимости:** `catalog` (полный индекс с саммари)

**Алгоритм:**
1. Прочитать `case.yaml`, `index.yaml`.
2. Прочитать вводную от клиента (из текста команды или файла).
3. Загрузить md-зеркала ключевых документов (договор, претензия, иск).
4. Через Opus-субагент выполнить анализ. **Каждый вывод** оформляется в аудируемом формате:
   ```
   **Вывод:** [заключение]
   **Фактическое основание:** [на чём основано]
   **Правовое основание:** [нормы, статьи]
   **Источники:** [doc-001, doc-003]
   **Уверенность:** высокая | средняя | низкая
   **Недостающие данные:** [чего не хватает для уверенного вывода]
   ```

   **a) Ответы на вопросы Сюзерена** (→ `.vassal/analysis/answers.md` и в case-root документ)
   - Краткие ответы на поставленные вопросы.
   - Если указано расписать подробно — развёрнутый ответ.

   **b) Пресекательные сроки** (→ `.vassal/analysis/deadlines.md` и в case-root документ)
   - Исковая давность (общая — 3 года, специальные).
   - Течение, приостановление, перерыв исковой давности.
   - Процессуальные сроки (обжалование, предъявление ИД).
   - Для каждого срока: начало, истечение, статус (действует/истёк/под вопросом).

   **c) Досудебный порядок** (→ `.vassal/analysis/pretrial.md` и в case-root документ)
   - Требуется ли обязательный досудебный порядок.
   - Если да — проверка соблюдения (претензия направлена? срок выдержан?).

   **d) Подсудность** (→ `.vassal/analysis/jurisdiction.md` и в case-root документ)
   - Определение компетентного суда.
   - Проверка пророгационных / дерогационных оговорок в договоре.
   - Исключительная подсудность.

   **e) Качество и полнота документов** (→ `.vassal/analysis/completeness.md` и в case-root документ)
   - Состояние документов (читаемость, полнота).
   - Отсутствующие документы, которые следует запросить.
   - Документы, требующие нотариального заверения / апостиля и т.д.

   **f) Вопросы клиенту** (→ `.vassal/analysis/questions-to-client.md` и в case-root документ)
   - Перечень вопросов.
   - Перечень документов, которые следует запросить.

5. Собрать весь анализ в один связный документ и сохранить в case root как `Правовое заключение.md`.
6. Обновить `history.md`.

**Тип:** скилл + команда (`/vassal-litigator:update-index`)
**Модель:** Sonnet
**Вход:** Текущее состояние папки дела
**Выход:** Обновлённый и верифицированный `index.yaml`

**Алгоритм:**

4. Обновить `index.yaml` и сообщить Сюзерену о результатах.

1. Загрузить полный контекст дела:
   - case.yaml, index.yaml
   - Все позиции из case root (файлы вида `{ГГГГ-ММ-ДД} {тип} позиция.md`)
   - Все отчёты из `.vassal/hearings/` (если были предыдущие заседания)
   - md-зеркала процессуальных документов обеих сторон
2. Учесть:
   - Ранее подготовленные процессуальные документы Сюзерена.
   - Позицию оппонента из его документов и выступлений.
   - Ремарки и позицию судьи (из отчётов о заседаниях).
3. Работа по принципу **red team / blue team**:
   - **Blue team**: аргументы в пользу нашего клиента.
   - **Red team**: контраргументы, которые выдвинет оппонент, и слабости нашей позиции.
   - **Ответ на red team**: как парировать контраргументы.
4. Сформировать текст процессуального документа.
5. **Оформление через arbitrum-docx** — передать текст скиллу `arbitrum-docx` для генерации .docx по шаблону.
6. Поместить .docx в соответствующую процессуальную папку.
7. Обновить `index.yaml` и `history.md`.

### 8.9. analyze-hearing — Анализ транскрипции заседания

**Тип:** скилл + команда (`/vassal-litigator:analyze-hearing`)
**Модель:** Sonnet (структурирование) + Opus (глубокий анализ)
**Вход:** Транскрипция заседания (текст или файл)
**Выход:** Отчёты в `.vassal/hearings/` + (опционально) `.docx` для клиента

**Зависимости:** `init-case`

**Алгоритм:**
1. Прочитать транскрипцию. Учитывать, что транскрипции могут содержать:
   - Артефакты распознавания речи.
   - Галлюцинации (особенно имена, даты, номера).
   - Пропуски и неразборчивые фрагменты.

2. Через Sonnet-субагент (model: "sonnet") — структурирование:
   - Дата заседания, состав участников.
   - Разбивка на блоки по выступающим.
   - Очистка от артефактов (пометить сомнительные места `[?]`).

3. Сформировать три вида отчётов:

   **a) Внутренний отчёт** (`hearings/{ГГГГ-ММ-ДД}-internal.md`) — через Opus-субагент (model: "opus"):
   - Подробная хронология заседания.
   - Позиция каждой стороны (с цитатами).
   - Вопросы и замечания судьи.
   - Процессуальные решения (отложение, назначение экспертизы и т.д.).
   - Выявленные слабости и сильные стороны (наши и оппонента).
   - Рекомендации к следующему заседанию.

   **b) Отчёт для клиента — краткий** (`hearings/{ГГГГ-ММ-ДД}-client-brief.md`)
   - Дата, суд, состав.
   - Что произошло (2-3 абзаца).
   - Позиция суда (если обозначена).
   - Что решено, следующие шаги.

   **c) Отчёт для клиента — красивый** (`.docx` через arbitrum-docx)
   - Фирменно оформленный вариант отчёта (b).
   - Тип документа: «заключение / аналитический документ» (без шапки).

4. **Preview обновлений case.yaml** — показать Сюзерену предлагаемые изменения:
   - Новые события в timeline (с пометкой `source: transcript`).
   - Дата следующего заседания (если озвучена).
   - Приобщённые документы.
   Транскрипции ненадёжны — обновление карточки дела только после подтверждения.
5. После подтверждения — обновить `case.yaml`, `index.yaml`.
6. Обновить `history.md`.

### 8.10. draft-judgment — Проект судебного акта

**Тип:** команда (`/vassal-litigator:draft-judgment`)
**Модель:** Opus
**Вход:** Полное состояние дела + (опционально) примеры судебных актов данного судьи
**Выход:** Проект решения суда (`.md` + опционально `.docx`)

**Алгоритм:**

2. Если Сюзерен загрузил примеры судебных актов данного судьи — проанализировать стиль изложения:

3. Сгенерировать проект решения, имитируя стиль судьи (если примеры есть) или в стандартном стиле (если нет).
4. Структура:
   - Вводная часть (суд, стороны, предмет).
   - Описательная часть (позиции сторон).
   - Мотивировочная часть (анализ суда).

### 8.11. appeal — Апелляционная жалоба

4. Сформировать текст жалобы.
5. Проверить сроки подачи.
6. Оформить через `arbitrum-docx`.
7. Поместить в процессуальную папку.

### 8.12. cassation — Кассационная жалоба

**Тип:** скилл + команда (`/vassal-litigator:cassation`)
**Модель:** Opus
**Вход:** Решение + постановление апелляции + состояние дела
**Выход:** `.docx` кассационной жалобы

**Зависимости:** `arbitrum-docx` (оформление)

**Алгоритм:**
1. Загрузить решение и постановление апелляции (md-зеркала).
2. Загрузить полный контекст дела.
3. Через Opus — анализ с учётом специфики кассации:
   - Кассация проверяет **только** правильность применения норм права.
   - Не переоценивает доказательства (за исключением существенных нарушений).
   - Основания: ст. 288 АПК РФ.
4. Сформировать текст жалобы.
5. Проверить сроки подачи.
6. Оформить через `arbitrum-docx`.
7. Поместить в процессуальную папку.

**Ветвление по target (case-schema v4):** при обжаловании определения об утверждении мирового (ст. 141 ч. 11 АПК) — особый путь: 1 месяц вместо 2, минуя апелляцию, материальные основания иные (нарушение порядка утверждения / противоречие закону / нарушение прав третьих лиц / неопределённость предмета). Подробности — `skills/cassation/references/ruling-appeals.md`.

### 8.13. settlement — Мировое соглашение / отказ от иска / признание иска

**Тип:** скилл + команда (`/vassal-litigator:settlement`)
**Модель:** Opus (трёхсторонняя оценка зоны компромисса + стресс-тест условий)
**Вход:** `case.yaml`, `index.yaml`, позиции, аналитика `build-position` (`strengths.md`, `weaknesses.md`, `risks.md`), глобальные профили (судья, оппонент), переписка вне процесса
**Выход:**
- Сводный документ для Сюзерена: `{case-root}/{ГГГГ-ММ-ДД} Мировое соглашение — анализ и проект.md` (либо аналоги для `withdrawal` / `admission` по `case.settlement.mode`)
- Документы для подачи в суд через `arbitrum-docx`: `{ГГГГ-ММ-ДД} Мировое соглашение.docx` (двусторонний документ для подписи обеими сторонами, ст. 140 АПК); `{ГГГГ-ММ-ДД} Ходатайство об утверждении мирового соглашения.docx` (по умолчанию одностороннее от нашего клиента); `{ГГГГ-ММ-ДД} Заявление об отказе от иска.docx`; `{ГГГГ-ММ-ДД} Заявление о признании иска.docx`
- Машинно-читаемые секции анализа: `.vassal/analysis/settlement-{ГГГГ-ММ-ДД}-{batna|zone-of-agreement|opponent-forecast|risks|terms-draft}.md`
- Обновление `case.yaml`: блок `case.settlement.*`, статус `pending_settlement → settled | withdrawn`, `timeline`

**Зависимости:** `build-position` (как источник аналитики позиции, желательна); `arbitrum-docx` (оформление, с feature detection + fallback на `.md`)

**Сценарии MVP (по `case.settlement.mode`):**
- `agreement` — мировое соглашение (ст. 138-142 АПК) на любой стадии + исполнительное производство.
- `withdrawal` — отказ от иска нашим истцом (ст. 49 ч. 2, ст. 150 ч. 1 п. 4 АПК).
- `admission` — признание иска нашим ответчиком (ст. 49 ч. 3 АПК; только в 1 инст.).

Зарезервированные значения `mode` (пост-MVP, по ст. 138.6 АПК) — `appeal_withdrawal`, `facts_agreement`, `circumstances_admission`, `tm_consent`. MVP-валидатор отказывает на них (см. `shared/case-schema.yaml` v4, блок `case.settlement`).

**Алгоритм (8 фаз):**

1. **Фаза 0** — Pre-flight + Phase 0 ФС↔индекс. Проверка совместимости `mode × stage` по матрице в case-schema (например, `admission × appeal` блокируется).
2. **Фаза 1** — Загрузка контекста. Помимо стандартного для аналитических — загрузка `build-position` секций (`strengths`, `weaknesses`, `risks`); глобальных профилей судьи (как принимает мировые) и оппонента (договороспособный / уходит от компромисса); расчёт возврата госпошлины по стадии (ст. 333.40 НК: 70% / 50% / 30%).
3. **Фаза 2** — Анализ зоны компромисса (Opus). Трёхсторонняя оценка: BATNA нашего клиента; zone of agreement; прогноз оппонента. Учёт последствий мирового — полное прекращение спора по этому правоотношению (п. 15 ППВАС № 50).
4. **Фаза 3** — Стресс-тест условий (Opus). 3 риска: юридический (чек-лист по пп. 13, 14, 16 ППВАС № 50 + ст. 140 АПК), исполнительный (астрент по Обзору ВС № 3 (2024) п. 32 + индексация по Обзору 18.12.2024 п. 4), тактический.
5. **Фаза 4** — Подготовка документов. Ветвление по `mode` и по `stage` (просительная часть ходатайства об утверждении различается: 1 инст. — прекращение производства; апелляция — отмена решения 1 инст. + прекращение; кассация — отмена актов нижестоящих + прекращение; исполнение — прекращение исполнительного производства). Через `arbitrum-docx` с fallback.
6. **Фаза 5** — Preview. Сводка зоны компромисса + риски + проекты документов + изменения case.yaml. Выбор Сюзереном формы подачи ходатайства об утверждении (одностороннее по умолчанию / два отдельных / совместное).
7. **Фаза 6** — Apply. Сводный `.md` + комплект `.docx` + машинно-читаемые секции + обновление `case.settlement.*` + `case.status`.
8. **Фаза 7** — Сводка для Сюзерена (что готово / следующий шаг / deadlines / постусловия Notion-хука).

**Хук в `prepare-hearing`:** в фазе 2 п.13 (Стратегическая оценка) `prepare-hearing` условно предлагает `/vassal-litigator:settlement` при (а) слабой позиции по анализу; (б) `case.settlement.in_progress = true`; (в) явном упоминании мирового Сюзереном. Не блокирующий, не дублирует логику.

**Подключение к downstream:** после утверждения судом — `case.settlement.confirmed_date` фиксирует факт. Если обжалуется определение по ст. 141 ч. 11 АПК — путь через расширенный `cassation` (`cassation.target = 'ruling_settlement'`, 1 месяц со дня вынесения, минуя апелляцию).

**Out of scope MVP** (зарезервировано в OPEN-ITEMS): мировое в банкротстве (гл. VIII ФЗ-127); процессуальные основания ст. 150 АПК (это не примирение); соглашение по обстоятельствам ст. 70 ч. 2 АПК; отказ от жалобы как самостоятельный режим.

## 9. Взаимосвязи скиллов

```
init-case
    │
    ▼
  intake ──────────► catalog
    │                   │
    ▼                   ▼
add-evidence        legal-review
add-opponent            │
    │                   ▼
    └──► update-index   build-position ────┐
              │              │             │
              ▼              ▼             ▼
         prepare-hearing ◄───┘         settlement
              │   │                    (мировое /
              │   │ (хук фаза 2 п.13)   отказ /
              │   └──────────────────►  признание)
              ▼                              │
       analyze-hearing                       │
              │                              │
              ▼                              ▼
    ┌─── draft-judgment            case.status =
    │                              settled | withdrawn
    ▼                              (или → cassation
  appeal                           по ст. 141 АПК)
    │
    ▼
  cassation
```

**Критический путь:** init-case → intake → catalog → legal-review → build-position → prepare-hearing

**Циклические:** add-evidence, add-opponent, update-index, prepare-hearing, analyze-hearing — вызываются многократно по ходу дела.

**Ответвления (примирительный трек):** `settlement` может быть запущен на любой стадии — из `prepare-hearing` (через хук в фазе 2 п.13), из `build-position` (когда анализ позиции даёт «слабо, лучше уступить»), напрямую (когда оппонент прислал проект мирового). Возможные стадии — `1inst / appeal / cassation / enforcement` (ст. 139 АПК). Поток замыкается на `case.status` (settled / withdrawn) или, при обжаловании определения об утверждении мирового, уходит в `cassation` с `target='ruling_settlement'`.

**Глобальная память (кросс-дельная)** -- ортогональный слой, общий для всех дел (см. раздел 15). Основные потоки данных через неё:
- `analyze-hearing`, `draft-judgment` → `$VASSAL_GLOBAL_DIR/judges/` (двойная запись профиля судьи).
- `add-opponent`, `analyze-hearing` → `$VASSAL_GLOBAL_DIR/counterparties/` (двойная запись профиля оппонента).
- `prepare-hearing`, `build-position`, `appeal`, `cassation`, `analyze-hearing`, `draft-judgment` ← читают глобальные профили **до** анализа как фон.

**Notion-слой (опционально)** -- если настроен `~/.vassal/notion-config.yaml`, скилл `notion-sync` (вызывается вручную через `/vassal-litigator:sync-notion` или предлагается хуками после `init-case` / `analyze-hearing` / `appeal` / `cassation`) пушит метаданные дела и глобальный профиль судьи в Notion-базы `Cases` и `Judges`. Односторонне, без обратного синка. См. раздел 15.7 и [skills/notion-sync/SKILL.md](skills/notion-sync/SKILL.md).

## 10. Интеграция с arbitrum-docx

**Зависимость, не встраивание.** Плагин `arbitrum-docx` — внешний скилл, который vassal-litigator использует для оформления процессуальных документов в .docx. Логика генерации документов не дублируется внутри vassal-litigator.

**Как это работает:** Claude — единый рантайм. Когда скилл vassal-litigator (например, `appeal`) формирует содержание документа, он указывает в инструкциях: «оформи результат через скилл arbitrum-docx». Claude читает оба SKILL.md и комбинирует: vassal-litigator даёт контекст дела (стороны, суд, содержание), arbitrum-docx — шаблон, стили, форматирование.

**Скиллы, использующие arbitrum-docx:**
- `appeal` — апелляционная жалоба (Тип 1: процессуальный с шапкой)
- `cassation` — кассационная жалоба (Тип 1)
- `prepare-hearing` — ходатайства, письменные пояснения, отзывы (Тип 1)
- `analyze-hearing` — отчёт по заседанию для клиента (Тип 2: аналитический без шапки), замечания на протокол (Тип 1)
- `settlement` — мировое соглашение (Тип 4: двусторонний документ для подписи обеими сторонами — без шапки в формате процессуальной жалобы; см. примечание ниже); ходатайство об утверждении мирового, заявления об отказе от иска и о признании иска (Тип 1: процессуальный с шапкой)

> **Тип 4: двусторонний документ.** Текст мирового соглашения по ст. 140 АПК — гражданско-правовая сделка с подписями обеих сторон. Это материально иной формат, чем процессуальные жалобы/ходатайства (нет «отправитель → адресат», есть «Сторона 1» / «Сторона 2», подписи без указания представителя одной стороны). Если в `arbitrum-docx` нет соответствующего шаблона — `settlement` уходит в fallback на `.md` с явным предупреждением (см. [shared/conventions.md](shared/conventions.md) → «Внешние зависимости и fallback»).

**Скилл, не использующий arbitrum-docx:**
- `draft-judgment` — проект судебного акта. Генерирует `.docx` напрямую, потому что формат судебного акта (вводная / описательная / мотивировочная / резолютивная части, состав суда вместо «отправителя», без подписи представителя) **не покрывается шаблонами arbitrum-docx**, заточенными под процессуальные документы сторон. Если в `arbitrum-docx` появится «Тип 3: судебный акт» — `draft-judgment` следует перевести на него для единообразия (см. SKILL.md скилла, раздел «Обоснование прямой генерации .docx»).

**Feature detection и fallback** для всех скиллов, зависящих от внешних плагинов / библиотек (включая `arbitrum-docx` и `python-docx` в `draft-judgment`) — единый паттерн в [shared/conventions.md](shared/conventions.md) → раздел «Внешние зависимости и fallback».

## 15. Кросс-дельная память (глобальное хранилище)

**Проблема, которую закрывает.** До этапа 6 ценные кросс-дельные наблюдения (профиль судьи, паттерны оппонента) хранились внутри одного дела:
- 5 дел у судьи Иванова → 5 пустых стартовых `judge-profile.md`, ни один не знает про остальных.
- Тот же оппонент в трёх делах → три отдельных экспресс-анализа в трёх `.vassal/`, четвёртое дело снова с нуля.

**Решение.** Глобальное хранилище `~/.vassal-global/` (с переопределением через `$VASSAL_GLOBAL_DIR`). Скиллы пишут в **два места одновременно** (локальный файл дела + глобальный файл), и читают глобальный профиль до анализа.

Полные правила -- в [shared/conventions.md](shared/conventions.md) → раздел «Глобальная память (кросс-дельная)».

### 15.1. Структура хранилища

```
$VASSAL_GLOBAL_DIR/                              # по умолчанию ~/.vassal-global/
├── judges/
│   └── {ФИО-slug}--{court-slug}.md             # один файл на пару (судья × суд)
├── counterparties/
│   ├── inn-{ИНН}.md                            # для юрлиц с ИНН
│   └── noinn-{slug-name}.md                    # fallback для физлиц / без ИНН
└── README.md                                    # опционально
```

`templates/` -- зарезервирована, вне scope этапа 6 (наполняется вручную в этапе 7).

### 15.2. Кто пишет, кто читает

| Файл | Пишет (двойная запись) | Читает (фон до анализа) |
|---|---|---|
| `judges/{judge}--{court}.md` | `analyze-hearing` (устные паттерны), `draft-judgment` (письменный стиль) | `analyze-hearing`, `draft-judgment`, `prepare-hearing`, `appeal`, `cassation`, `settlement` (как принимает мировые: формально / с пристрастием) |
| `counterparties/{key}.md` | `add-opponent` (письменные паттерны), `analyze-hearing` (устные паттерны) | `add-opponent`, `prepare-hearing`, `build-position`, `appeal`, `cassation`, `settlement` (договороспособный / уходит от компромисса) |

### 15.3. Двойная запись и обратные ссылки

- **Локальный файл** в `.vassal/` дела (`judge-profile.md`, `opponent-filing-*.md`, `opponent-hearing-*.md`) -- наблюдения только этого дела, привязка к конкретным `[doc-NNN]`.
- **Глобальный файл** -- агрегированные наблюдения по всем делам Сюзерена с этой парой/сущностью; каждое наблюдение помечено `(дело {номер}, {дата})`.
- Локальный связан с глобальным через `shared_profile:` во frontmatter -- путь относительно `$VASSAL_GLOBAL_DIR`.

### 15.4. Правило накопления (аппенд-only)

- Новые наблюдения **дописываются** в соответствующий раздел.
- Старые **никогда не удаляются и не редактируются**, даже если новое им противоречит -- противоречие добавляется как «уточнение» с новой датой.
- Счётчики (`hearings_total`, `written_decisions_total`, `filings_analyzed`, `hearings_analyzed`) инкрементируются ровно на число добавленных в запуске.
- Список `cases:` пополняется без дублей по номеру.
- `last_updated` и `last_case` обновляются.

### 15.5. Конфигурируемый путь

- По умолчанию: `~/.vassal-global/`.
- Переопределение через `$VASSAL_GLOBAL_DIR`.
- Сценарии переопределения:
  - Хранение в облачном диске (Яндекс.Диск / Dropbox / OneDrive) -- кросс-машинный синк между рабочей и поездочной машинами.
  - Изоляция глобальной памяти разных юристов команды на одной рабочей станции.

### 15.6. Конфиденциальность

Глобальная память -- **производная аналитика**, не копия документов. Допустимо: паттерны судьи, типовые формулировки оппонента, статистика, ФИО представителей, ИНН/ОГРН (публичные). Недопустимо: тексты документов клиентов, конкретные суммы по делам, чувствительные ПДн третьих лиц без необходимости.

### 15.7. Связь с Notion-слоем

Глобальная память -- **локальный** аналог баз `Judges` и `Counterparties` Notion-слоя. После настройки Notion (этап 6 п.23, см. ниже) данные синхронизируются в Notion для дашборда и более удобного UX, но локальная папка остаётся **источником правды** -- Notion получает push односторонне (см. раздел «Notion-слой» в conventions и [NOTION-INTEGRATION-PROPOSAL.md](NOTION-INTEGRATION-PROPOSAL.md)).
