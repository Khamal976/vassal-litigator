# Ревизия скилла `legal-review`

**Дата:** 2026-04-20
**Файлы:** [skills/legal-review/SKILL.md](../skills/legal-review/SKILL.md), [commands/legal-review.md](../commands/legal-review.md), [references/deadlines-reference.md](../skills/legal-review/references/deadlines-reference.md), [references/pretrial-reference.md](../skills/legal-review/references/pretrial-reference.md), [references/jurisdiction-reference.md](../skills/legal-review/references/jurisdiction-reference.md)
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** первичный комплексный правовой анализ — квалификация правоотношений, сроки (исковая давность + процессуальные), досудебный порядок, подсудность, полнота доказательств, вопросы клиенту.

**Позиция в workflow:** первый аналитический скилл фазы «Анализ». Запускается после `intake` (+`catalog` рекомендован). Результаты читают почти все последующие: `build-position` — основа позиции, `prepare-hearing` — проверка аргументов, `appeal`/`cassation` — повторная сверка сроков и подсудности.

**Парные / последующие скиллы:** `build-position` — стратегия (следующий шаг), `catalog` — рекомендован перед запуском.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Стороны, предмет, суд |
| `.vassal/index.yaml` | да | Реестр документов + summary |
| `.vassal/mirrors/*.md` | да (выборочно) | Зеркала ключевых документов — модель сама выбирает какие |
| `references/deadlines-reference.md` | да | Справочник по срокам |
| `references/pretrial-reference.md` | да | Справочник по досудебному порядку |
| `references/jurisdiction-reference.md` | да | Справочник по подсудности |
| Вводная Сюзерена | нет | Конкретные вопросы через `$ARGUMENTS` |

### 2.2. Выход

| Файл | Путь | Для кого |
|---|---|---|
| Сводный анализ | `{корень дела}/{ГГГГ-ММ-ДД} Предварительный анализ документов.md` | Сюзерен |
| Квалификация | `.vassal/analysis/qualification.md` | агент (build-position) |
| Сроки | `.vassal/analysis/deadlines.md` | агент |
| Досудебный порядок | `.vassal/analysis/pretrial.md` | агент |
| Подсудность | `.vassal/analysis/jurisdiction.md` | агент |
| Полнота доказательств | `.vassal/analysis/completeness.md` | агент |
| Вопросы клиенту | `.vassal/analysis/questions-to-client.md` | агент |
| Ответы на вопросы Сюзерена | `.vassal/analysis/answers.md` | агент (если были вопросы) |
| Обновление карточки дела | `.vassal/case.yaml` (поля: подсудность, досудебный порядок, квалификация) | агент |
| Запись в истории | `.vassal/history.md` | агент |

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/case.yaml` существует.
- В `index.yaml` не менее 1 документа.
- **Рекомендуется** `catalog` (summary в индексе) — но это не hard-требование.

### 2.4. Постусловия — ❌ нет

Должны быть:
- Все 7 файлов в `.vassal/analysis/` существуют (или явно помечены «пропущено» в сводке).
- Сводный файл в корне дела создан и соответствует сумме файлов из `analysis/`.
- `case.yaml` остался валидным YAML и содержит новые поля.
- Все выводы в аудируемом формате имеют заполненные поля (Вывод, Основание, Источники, Уверенность).

## 3. Качество промпт-инженерии

| Критерий | Оценка | Комментарий |
|---|---|---|
| Description (триггеры) | ✅ | Отличный — много позитивных + негативные («НЕ для позиции — build-position», «не для процдокумента — prepare-hearing») |
| Pre-conditions | ✅ | Есть |
| Post-conditions | ❌ | Нет |
| Аудируемый формат выводов | ⭐ | **Лучшая сторона скилла.** Обязательный формат (Вывод / Основание / Источники / Уверенность / Недостающие данные) — ценно для юриста, снижает риск галлюцинаций |
| Подключение справочников | ✅ | Три справочника по ключевым вопросам, внутри скилла — это хороший паттерн |
| Декомпозиция на блоки | ✅ | 7 блоков (a–g), каждый с отдельным файлом |
| Preview-формат | ✅ | Чёткий, компактный |
| Идемпотентность | ⚠️ | Описана только для `analysis/` (перезапись с diff), **ничего про корневой файл** и `case.yaml` |
| Валидация перед apply | ❌ | Нет |
| Error-handling | ❌ | Что делать, если один из 7 блоков не получился? Политики нет |
| Критерии выбора ключевых документов | ⚠️ | Список шаблонный (договор, претензия, иск, отзыв), но без правила для нестандартных документов |
| Лимиты контекста для Opus | ❌ | Не описано, сколько зеркал загружать. При деле с 200+ документами — риск переполнения |

## 4. Противоречия и баги

### 4.1. Поля для обновления `case.yaml` не существуют в схеме

[SKILL.md:197](../skills/legal-review/SKILL.md): «Обнови `case.yaml` (заполни поля: **подсудность**, **досудебный порядок**, **квалификация**).»

Сверка с [case-schema.yaml](../shared/case-schema.yaml):
- `pretrial` — ✅ есть в схеме
- «подсудность» — ❌ поля нет в схеме
- «квалификация» — ❌ поля нет в схеме

Либо SKILL.md ссылается на несуществующие поля (и при исполнении Claude импровизирует — создаёт произвольные имена), либо `case-schema.yaml` неполна. **Как минимум, нужно расширить схему:**

```yaml
qualification:
  relation_type: null        # поставка | подряд | услуги | аренда | заём | ...
  applicable_norms: []       # список статей (напр., ["ст. 506 ГК", "ФЗ о контрактной системе"])
  qualified_by: null         # agent | user
  qualified_at: null         # дата анализа

jurisdiction:
  competent_court: null      # арбитражный | СОЮ
  territorial_basis: null    # общая | договорная | исключительная | альтернативная
  correct: null              # true | false | null (при анализе уже поданного иска)
  prorogation_clause: null   # текст оговорки из договора или null
```

### 4.2. Неопределённое поведение для корневого файла при повторном запуске

[SKILL.md:226](../skills/legal-review/SKILL.md): «Повторный запуск перезаписывает файлы в `analysis/`. Перед перезаписью покажи diff».

Про **корневой файл** `{дата} Предварительный анализ документов.md` — ничего. Два возможных сценария:
- **Новый файл с новой датой** каждый запуск → в корне накапливаются 3-5 анализов за разные даты.
- **Перезапись старого файла** → теряется история.

Оба поведения допустимы, но разные. Без явного правила — определяется импровизацией модели, т.е. непредсказуемо.

**Предлагаемое правило:** новый файл с новой датой (хронологическое версионирование, как у позиций). В preview показывать: «Обнаружен предыдущий анализ от {дата}. Создать новый файл или обновить старый?».

### 4.3. Аудируемый формат: потеря машинной читаемости

[SKILL.md:55](../skills/legal-review/SKILL.md): «**Источники:** [названия документов с датами — НЕ doc-ID; человек читает названия, а не идентификаторы]».

Задумка правильная — юристу удобнее читать «Договор поставки №47 от 01.06.2025», чем «doc-042». Но:

1. **Последующие скиллы** (`build-position`, `appeal`) читают `analysis/deadlines.md` и хотят найти первоисточник в `index.yaml`. Если ссылка только по имени — им приходится реконструировать doc-id через fuzzy match (имя из индекса может слегка отличаться, содержать опечатки, иметь дубль с суффиксом `(2)`).
2. **При переименовании/удалении документа** (в будущем) ссылка в `deadlines.md` становится «висячей» — никто автоматически не проверит.

**Решение:** двойная ссылка в аудируемом формате:
```
**Источники:** Договор поставки №47 от 01.06.2025 [doc-042]
```
Для Сюзерена читается игнорируя `[doc-042]`, для агента — точная ссылка.

### 4.4. Конвенция имени корневого файла не соответствует общему правилу

[shared/conventions.md:24](../shared/conventions.md): «Формат: `ГГГГ-ММ-ДД Отправитель Описание.расширение`».

Файл legal-review: `{ГГГГ-ММ-ДД} Предварительный анализ документов.md` — **без «Отправителя»**.

Это внутренний продукт плагина, а не документ от стороны — тут «отправитель» не к месту. Но правило в conventions.md это не оговаривает. Нужно либо:
- Уточнить conventions.md: «правило касается документов от сторон (client/opponent/court). Продукты плагина именуются `{дата} {Тип продукта}.md`.»
- Либо добавить фиктивного отправителя (напр., «Представитель»): `2026-04-20 Представитель Предварительный анализ.md`.

Первый вариант честнее.

### 4.5. Поведение `answers.md` при повторном запуске с новыми вопросами

Сценарий: Сюзерен задал вопросы 1-2-3 → скилл ответил → сохранил в `analysis/answers.md`. Через месяц Сюзерен спросил новые 4-5-6 → запуск → по правилу идемпотентности **перезапись** → ответы 1-2-3 потеряны.

**Решение:** `answers.md` — аппенд-only. Каждая сессия вопросов — своя секция с датой. Никогда не перезаписываем.

### 4.6. Нет политики слияния `case.yaml` между скиллами

Поля `case.yaml` пишут: `init-case` (структурные), `legal-review` (аналитические), `build-position` (стратегические), `analyze-hearing` (хронология), `appeal`/`cassation` (статус). Нет документированной матрицы «кто какие поля имеет право трогать». Риск: `legal-review` случайно перезапишет поле, которое заполнено через `init-case` вручную.

**Решение:** в [shared/case-schema.yaml](../shared/case-schema.yaml) для каждого поля добавить комментарий `# writes: <список скиллов>`. Или отдельный файл `shared/case-schema-ownership.md`.

## 5. Непокрытые edge-cases

### 5.1. Критерий выбора «ключевых документов» размыт

[SKILL.md:37-42](../skills/legal-review/SKILL.md) перечисляет типовой список (договор, претензия, иск, отзыв, гарантийные письма, акты сверки). Но:

- Если акт сверки есть, но плохого качества (summary неинформативен) — загружать зеркало целиком?
- Если нет договора, но есть счета-оферты и акты (реально в делах о неосновательном обогащении) — что считать ключевыми?
- Если претензий несколько (первая, повторная) — все или последнюю?

Модель импровизирует. Для юриста это значит — **непредсказуемая глубина анализа** между разными делами.

### 5.2. Большое дело — переполнение контекста

Сценарий: дело на 300 документов (типично для строительных подрядов с актами КС-2/КС-3). Модель должна выбрать «ключевые» — но если под «ключевыми» понимаются все 30 договоров и допсоглашений, все акты сверки, вся переписка — это гигабайты зеркал для Opus. Скилл не ограничивает и не инструктирует.

**Решение:** ввести верхний лимит «не более N страниц зеркал за раз, приоритизируй по свежести и по критичности для аргумента». Плюс предупреждение в preview: «Загружено {X} MB контекста, {Y} документов не вошли — анализ по ним через summary только».

### 5.3. Устаревающие сроки

Сроки рассчитаны на дату анализа. Через 3 месяца `analysis/deadlines.md` может содержать уже недействительные оценки (срок истёк, процессуальный дедлайн пропущен). `build-position`, прочитав устаревший `deadlines.md`, построит ложную стратегию.

Ни SKILL.md, ни схема не содержат TTL или флага «устарело».

**Решение:** в frontmatter каждого файла `analysis/*.md` — поле `generated_at` и `expires_at`. При использовании другими скиллами — проверка; если срок прошёл — предупредить пользователя или триггернуть перезапуск.

### 5.4. Триаж для `completeness.md`

Блок (f) предполагает оценку каждого документа из `index.yaml`. При 200+ документах — 200 оценок. Большинство банальных (счёт, накладная — ок, читаемо, подписано). Реально важны только спорные (скан плохого качества, нет подписи, копия без печати).

SKILL.md не даёт критерий — оценивать все или триаж. В итоге — Opus прожигает бюджет на формальную оценку.

**Решение:** явно — оценивать только (а) документы с `needs_manual_review: true`, (b) документы с `quality: low`, (c) критичные доказательства (договор, первичные акты, платёжки по спорным суммам), (d) документы из бандла с иском.

### 5.5. Нестандартные споры

[SKILL.md:234](../skills/legal-review/SKILL.md): «Нестандартный спор (не из ГК) → примени ту же структуру анализа, адаптируй нормативную базу».

Однако справочники (`deadlines-reference.md`, `pretrial-reference.md`, `jurisdiction-reference.md`) покрывают только гражданско-правовые споры в арбитражных судах. **Не покрыто:**
- Банкротные обособленные споры (свои сроки, свои правила обжалования).
- Административные дела (КоАП, КАС).
- Споры в СОЮ (ГПК — другие сроки).
- Налоговые споры (НК РФ — свои досудебные механизмы).
- Трудовые споры (в частности, сроки — в deadlines-reference упомянуты кратко).

Для банкротства (которое явно указано в [shared/case-schema.yaml:category](../shared/case-schema.yaml:11) как одна из категорий) — **справочников нет вообще**. Модель будет импровизировать → риск ошибки в специализированной области.

### 5.6. Противоречия между документами

[SKILL.md:233](../skills/legal-review/SKILL.md): «Противоречия между документами → зафиксируй в `completeness.md` и в `questions-to-client.md`».

Но что за противоречия? Конкретные примеры:
- Разные суммы в разных документах (претензия на 1 млн, иск на 1.2 млн).
- Разные даты исполнения.
- Противоречивые формулировки условий.

SKILL.md не даёт чек-лист — что искать как противоречие. Модель найдёт очевидные, пропустит тонкие.

## 6. Качество справочников

### 6.1. `deadlines-reference.md` — ⭐ сильно

Покрывает общий срок, специальные сроки, приостановление/перерыв/восстановление, процессуальные сроки АПК, упрощённое производство, чек-лист. **Цитирует правильные статьи ГК и АПК**, ссылается на Постановление Пленума ВС № 43. Актуально для гражданско-правовых споров.

Пробелы:
- Нет упоминания сроков по налоговым спорам (НК).
- Нет сроков по банкротству (специфика сроков на оспаривание сделок, возражения на требования кредиторов).
- Сроки по ГПК / КАС не упомянуты (хотя case-schema допускает категории кроме «экономические споры»).

### 6.2. `pretrial-reference.md` — ⭐ сильно

Очень подробно. Чек-лист претензии практичный. Упомянуты договорный порядок, частые ошибки, влияние на исковую давность. **Точно и полезно.**

Пробел: нет главы про **досудебный порядок по КАС** (административные дела) и **обязательный претензионный порядок в трудовых спорах**.

### 6.3. `jurisdiction-reference.md` — хорошо

Компетенция арбитража, территориальная, договорная, исключительная — всё по существу. Третейская оговорка упомянута. Чек-лист практичный.

Пробелы:
- Нет **СОЮ-подсудности** (если спор уходит в общую юрисдикцию — какие правила).
- Не указаны **изменения 2019 года** (объединение СОЮ и арбитражных судов в некоторых вопросах подсудности).
- Нет главы про **международную подсудность** (если контрагент — иностранный).

## 7. Реальные проблемы на практике

### 7.1. Дубль «для Сюзерена» vs «для агента» требует синхронизации

Сводный файл в корне и 7 файлов в `.vassal/analysis/` — два представления одного. Если юрист правит корневой вручную (дописал комментарий к квалификации) — `analysis/qualification.md` отстаёт. `build-position` в следующий раз прочитает отставший — получит неактуальные данные.

**Решение:** либо корневой файл строится автоматически из `analysis/*.md` (не редактируется вручную, есть disclaimer), либо `analysis/*.md` автоматически пересинхронизируются с корневым при его изменении.

### 7.2. Юрист забудет пересчитать сроки

Срок — живая величина. Но в плагине он фиксируется на момент анализа. Нет автотриггера «прошло 60 дней с legal-review, сроки могли измениться».

### 7.3. Вопросы клиенту не доставлены

`questions-to-client.md` — собирает вопросы, но **как** они доставляются клиенту? В сводном файле они есть, но сводный — «Предварительный анализ документов», который читает только юрист. Нужен отдельный экспорт «Вопросы клиенту.docx» (для отправки), но его нет.

**Решение:** при apply предлагать: «Сохранить вопросы клиенту как отдельный .docx для отправки?».

### 7.4. Зависимость от `catalog` мягкая, но реальная

SKILL.md: «рекомендуется `catalog`». Если `catalog` не запущен — в `index.yaml` поле `summary` у документов будет пустое или краткое. Legal-review будет вынужден открывать зеркала целиком → больше токенов, медленнее, дороже. Но он не сказал об этом пользователю явно в preview.

**Решение:** в фазе 1 проверять: сколько документов в индексе имеют summary. Если <50% — явно предупреждать: «рекомендую сначала `catalog`, иначе анализ будет дольше и дороже на ~X%».

## 8. Рекомендации (приоритизированно)

### P0 (критично)

1. **Исправить схему `case.yaml`:** добавить поля `qualification` и `jurisdiction` согласно §4.1.
2. **Двойные ссылки в аудируемом формате** (§4.3): `Название документа [doc-NNN]`. Правит SKILL.md + shared/conventions.md.
3. **Постусловия + валидация** перед apply (§2.4).
4. **Политика повторного запуска:**
   - Корневой файл — новый с датой (§4.2).
   - `answers.md` — аппенд-only (§4.5).

### P1 (важно)

## 9. Резюме

Legal-review — **содержательно самый сильный** скилл плагина за счёт трёх качественных справочников и аудируемого формата выводов. Это редкий пример, где LLM-скилл сделан не просто как промпт, а как знаниевый продукт.

Проблемы:
- Технические: несогласованность со схемой `case.yaml`, нет постусловий, неявная политика повторного запуска.
- Архитектурные: дубль сводного файла и `analysis/*.md` без правила синхронизации, нет защиты от устаревания сроков.
- Покрытие: справочники охватывают стандартные ГК-споры в арбитраже, но банкротство, КАС, ГПК, трудовые — белые пятна. При том что схема дела явно допускает эти категории.

**Ключевая правка** для надёжности: двойные ссылки в audit-формате + TTL на сроки + расширение схемы `case.yaml`. Это три небольших изменения, которые закрывают большую часть системных рисков.
