# Ревизия скилла `add-evidence`

**Дата:** 2026-04-21
**Файлы:** [skills/add-evidence/SKILL.md](../skills/add-evidence/SKILL.md), [commands/add-evidence.md](../commands/add-evidence.md)
**Связанные:** [skills/intake/SKILL.md](../skills/intake/SKILL.md), [skills/add-opponent/SKILL.md](../skills/add-opponent/SKILL.md), [reviews/intake.md](intake.md)
**Версия плагина:** 0.4.0

## 1. Назначение и роль в workflow

**Декларация:** приём дополнительных доказательств от клиента **по уже инициализированному делу** — когда intake уже отработал, в `index.yaml` есть записи, и клиент присылает новые файлы по ходу процесса.

**Позиция в workflow:** циклический скилл фазы «Ведение дела», аналог `intake` во времени жизни дела. Запускается многократно по мере поступления новых материалов. Результат читают почти все последующие скиллы через shared state (`index.yaml`, `mirrors/`): `legal-review` (пересчёт полноты доказательств), `build-position` (обновление позиции), `prepare-hearing` (включение в приложения к процдокументам).

**Парные скиллы:**
- `intake` — **первичный** приём (алгоритмически почти идентичен — см. §4.1).
- `add-opponent` — приём от оппонента (отличается папкой назначения, source-пометкой и наличием фазы экспресс-анализа).

**Критическая особенность позиции:** add-evidence — **единственный** ко-пилотский вход для новых доказательств клиента после intake. Если он работает плохо — либо часть новых документов не попадёт в индекс (и станет невидимой для анализа), либо будет дублировать уже имеющиеся (засоряя дело).

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Контекст дела, стороны |
| `.vassal/index.yaml` | да | Уже известные документы + `next_id` + дедуп |
| `Входящие документы/**` | да | Источник файлов |
| `shared/conventions.md` | неявно | Таксономия, правила именования, подпапки |
| `shared/index-schema.yaml` | неявно | Структура записи |
| `shared/mirror-template.md` | неявно | Формат зеркал |

### 2.2. Выход

| Результат | Путь |
|---|---|
| Оригиналы | `.vassal/raw/evidence-{ГГГГ-ММ-ДД}/` |
| Переименованные копии | `Материалы от клиента/{подпапка}/` или процессуальная папка |
| Md-зеркала | `.vassal/mirrors/doc-NNN.md` |
| Обновлённый индекс | `.vassal/index.yaml` (новые записи с `source: client`) |
| Запись в истории | `.vassal/history.md` |
| Очистка | «Входящие документы/» → «На удаление/» |

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/case.yaml` существует.
- `.vassal/index.yaml` существует **и непуст** (есть записи от intake).
- `Входящие документы/` существует.

### 2.4. Постусловия — ❌ нет в SKILL.md

Как и у intake, не формализованы. Должны быть:
- Каждый файл из «Входящих» либо обработан (попал в индекс + зеркало + на удаление), либо явно оставлен с причиной.
- Новые записи в `index.yaml` ссылаются на физически существующие файлы и зеркала.
- `next_id` инкрементирован ровно на количество добавленных документов.
- `index.yaml` остался валидным YAML, `origin.intake_batch` = `evidence-{дата}`.
- Для дубликатов (совпадение с уже имеющимся доком) — явное действие (пропуск / создание / пометка связи).

## 4. Противоречия и баги

### 4.1. **Главный архитектурный дефект:** add-evidence = обрезанный intake, но не упрощённый

SKILL.md позиционирует add-evidence как **частный случай** intake: «Отличие от intake: дело уже содержит материалы, индекс заполнен, контекст дела известен». Но в реализации скилл **пропускает фазы intake**, которые нужны и для дополнительных доказательств:

| Шаг intake | Есть в add-evidence? | Должен быть? |
|---|---|---|
| Чтение `case.yaml` | ✅ | да |
| Проверка «Входящие» пуста | ✅ | да |
| Копия в `.vassal/raw/` | ✅ | да |
| Фиксация `origin_name`, `received`, `batch` | ✅ | да |
| **Рекурсивный обзор (вложенные папки)** | ❌ | **да** — клиент часто присылает «папку со сканами» |
| **Распаковка архивов** (zip/rar/7z/tar.gz) | ❌ | **да** — типичный сценарий: пришёл `accerts-april.zip` |
| OCR pipeline | ✅ | да |
| **Анализ имён файлов (три категории)** | ❌ | **да** — имена-«мусор» («IMG_2031.pdf») неизбежны |
| Категоризация + таксономия | ✅ | да |
| Делегирование Haiku | ⚠️ (один пункт в фазе 3 «делегируй Haiku-субагентам», без инструкций) | да |
| Preview | ✅ | да |
| Apply: создание папок, копирование, зеркала | ✅ | да |
| Обновление индекса | ✅ | да |
| Обновление `history.md` | ✅ | да |
| Очистка «Входящих» → «На удаление» | ✅ | да |

**Итог:** 3 шага intake **молча утеряны** в add-evidence. При этом их подключение не меньше, чем в первичном приёме: при первичном клиент тоже может прислать архив, и там тоже нужен анализ имён.

**Правка:** либо перенести недостающие шаги в SKILL.md add-evidence явно, либо сделать SKILL.md add-evidence тонкой обёрткой над intake («выполни intake pipeline с префиксом `evidence-{дата}` и `source: client`, плюс дополнительно: шаг N — проверь связь с существующими документами»).

### 4.2. **Неоднозначный триггер: intake vs add-evidence**

В description обоих скиллов — отсылки друг к другу («хотя алгоритм похож»). Но **как модель различает**, какой запускать?

Сценарии:
- Юрист говорит: «Разбери эти файлы от клиента». `index.yaml` пуст. → intake. ✅
- Юрист говорит: «Разбери эти файлы от клиента». `index.yaml` уже содержит 20 документов. → должен быть add-evidence, но Claude может запустить intake по формулировке. **Неопределённость.**
- Юрист говорит: «Приобщи к делу» (явный маркер дополнения). → add-evidence. ✅
- Юрист не говорит ничего — просто `/vassal-litigator:intake` при непустом индексе. → intake стартует **в условиях, когда уже есть записи**. Что произойдёт?

Последний сценарий — реальная проблема. SKILL.md intake в разделе «Идемпотентность» говорит: «Если intake уже запускался — проверь по `origin.name + origin.intake_batch`». То есть intake **не падает**, а продолжает работу с дедупом. Но:
- `intake_batch` будет `intake-{дата}`, а не `evidence-{дата}` → раскладка по папкам, но source тот же (client) → **функционально равно add-evidence**, но с отметкой «intake-batch» в индексе.
- Все последующие фильтры по `intake_batch` (если они появятся) начнут путать первичные и последующие пакеты.

**Правка:**
1. В commands/intake.md и commands/add-evidence.md сделать **hard-check**:
   - intake: если `index.yaml` содержит документы с `source: client` → спросить: «Дело уже содержит N документов клиента. Запустить add-evidence вместо intake?» и переадресовать.
   - add-evidence: если `index.yaml` пуст → спросить: «Индекс пуст. Запустить intake?».
2. В SKILL.md обоих — явная инструкция «проверь наличие записей до принятия решения».

### 4.3. **Правило папки назначения размыто, противоречит add-opponent**

SKILL.md add-evidence фаза 2 п.7:
- «Процессуальный документ для приобщения → `{ГГГГ-ММ-ДД} {Сторона} - {действие}/`»
- «Дополнительное доказательство к существующей категории → `Материалы от клиента/{подпапка}/`»
- «Назначение неясно → `Материалы от клиента/Иное/`»

Вопросы:
1. «Процессуальный документ для приобщения» от **клиента** — это что? Клиент не подаёт в суд напрямую — подаёт его юрист. Если клиент прислал файл «Заявление о …» — это материал для юриста, который юрист **может** положить в процессуальную папку при подготовке. То есть — это решение юриста, а не клиента.
2. [skills/add-opponent/SKILL.md:39-43](../skills/add-opponent/SKILL.md) — документы оппонента идут в процессуальную папку `{ГГГГ-ММ-ДД} {Сторона оппонента} - {тип документа}/` (например `2026-04-10 Ответчик - отзыв на иск/`). То есть **процессуальная папка от add-opponent = документы оппонента**. Если add-evidence тоже кладёт в процессуальную папку — в ней смешаются документы клиента и оппонента. Непредусмотренная коллизия.
3. В [reviews/prepare-hearing.md §5.4](prepare-hearing.md) уже зафиксировано расхождение между SKILL.md и ARCHITECTURE.md про «корень vs процессуальная папка». add-evidence добавляет третью точку зрения на то же место.

**Правка:** убрать из add-evidence раскладку в процессуальные папки. Всё от клиента → `Материалы от клиента/*`. Процессуальные папки формирует `prepare-hearing` (или операции приобщения), а не входной пайплайн.

### 4.4. Дубликат содержимого с уже имеющимся документом — нет сценария

SKILL.md фаза 3 п.11: «Скопируй файлы с новыми именами в целевые папки».
Граничные случаи: «Документ дублирует уже имеющийся (по содержимому) → сообщи в preview, спроси Сюзерена.»

Но **как** определять дубликат? В index.yaml нет поля `content_hash` (это уже отмечено в [reviews/intake.md §4.5](intake.md) как P1-рекомендация). Без хэша — сравнивать по:
- Имени? Не сработает (новое имя после intake часто отличается от оригинала).
- Типу + датe + сторонам? Ложные совпадения на типовых документах.
- Первым N символам текста? Дорого и неточно.

В результате SKILL.md декларирует фичу, которую сам же не реализует. На практике дубликаты пройдут как новые документы, `index.yaml` распухнет, аналитика станет путаться.

**Правка:** либо ввести `content_hash: sha256:...` в схему и считать при приёме, либо в SKILL.md честно указать: «Дедуп по содержимому не выполняется, положитесь на ручную проверку Сюзереном в preview. Показывай в preview ФИО + дату + тип — если совпадают с существующим, подсвети».

### 4.5. Унаследованная хрупкая идемпотентность

**Унаследовано из intake ([reviews/intake.md §4.5](intake.md)):** add-evidence использует ту же проверку «по `origin.name + origin.intake_batch`». В add-evidence — даже **хуже**, потому что:
- `intake_batch` теперь `evidence-{дата}`. При **двух запусках в один день** (классика для add-evidence: клиент прислал в 10:00, потом ещё что-то в 14:00) — батч один и тот же `evidence-2026-04-05`. Дедуп по имени+батчу **сработает корректно только если** файлы точно с теми же именами, что уже обработаны.
- При повторной присылке того же файла с другим именем (частый сценарий: «переотправил, предыдущее не открылось») — дедуп не поймает. Новый документ, новый `doc-id`, дубль в индексе.

**Правка:** как в [reviews/intake.md](intake.md) P1 — дедуп по `content_hash`.

### 4.6. Идемпотентность именования `evidence-{дата}`

SKILL.md использует `evidence-{ГГГГ-ММ-ДД}` как имя папки в `.vassal/raw/` и значение `origin.intake_batch`. Но:
- Дата без времени → два пакета в один день попадают в одну папку → коллизия имён файлов внутри батча.
- Если первый пакет (утро) был обработан и файлы из `Входящие` очищены, а второй (вечер) пришёл — `cp` во второй батч в ту же папку `evidence-2026-04-05/` **перезапишет** одноимённые оригиналы.

У intake такая же проблема, но реальный риск меньше (intake один раз в начале дела). Для add-evidence, который запускается часто, — актуально.

**Правка:** для add-evidence использовать `evidence-{ГГГГ-ММ-ДД}-{ЧЧММ}` при повторе, или глобальный счётчик `evidence-{дата}-{N}`.

### 4.7. Команда и скилл — разная степень подробности на init-checks

[commands/add-evidence.md:11](../commands/add-evidence.md): «Убедись, что `.vassal/case.yaml` и `.vassal/index.yaml` существуют. Если нет — предложи сначала `/vassal-litigator:init-case` и `/vassal-litigator:intake`.»

[SKILL.md:16-18](../skills/add-evidence/SKILL.md): те же три проверки (`case.yaml`, записи в `index.yaml`, папка `Входящие документы/`).

Частично дублируется. Не страшно, но: команда предлагает при пустом `index.yaml` запустить **intake**. SKILL.md не формализует, что делать при **намеренно** пустом индексе (клиент удалил все документы, но папки на месте). Минорный edge-case, но показывает рассинхронизацию уровней.

### 4.8. Нет инструкции о перезапуске зависимых скиллов после приёма

Новые доказательства по ходу дела могут:
- Изменить оценку полноты доказательств → требует перезапуска `legal-review` (блок `completeness.md`).
- Изменить позицию → требует перезапуска `build-position` (если аргумент меняется принципиально).
- Должны попасть в приложения следующего процдокумента → `prepare-hearing` должен учесть.

SKILL.md add-evidence **ничего** не говорит о пост-интеграции. Юрист должен сам помнить, что надо сделать после. На практике — не будет помнить, и новое доказательство попадёт только в индекс, но не в анализ.

**Правка:** в финальном сообщении add-evidence выводить рекомендации по триггерам:
```
Приём завершён. Добавлено N документов.
Возможные следующие шаги:
- Пересчитать полноту доказательств: /vassal-litigator:legal-review (в части completeness)
- Обновить позицию: /vassal-litigator:build-position
- Учесть в следующем процдокументе: /vassal-litigator:prepare-hearing (приложения)
```

Это явно согласуется с рекомендациями [WORKFLOW-REVIEW.md §5](../WORKFLOW-REVIEW.md) про формализацию soft-зависимостей.

### 4.9. Унаследованные противоречия из intake

Ниже — пометка «унаследовано» по проблемам [reviews/intake.md](intake.md), актуальным и для add-evidence:

| Из intake | Актуально? | Комментарий |
|---|---|---|
| §4.1 «custom path в команде vs «Входящие» в скилле» | **Нет** | commands/add-evidence.md не поддерживает custom path, согласовано с SKILL.md |
| §4.2 «мёртвая папка `.vassal/positions/`» | Нет | не касается add-evidence напрямую |
| §4.3 «неоднозначность отправителя для двустронних» | **Да, унаследовано** | add-evidence имеет ту же таксономию и тот же формат имени; дополнительная специфика — см. §5.4 ниже |
| §4.4 «OCR-fallback декларативен» | **Да, унаследовано** | даже усугублено: фаза 1 add-evidence упоминает OCR одним пунктом без деталей |
| §4.5 «идемпотентность по имени+пакету — хрупкая» | **Да, унаследовано и усилено** | см. §4.5 выше — два запуска в день усугубляют |

## 5. Непокрытые edge-cases

### 5.1. Уточнение / исправление ранее принятого документа

Реальный сценарий: клиент прислал акт сверки, его обработали, затем клиент прислал **исправленный** акт сверки с тем же номером — «предыдущий был с опечаткой в сумме». Что делать?

- Создать новую запись? → в индексе два похожих документа, в анализе путаница.
- Перезаписать старую? → потеряем оригинал с опечаткой, хотя его могут попросить суд.
- Связать как версии? → в [index-schema.yaml](../shared/index-schema.yaml) есть `parent_id`, но он задуман для приложений, не для версий.

SKILL.md молчит. В реальной практике юрист будет вынужден решать вручную каждый раз.

**Правка:** ввести поле `supersedes: doc-NNN` (как у позиций) или `version: 2, previous: doc-NNN`. В SKILL.md add-evidence — явный блок «Если документ заменяет предыдущий версию».

### 5.2. Документ из ранее пропущенных (забытый документ)

Клиент прислал в intake пакет из 10 документов, один потерялся. Спустя неделю вспомнил и дослал. С точки зрения дела это **первичный** документ (к тому же договорному событию), но попадает через add-evidence. Это нормально физически, но:
- В preview нужно бы отразить «это, вероятно, дополняет пакет первичных материалов от intake-{дата}».
- Связать с существующим бандлом договора?

Нет инструкций.

### 5.3. Документ для замены процессуальной позиции

Клиент прислал письмо с указанием: «Использовать в качестве приложения к кассационной жалобе». Это метаинформация, которую нельзя извлечь из содержимого — только из сопроводительного сообщения клиента.

SKILL.md **не фиксирует назначение/пометку документа** (`intended_for: cassation`). Модель может эту инструкцию проигнорировать, потому что она не прописана как поле. Когда начнётся работа с кассацией — юрист должен сам помнить про письмо.

**Правка:** в preview спрашивать: «Есть ли у этих документов специальное назначение (к какой-то жалобе, заседанию, для клиента для подписи)?». Поле в индексе `intended_for: null | {описание}`.

### 5.4. Отправитель «клиент» vs контрагент в двустороннем документе

Специфика add-evidence: клиент может прислать документ, исходящий **не от него** (например, ответ контрагента на претензию). Для intake это тоже актуально (унаследовано из [reviews/intake.md §4.3](intake.md)), но для add-evidence особенно:

- Файл: `otvet-otvetchika.pdf` (прислал клиент, но документ от ответчика).
- Имя по конвенции: `2026-03-20 Ответчик Ответ на претензию.pdf` — но source = `client` (мы получили от клиента, не от оппонента).
- Юрист может позже искать «что у нас есть от ответчика?» — `source: opponent` не покажет этот документ. «Что у нас есть от клиента?» — покажет, но это путает (физически документ от ответчика).

Это **структурный пробел схемы**: `source` отражает канал получения, а не авторство документа. Для полноты нужно два поля:
- `received_from: client | opponent | court | public`
- `authored_by: {сторона}`

**Правка:** расширить [index-schema.yaml](../shared/index-schema.yaml) или в SKILL.md зафиксировать: «`source: client` означает канал получения, не авторство документа. Авторство кодируется в `parties`».

### 5.5. Синхронность `update-index` и add-evidence

[WORKFLOW-REVIEW.md §2-3](../WORKFLOW-REVIEW.md) поднимает проблему: между запусками модифицирующих скиллов индекс может устаревать (ручное попадание файлов мимо пайплайна). Для add-evidence это **усугубляется**:
- Клиент прислал 5 документов в «Входящие документы/».
- Одновременно юрист **вручную** скопировал 2 других документа в «Материалы от клиента/Договоры/» (по привычке или потому что не дочитал инструкцию).
- add-evidence видит только 5 из «Входящих», 2 остальных не попадут в индекс.

SKILL.md не инструктирует запускать `update-index` до/после приёма. Риск рассинхронизации высокий.

**Правка:** перед фазой 1 — быстрая проверка: «Обнаружены N файлов в `Материалы от клиента/`, которых нет в `index.yaml`. Включить их в обработку? (да/нет/запустить update-index)».

### 5.6. Очень частые пакеты (1 файл за приход)

В отличие от intake (большой первичный пакет), add-evidence может запускаться за 1 файл. Это меняет cost/benefit:
- OCR-pipeline запуск, копирование в `raw/`, фаза preview — всё это overhead.
- Юрист ожидает, что для одного документа процесс будет быстрым.

SKILL.md не предусматривает «лёгкий режим» — все шаги выполняются одинаково для 1 и 100 файлов. Дорого для маленьких приращений.

**Правка:** для пакетов ≤ 3 файлов — сокращённый preview без таблиц, inline подтверждение.

### 5.7. Что если `Входящие` содержит файлы, предназначенные для add-opponent

Сценарий: в «Входящие документы/» лежат 5 файлов — 3 от клиента, 2 от оппонента (клиент переслал копии). Юрист запустил add-evidence. Все 5 попадают как `source: client`, хотя два из них нужно было обработать через add-opponent.

SKILL.md add-evidence **не делает анализа авторства**. На уровне description сказано «НЕ используй для оппонента», но если модель в середине обработки увидит документ, авторство которого не клиент — что делать?

**Правка:** в фазе 2 (категоризация) проверять авторство по содержимому:
- Если среди обрабатываемых есть документ с отправителем = оппонент → в preview явно отметить: «Обнаружен документ от ответчика среди присланных клиентом. Обработать отдельно через /add-opponent?».

### 5.8. «Недостающие данные» после приёма

Клиент прислал договор без последней страницы (с подписями). add-evidence это обработает как обычный документ, но юрист не получит сигнала «этот документ неполный».

В [shared/index-schema.yaml](../shared/index-schema.yaml) есть `completeness: full | partial | fragment`. Но SKILL.md add-evidence не инструктирует проверять полноту (нет чек-листа: есть ли подписи, печати, все ли страницы).

**Правка:** в фазе 2 (категоризация) — обязательная оценка `completeness` и `signature`/`seal` (уже есть в схеме). При `completeness: partial` — флаг в preview: «Документ неполный. Запросить у клиента недостающее?».

### 5.9. Унаследованные edge-cases из intake

Все 11 сценариев из [reviews/intake.md §5](intake.md) применимы и здесь:
- PDF с паролем
- Архив с паролем (**особенно актуально** при добавлении — клиенты часто шлют защищённые архивы)
- Дубликаты по содержимому (см. §4.4)
- Один PDF — несколько документов
- Тип документа вне таксономии
- Документ без номера и без даты
- Parallel/duplicate запуск (две сессии)
- Partial failure в фазе 3
- Кириллица в архивах
- Триаж `needs_manual_review`

И при этом в SKILL.md add-evidence граничные случаи свёрнуты одной строкой: «Те же, что в intake». Это **недостаточно** — модель, читающая SKILL.md add-evidence, может не прочесть intake SKILL.md в той же сессии. Правильнее дублировать перечень явно или хотя бы сослаться на конкретный раздел.

## 6. Реальные проблемы на практике

### 6.1. Главная практическая проблема: «тонкая» SKILL.md провоцирует упрощение

SKILL.md add-evidence короче intake в ~3 раза. Для читающего промпта это сигнал «скилл проще». Модель при запуске add-evidence — даже если сама пришла бы к шагам архивов, анализа имён, распаковки — **с меньшей вероятностью их выполнит**, потому что инструкция их не упоминает.

Это особенно плохо, потому что реальные пакеты клиента могут быть такими же сложными, как первичные (архив на 50 файлов с корявыми именами). Add-evidence на тонкой SKILL.md обработает их хуже intake.

### 6.2. Дубли между intake и add-evidence в одном деле

Если юрист по ошибке запустил **intake дважды** (вместо add-evidence второй раз) — либо:
- Intake корректно отфильтрует уже обработанные по `origin.name + intake_batch` (не сработает — разные батчи → дубли) или
- Intake перезапустит обработку всех файлов в «Входящие» (но там только новые — не страшно).

А если запустил **add-evidence, а должен был intake** (на первом заходе) — add-evidence упадёт на предусловии «index.yaml должен быть непустым» → юрист получит ошибку, запустит intake. Тут поведение ок.

Асимметрия: ошибка «запустил intake вместо add-evidence» не ловится, ошибка «запустил add-evidence вместо intake» ловится. Лучше бы ловились обе (см. §4.2).

### 6.3. Статус приобщения не меняется

В [index-schema.yaml](../shared/index-schema.yaml) есть поле `filing_status: null | draft | submitted | filed`. Когда клиент прислал доказательство «для приобщения к следующему заседанию» — add-evidence НЕ меняет это поле, оставляет `null`. Потом `prepare-hearing` формирует приложения — но как он узнаёт, какие документы новые и нужны в приложениях?

В реальности юристу приходится вручную помечать. SKILL.md add-evidence не предлагает диалог: «Вы планируете приобщать эти документы к следующему заседанию? Пометить `filing_status: draft`?».

### 6.4. Приглушённые «Связи с существующими документами» в preview

SKILL.md фаза 2 п.8 preview-пример содержит:
```
Связь с существующими документами:
  - Выписка может подтвердить/опровергнуть факт оплаты по Спецификации №2
```
Это **отличная фича** (отличие от intake в preview-формате — плюс add-evidence). Но:
- В алгоритме нет инструкции, **как** модель должна вычислять связь. Через embedding? Через LLM-сравнение summary? Через ручной просмотр?
- На больших делах (200+ документов) сравнивать каждый новый со всеми — дорого.
- Нет поля в схеме для сохранения связей (`related_to: [doc-NNN]`).

В результате в preview будут либо тривиальные связи («та же сторона»), либо придуманные моделью, либо никаких. Непредсказуемое качество.

**Правка:** формализовать либо правило «связь = совпадение 2+ параметров (сторона, тип, период, номер договора)», либо отменить эту фичу.

### 6.5. Нет сигнала о том, что индекс вырос существенно

После 10 запусков add-evidence в деле с первичными 20 документами — 120 документов. Все зависимые скиллы (`legal-review`, `build-position`, `prepare-hearing`) начнут задыхаться от контекста. SKILL.md add-evidence не сигнализирует: «в деле уже N документов, рекомендую ревизию через catalog/update-index».

## 7. Рекомендации (приоритизированно)

### P0 (критично)

1. **Согласовать алгоритм с intake.** Перенести из intake явно: рекурсивный обзор, распаковку архивов, анализ имён файлов. Либо сделать SKILL.md add-evidence тонкой обёрткой, вызывающей intake pipeline с другими параметрами (§4.1).
2. **Hard-check триггера:** в SKILL.md и командах intake / add-evidence — явная проверка «запущен ли правильный скилл?» по состоянию `index.yaml` (§4.2).
3. **Убрать раскладку в процессуальные папки** из add-evidence. Всё от клиента → `Материалы от клиента/`. Процессуальные папки — удел `prepare-hearing` и `add-opponent` (§4.3).
4. **Постусловия + валидация** (§2.4). Унаследовано из intake P0.
5. **Разрешить `evidence-{дата}`-коллизию** для нескольких запусков в один день (§4.6).

### P1 (важно)

6. **Дедуп по `content_hash: sha256:...`** в схеме индекса + реализация в add-evidence (§4.4, §4.5). Унаследовано из intake P1.
7. **Рекомендации о перезапуске зависимых скиллов** в финальном сообщении (§4.8).
8. **Раскрыть граничные случаи явно** (не ссылкой «как в intake») — особенно архив с паролем, дубликаты, partial failure (§5.9).
9. **Правило «отправитель vs автор» в двусторонних документах** — разграничить `source` (канал) и `parties` (авторство) в SKILL.md (§5.4). Можно вместе с разделением схемы.
10. **Проверка авторства** в фазе 2: если среди файлов обнаружен документ от оппонента → предложить переадресацию на add-opponent (§5.7).
11. **Проверка `completeness` + `signature`/`seal`** обязательна для каждого документа в фазе 2 (§5.8).
12. **Интеграция с `update-index`:** перед фазой 1 — быстрая проверка расхождения ФС и индекса (§5.5).

### P2 (UX)

13. **Диалог о `filing_status` и `intended_for`** в preview: «Эти документы — для приобщения к заседанию? Для подготовки к кассации?» (§5.3, §6.3).
14. **«Лёгкий» режим для маленьких пакетов** (≤3 файлов) — минимальный preview (§5.6).
15. **Пометка версий документа** (`supersedes` / `version` в схеме) для сценария замены (§5.1).
16. **Формализация «связь с существующими документами»** в preview (§6.4).
17. **Progress reporting** для больших пакетов — унаследовано из intake P2.

### P3 (архитектура)

18. **Разделение схемы:** `received_from` + `authored_by` (§5.4). Потребует миграции существующих данных.
19. **Автотриггер ревизии дела** при превышении порога (например, 100 документов) — предложить `catalog` / триаж (§6.5).
20. **Единый `Ingest`-пайплайн** для intake / add-evidence / add-opponent как общий базовый скилл — устранит дублирование инструкций.

## 8. Резюме

Add-evidence — **алгоритмический клон intake с урезанными шагами**. На уровне декларации это «продолжение intake для дополнительных доказательств», но на уровне реализации — ближе к «сокращённому intake», у которого явно пропущены важные шаги (распаковка архивов, рекурсивный обзор, анализ имён файлов, делегирование Haiku с промптами).

**Главная проблема — архитектурная:** add-evidence создан как «лёгкая версия» intake, но реальные пакеты от клиента в ходе процесса могут быть такими же сложными, как первичные. Краткость SKILL.md → модель упускает шаги → качество приёма падает.

**Существенные отличия от intake, которые нужно усилить:**
- Преимущество preview-раздела «Связь с существующими документами» — хорошая идея, но не формализована в алгоритме.
- Добавлено предусловие «index.yaml непуст».
- Сокращены фазы (5 → 3) — но это не упрощение, а **недосказанность**.

**Не разрешённые по сравнению с intake:**
- Неоднозначность триггера intake/add-evidence (две стороны одной монеты — ни одна не ловит ошибочный запуск).
- Унаследованы все идемпотентности-баги intake, плюс добавлены свои (одинаковый `evidence-{дата}` при двух запусках в день).
- Унаследована хрупкая OCR-fallback-инструкция.
- Унаследовано отсутствие постусловий.

**Добавлены свои проблемы:**
- Противоречивая раскладка в процессуальные папки (конфликт с add-opponent и конвенциями).
- Нет интеграции с зависимыми скиллами после приёма — новые доказательства не триггерят переоценку в legal-review / build-position.
- Декларирован «дедуп дубликатов по содержимому» — без механизма реализации.

**Две правки закрывают большую часть проблем:**
1. Явно согласовать алгоритм с intake (либо дублировать шаги, либо обернуть intake pipeline) → сразу закрываются §4.1, §4.2, §5.9.
2. Ввести `content_hash` + правило «source ≠ authored_by» в схему → закрывается §4.4, §4.5, §5.4.

Remaining — поступательные улучшения, большинство унаследовано из [reviews/intake.md](intake.md).
