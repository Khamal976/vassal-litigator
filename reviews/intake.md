# Ревизия скилла `intake`

**Дата:** 2026-04-20
**Файлы:** [skills/intake/SKILL.md](../skills/intake/SKILL.md), [commands/intake.md](../commands/intake.md), [scripts/extract_text.py](../scripts/extract_text.py), [shared/mirror-template.md](../shared/mirror-template.md)
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** приём файлов клиента из «Входящие документы/», извлечение текста, переименование по конвенции, раскладка по тематическим папкам, создание md-зеркал, обновление `.vassal/index.yaml`.

**Позиция в workflow:** первый скилл после `init-case`. На нём держится всё остальное — любой аналитический скилл (`legal-review`, `build-position`) работает через зеркала и индекс, которые создаёт `intake`. Если он не запущен или отработал плохо — каскадно страдает весь дальнейший анализ.

**Парные скиллы:** `add-evidence` (тот же алгоритм для последующих поступлений от клиента), `add-opponent` (от оппонента).

## 2. Контракт скилла

### 2.1. Вход (что читает)

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Контекст: стороны, предмет спора |
| `Входящие документы/**` | да | Источник файлов (рекурсивно) |
| `.vassal/index.yaml` | да | `next_id`, проверка идемпотентности |
| `shared/conventions.md` | неявно | Таксономия типов, именование, подпапки |
| `shared/mirror-template.md` | неявно | Формат зеркал |
| `shared/index-schema.yaml` | неявно | Структура записи в индексе |

### 2.2. Выход (что пишет)

| Результат | Путь |
|---|---|
| Оригиналы (бит-в-бит) | `.vassal/raw/intake-{ГГГГ-ММ-ДД}/` |
| Переименованные копии | `Материалы от клиента/{Договоры,Переписка,...}/` |
| Md-зеркала | `.vassal/mirrors/doc-NNN.md` |
| Обновлённый индекс | `.vassal/index.yaml` (новые записи + обновлённый `next_id`) |
| Запись в истории | `.vassal/history.md` |
| Очистка | «Входящие документы/» → «На удаление/» |

### 2.3. Предусловия (pre-conditions) — есть в SKILL.md

- `.vassal/case.yaml` существует.
- `Входящие документы/` существует (создать, если нет).
- OCR-зависимости установлены (`scripts/setup.sh`).

### 2.4. Постусловия (post-conditions) — ❌ **нет в SKILL.md**

Должны быть, но не формализованы:
- Каждый файл из «Входящих» либо обработан и внесён в индекс, либо явно оставлен там с пометкой причины.
- `.vassal/index.yaml` — валидный YAML.
- Для каждой новой записи в индексе существует md-зеркало и физический файл по указанному `file:`.
- `next_id` инкрементирован ровно на количество добавленных документов.
- `.vassal/raw/intake-{дата}/` содержит копии всех обработанных оригиналов.

Без этих постусловий скилл не может проверить себя после выполнения.

## 4. Противоречия и баги

### 4.1. Конфликт команды и скилла по источнику файлов

- [commands/intake.md:7-8](../commands/intake.md): «Если `$ARGUMENTS` содержит путь к папке — используй её как источник файлов.»
- [skills/intake/SKILL.md:28](../skills/intake/SKILL.md): «Источник файлов — `Входящие документы/`. Это **единственное** место, откуда intake забирает файлы.»

Команда разрешает custom path, скилл — запрещает. Если юрист вызовет `/vassal-litigator:intake ~/downloads/новые-доки/`, поведение непредсказуемо: модель решит сама, какой источник приоритетнее.

### 4.2. Мёртвая папка `.vassal/positions/`

[commands/init-case.md:36](../commands/init-case.md) создаёт `.vassal/positions/`, но **ни один скилл туда не пишет**. `build-position` кладёт позиции в корень дела (`{case-root}/{дата} Начальная позиция.md`), как правильно описано в [shared/conventions.md:10-11](../shared/conventions.md). Папка `.vassal/positions/` — артефакт устаревшего дизайна.

Не вина `intake`, но всплывает при анализе его окружения.

### 4.3. Неоднозначность «Отправителя» в имени файла

Конвенция: `ГГГГ-ММ-ДД Отправитель Описание.ext`.

Пример из SKILL.md: `2025-06-01 ООО Ромашка Договор поставки №47.pdf`.

Но договор — двусторонний документ. Почему «Ромашка», а не «Лютик»? По контексту — «Ромашка» это клиент. Но в правиле это не сформулировано. А если клиент в деле — «Лютик», а договор между ним и Ромашкой, то имя должно быть `2025-06-01 ООО Лютик Договор поставки №47.pdf` или `2025-06-01 ООО Ромашка Договор поставки №47.pdf`?

В SKILL.md нет правила «кто отправитель для двусторонних документов». На практике разные файлы одного дела могут получить разные «отправители» в зависимости от импровизации модели.

**Предлагаемое правило** (проверить с юристом): для двусторонних документов — наш клиент всегда на первом месте (если он сторона документа). Для односторонних (претензии, уведомления) — явный отправитель.

### 4.4. OCR-fallback декларативен, не операционен

[scripts/extract_text.py:100](../scripts/extract_text.py) при провале OCR возвращает warning `"Не удалось извлечь текст. Требуется LLM fallback (Read tool)."`

SKILL.md фаза 2 п.7: «Изображения/сканы → `scripts/extract_text.py` (tesseract), при низком качестве — Read tool для визуального анализа (LLM fallback)».

Проблема: **нет чёткого триггера**, когда модель должна переключиться на LLM fallback. «Низкое качество» — по `confidence: low`? По warning'у? По пустому тексту? Скилл не инструктирует явно. Получаем ситуацию «зависит от модели».

Дополнительно: Read tool в Claude Code умеет читать изображения визуально. Но tesseract запущен в subprocess и его ошибки (не установлен, упал) — не пробрасываются как warning в confidence. Скилл их не «увидит».

### 4.5. Повторная обработка по `origin.name + intake_batch` — хрупкая идемпотентность

[SKILL.md:129-131](../skills/intake/SKILL.md): «Проверь, какие файлы из нового пакета уже обработаны (по `origin.name` + `origin.intake_batch`)».

Если юрист **повторно** положит тот же файл (с тем же именем) в «Входящие документы/» через неделю — это будет новый `intake-batch` (другая дата), и `origin.name` совпадёт, но `intake_batch` — нет. Проверка не сработает → дубль в индексе.

Корректнее было бы дедуплицировать **по хэшу содержимого** (SHA-256), не по имени+пакету. Это дорого в рантайме, но избавит от дублей.

## 5. Непокрытые edge-cases

Что есть в SKILL.md (граничные случаи):
- Пустой архив
- Файл нулевого размера
- Повреждённая кодировка
- >50 страниц
- Дубликат имени

### 5.1. Чего не хватает

| Сценарий | Что произойдёт сейчас | Риск |
|---|---|---|
| **PDF с паролем** | `pymupdf` упадёт, LLM Read тоже не откроет | Молчаливый пропуск или ступор |
| **Архив с паролем** | Распаковка упадёт | Молчаливый пропуск |
| **Дубликаты по содержимому** (тот же документ с другим именем) | Оба попадают в индекс как разные | Засорение дела, путаница в анализе |
| **Файл уже в `Материалы от клиента/`**, не во «Входящих» | Intake не видит | Документ не попадёт в индекс |
| **Один PDF — несколько документов** (сканированная папка со всеми доками дела одним файлом) | Обрабатывается как один документ | Неверная категоризация и именование |
| **Тип документа вне таксономии** | Категоризация не определена в SKILL.md — модель импровизирует | Разъезжающаяся таксономия между делами |
| **Документ без номера и без даты** | Имя `0000-00-00 Неизвестный Документ.ext` | Неинформативное имя, тяжело потом найти |
| **Parallel/duplicate intake** (две сессии в одной папке дела) | Race на `next_id`, дубли записей | Порча индекса |
| **Partial failure в фазе 4** (создали 10 зеркал из 15, потом ошибка) | Не определено — индекс может разъехаться с ФС | Порча состояния |
| **Кириллические имена в архивах (cp1251/cp866)** | Распаковка может побить имена | Обходимо, но не описано |
| **Многостраничный скан с низким OCR**: `needs_manual_review: true` | Зеркало создано, но плохое; нет триажа | Юрист не знает, какие зеркала перечитать |

## 6. Реальные проблемы на практике

Оцениваю с точки зрения реального использования юристом:

### 6.1. Скан «папки с документами» как один PDF

Типовой сценарий: клиент прислал архив или PDF-файл, в который отсканированы подряд — договор, акты, платёжки, переписка. Intake увидит это как **один** файл типа «договор» (по первой странице), и всё остальное будет в том же зеркале.

Скилл не предусматривает **разделение одного файла на логически разные документы**. В реальной практике это частый сценарий.

### 6.2. Дубли форматов

Юрист часто получает один документ в двух форматах — `.docx` (исходник) и `.pdf` (подписанный скан). Intake создаст два зеркала, две записи в индексе. Потом в анализе Claude будет ссылаться то на один, то на другой.

Нужен либо дедуп по содержимому, либо явная связь «это одна и та же претензия, просто в двух форматах».

### 6.3. Длинный OCR-pipeline на больших пакетах

Если клиент сразу прислал 200 файлов — intake будет работать долго. Progress reporting не предусмотрен. В Cowork-сессии это может упереться в таймауты.

### 6.4. Нет триажа документов, требующих ручной проверки

Поле `needs_manual_review: true` проставляется, но нигде не используется. Юрист должен сам помнить/проверять, какие зеркала плохие. Нет отчёта «вот список документов, которые я обработал плохо, проверь их».

## 7. Рекомендации (приоритизированно)

### P0 (критично, блокирует надёжную работу)

1. **Определить постусловия** и добавить в SKILL.md валидацию после apply:
   - YAML валиден.
   - Для каждой новой записи `file:` существует, `mirror:` существует.
   - `next_id` инкрементирован корректно.
   - Сверить количество файлов в «На удаление/» с количеством обработанных.
2. **Ввести rollback/recovery для фазы 4.** Если создание зеркал или обновление индекса упало на полпути — чёткая политика: «удали созданные зеркала, восстанови index.yaml из snapshot-а, оригиналы в raw/ не трогай». Сейчас — undefined behavior.
3. **Разрешить конфликт [§4.1](#41-конфликт-команды-и-скилла-по-источнику-файлов):** либо команда должна настаивать на «Входящие», либо SKILL.md должен принимать custom path.

### P1 (важно, влияет на качество данных)

4. **Дедуп по хэшу содержимого** вместо `origin.name + intake_batch`. Добавить поле `content_hash: sha256:...` в схему index.yaml. При intake — считать хэш и сверять.
5. **Правило «отправителя» для двусторонних документов.** Зафиксировать в `shared/conventions.md` и сослаться из SKILL.md.
6. **Явный триггер LLM-fallback для OCR:** если `confidence: low` ИЛИ есть warning про «требуется LLM fallback» — обязательно прогнать файл через Read tool перед созданием зеркала.
7. **Обработка защищённых PDF/архивов:** явно ловить и возвращать в preview как «требует пароль» — пользователь либо снимает пароль и кладёт файл повторно, либо пропускаем.

### P2 (улучшения UX)

8. **Progress reporting** для больших пакетов (>20 файлов): «обработано 12/200, текущий файл: ...».
9. **Финальный отчёт с триажем:** список документов с `needs_manual_review: true` + список с `confidence: low` + явный призыв «проверь эти N зеркал перед `legal-review`».
10. **Возможность split'а одного PDF на несколько документов:** опциональный режим интерактивности — «в файле X.pdf распознано 4 разных документа, хотите разделить?».
11. **Lock-файл** `.vassal/.intake.lock` на время работы, чтобы параллельный запуск не убил индекс.

### P3 (архитектурные, связанные с другими скиллами)

12. **Расширение схемы индекса:** поле `duplicates: [doc-N, ...]` для документов в разных форматах одного содержимого.
13. **Автотриггер `update-index` после intake** как самопроверка целостности (снимает часть проблем из [WORKFLOW-REVIEW.md](../WORKFLOW-REVIEW.md) п. 2-3).

## 8. Резюме

Скилл в целом хорошо спроектирован на макроуровне — понятное разделение на фазы, preview→apply, делегирование Haiku-субагентам. Но **хрупок в деталях**: идемпотентность через имена (легко ломается), нет постусловий/валидации, нет rollback, не покрыты частые реальные сценарии (PDF с паролем, многодокументные сканы, дубли форматов, split файлов).

Критически важный скилл — от него зависят все остальные, — поэтому **приоритет укрепления `intake` выше** большинства других улучшений плагина.

Главное техническое решение, которое сильно улучшит надёжность: **дедуп по хэшу + постусловия + recovery-политика**. Это небольшие правки SKILL.md и схемы, дают большой выигрыш в стабильности данных дела.
