# Ревизия скилла `prepare-hearing`

**Дата:** 2026-04-20
**Файлы:** [skills/prepare-hearing/SKILL.md](../skills/prepare-hearing/SKILL.md), [commands/prepare-hearing.md](../commands/prepare-hearing.md)
**Связанные:** [ARCHITECTURE.md §8.8](../ARCHITECTURE.md), внешний плагин `arbitrum-docx`
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** формирование обновлённой правовой позиции перед заседанием + генерация процессуального документа (отзыв, ходатайство, пояснения, возражение, дополнение) в `.docx` через внешний плагин `arbitrum-docx`. Включает red team / blue team анализ и отдельный шаг стресс-теста.

**Позиция в workflow:** центральный циклический скилл фазы «Ведение дела». Запускается **перед каждым** заседанием (или каждый раз, когда нужен процессуальный документ). Ключевой потребитель результатов `build-position`, `legal-review`, `analyze-hearing`, `add-opponent`.

**Парные / последующие скиллы:**
- Вход: `build-position` (позиция), `analyze-hearing` (отчёты о прошлых заседаниях), `add-opponent` (анализ документов оппонента), `legal-review` (сроки, квалификация).
- Выход: процессуальный документ (`.md` + `.docx`) → подаётся в суд → после заседания → `analyze-hearing`.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Стороны, суд, хронология |
| `.vassal/index.yaml` | да | Реестр документов |
| `{корень}/*позиция.md` | да | Последняя позиция (какая — см. §4.4) |
| `.vassal/mirrors/*.md` (ключевых) | да | Зеркала доказательств обеих сторон |
| `.vassal/hearings/*.md` | нет | Отчёты о прошлых заседаниях |
| `.vassal/analysis/opponent-*.md` | нет | Экспресс-анализ оппонента |
| Процессуальные документы обеих сторон (зеркала) | да | Иск, отзыв, возражения |
| Вводная от Сюзерена | да | Тип документа + стратегия |
| `arbitrum-docx` SKILL.md | да (phase 6) | Правила оформления `.docx` |

### 2.2. Выход

| Файл | Путь |
|---|---|
| Процессуальный документ (markdown) | `{корень}/{ГГГГ-ММ-ДД} {тип}.md` |
| Процессуальный документ (docx) | `{корень}/{ГГГГ-ММ-ДД} Клиент {тип}.docx` через `arbitrum-docx` |
| Обновление истории | `.vassal/history.md` |
| Обновление индекса | `.vassal/index.yaml` (если приобщены новые документы) |

**Что НЕ сохраняется** (но создаётся в ходе работы — см. §4.1):
- Red team / Blue team анализ
- Результаты стресс-теста
- «Обновление ситуации» из фазы 2 п.10
- Промежуточные итерации текста

### 2.3. Предусловия — есть в SKILL.md

- Дело инициализировано, `intake` выполнен.
- Есть хотя бы одна позиция в корне.
- Для последующих заседаний — желательны отчёты из `.vassal/hearings/`.

### 2.4. Постусловия — ❌ нет

Должны быть:
- `.md` и `.docx` файлы созданы и ссылаются друг на друга.
- В `index.yaml` есть запись о новом процессуальном документе.
- В `history.md` — запись со ссылкой на документ.
- Если приобщались новые доказательства — они в индексе и в соответствующей папке.

## 3. Качество промпт-инженерии

| Критерий | Оценка | Комментарий |
|---|---|---|
| Description (триггеры) | ✅ | Много позитивных триггеров, покрывают разные формулировки |
| Pre-conditions | ✅ | Есть, явно |
| Post-conditions | ❌ | Нет |
| **Методологическая основа** | ⭐ | Ссылка на метод Мошкина С.А. — редкий случай, когда скилл опирается на оформленную внешнюю практику |
| **Порядок загрузки контекста** | ⭐ | Доказательства → процессуальные документы ПОСЛЕДНИМИ. Тонкий и правильный приём: модель не попадает сразу под рамку оппонента |
| **Отдельный шаг стресс-теста** | ⭐ | Фаза 3 — не часть анализа, а независимый этап с собственной задачей. Цитата Мошкина прямо объясняет зачем |
| Red team / Blue team структура | ✅ | Есть, с парированием |
| Приоритизация аргументов | ✅ | Явное правило: «сильный — первым, ответ на опасный — вторым» |
| Preview + итерации | ✅ | Возврат к нужной фазе при корректировке |
| Валидация перед phase 5-6 | ⚠️ | Есть confirmation от Сюзерена, но нет программной валидации (YAML, ссылки) |
| Error-handling / fallback | ❌ | Особенно для `arbitrum-docx` (см. §4.5) |
| Сохранение промежуточных артефактов | ❌ | Red team / стресс-тест теряются после сессии (см. §4.1) |
| Лимиты контекста | ❌ | Не описаны |
| Разрешение «какая позиция последняя» | ❌ | Glob `*позиция.md` без правила выбора (см. §4.4) |

## 4. Противоречия и баги

### 4.1. Ценные аналитические артефакты эфемерны

В фазах 2-3 скилл создаёт:
- **Обновление ситуации** (фаза 2 п.10) — что изменилось с последней позиции
- **Red team / Blue team анализ** (п.11)
- **Стратегическая оценка** (п.13)
- **Стресс-тест: найденные уязвимости, расхождения** (фаза 3)

Всё это — **работа Opus**, дорогая. Но SKILL.md **нигде не инструктирует сохранить** эти артефакты. Единственный выход — текст финального документа, где они частично «зашиты» в аргументацию.

Последствия:
1. При следующем `prepare-hearing` — заново прогоняем phase 2-3 → повторная оплата Opus.
2. При сбое сессии — работа потеряна.
3. Нет накопления «что мы думали год назад» — исторической аналитики нет.

**Предлагаемое решение:** сохранять в `.vassal/hearings/{ГГГГ-ММ-ДД}-prep.md` структурированный артефакт:
```markdown
---
type: prep-analysis
for_hearing: 2026-04-20
generated_at: 2026-04-15
---
# Подготовка к заседанию 2026-04-20

## Обновление ситуации
...

## Blue team — наши аргументы
...

## Red team — аргументы оппонента
...

## Стресс-тест — уязвимости и расхождения
...

## Финальная структура документа
...
```

### 4.2. Дублирование workflow с `analyze-hearing`

Оба скилла пишут в `.vassal/hearings/`:
- `analyze-hearing` — `{дата}-internal.md`, `{дата}-client-brief.md`, `{дата}-judge-profile.md` (постфактум)
- `prepare-hearing` — сейчас никуда не пишет (см. §4.1), но должен писать аналогично (префактум)

Если ввести `prep.md` (§4.1), естественно дополняется набор: **до заседания** — `prep.md`, **после** — `internal.md` и `client-brief.md`. Получится полный цикл на одну дату.

### 4.3. Жёсткая зависимость от `arbitrum-docx` без fallback

[SKILL.md:161-165](../skills/prepare-hearing/SKILL.md): «прочитай `arbitrum-docx` SKILL.md и оформи».

Проблемы:
1. Если `arbitrum-docx` не установлен / не найден → скилл падает / импровизирует.
2. Нет feature detection: «проверь, доступен ли arbitrum-docx. Если нет — сохрани только `.md` и предупреди Сюзерена».
3. Версионирование контракта: `arbitrum-docx` — внешний плагин. Его SKILL.md может меняться. Как `prepare-hearing` узнаёт, что версии совместимы?

В [WORKFLOW-REVIEW.md §4](../WORKFLOW-REVIEW.md) это отмечено как общая проблема для 4 скиллов плагина. Для `prepare-hearing` — особенно критично, так как это самый частый скилл в цикле.

**Минимальный fallback:** всегда сохранить `.md` в корне; если `arbitrum-docx` недоступен → явное сообщение «Оформление в .docx пропущено, документ сохранён как .md».

### 4.4. Неоднозначный выбор «последней позиции»

[SKILL.md:49](../skills/prepare-hearing/SKILL.md): «Загрузи последнюю позицию из корня дела (файл `*позиция.md`)».

При 3-4 позициях в корне (начальная + обновлённая + ответ + ...) — какая «последняя»?

Варианты интерпретации:
- По дате в имени файла.
- По `mtime` файла.
- По алфавиту (что будет означать зависимость от нейминга).

Нет правила → зависит от реализации модели.

**Правильный ответ:** по дате в префиксе имени (`ГГГГ-ММ-ДД`) — это согласуется с конвенцией. Нужно явно записать.

### 4.5. Стресс-тест без внешнего взгляда

Фазы 2-3 проводятся в рамках **одной сессии одной модели**. Она же формирует blue team → она же атакует с red team → она же делает стресс-тест.

Проблема: LLM склонен к consistency. Построив blue team, он подсознательно смягчит red team — чтобы не дестабилизировать свой же план. Это известный эффект.

**Правильная архитектура:** red team и стресс-тест — **отдельные субагенты** с системным промптом «Ты — юрист оппонента. Твоя задача — разбить эту позицию». Они получают документ в вакууме (без наших рассуждений) и атакуют.

SKILL.md делегирует Opus-субагенту общий анализ, но не разводит роли.

### 4.6. Стратегическая оценка навязывается

[SKILL.md:96-98](../skills/prepare-hearing/SKILL.md) п.13: «Если анализ показывает слабую позицию — сообщить Сюзерену прямо, предложить альтернативные стратегии (мировое, минимизация последствий)».

Формулировка подразумевает, что это делается **всегда**. Но:
- Сюзерен может уже обсудил стратегию с клиентом, мировое обсуждается отдельно.
- Навязчивые рекомендации «подумайте о мировом» посреди подготовки возражения — раздражают.
- Нет триггера «только если Сюзерен попросил оценить перспективы».

**Решение:** стратегическая оценка — в preview как опциональный блок: «Хотите, чтобы я оценил перспективы / предложил альтернативы?». По умолчанию — не предлагать.

### 4.7. Нет шаблона для мирового соглашения

В таблице «Типы процессуальных документов» ([SKILL.md:175-183](../skills/prepare-hearing/SKILL.md)) **мировое соглашение отсутствует**. Но в п.13 оно упоминается как стратегия.

Если Сюзерен ответит на вопрос «какой документ готовим?» → «мировое соглашение», scilл не готов. Будет импровизировать.

Это связано с [WORKFLOW-REVIEW.md §7](../WORKFLOW-REVIEW.md) («нет скилла мирового соглашения») — либо отдельный скилл, либо отдельный тип в этой таблице.

### 4.8. `update-index` не вызывается автоматически

Если Сюзерен перед подготовкой положил новые документы мимо `add-evidence` (частый реальный сценарий: «пришёл свежий акт, кладу в папку») → их нет в `index.yaml` → `prepare-hearing` их не увидит.

В SKILL.md нет инструкции «перед phase 1 — проверь актуальность индекса». Это связано с [WORKFLOW-REVIEW.md §2-3](../WORKFLOW-REVIEW.md).

### 4.9. Нет реакции на срок до заседания

Вопрос 3 во входных данных: «Дата заседания (если известна — для контекста срочности)». Но в алгоритме **дата никак не используется**. Ни одна фаза не адаптируется:
- 30 дней до заседания → можно углубить анализ, потратить больше Opus.
- 2 дня до заседания → минимум, успеть подать.

**Решение:** в phase 2 проверять срок. Если < 3 дней — сокращать red team до минимума, стресс-тест упрощать, сразу писать. Явное правило в SKILL.md.

### 4.10. Конфликт команды и скилла

[commands/prepare-hearing.md:10-11](../commands/prepare-hearing.md):
- «Если `$ARGUMENTS` содержит тип документа — используй как вводную.»
- «Если `$ARGUMENTS` содержит стратегические указания — учти при анализе.»

SKILL.md ([входные данные:31-38](../skills/prepare-hearing/SKILL.md)): «Если Сюзерен сразу дал конкретные указания в аргументах команды — не переспрашивай».

В целом согласовано, но парсинг `$ARGUMENTS` — неявный. Если юрист напишет `/vassal-litigator:prepare-hearing отзыв, акцент на срок давности` — что отдельное? Тип документа = «отзыв», стратегия = «акцент на срок давности». Но если `отзыв на срок давности` — это тип или стратегия? Нет правила разбора.

## 5. Непокрытые edge-cases

### 5.1. Повторный запуск в той же подготовке (итерация)

Сюзерен увидел preview, попросил «усилить довод №3». Скилл должен вернуться к phase 2 ограниченно (только для этого довода), не перезагружая весь контекст.

Нет инструкции для инкрементальных итераций. Модель либо начнёт заново (дорого), либо сделает мелкую правку без новой аналитики.

### 5.2. Очень большое дело — переполнение контекста

Загружаются зеркала ключевых доказательств + все процессуальные + все отчёты о заседаниях. Для дела со 100+ документами и 3-4 прошедшими заседаниями — легко 100k+ токенов. Нет явной политики.

### 5.3. Нет механизма сохранения «текущего черновика» между сессиями

Сценарий: юрист начал `prepare-hearing` утром, дошёл до phase 5, прервал сессию. Вернулся через 2 часа. Всё начинается с phase 1. Вся работа в контексте потеряна.

**Решение:** в конце каждой фазы сохранять snapshot в `.vassal/hearings/{дата}-prep-wip.md`. При повторном запуске предлагать: «Обнаружен незавершённый prep от 14:30. Продолжить или начать заново?».

### 5.4. Документ в процессуальную папку vs в корень — нет правила

По [conventions.md:10-12](../shared/conventions.md):
- Позиции, анализ → корень.
- Процессуальные документы → процессуальная папка `{дата} Клиент - {тип}/`.

`prepare-hearing` phase 6 п.23 говорит «поместить .docx в **корень** дела». Это **противоречит** конвенции. Если мы отвечаем отзывом — должна быть папка `{дата} Клиент - отзыв на иск/` с основным файлом + приложениями.

Кроме того, [ARCHITECTURE.md:759](../ARCHITECTURE.md) говорит: «Поместить .docx в **соответствующую процессуальную папку**». SKILL.md и ARCHITECTURE **расходятся**.

### 5.5. Приложения к процессуальному документу — неясно кто формирует

SKILL.md п.12: «Приложения (список документов для приобщения)» — часть структуры документа. Но:
- Откуда список? Автоматически из `index.yaml`? Или Сюзерен диктует?
- Помечаются ли приобщаемые документы в `index.yaml` (`filing_status: submitted`)?
- Если документ уже приобщался ранее — нужна пометка «(приобщён ранее)» в списке.

Ничего не описано.

### 5.6. Генерация «целиком или по частям» — без правила

Phase 5 п.18: «Спроси Сюзерена: «Генерировать целиком или по частям?». По умолчанию — целиком».

Для большого возражения (10+ доводов) «целиком» = огромный выхлоп, сложно итерировать. «По частям» = контроль на уровне довода. Нет правила, когда что.

### 5.7. Просительная часть — нет чек-листа

Наиболее частая ошибка в процессуальных документах — плохо сформулированная просительная часть (не указаны проценты, не указана солидарность, не включены судебные расходы). SKILL.md просто говорит «четко и однозначно» без чек-листа.

Пример нужного чек-листа:
- Указана точная сумма и её состав (основной долг + проценты + неустойка + пошлина).
- Если несколько должников — явно «солидарно» или «в долях».
- Проценты указаны либо твёрдой суммой на дату, либо с формулой «по день фактической оплаты».
- Судебные расходы заявлены.
- Для ходатайств — конкретная просьба, а не рассуждение.

### 5.8. Копирование стиля — без метрик

Phase «Копирование стиля» (опционально, после §7 в SKILL.md): «Генерировать документ максимально близко к этому стилю».

«Близко» — что значит? Структура? Обороты? Шрифт? Длина абзацев? Юрист получит — и сам поймёт, удачно или нет. Но критерий не задан.

### 5.9. Нет поддержки встречного иска и третьих лиц

Типы документов не покрывают:
- Встречный иск (свои правила составления).
- Заявление о вступлении третьего лица.
- Отзыв третьего лица.

Для больших дел с несколькими ответчиками — реально.

## 6. Реальные проблемы на практике

### 6.1. Разрастание контекста и стоимость

Opus + весь контекст дела = дорого. Для одного заседания запуск может стоить заметных денег. Скилл не показывает «сколько токенов я загружаю» — юрист не видит цену.

### 6.2. Красивый метод Мошкина vs реальность

Порядок «сначала факты, потом процессуальные» — правильный принцип. Но в Claude Code контекст «видит всё сразу», и порядок в промпте имеет меньшее значение, чем при работе с живым экспертом. Это не убивает метод, но эффект слабее, чем обещано.

### 6.3. Срок подачи ≠ срок на стресс-тест

На практике юрист часто заходит за 1-2 дня до заседания. Все фазы скилла (особенно 3) — роскошь. Нужна «turbo-версия»: быстрый проход, минимальный анализ, сразу в текст. Сейчас такого режима нет.

### 6.4. Юрист может захотеть свою структуру

SKILL.md предлагает структуру из 6 разделов (вводная / обстоятельства / правовое / ответ / просительная / приложения). Но разные юристы пишут по-разному: кто-то сначала правовое, потом факты. Скилл на это не адаптируется.

### 6.5. Финальная правка руками неминуема

Ни один процессуальный документ не сдаётся без ручной доводки. Но скилл не облегчает возврат: `.docx` открывается в Word, правки вносятся там, в `.md` в `.vassal/` они не попадают. При следующем `prepare-hearing` скилл увидит устаревшую `.md`-версию.

**Решение:** при следующем запуске проверять `mtime` `.docx` и `.md` — если `.docx` свежее, предупредить: «Финальная версия правилась вне плагина. Подтянуть её в .md?».

## 7. Рекомендации (приоритизированно)

### P0 (критично)

3. **Fallback для `arbitrum-docx`**: feature detection + сохранение `.md` с предупреждением (§4.5).
4. **Разрешить противоречие с `conventions.md`** по размещению документа (корень vs процессуальная папка) (§5.4).
5. **Постусловия и валидация** (§2.4).

### P1 (важно)

6. **Red team как отдельный субагент** с системным промптом «юрист оппонента» (§4.5 про стресс-тест).
7. **Адаптация под срок до заседания** — turbo-режим при <3 дней (§4.9, §6.3).
8. **Сессионный state + recovery** из незавершённой подготовки (§5.3).
9. **Автовызов `update-index`** перед phase 1 или хотя бы предупреждение (§4.8).
10. **Чек-лист просительной части** — отдельный блок в SKILL.md (§5.7).
11. **Шаблон мирового соглашения** — добавить тип в таблицу или вынести в отдельный скилл (§4.7).
12. **Регламент приложений:** формирование списка, проставление `filing_status` в индексе (§5.5).
13. **Стратегическая оценка — опциональная**, по запросу (§4.6).

### P2 (UX)

14. **Метрики контекста**: сколько токенов загружено, стоимость запуска (§6.1).
15. **Sync `.docx` ↔ `.md`** при повторных запусках (§6.5).

21. **Отдельный subagent для blue team** (симметрично red team) — для чистоты эксперимента.

## 8. Резюме

Prepare-hearing — **самый методологически продуманный** скилл плагина. Редкие сильные стороны:
- Опора на оформленный метод (Мошкин С.А., runetlex.academy).
- Порядок загрузки контекста (доказательства → процессуальные последними).
- Стресс-тест как отдельный этап, а не часть анализа.
- Red team / Blue team с парированием.
- Приоритизация аргументов (сильный — первым).

Главная архитектурная слабость — **эфемерность дорогой работы**. Red team, стресс-тест, «обновление ситуации» — Opus-работа, но нигде не сохраняется. Это не просто неудобно — это **основной антипаттерн скилла**, который убивает ROI от метода Мошкина: при каждом новом заседании всё делается заново.

Две правки закроют 70% проблем:
1. `.vassal/hearings/{дата}-prep.md` со всей промежуточной аналитикой.
2. Feature detection + fallback для `arbitrum-docx`.

Остальное — обширный, но поступательный список улучшений.
