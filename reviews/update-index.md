# Ревизия скилла `update-index`

**Дата:** 2026-04-20
**Файлы:** [skills/update-index/SKILL.md](../skills/update-index/SKILL.md), [commands/update-index.md](../commands/update-index.md)
**Связанные:** [shared/index-schema.yaml](../shared/index-schema.yaml), [shared/conventions.md](../shared/conventions.md), [WORKFLOW-REVIEW.md §2-3, §9](../WORKFLOW-REVIEW.md)
**Версия плагина:** 0.4.0

## 1. Назначение и роль в workflow

**Декларация:** сканирование файловой системы дела и приведение `index.yaml` в соответствие с реальным состоянием файлов. Находит три класса рассинхронизаций:
- новые файлы без записи в индексе (ручное попадание мимо intake/add-evidence/add-opponent);
- «осиротевшие» записи — файл из index.yaml отсутствует на диске (ручное удаление/перемещение);
- устаревшие зеркала — файл изменился после `last_verified`.

**Позиция в workflow:** теоретически — страж целостности shared state, единственная точка, где плагин ловит разрыв между ФС и индексом. Фактически — скилл-сирота. По [ARCHITECTURE.md:896](../ARCHITECTURE.md) он стоит «после add-evidence/add-opponent», но ни один другой SKILL.md его не зовёт, ни один из них не делегирует к нему, в описаниях intake/catalog его упоминают только в негативной форме («НЕ используй для обновления индекса»).

**Триггер:** формально — команда `/vassal-litigator:update-index` или фразы юриста типа «обнови индекс», «я вручную положил файлы». Неформально — никак. Триггерных точек в алгоритмах других скиллов нет. См. [WORKFLOW-REVIEW.md §2](../WORKFLOW-REVIEW.md). Это **критическая дыра в архитектуре целостности**.

**Парные / последующие скиллы:** формально нет. Читают `index.yaml` все, но никто не ожидает, что он синхронизирован с ФС.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/index.yaml` | да | Текущие записи, `next_id` |
| `.vassal/case.yaml` | да (в пре-условиях) | Фактически нигде в алгоритме не читается — см. §4.1 |
| `Материалы от клиента/**` | да | Рекурсивный скан |
| `{ГГГГ-ММ-ДД} *` (процессуальные папки) | да | Рекурсивный скан |
| Корень дела (кроме `.vassal/` и служебных) | да | Плоский скан |
| `shared/conventions.md` | неявно | Таксономия типов (при добавлении новых) |
| `shared/index-schema.yaml` | неявно | Структура записи |
| `shared/mirror-template.md` | неявно | Формат зеркала при пересоздании |

### 2.2. Выход

| Результат | Путь |
|---|---|
| Обновлённый индекс | `.vassal/index.yaml` (новые записи + удаление/обновление старых + `last_updated`, `last_verified`) |
| Новые зеркала | `.vassal/mirrors/doc-NNN.md` (для новых файлов) |
| Пересозданные зеркала | `.vassal/mirrors/doc-NNN.md` (для изменённых файлов) |
| Удалённые зеркала | `На удаление/doc-NNN.md` (для осиротевших записей, по подтверждению) |
| Запись в истории | `.vassal/history.md` |

**Чего НЕТ на выходе** (но должно быть — см. §4.7): snapshot'а `index.yaml` перед правками (бэкап), отчёта о расхождениях для ручного разбора, пометки «этот файл попал мимо intake» в самой записи индекса.

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/index.yaml` существует.
- `.vassal/case.yaml` существует (проверяется, но не используется — см. §4.1).

### 2.4. Постусловия — ❌ нет в SKILL.md

Шаг 5 «Верификация целостности» отчасти играет роль постусловий, но не формализован как контракт (что делать, если проверка провалилась после apply — неясно) и не покрывает обратное отношение «зеркало → запись» (orphan mirrors, §4.4) + интегрити-чексуммы. Полный список должных постусловий:
- `index.yaml` — валидный YAML и соответствует [index-schema.yaml](../shared/index-schema.yaml).
- Для каждой записи `file:` и `mirror:` существуют физические файлы.
- Для каждого зеркала в `.vassal/mirrors/` существует запись в индексе (обратное отношение).
- `next_id` > max(ids), нет дубликатов ID и `file:`.
- Backup снят до изменений и хранится минимум 1 операцию назад.

| Критерий | Оценка | Комментарий |

| Description (триггеры) | ✅ | Хорошие позитивные и негативные триггеры, явно отделён от intake/catalog |
| Pre-conditions | ✅ | Есть, минимальные |
| Post-conditions | ❌ | Не формализованы (см. §2.4) |

| Идемпотентность | ⚠️ | Декларирована «повторный запуск безопасен» в одном абзаце, без деталей **что именно** повторный запуск не сделает (см. §5.6) |

| Критерии «файл изменился» | ❌ | «После `last_verified`» — по какому признаку? mtime? hash? (см. §4.2) |

| Валидация перед apply | ❌ | Нет чек-листа на план изменений (сколько записей предполагается тронуть, пределы безопасности) |
| Backup / rollback | ❌ | Нет снапшота `index.yaml` до правки (см. §4.7 и [WORKFLOW-REVIEW.md §9](../WORKFLOW-REVIEW.md)) |

## 4. Противоречия и баги

### 4.1. `case.yaml` декларируется предусловием, но не используется

[SKILL.md:20](../skills/update-index/SKILL.md): «Существует `.vassal/case.yaml`».

В алгоритме `case.yaml` не читается ни в одной фазе. Проверка сводится к «файл существует». Зачем это проверяется — непонятно. Либо алгоритм должен использовать `case.yaml` (для контекста при категоризации новых файлов — см. §5.1), либо предусловие — избыточное.

Согласованно с intake/add-evidence, которые читают `case.yaml` для контекста сторон и предмета спора. `update-index` потенциально создаёт новые записи в индексе тем же маршрутом — но без контекста дела. Модель будет импровизировать.

### 4.2. «Файл изменился» без определения способа сверки

[SKILL.md:37](../skills/update-index/SKILL.md): «проверь, не устарело ли зеркало (файл изменился после `last_verified`)». Нет определения, что значит «изменился»: mtime (уязвимо — `touch` даёт ложное срабатывание, `cp` без `-p` тоже)? Размер (не ловит переписывание одинаковым размером)? SHA-256 хэш (надёжно, но поля `content_hash` в [index-schema.yaml](../shared/index-schema.yaml) нет)? Пересравнение через пересоздание зеркала (дорого — надо пересоздать, чтобы понять, надо ли пересоздавать)?

На практике модель выберет mtime, но SKILL.md это не фиксирует. При файловых операциях Cowork (копирование/перемещение) mtime может не отражать реальное изменение.

**Правка:** ввести в схеме поле `content_hash: "sha256:..."`, использовать как признак изменения. Совпадает с рекомендацией [reviews/intake.md §4.5, §7.P1.4](intake.md) — единое решение для обеих проблем.

### 4.3. Удаление осиротевших записей — размытый Preview→Apply

Формально preview показан в шаге 7 (перечислены осиротевшие записи), но решение «удалить или связать с новым файлом» должно приниматься по каждой записи индивидуально. В SKILL.md нет шаблона: единое решение на все (теряется контроль) vs. отдельный вопрос по каждой (20 туров диалога при 20 осиротевших) vs. автосвязки (нет алгоритма). По [conventions.md:119-125](../shared/conventions.md) Preview→Apply для удалений задекларирован, но в SKILL.md update-index его реализация размазана.

**Правка:** алгоритм «автомэтч по хэшу/размеру/имени → таблица связок-удалений-непонятных → подтверждение всей таблицы».

### 4.4. Orphan mirrors не детектируются

Шаг 5 «Верификация целостности» ([SKILL.md:94-98](../skills/update-index/SKILL.md)) проверяет только **одну сторону** отношения «запись ↔ зеркало»:
- ✅ «Каждая запись в индексе имеет зеркало»
- ❌ **Не проверяет:** «Каждое зеркало в `.vassal/mirrors/` имеет запись в индексе».

Шаг 2 «Сканирование» ([SKILL.md:26-31](../skills/update-index/SKILL.md)) явно игнорирует `.vassal/` — то есть зеркала вообще не сканируются.

Сценарий: в indexe кто-то удалил запись `doc-007` (скажем, intake откатился наполовину, в §4.7 [reviews/intake.md](intake.md) описано partial failure), но `mirrors/doc-007.md` остался. Update-index не увидит его. Зеркало-призрак сохранится навсегда, будет всплывать в grep'ах и вводить в заблуждение другие скиллы (legal-review, prepare-hearing читают зеркала по glob).

**Правка:** сканировать `.vassal/mirrors/` отдельно. Зеркало без записи в индексе → классифицировать как «orphan mirror», предложить удалить (переместить в `На удаление/`) или восстановить запись.

### 4.5. `Входящие документы/` игнорируются в скане — правильно, но «На удаление/» тоже — ошибка?

[SKILL.md:31](../skills/update-index/SKILL.md): «Игнорируй: `.vassal/` (внутренности), `На удаление/`, `Входящие документы/`, `.DS_Store`, `Thumbs.db`, `Таблица документов.xlsx`».

- `Входящие документы/` — верно, там полуобработанные/необработанные файлы.
- `На удаление/` — спорно. Если файл, который раньше был в индексе, был перемещён туда (человеком или предыдущим update-index), запись индекса становится «осиротевшей». Но update-index не видит, куда файл делся. Нет возможности сказать «этот файл не исчез, он в корзине, можешь удалить запись».

На практике логика должна быть: если файл есть в `На удаление/` с именем, соответствующим записи → запись → delete, тихо, без вопросов.

Сейчас каждое «удалённое» через перемещение в корзину реально становится «осиротевшим» и будет подниматься в вопросах к Сюзерену при каждом update-index. Раздражающий UX.

**Правка:** при детекции осиротевшей записи — проверять, нет ли файла с тем же именем в `На удаление/`. Если есть — предлагать «удалить запись (файл в корзине)».

### 4.6. Файлы `source: opponent` и процессуальные папки

Процессуальные папки `{ГГГГ-ММ-ДД} *` могут содержать документы обеих сторон: наши (при `prepare-hearing`) и оппонента (через `add-opponent` со структурой бандла). У каждого — запись в индексе с `source: client | opponent`.

Если update-index встретит новый файл в процессуальной папке — **как он определит `source`**? В SKILL.md (шаг 10) просто сказано «делегируй Haiku при необходимости для типа/даты/сторон». Но `source` — не тип, это орган происхождения. Модель либо угадает по контексту папки (`{Ответчик} - отзыв/` → opponent), либо предположит `client` по умолчанию.

Риск неверного `source` → `legal-review` и `prepare-hearing` используют это поле для отделения «нашего» от «оппонентского». Неверный `source` искажает red/blue team анализ.

**Правка:** в SKILL.md явно: «Для нового файла в процессуальной папке — определи `source` по имени папки: если сторона в имени = наш клиент → client, если оппонент → opponent. Если папка общая (без стороны) — спроси Сюзерена». А лучше — вообще всегда спрашивать по каждому новому файлу.

### 4.7. Нет backup'а `index.yaml` до правки — сломанный sync откатить нечем

[WORKFLOW-REVIEW.md §9](../WORKFLOW-REVIEW.md) прямо отмечает эту проблему. Update-index пишет в `index.yaml` без снятия снапшота.

Сценарий:
1. Update-index детектировал 15 осиротевших записей (например, пользователь переименовал папку вручную, все пути сломались разом).
2. Сюзерен в спешке подтвердил «удалить все 15».
3. Скилл удалил 15 записей, переместил 15 зеркал в `На удаление/`.
4. Сюзерен: «стоп, я хотел переименовать, а не удалить».
5. Откатить нечем. Записи стёрты, зеркала в корзине (потенциально очищены пользователем).

Аналогичный сценарий — partial failure в apply (скилл упал на 8-й записи из 15): index.yaml в промежуточном состоянии, часть зеркал уже удалены.

**Правка (P0):**
- Перед любым изменением `index.yaml` копировать текущий в `.vassal/index.yaml.bak.{timestamp}`.
- Хранить последние N бэкапов (например, 5) ротацией.
- В истории фиксировать имя бэкапа: «откат возможен через `cp index.yaml.bak.{timestamp} index.yaml`».
- В SKILL.md — явный шаг «Snapshot» как фаза 3.5.

### 4.8. Смешаны `last_updated` и `last_verified` + политика `next_id`

Шаг 13: «Обнови `last_updated` и `last_verified` в index.yaml». Но это два разных уровня: `last_updated` (top-level) — модификация файла индекса; `last_verified` — поле **каждой записи** ([index-schema.yaml:43](../shared/index-schema.yaml)). SKILL.md их смешивает.

Правильное поведение — `last_verified` каждой записи должен обновляться для **ВСЕХ прошедших верификацию**, не только модифицированных. Иначе поле никогда не растёт, и детект устаревших зеркал (§4.2) невозможен в принципе.

Похожая проблема с `next_id`: шаг 5 проверяет «корректен ли», но **что делать при некорректном** не описано (автоисправлять? падать? спрашивать?). И скачущие ID (doc-001, doc-003, doc-010 при next_id=11) валидны, но модель может принять за баг.

### 4.9. ARCHITECTURE.md ссылается на несуществующие скрипты

[ARCHITECTURE.md:93-97](../ARCHITECTURE.md) перечисляет в `scripts/`:
- `setup.sh` ✅ существует
- `extract_text.py` ✅ существует
- `create_mirror.py` ❌ отсутствует
- `update_index.py` ❌ отсутствует
- `generate_table.py` ❌ отсутствует

Из 5 заявленных скриптов реально есть только 2. Update-index, catalog и создание зеркал работают как чистые LLM-прогоны через файловые операции. Для больших дел (см. §5.3) это медленнее и дороже обещанной программной реализации.

(Эта же проблема упомянута в [reviews/catalog.md §4.3](catalog.md) применительно к `generate_table.py` — здесь же видно, что разрыв между ARCHITECTURE.md и реальностью шире и системнее.)

## 5. Непокрытые edge-cases

### 5.1. Новый файл без контекста — как категоризировать

Если файл появился в `Материалы от клиента/Иное/` — что это? Договор? Претензия? Update-index должен:
- Извлечь текст (OCR pipeline — как в intake).
- Определить тип документа по содержимому.
- Определить дату и стороны.

В SKILL.md (шаг 10) это одной строкой: «Определи тип, дату, стороны (делегируй Haiku при необходимости)». Но:
- Какой промпт для Haiku? В intake это расписано подробно с примерами, тут — нет.
- Какая таксономия? conventions.md — но нигде не сказано «используй ту же таксономию, что intake».
- Как обрабатывать файл, который в принципе непохож на документ (служебная заметка, скан случайный, черновик)?

**Риск:** категоризация разъезжается. intake и add-evidence создают консистентные записи, update-index при ручном добавлении — «на что модель угадает».

**Правка:** либо делегировать создание записи именно intake/add-evidence (update-index только показывает: «обнаружен необработанный файл, запустить add-evidence?»), либо явно инструктировать промпт и ссылаться на conventions.md и примеры в intake SKILL.md.

### 5.2. Файл переименован → 2 записи, орфан + новый, но это один и тот же файл

[SKILL.md:109](../skills/update-index/SKILL.md): «Файл переименован вручную → старая запись станет «осиротевшей», новое имя — «новым файлом». Предложи Сюзерену связать их (обновить путь в существующей записи)».

Хорошо, что случай упомянут. Но **как предложить связь?**
- По похожести имени (Левенштейн)?
- По размеру?
- По первым страницам содержимого?
- По хэшу?

Алгоритм не описан. На практике модель сравнит имена, но если переименование было кардинальным («скан.pdf» → «2025-06-01 ООО Ромашка Договор.pdf»), по имени не сматчится.

**Правка:** использовать `content_hash` (см. §4.2) — если осиротевшая запись и «новый файл» имеют совпадающий хэш, автоматически связывать без вопросов.

### 5.3. Большое дело — N файлов на ФС × M записей в индексе = N*M сравнений

Для дела со 300 документами шаг 2 («для каждого файла — проверь, есть ли запись») — квадратичный. Без индекса по `file:` в памяти это медленно. LLM сделает это последовательно, через чтение индекса + scan — но времени уйдёт много, и риск «потерять в контексте» часть файлов.

SKILL.md не описывает батчинг или быстрые структуры. Для интерактивного использования на больших делах это непригодно.

**Правка:** делегировать Python-скрипту (`scripts/update_index.py`, упомянутому в [ARCHITECTURE.md:96](../ARCHITECTURE.md), но не созданному) сверку множеств. LLM — только для интерпретации результата и вопросов Сюзерену.

### 5.4. Конфликт `file:` — один путь в двух записях

Верификация (шаг 5) упоминает: «Нет дубликатов путей в `file:`». Но **что делать**, если обнаружены? Автоматически убрать? Спросить Сюзерена?

Сценарий: intake при partial failure создал запись, повторный запуск пересоздал (см. [reviews/intake.md §5](intake.md)) — два doc-id на один путь. update-index это детектирует, но политики разрешения нет.

### 5.5. `filing_status: submitted` и фактическое расположение файла

В [index-schema.yaml:51-52](../shared/index-schema.yaml) есть `filing_status: submitted | filed` и `filing_folder`. Если документ фактически подан, но в процессуальной папке переименован/перемещён — update-index увидит это как «файл исчез» → осиротевшая запись. Не учитывая, что статус `submitted` должен защитить запись от автоматического удаления (поданный в суд документ нельзя «потерять» из реестра дела — он фактически есть в материалах суда).

**Правка:** при детекции осиротевшей записи с `filing_status: submitted | filed` — **не предлагать удаление**, предупреждать «документ подан в суд, но файл в локальной ФС не найден — восстановить или обновить путь?».

### 5.6. Идемпотентность декларативна

[SKILL.md:104-105](../skills/update-index/SKILL.md): «Повторный запуск безопасен — уже синхронизированные файлы не будут обработаны повторно».

В чём выражается «безопасность»? Сценарии:
- Запуск N+1 сразу после N на тех же данных → не должно быть изменений ни в индексе, ни в зеркалах. **Проверить нечем** — нет постусловия «если изменений нет, `last_updated` не меняется» (сейчас `last_updated` обновляется в любом случае).
- Запуск N+1 после того, как Сюзерен ответил «нет» на preview → должен закончиться без изменений. В SKILL.md нет явного early-exit при пустом плане.
- Запуск N+1, когда preview показал «0 новых, 0 осиротевших, 0 устаревших» — должен выйти сразу. Не описано.

### 5.7. Конкурентный запуск (lock)

Аналогично [reviews/intake.md §5.7](intake.md): две сессии в одной папке дела → гонка на `next_id`, дубликаты. `update-index` особенно уязвим, т.к. именно он выдаёт новые ID при добавлении файлов.

**Правка:** lock-файл `.vassal/.index.lock` на время работы.

### 5.8. Пустой индекс при существующих файлах на диске

[SKILL.md:110](../skills/update-index/SKILL.md): «Пустой индекс → предложи запустить intake». Но что делать, если **файлы на диске уже есть**, а индекс пустой (папка дела скопирована с другого диска без `.vassal/`)? Логически — это массовое «новых файлов без записи». Формулировка неопределённая, модель дождётся пояснений.

### 5.9. Техмусор, побитые зеркала, ручное редактирование зеркал

Мелкие пробелы одного характера — «что скилл считает документом и что с ним делает, если пользователь что-то изменил руками»:
- Blacklist в [SKILL.md:31](../skills/update-index/SKILL.md) покрывает только `.DS_Store`, `Thumbs.db`, `Таблица документов.xlsx`. Реальный техмусор шире: `.tmp`, `~$*.docx` (Word lock-files), `*.crdownload`, `*.swp`, IDE-файлы. Их скилл предложит индексировать.
- Верификация ищет валидный frontmatter зеркала, но при его повреждении (Сюзерен открыл в редакторе, случайно сломал YAML) — политика не описана.
- Если Сюзерен вручную дополнил зеркало своими заметками, оригинал не тронут — update-index (по mtime зеркала) может предложить «пересоздать» и затрёт правки.

**Правка:** расширить blacklist; объявить зеркала «read-only для юриста» (или добавить поле `user_notes:` в frontmatter с гарантией сохранения при пересоздании); при невалидном frontmatter — предупреждать, не молча пересоздавать.

### 5.10. Перемещение между папками не обрабатывается отдельно

Сценарий: юрист переложил файл из `Материалы от клиента/Иное/` в `Материалы от клиента/Договоры/` (улучшил классификацию). Для update-index это одновременно «осиротевшая запись» + «новый файл». Технически покрывается §5.2 (переименование), но семантика другая — не имя, а папка. При матчинге по хэшу (P1.11) решается автоматически.

## 6. Реальные проблемы на практике

### 6.1. Скилл вызывается крайне редко — риск рассинхрона накапливается

[WORKFLOW-REVIEW.md §2-3](../WORKFLOW-REVIEW.md) формулирует это как системную проблему. Юрист:
- Скопировал в процессуальную папку документ, пришедший в Telegram. Не через intake. Запись в индексе не появилась.
- Переименовал зеркало вручную (название не нравилось). `file:` в индексе указывает на старое имя.
- Переложил файл между папками. Путь сломался.

Все эти случаи — **невидимы** для всех скиллов, кроме update-index. Но update-index не триггерится ниоткуда. Юрист забывает его вызывать. Дело постепенно «расходится» с индексом. Legal-review, prepare-hearing читают уже неактуальный индекс — делают выводы на неполных данных.

**Правка (системная, не чисто update-index):**
- Автовызов update-index перед каждым аналитическим скиллом (legal-review, build-position, prepare-hearing).
- Или: хук на начало любого скилла «проверь консистентность — если есть новые файлы в папке дела, пометь красным и предложи run update-index».

### 6.2. UX preview не масштабируется на большие объёмы

Пример preview в SKILL.md — 3 категории × 1-2 строки. Реальность большого дела, куда пользователь только что слил 30 файлов вручную: 30 новых + 5 осиротевших + 8 устаревших зеркал. Это уже длинный список для ручного разбора. В SKILL.md нет ни группировки (по папке, типу), ни суммаризации («30 файлов — партия платёжек, применить intake-скрипт?»), ни приоритизации (сначала процессуальные).

### 6.3. Тихое изменение зеркал = молчаливое изменение выводов

Пересоздание зеркала (§4.2, §5.9) — это тихое переписывание данных, на которых ранее строились выводы. Если `legal-review.md` уже ссылается на «Договор поставки №47» с конкретной суммой, а update-index пересоздал зеркало — скилл должен сигнализировать: «зеркало `doc-042` пересоздано; на него ссылаются `analysis/deadlines.md` и `2026-04-10 Обновленная позиция.md`; рекомендую перезапустить legal-review / build-position». Сейчас такого оповещения нет.

### 6.4. «Удалить запись» != «забыть про документ» + нет итогового отчёта

Даже если файл реально удалён вручную, документ мог быть важен (подан в суд, фигурирует в позиции). Полное удаление записи из индекса = потеря истории. Хотя бы soft-delete с `status: deleted, deleted_at: YYYY-MM-DD` был бы безопаснее; в [index-schema.yaml](../shared/index-schema.yaml) поля `status` нет.

Плюс: после apply SKILL.md ограничивается записью в `history.md`. Нет отчёта Сюзерену в чате («добавлено N, удалено M, пересоздано K зеркал, предупреждения: ...»). В сравнении с intake — UX беднее.

## 7. Рекомендации (приоритизированно)

### P0 (критично, без этого ненадёжно)

1. **Backup `index.yaml` перед apply** (§4.7): `index.yaml.bak.{timestamp}`, ротация N=5. Решает [WORKFLOW-REVIEW.md §9](../WORKFLOW-REVIEW.md). Без этого любая ошибка скилла необратима.
2. **Постусловия + валидация после apply** (§2.4): не просто список проверок, но explicit fail-at-violation + rollback из бэкапа.
3. **Определить «изменение файла» через `content_hash`** (§4.2): добавить поле `content_hash` в [index-schema.yaml](../shared/index-schema.yaml), использовать SHA-256. Единое решение с intake (см. [reviews/intake.md §7.P1.4](intake.md)).
4. **Детект orphan mirrors** (§4.4): сканировать `.vassal/mirrors/` отдельно, находить зеркала без записи.
5. **Политика rollback при partial failure в apply**: если 8 из 15 записей добавлены и скилл упал — восстановить из backup, откатить частично добавленные зеркала.
6. **Автотриггер update-index** (решается на уровне плагина, но критично для ценности именно этого скилла): перед любым аналитическим скиллом — проверка консистентности. Без этого update-index остаётся «мёртвой функцией».

### P1 (важно, влияет на корректность)

7. **Определение `source` для новых файлов в процессуальных папках** (§4.6): по контексту папки + обязательно спрашивать Сюзерена.
8. **Защита записей с `filing_status: submitted|filed`** от авто-удаления (§5.5).
9. **Soft-delete для осиротевших записей** (§6.4): поле `status`, `deleted_at` в [index-schema.yaml](../shared/index-schema.yaml).
10. **Сверка с «На удаление/»** перед предложением удалить запись (§4.5).
11. **Связывание переименований по хэшу** (§5.2) — автоматически, без вопросов.
12. **Оповещение о пересоздании зеркал, на которые ссылаются аналитические файлы** (§6.3).
13. **Обновление `last_verified` для всех проверенных записей**, не только модифицированных (§4.8).
14. **Lock-файл** для защиты от конкурентных запусков (§5.7).
15. **Промпт-шаблон для Haiku при категоризации новых файлов** — согласованный с intake (§5.1).

### P2 (UX)

16. **Группировка и суммаризация preview** на больших объёмах (§6.2).
17. **Финальный отчёт в чате** после apply (§6.4).
18. **Расширение blacklist + сохранение заметок в зеркалах + политика при побитом frontmatter** (§5.9).
19. **Early-exit при пустом плане изменений** (§5.6).
20. **Обработка перемещения между папками** (§5.10).

### P3 (архитектура)

21. **Программная реализация сверки** через `scripts/update_index.py` — ARCHITECTURE.md уже обещает скрипт, но его нет (§4.9, §5.3).
22. **Убрать `case.yaml` из предусловий** (или наполнить его использованием) — §4.1.
23. **Формализовать роль update-index в workflow**: либо сделать обязательным перед analytics-скиллами, либо описать как утилиту «по запросу». Сейчас — ни то, ни другое.

## 8. Резюме

Update-index — **страж, о котором все забывают**. Архитектурно правильная идея (единственная точка синхронизации ФС с `index.yaml`), но без триггера — не имеет ценности. Это не чисто проблема самого SKILL.md, а системный провал плагина: между «intake положил файлы в индекс» и «legal-review читает индекс» может пройти месяц ручных операций, за которые реальность и индекс разойдутся без единого сигнала.

Критические технические проблемы самого скилла:

1. **Нет backup'а** перед правкой — сломанный apply откатить нечем.
2. **Не определён способ детекции изменения файла** — модель импровизирует, на практике mtime будет ложно срабатывать или пропускать реальные изменения.
3. **Не детектируются orphan mirrors** — зеркала-призраки накапливаются, ломают выборки других скиллов.
4. **Нет политики rollback** при partial failure.
5. **`source` для новых файлов в процессуальных папках** определяется «на глазок» — это прямо ломает red/blue team анализ в prepare-hearing.

Главная архитектурная правка **вне самого скилла**: автоматический вызов update-index перед каждым аналитическим скиллом (legal-review, build-position, prepare-hearing). Без этого update-index остаётся факультативной утилитой, а целостность shared state — зависящей от дисциплины пользователя. Это прямое следствие [WORKFLOW-REVIEW.md §2-3](../WORKFLOW-REVIEW.md).

Главная техническая правка **внутри скилла**: backup перед apply + content-hash в схеме + постусловия с rollback. Это три небольших изменения, после которых update-index перестаёт быть «риском» и становится **надёжным инструментом**, которому можно безопасно доверить массовую операцию на большом деле.
