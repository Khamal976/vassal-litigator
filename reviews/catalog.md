# Ревизия скилла `catalog`

**Дата:** 2026-04-20
**Файлы:** [skills/catalog/SKILL.md](../skills/catalog/SKILL.md), [commands/catalog.md](../commands/catalog.md)
**Связанные:** [shared/conventions.md](../shared/conventions.md), [shared/index-schema.yaml](../shared/index-schema.yaml), [shared/mirror-template.md](../shared/mirror-template.md), [ARCHITECTURE.md §8.2 и §11](../ARCHITECTURE.md)
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** генерация человекочитаемой xlsx-таблицы всех документов дела + обогащение `index.yaml` краткими описаниями (`summary`) там, где они пусты или недостаточны. Попутно обещается заполнение `seal`, `signature`, `completeness`, `quality`.

**Позиция в workflow:** утилитарный скилл, формально стоит сразу после `intake` на критическом пути ([ARCHITECTURE.md:914](../ARCHITECTURE.md)). Два назначения одновременно:
1. **Для Сюзерена** — читаемая опись в `.xlsx` (главное — для юриста).
2. **Для агента** — обогащение `index.yaml` полем `summary`, от которого сильно выигрывают `legal-review` и `build-position` (меньше зеркал приходится читать целиком).

**Парные / последующие скиллы:**
- Вход: `intake` / `add-evidence` / `add-opponent` (они кладут документы в индекс).
- Выход: `legal-review` (формально зависит от `catalog` — см. [ARCHITECTURE.md:585](../ARCHITECTURE.md)), все остальные аналитические — косвенно через `summary` в индексе.
- Конкурент: `update-index` — тоже поддерживает индекс, но по другому измерению (соответствие файлам, а не качество описаний).

**Триггер запуска:** не формализован. В декларации — по запросу Сюзерена («сделай таблицу»). На критическом пути — «после intake». Автоматически ни одним скиллом не вызывается, хотя зависимость у `legal-review` прописана.

## 2. Контракт скилла

### 2.1. Вход (что читает)

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Контекст дела (SKILL.md §Шаг 1 п.1) — но фактически нигде не используется в алгоритме |
| `.vassal/index.yaml` | да | Все записи о документах |
| `.vassal/mirrors/{id}.md` | да (выборочно) | Для документов с пустым/коротким summary |
| `shared/conventions.md` | неявно | Таксономия типов, подпапки «Материалы от клиента» |
| `shared/index-schema.yaml` | неявно | Какие поля пишутся обратно в индекс |
| Внешний скилл `xlsx` | опционально | Альтернатива прямому вызову openpyxl |

### 2.2. Выход (что пишет)

| Результат | Путь | Для кого |
|---|---|---|
| Xlsx-таблица | `{корень дела}/Таблица документов.xlsx` | Сюзерен |
| Обогащённый индекс | `.vassal/index.yaml` (поля `summary`, `seal`, `signature`, `completeness`, `quality`) | Агент |
| Запись в историю | `.vassal/history.md` | Агент |
| Отчёт для Сюзерена | Ответ в чат (счётчики, ссылка на файл) | Сюзерен |

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/index.yaml` существует, не пуст (иначе — предложить `intake`).
- `.vassal/mirrors/` существуют (общие).

Не хватает:
- `.vassal/case.yaml` — упомянут в шаге 1, но не заявлен как предусловие.
- Проверка, что `openpyxl` установлен / доступен скилл `xlsx` (см. §4.6).
- Проверка, что индекс синхронизирован с ФС (см. §4.4).

### 2.4. Постусловия — отсутствуют

Должны быть, но не сформулированы:
- Файл `Таблица документов.xlsx` создан, открывается без ошибок, содержит ровно столько строк, сколько записей в индексе (+ строки-заголовки групп).
- `.vassal/index.yaml` остался валидным YAML; поле `last_updated` обновлено.
- Для каждой записи, где `summary` был пуст и зеркало существует — `summary` теперь не пуст.
- `next_id` не изменён (catalog ничего не создаёт, только правит существующие записи).
- Ни одна существующая непустая `summary` не перезаписана (заявлено в §Идемпотентность, но не проверяется).

Без формальных постусловий нельзя убедиться, что запуск отработал корректно — а значит `legal-review`, который зависит от качества `summary`, может получить «полу-обогащённый» индекс и не узнать об этом.

## 3. Качество промпт-инженерии

| Критерий | Оценка | Комментарий |
|---|---|---|
| Description (триггеры) | ✅ | Хорошо: позитивные («покажи все документы», «таблица», «опись») и негативный («НЕ для legal-review», «НЕ для обновления индекса без таблицы — это update-index») |
| Pre-conditions | ⚠️ | Указаны поверхностно — не хватает проверки зависимостей (openpyxl), актуальности индекса, существования case.yaml |
| Post-conditions | ❌ | Нет |
| Алгоритм структурирован | ✅ | Четыре шага, нумерованные пункты внутри |
| Идемпотентность | ⭐ | Есть отдельный раздел с ясным правилом: xlsx перезаписывается, существующие summary не трогаются |
| Граничные случаи | ⚠️ | Три кейса (нет зеркала, >100 документов, таблица существует). Реальных сценариев больше — см. §5 |
| Preview → Apply | ❌ | Скилл меняет `index.yaml` без показа плана Сюзерену, хотя [conventions.md:120-125](../shared/conventions.md) требует preview для всех изменений файлов дела |
| Делегирование Haiku | ⚠️ | Есть инструкция «делегируй Haiku», но нет шаблона промпта для субагента + нет согласования с Sonnet (см. §4.1) |
| Error-handling | ❌ | Что делать, если openpyxl упал, xlsx не записался, Haiku вернул пустой ответ — не описано |
| Валидация перед записью | ❌ | Нет чек-листа |
| Отчёт Сюзерену | ✅ | Чёткий список метрик: сколько в реестре, сколько обновлено, триаж по needs_manual_review, ссылка |
| Лимит контекста | ⚠️ | Правильно названо «пакеты по 10» при >100 документах, но нет инструкции, что грузить в Haiku-субагент — всё зеркало или только начало |
| Размер зеркала для саммари | ❌ | Нет ограничения на объём входа в Haiku: 300-страничный договор будет целиком подан → дорого, риск превышения контекста |

## 4. Противоречия и баги

### 4.1. Рассогласование моделей: Haiku пишет summary, Sonnet делает таблицу

[ARCHITECTURE.md:553](../ARCHITECTURE.md): «Модель: Haiku (саммаризация) + Sonnet (таблица)». В [SKILL.md:32](../skills/catalog/SKILL.md) явно: «Делегируй Haiku-субагенту».

Проблемы:
1. Haiku — более слабая модель. В шаблоне промпта для субагента ([SKILL.md:32](../skills/catalog/SKILL.md)) требуется «ключевые суммы, даты, стороны, обязательства» — нормальная задача для Haiku, но на сложных юридических текстах (многостраничный договор с кросс-ссылками, претензия со встроенным расчётом) она будет упрощать.
2. Sonnet, который собирает таблицу, **не перечитывает зеркало** — он берёт результат Haiku как готовый и переливает в ячейку. Если Haiku пропустил важное — Sonnet этого не заметит.
3. **Нет контроля согласованности.** Описание в summary может не совпадать с `type`, `parties`, `date` из индекса (Haiku может «увидеть» другую сумму или дату в тексте, но не имеет права править эти поля — только пишет в `summary`).

**Правка:** добавить пост-проверку «summary vs structured fields» — если Haiku упомянул сумму/дату, отличную от `index.yaml`, помечать `needs_manual_review: true`.

### 4.2. Поля, которые catalog обещает заполнять — не всегда в его юрисдикции

[SKILL.md:35](../skills/catalog/SKILL.md): «также убедись, что заполнены: `seal`, `signature`, `completeness`, `quality`. Если нет — извлеки из метаданных зеркала или прочитай начало зеркала».

Но:
- Ни `seal`, ни `signature` не входят в обязательные поля frontmatter [mirror-template.md:34-45](../shared/mirror-template.md). Там есть `confidence`, но не «подпись/печать». Значит, из «метаданных зеркала» этого извлечь нельзя — только из тела.
- `completeness` и `quality` по схеме [index-schema.yaml:26-27](../shared/index-schema.yaml) — `full/partial/fragment` и `high/medium/low` соответственно. В зеркалах нет этих полей. Определять через «прочитай начало зеркала» — слишком слабый критерий (качество печати на странице 1 не гарантирует качество на странице 20).
- `intake` формально обязан заполнять «все поля схемы» ([skills/intake/SKILL.md:93](../skills/intake/SKILL.md) — «Добавь запись для каждого файла со всеми полями (см. shared/index-schema.yaml)»), но явные инструкции там только про `origin`, `mirror_stale: false` и `needs_manual_review`. `seal / signature / completeness / quality` остаются на усмотрение модели → на практике заполняются неравномерно или не заполняются вовсе.

Итог: catalog — **единственный источник** этих полей, но его инструкции слишком размыты. В реальности данные будут заполнены произвольно, и юрист получит в таблице «Печать + подпись» для документа, где подписи нет.

**Правка:** либо эти поля заполняет `intake` на этапе создания зеркала (у него уже есть полный текст под рукой), либо вынести их в отдельный под-шаг catalog с явным чек-листом: «ищи слова `м.п.`, `подпись`, подпись-изображения (`[Изображение: подпись]`)».

### 4.3. Скрипт `scripts/generate_table.py` упомянут в ARCHITECTURE, но его нет

[ARCHITECTURE.md:97](../ARCHITECTURE.md) в древе скриптов плагина: «`generate_table.py` ← Генерация xlsx-таблицы из индекса».

Фактически в `scripts/` лежат `extract_text.py` и `setup.sh` — `generate_table.py` отсутствует. SKILL.md catalog **не ссылается** на него — предлагает альтернативу «скилл xlsx ИЛИ python (openpyxl)».

Последствия:
- ARCHITECTURE обещает скрипт, которого нет — дезинформирует читателя.
- Логика генерации живёт внутри SKILL.md (в голове модели каждый раз заново), а не в детерминированном скрипте. Значит форматирование, группировка, сортировка будут **отличаться от запуска к запуску** — юрист увидит визуальную разницу.

**Правка:** либо реализовать `scripts/generate_table.py` и ссылаться на него из SKILL.md (детерминированно), либо удалить упоминание из ARCHITECTURE.md.

### 4.4. Catalog не вызывает и не предполагает `update-index`

[SKILL.md:15-19](../skills/catalog/SKILL.md) требует только «существует index.yaml». Если юрист вручную добавил документ в папку дела (типовой сценарий из [WORKFLOW-REVIEW.md §3](../WORKFLOW-REVIEW.md)) — catalog его не увидит. Таблица будет неполной, summary — устаревшим.

При этом catalog **знает про таблицу на всех документах дела**, а не про «документы в индексе» — пользователь ожидает, что раз в папке есть файл, он будет в таблице.

**Правка:** в предусловиях вызывать `update-index` или хотя бы явно предупреждать: «я строю таблицу по index.yaml. Если вы клали файлы мимо `intake/add-*`, сначала запустите `update-index`». Это перекликается с [WORKFLOW-REVIEW.md §2](../WORKFLOW-REVIEW.md).

### 4.5. Имя файла `Таблица документов.xlsx` нарушает базовую конвенцию без фиксации исключения

[conventions.md:7](../shared/conventions.md): «Все файлы и папки в корне дела начинаются с даты ГГГГ-ММ-ДД». Пример там же: `{case-root}/2026-03-26 Предварительный анализ документов.md`.

Catalog пишет `Таблица документов.xlsx` — **без даты**. Исключение нигде не зафиксировано.

Возможные обоснования:
- Файл живой, переписывается при каждом запуске — дата теряет смысл.
- Юристу удобнее, когда таблица всегда по одному пути.

Оба обоснования разумны, но они должны быть **записаны** в conventions.md как исключение: «для постоянно обновляемых артефактов без версионирования — имя без даты (список на сейчас: `Таблица документов.xlsx`)».

Без такого исключения — следующий скилл (или ручная правка) может «поправить» имя по общему правилу → рассогласование.

Это продолжение той же проблемы, что отмечена в [reviews/legal-review.md §4.4](legal-review.md) и [reviews/build-position.md §4.2](build-position.md): **плагин систематически отступает от собственной конвенции** и не документирует это. Но здесь отступление ещё радикальнее — не только без «Отправителя», а вообще без даты.

### 4.6. Жёсткая зависимость от `openpyxl` / внешнего скилла без fallback

[commands/catalog.md:12](../commands/catalog.md): «Для генерации xlsx используй скилл `xlsx` (если доступен) или `openpyxl` напрямую».

Проблема: что если **ни того, ни другого нет**? Скилл либо упадёт, либо молча сгенерирует пустой файл, либо предложит csv — поведение не определено.

Это прямая параллель с жёсткой зависимостью `prepare-hearing` от `arbitrum-docx` ([WORKFLOW-REVIEW.md §4](../WORKFLOW-REVIEW.md)): оба скилла используют внешние инструменты для конечной сериализации, но:
- `prepare-hearing` хотя бы формально ссылается на один внешний плагин.
- `catalog` допускает два пути (`xlsx` или `openpyxl`), но без feature detection и без fallback на CSV/markdown-таблицу.

**Правка (минимум):** в предусловиях проверить доступность. Если нет — сохранить таблицу как `Таблица документов.md` (markdown-таблица, читается напрямую) или `.csv` с предупреждением: «openpyxl недоступен, таблица сохранена в markdown». Юрист хотя бы получит данные.

### 4.7. Словарь полей — русский в таблице, английский в схеме

Таблица ([SKILL.md:41-54](../skills/catalog/SKILL.md)) — русские заголовки («№», «Дата», «Тип», «Краткое содержание»).
Значения [index-schema.yaml:26-27](../shared/index-schema.yaml) — английские константы: `completeness: full | partial | fragment`, `quality: high | medium | low`.

SKILL.md предлагает в примерах «Полный», «Читаемый» — русские эквиваленты. Но **нигде не зафиксирована карта перевода** (`full → Полный`, `partial → Частичный`, `fragment → Фрагмент`? или «Полный / Частично / Фрагмент»? или что-то ещё?).

Результат: в таблице у одного документа будет «Полный», у другого «Полнота: full», у третьего «Полный документ» — зависит от импровизации Sonnet.

**Правка:** зафиксировать таблицу-перевод в SKILL.md или в `shared/conventions.md`:
```
completeness: full → "Полный", partial → "Частичный", fragment → "Фрагмент"
quality:      high → "Читаемый", medium → "Средний", low → "Плохой"
seal:         true → "Есть", false → "Нет"
signature:    true → "Есть", false → "Нет"
```

### 4.8. Колонка «Файл» показывает только путь, ничего про `doc-id`

[SKILL.md:54](../skills/catalog/SKILL.md) последняя колонка: «Файл — `file` (относительный путь)».

При этом вся связь индекса с другими скиллами идёт по `doc-id`. Если юрист увидит в таблице строку «Договор поставки №47», захочет сослаться на неё в переписке с Claude — он назовёт имя, Claude fuzzy-match'ом будет искать в индексе. В делах с 200+ документами и двумя редакциями одного договора это **реальный источник ошибок**.

**Правка:** добавить колонку `ID` (скрытая или узкая, `doc-042`) сразу после «№». Это ровно та же проблема двойной ссылки, что и в [reviews/legal-review.md §4.3](legal-review.md) — юристу удобно имя, агенту нужен id.

### 4.9. Нет preview перед обновлением `index.yaml`

[conventions.md:120-125](../shared/conventions.md): «Все команды, изменяющие файлы или данные дела, работают в два шага: Preview → Apply. Касается: intake, add-evidence, add-opponent, update-index (при удалении), обновлений case.yaml».

Casalog изменяет `index.yaml` (добавляет/обновляет `summary`, `seal`, `signature`, `completeness`, `quality`) и не показывает план Сюзерену. Формально это — нарушение принципа.

Контраргумент: изменения чисто аддитивные (пустое → заполненное), юрист не теряет данные. Но:
- Haiku может написать неправильный summary (модель не видит предмет спора, только текст документа).
- Если `seal/signature` проставляются автоматически по начальным страницам — юрист получит ложную информацию в таблице и будет её использовать.

**Правка:** preview в стиле «обновить summary для N документов, вот примеры первых 3 — подтвердить?». Для массовых запусков — опция «доверяю, выполни сразу».

### 4.10. Предусловие «summary < 20 символов» — слабый порог

[SKILL.md:30](../skills/catalog/SKILL.md): «где `summary` пуст или слишком короткий (< 20 символов)».

20 символов — произвольное число, не обосновано. Примеры на грани:
- «Договор поставки» — 16 символов → триггерит перегенерацию.
- «Платёжное поручение №123» — 24 символа → не триггерит, хотя информации столько же.

Если `intake` в будущем начнёт писать плейсхолдер «Документ типа X» (18-22 символа), поведение catalog будет непредсказуемо.

**Правка:** либо более содержательный критерий («summary не содержит ни одной суммы/даты/стороны, когда они есть в документе»), либо явный флаг в индексе `summary_status: draft | verified | stale`.

### 4.11. Группировка по типам — жёсткий список, не соответствует таксономии

[SKILL.md:59](../skills/catalog/SKILL.md): «Группировка по типам документов (договорные, переписка, процессуальные, платёжные, прочие)».

[conventions.md:87-101](../shared/conventions.md) содержит 6 групп: **Договорные**, **Переписка**, **Процессуальные**, **Судебные**, **Платёжные**, **Прочие**.

Catalog теряет группу **«Судебные»** (определения, решения, постановления, протоколы). В деле, где уже есть решение суда первой инстанции, оно попадёт либо в «Процессуальные» (неправильно — это не сторона дела), либо в «Прочие» (тоже неправильно).

**Правка:** синхронизировать список групп с `shared/conventions.md` или ссылаться на него как на единый источник.

## 5. Непокрытые edge-cases

### 5.1. Таблица открыта в Excel во время запуска

Типовая реальная ситуация: юрист смотрит таблицу, просит обновить. На Windows — файл залочен процессом Excel, `openpyxl` выдаст `PermissionError`. Catalog этого не ловит.

**Правка:** в начале Шага 3 — пробная запись. Если ошибка — попросить закрыть Excel и повторить.

### 5.2. Очень большой `summary` в существующей записи

По схеме `summary` — «1-3 предложения». Но ограничения нет. Если в ранее записи лежит 20-строчный текст — он попадёт в ячейку Excel как есть, разрушая визуальную структуру.

**Правка:** при вставке в xlsx — обрезать до N символов с многоточием и добавить wrapping.

### 5.3. Бандлы: упомянуты в требовании, не реализованы в алгоритме

[SKILL.md:60](../skills/catalog/SKILL.md): «Если есть бандлы — документы одного бандла идут подряд, приложения с отступом».

По схеме [index-schema.yaml:46-48](../shared/index-schema.yaml) бандл определяется полями `bundle_id`, `parent_id`, `role_in_bundle` + отдельная секция `bundles:` в индексе.

Но алгоритм catalog **нигде не читает эти поля**. Шаг 3 описывает колонки, включающие только `date`, `type`, `parties`, `title`, `summary`, `seal`, `signature`, `completeness`, `quality`, `file`. Бандлы в этом списке отсутствуют. Инструкция «документы одного бандла идут подряд» без чтения `bundle_id` — пустая декларация.

**Правка:** в Шаге 3 явно: «сначала сгруппируй документы по `bundle_id` (main перед appendix). Внутри группы — по `parent_id`. Для документов с `role_in_bundle: appendix` — отступ в колонке `Название` (префикс `→` или indent на уровне xlsx)».

### 5.4. Документы с `date: null` (дата неизвестна)

По конвенции [conventions.md:33](../shared/conventions.md) таким документам присваивается имя `0000-00-00 Неизвестный ...`. В индексе поле `date: null`. Catalog должен решить:
- Куда в отсортированной таблице их поставить? (В начало как «неизвестные»? В конец?)
- Как отобразить в колонке «Дата»? Пустое? «н/д»?

Ничего не описано.

### 5.5. Документ от двустороннего взаимодействия — кто «Отправитель → Адресат»

Колонка «Отправитель → Адресат» из `parties`. Но [index-schema.yaml:15-20](../shared/index-schema.yaml) говорит: `parties` может быть списком строк (для договоров) или структурой `{from, to}` (для писем).

Для двустороннего договора список — `["ООО «Ромашка»", "ООО «Лютик»"]`. Куда это вывести в «Отправитель → Адресат»?

Алгоритм catalog не описывает обработку двух форматов.

**Правка:** либо ввести отдельную колонку «Стороны» для договоров и «Отправитель → Адресат» для переписки, либо явное правило: «если `parties` — список: соедини через ` — `; если `{from, to}` — как есть через ` → `».

### 5.6. Конфликт двух запусков в один день

Два запуска подряд → один и тот же файл `Таблица документов.xlsx` перезаписан, но **Excel у юриста открыт**: старая версия видна, новая на диске — рассинхрон.

### 5.7. Генерация пакетами по 10 — сохранение между пакетами

[SKILL.md:79](../skills/catalog/SKILL.md): «>100 документов → генерируй summary пакетами по 10».

Если пакет 3 из 10 упал на полпути — что с пакетами 1-2? Сохранены ли они в `index.yaml`? Ничего не сказано. Вероятный риск: модель сначала собирает все summary в память, потом пишет всё за раз → при сбое теряем всё.

**Правка:** явно «после каждого пакета — сохраняй index.yaml». Тогда при сбое — просто повторный запуск доделает.

### 5.8. Документ в процессуальной папке (подан в суд)

Документы с `filing_status: submitted/filed` живут в процессуальных папках ([conventions.md:37-43](../shared/conventions.md)), не в «Материалы от клиента/». Colonка «Файл» покажет путь, но таблица не подсвечивает процессуальный статус.

**Правка:** либо отдельная колонка «Статус» (подан / черновик / не подан), либо визуальное отделение — процессуальные документы в отдельной группе в начале.

### 5.9. Саммари-сессии не помнят друг друга

При повторном запуске catalog (скажем, после `add-evidence`) Haiku пишет summary для новых документов **в вакууме** — не учитывая уже известные стороны, предмет спора, существующие описания по соседним документам. Описания получаются разностильные.

**Правка:** в промпт субагента включать выдержку из `case.yaml` (стороны, предмет, суммы) + 2-3 примера хороших summary уже существующих документов. Консистентность стиля важна — это же читает юрист.

### 5.10. Нестрого monotonic `last_updated`

`index.yaml` имеет поле `last_updated`. Catalog обязан его обновить, если меняет поля. Не описано.

## 6. Реальные проблемы на практике

### 6.1. Непонятно, когда юрист должен вызывать catalog

На критическом пути — «после intake». Но:
- `add-evidence` и `add-opponent` не триггерят catalog автоматически. Таблица устареет сразу после первого доп. приёма документов.
- Если юрист получает 5 партий документов подряд и каждый раз запускает `add-evidence`, потом `catalog`, потом следующий `add-evidence`, потом `catalog` — это много ручных шагов.

**Правка (архитектурная):** либо добавить автовызов catalog в конце `intake/add-evidence/add-opponent` (с флагом пропуска), либо оставить ручным, но честно писать в истории «таблица устарела».

### 6.2. Дорогое обогащение — не сохраняется между плагиновыми сессиями

Haiku для 50 документов — 50 API-вызовов. Если юрист случайно запускает catalog дважды за неделю с небольшими дополнениями, второй запуск **не дублирует** работу (идемпотентность описана) — это хорошо. Но:
- Если юрист удалил строку summary в `index.yaml` (например, хотел улучшить формулировку) — catalog её перегенерирует. Ручные правки теряются.
- Нет поля `summary_source: haiku | user`, чтобы пометить «это юрист правил, не трогай».

Это та же проблема потери дорогой работы, что и в [reviews/prepare-hearing.md §4.1](prepare-hearing.md), но в мягкой форме.

### 6.3. Вывод в xlsx для юриста ≠ удобная работа

Таблица из 10 колонок при 100+ строках — это не самый удобный формат. Юристу часто нужно:
- Фильтр по типу документа (чтобы посмотреть всю переписку).
- Сортировку по дате.
- Пометки «это я уже прочитал» / «это спорное».

xlsx форма умеет фильтры и AutoFilter — но SKILL.md не требует их включать. [SKILL.md:56-58](../skills/catalog/SKILL.md) просит только жирный заголовок, freeze panes, автоширину. **AutoFilter** — стандартный для Excel инструмент, без него таблица слепая.

**Правка:** в форматирование — включить `ws.auto_filter.ref = "A1:J1"`.

### 6.4. Колонки «Полнота / Качество» — юристу бесполезны, но занимают место

Юрист не работает с документом в терминах «high/medium/low читаемости» — он работает с конкретикой («пропущена страница», «подпись нечитаема»). Поля в таблице — артефакт агентской кухни, вытекающий напрямую в UX.

**Правка:** либо заменить две колонки на одну объединённую «Замечания» (короткая человеческая формулировка: «Нечитаем хвост», «Без подписи», «Скан низкого качества»), либо спрятать за ширмой (отдельный лист / фильтр «только проблемные»).

### 6.5. Нет экспорта «для клиента»

Юристу часто нужен **сокращённый** реестр документов для клиента (без внутренних оценок completeness/quality). Catalog этого не умеет — только полная таблица.

### 6.6. Переход между Excel-файлом и вложенными папками невозможен

В колонке «Файл» лежит текстовый путь. В xlsx можно сделать hyperlink → клик, откроется. SKILL.md об этом не говорит.

**Правка:** превратить `file` в `=HYPERLINK("...")` или использовать `cell.hyperlink` в openpyxl.

## 7. Рекомендации (приоритизированно)

### P0 (критично — блокирует надёжность или корректность данных)

1. **Определить постусловия и валидацию** (см. §2.4). Минимум: xlsx существует и парсится; index.yaml валиден; количество строк в таблице = количество документов в индексе; `last_updated` обновлён.
2. **Fallback для отсутствующего openpyxl** (§4.6): сохранять markdown-таблицу (или csv) с предупреждением.
3. **Разграничить ответственность с `intake` по полям `seal/signature/completeness/quality`** (§4.2). Предложение: перенести в `intake`, так как там доступ к полному тексту зеркала.
4. **Добавить preview перед обновлением `index.yaml`** или явное исключение в conventions.md (§4.9).
5. **Синхронизировать группировку в таблице с `shared/conventions.md`** (§4.11) — добавить «Судебные».

### P1 (важно — влияет на качество и согласованность)

6. **Зафиксировать исключение имени `Таблица документов.xlsx`** в conventions.md как единый источник правды (§4.5).
7. **Добавить колонку `ID` с doc-id** (§4.8). Узкая, опциональная, но обязательная в данных — для надёжной адресации.
8. **Карта перевода enum-полей** (full/partial/fragment, high/medium/low) — в conventions.md (§4.7).
9. **Реализовать `scripts/generate_table.py`** или удалить упоминание из ARCHITECTURE.md (§4.3) — сейчас это ложная документация.
10. **Автовызов `update-index` в предусловиях** или явное предупреждение (§4.4).
11. **Реализация бандлов в алгоритме** (§5.3): группировка по `bundle_id`, отступы для приложений.
12. **Обработка двух форматов `parties`** — список vs `{from, to}` (§5.5).
13. **Сохранение после каждого пакета** при >100 документов (§5.7).
14. **Промпт для Haiku с контекстом дела** (стороны, предмет) для согласованности стиля (§5.9).
15. **Проверка на занятость файла Excel'ом** перед записью (§5.1).

### P2 (UX)

17. **Объединить `completeness/quality` в человечную колонку «Замечания»** (§6.4).
18. **Отдельная колонка / группа для документов с `filing_status`** (§5.8).
19. **Ограничение длины ячейки `summary`** с многоточием (§5.2).
20. **Защита ручных правок `summary`**: поле `summary_source: haiku|user` (§6.2).
21. **Критерий «нужно ли обновлять summary»** — от порога по символам к содержательному (§4.10).
22. **Обработка `date: null`** — отдельная группа «Без даты» в таблице (§5.4).

### P3 (архитектурные, связанные с другими скиллами)

23. **Автовызов catalog в конце `intake/add-evidence/add-opponent`** (с возможностью отключить) — чтобы таблица не устаревала (§6.1).
24. **Экспорт «для клиента»** — сокращённая таблица без внутренних полей (§6.5).
25. **Нотация согласованности «summary vs structured fields»** — сверка Haiku-саммари с type/date/parties (§4.1).

## 8. Резюме

Catalog — формально **самый простой** скилл плагина: читает индекс, зовёт Haiku, рисует xlsx. Но за простотой скрывается ряд системных огрехов. Сильные стороны — всего две: чёткая идемпотентность (описана как отдельный раздел, правило «не перезаписываем непустой summary» правильное) и хороший description с позитивными/негативными триггерами.

Проблемы более существенные, чем кажется:

- **Размытая юрисдикция.** `seal / signature / completeness / quality` — catalog обещает заполнять их из зеркал, но в зеркалах этих данных нет; правильное место — `intake`, где есть полный текст. Сейчас эти поля заполняются наугад.
- **Бандлы и судебные документы — «на бумаге».** Форматирование таблицы обещает группировку по бандлам и по типам, но алгоритм ни того, ни другого не реализует: секция про бандлы ничего не читает из `bundle_id/parent_id`, а список групп расходится с таксономией в conventions.md.
- **Нарушение preview→apply** и произвольное имя файла **без фиксации исключения** — ровно тот же паттерн, что и в legal-review / build-position, но здесь особенно заметно.
- **Жёсткая зависимость от openpyxl / внешнего `xlsx`** без fallback.
- **Связь с `update-index` не установлена**: catalog работает по индексу, но сам не проверяет синхронизацию индекса с ФС.

Три ключевые правки, закрывающие критическую массу:
1. **Перенести `seal/signature/completeness/quality` в `intake`**, убрать эту ответственность из catalog (однозначная юрисдикция вместо двух полу-реализаций).
2. **Fallback openpyxl → markdown/csv** + постусловия + реальная реализация бандлов и единая группировка с conventions.md.
3. **Колонка `ID`, AutoFilter, hyperlinks и карта перевода enum → русский** — минимальные правки, делающие таблицу реально полезной юристу и машиночитаемой одновременно.
