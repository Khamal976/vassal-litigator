# Ревизия скилла `build-position`

**Дата:** 2026-04-20
**Файлы:** [skills/build-position/SKILL.md](../skills/build-position/SKILL.md), [commands/build-position.md](../commands/build-position.md)
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** выработка структурированной правовой позиции — требования, доказательственная база, сильные/слабые стороны, прогноз аргументов оппонента, оценка рисков, стратегия усиления.

**Позиция в workflow:** второй (и последний) скилл фазы «Анализ», после `legal-review`. Результаты читают `prepare-hearing` (перед каждым заседанием), `appeal` (при обжаловании), `cassation`.

**Парные / последующие скиллы:**
- Вход: `legal-review` (мягкая зависимость — работает без, но хуже).
- Выход: `prepare-hearing` (основной потребитель), `appeal`, `cassation`.

**Циклический скилл:** запускается не один раз, а при каждом существенном изменении — новые документы от клиента/оппонента, решение суда, результат заседания. Цепочка позиций (`supersedes`) отражает эволюцию стратегии.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Стороны, роль клиента, предмет |
| `.vassal/index.yaml` | да | Реестр документов |
| `.vassal/analysis/*.md` | нет (рекомендовано) | Результаты legal-review |
| `.vassal/mirrors/*.md` (ключевых) | да | Зеркала договоров, актов, претензии, процдокументов |
| Предыдущие позиции | нет | Для обновления — **путь противоречив, см. §4.1** |
| Вводная Сюзерена | нет | Роль, акценты, конкретные требования |

### 2.2. Выход

| Файл | Путь |
|---|---|
| Позиция (для Сюзерена) | `{корень}/{ГГГГ-ММ-ДД} {тип} позиция.md` (тип: Начальная / Обновленная / Ответ) |
| Обновление карточки дела | `.vassal/case.yaml` (поля «если стали ясны» — см. §4.3) |
| Запись в истории | `.vassal/history.md` |

**Чего НЕТ на выходе** (но должно быть — см. §4.4):
- Отдельных файлов для агента по разделам (доказательственная база, риски и т.д.).
- Структурированного списка приложений для prepare-hearing.
- Diff-файла между позициями.

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/case.yaml` существует.
- Документы приняты и проиндексированы.
- Рекомендуется `legal-review` (мягкая зависимость).

### 2.4. Постусловия — ❌ нет

Должны быть:
- Файл позиции создан и валиден (YAML frontmatter парсится).
- Все ссылки на документы в позиции соответствуют записям в `index.yaml`.
- `supersedes` корректно указывает на файл предыдущей позиции (если обновление).
- Обновлённая позиция читается `prepare-hearing` как «последняя» по правилу выбора.

## 3. Качество промпт-инженерии

| Критерий | Оценка | Комментарий |
|---|---|---|
| Description (триггеры) | ✅ | Позитивные + негативные («НЕ для legal-review, НЕ для prepare-hearing») |
| Pre-conditions | ✅ | Есть, с явным указанием мягкой зависимости |
| Post-conditions | ❌ | Нет |
| Аудируемый формат выводов | ⚠️ | Декларирован, но в шаблоне используются таблицы (без audit-полей) — нестыковка (см. §4.5) |
| Декомпозиция на блоки | ✅ | 7 блоков (a-g) — предмет / доказательства / сильные / слабые / прогноз / риски / стратегия |
| Preview-формат | ✅ | Понятный, компактный |
| Идемпотентность | ⚠️ | «Каждый запуск — новый файл с датой», но нет правила для двух запусков в один день (§4.6) |
| Цепочка позиций (`supersedes`) | ⭐ | Хорошая идея — chain of evolution. Но никто не читает (§4.7) |
| Граничные случаи | ⚠️ | 5 случаев, но не покрыт встречный иск, солидарные стороны, нестандартные типы |
| Error-handling | ❌ | Что делать, если `analysis/deadlines.md` пропущен — не детализировано |
| Двухуровневое сохранение (для агента + для человека) | ❌ | В отличие от legal-review — только один файл для обоих |

## 4. Противоречия и баги

### 4.1. **Главный баг:** расхождение путей к позициям внутри SKILL.md

В **одном и том же SKILL.md** — два противоречащих указания:

- [SKILL.md:48](../skills/build-position/SKILL.md) (фаза 1 п.5): «Загрузи предыдущие позиции из **`.vassal/positions/`**»
- [SKILL.md:213](../skills/build-position/SKILL.md) (обновление позиции, п.1): «Загрузи предыдущую позицию из **case root folder**».
- [SKILL.md:151](../skills/build-position/SKILL.md) (фаза 4): сохраняет в **case root folder** `{ГГГГ-ММ-ДД} {тип} позиция.md`.

Факты:
- `.vassal/positions/` создаётся [init-case:36](../commands/init-case.md), но **никто туда не пишет**.
- Корень дела используется для позиций всеми остальными скиллами (`prepare-hearing` ищет `{корень}/*позиция.md`).

Значит:
1. Папка `.vassal/positions/` — **мёртвая** (артефакт устаревшего дизайна).
2. Фаза 1 п.5 `build-position` ссылается на мёртвую папку → при повторном запуске ничего не найдёт → не увидит предыдущей позиции → построит с нуля, а не обновит.

**Правка:** заменить `.vassal/positions/` на корень дела во всех упоминаниях в SKILL.md + удалить создание `.vassal/positions/` из init-case.

Это не просто ошибка в тексте — это **реальный функциональный баг идемпотентности**.

### 4.2. Конвенция имени отклоняется от общего правила, но не зафиксирована

[conventions.md:24](../shared/conventions.md): общий формат `ГГГГ-ММ-ДД Отправитель Описание.ext`.

Имя позиции: `{ГГГГ-ММ-ДД} {тип} позиция.md` — где `{тип}` = `Начальная | Обновленная | Ответ`. Нет «Отправителя».

Пример в [conventions.md:11](../shared/conventions.md): «`2026-03-26 Начальная позиция.md`» — показан как пример, но правило именования продуктов плагина в conventions.md не описано.

Это третий случай (после `{дата} Предварительный анализ.md` в legal-review и `{дата} {тип документа}.md` в prepare-hearing), где имена продуктов плагина отличаются от общего правила. **Нужно зафиксировать отдельный раздел в conventions.md:**

> «Продукты плагина (анализ, позиция, процессуальные документы) именуются по шаблонам per-skill. Общее правило `Отправитель Описание` применяется только к документам сторон (client / opponent / court).»

### 4.3. Обновление `case.yaml` — какие поля?

[SKILL.md:160](../skills/build-position/SKILL.md): «Обнови `case.yaml` — заполни оценочные поля, если они стали ясны».

В [case-schema.yaml](../shared/case-schema.yaml) **оценочных полей нет**. Есть факты (pretrial.complied, суммы, timeline). Что именно обновляет `build-position`?

Возможные кандидаты:
- `amounts.claim` (если позиция скорректировала сумму).
- `pretrial.required/complied` (если анализ подтвердил/опроверг).
- Что-то ещё?

Как и в legal-review (§4.1 ревизии), SKILL.md ссылается на поля, которые не формализованы в схеме. В результате модель импровизирует.

**Правка:** либо расширить схему полями вида `strategy.prospects`, `strategy.key_risks`, либо в SKILL.md явно перечислить — какие существующие поля он трогает.

### 4.4. Нет двухуровневого сохранения (для человека + для агента)

[legal-review](../skills/legal-review/SKILL.md) сохраняет:
- **Для Сюзерена:** один сводный файл в корне.
- **Для агента:** 7 отдельных файлов в `.vassal/analysis/`.

`build-position` сохраняет **только один файл в корне** — он одновременно и для юриста, и для агента.

Проблема: при `prepare-hearing` агенту нужны конкретные блоки (доказательственная база, прогноз оппонента, риски). Он должен парсить «человечный» `.md` → ненадёжно, зависит от форматирования.

**Правка:** добавить параллельное сохранение в `.vassal/position/` (или `.vassal/analysis/position-*.md`):
- `evidence-base.md` — таблица доказательств
- `strengths.md`
- `weaknesses.md`
- `opponent-forecast.md`
- `risks.md`
- `strategy.md`

Тогда `prepare-hearing` читает машинно-структурированные файлы, а не парсит человечный.

### 4.5. Аудируемый формат vs таблицы

[SKILL.md:53](../skills/build-position/SKILL.md): «Каждый вывод — в аудируемом формате».

Но в шаблонах:
- **Доказательственная база** — таблица: Факт / Бремя / Доказательства / Достаточность / Что усилит.
- **Оценка рисков** — таблица: Требование / Вероятность / Обоснование / Риск.

В таблице нет полей «Уверенность», «Недостающие данные», и таблица плохо вмещает «Фактическое основание» + «Правовое основание» одновременно.

Что реально делать: держать аудируемый формат для **текстовых выводов** (сильные стороны, стратегия, прогноз), а таблицы — для **структурных сводок** (доказательства, риски). Но это нужно явно разрешить в SKILL.md, иначе модель путается.

### 4.6. Идемпотентность: что при двух запусках в один день

[SKILL.md:220](../skills/build-position/SKILL.md): «Каждый запуск создаёт новый файл с датой».

Но если запуск повторён тем же днём (скорректирована вводная) — файлы `2026-04-20 Начальная позиция.md` совпадут. Что происходит:
- Перезапись? (теряется первая)
- Суффикс `(2)`?
- Добавление времени в имя (`2026-04-20 14-30 ...`)?

Не описано.

### 4.7. `supersedes` в frontmatter — никто не читает

Позиция содержит `supersedes: "{путь к предыдущей}"`. Хорошая идея — цепочка эволюции.

Но:
- `prepare-hearing` ищет позиции по glob `*позиция.md` и берёт «последнюю» — не по `supersedes`, а по дате в имени.
- `appeal`, `cassation` — «все позиции из case root».
- Нет ни одного скилла, который следовал бы цепочке `supersedes`.

`supersedes` — **неиспользуемые метаданные**. Либо добавить скилл-навигатор по цепочке, либо убрать поле.

### 4.8. Словарь `{тип}` русско-английский

- Имя файла: `Начальная | Обновленная | Ответ` (русский).
- Frontmatter `subtype`: `initial | updated | response-to-{документ}` (английский).

Неконсистентно. Легче для людей — русский, легче для машины — английский. Нужно выбрать одно или явно мапить.

### 4.9. Оценка вероятностей в процентах — опасно

[SKILL.md:106-110](../skills/build-position/SKILL.md):

| Требование | Вероятность удовлетворения |
|---|---|
| Основной долг | высокая (80%) |
| Неустойка | средняя (50%) |

LLM не калиброван как вероятностная модель. Давая цифру `80%`, создаёт ложное впечатление точности. Юрист может скопировать цифру в консультацию клиенту → клиент воспринимает как гарантию.

**Правка:** убрать числовые проценты из шаблона. Оставить только словесные шкалы («высокая / средняя / низкая») + обязательное обоснование. Если нужна калибровка — отдельный инструмент.

### 4.10. Прогноз аргументов оппонента дублируется с `prepare-hearing`

[SKILL.md:97-100](../skills/build-position/SKILL.md) п.e: прогноз аргументов оппонента, на чём основаны, как отвечать.

[prepare-hearing](../skills/prepare-hearing/SKILL.md) фаза 2 п.11 — Red team: контраргументы, слабости, доказательства оппонента.

Пересекающиеся функции. Непонятен контракт:
- `build-position` = рамочный стратегический прогноз.
- `prepare-hearing` = тактический прогноз к конкретному заседанию.

Но это различие нигде не прописано. Скиллы дублируют работу.

**Правка:** в SKILL.md явно: «в `build-position` — **стратегический** прогноз (какие аргументы оппонент может выдвинуть в принципе); в `prepare-hearing` — **тактический** (какие аргументы оппонент **уже** выдвинул в процдокументах + что ещё может добавить к конкретному заседанию)».

### 4.11. Расчёт суммы требования — Opus считает

[SKILL.md:62](../skills/build-position/SKILL.md) п.a: «Расчёт суммы требования (если возможно из документов)».

Opus — плохой калькулятор. Суммы по УПД, проценты по 395 ГК, неустойка — арифметика. Делегировать Python-скрипту или минимум явно сказать «показать формулы и промежуточные суммы, не только итог».

## 5. Непокрытые edge-cases

### 5.1. Встречный иск

В граничных случаях упомянут: «Встречный иск → включи анализ и требований, и встречных требований». Но:
- Одна позиция или две?
- Цепочка `supersedes` — как?
- Имя файла — `{дата} Встречный Позиция.md`?

Без правил — путаница в реальном деле с встречкой.

### 5.2. Солидарные должники / соистцы

Наш клиент — один из 3 ответчиков (например, поручитель). Позиция должна:
- Учитывать позицию других ответчиков (они могут быть против клиента).
- Понимать, что часть аргументов — «общая защита всех ответчиков», часть — «только наша».

SKILL.md не различает.

### 5.3. Третьи лица

Наш клиент — третье лицо. Позиция устроена иначе (нет требований, есть заявления о правах/возражения).

### 5.4. Ответ на конкретный документ

`subtype: response-to-{документ}` — указывает, на какой документ отвечает. Но:
- В имени файла — `{тип} позиция.md` → `Ответ позиция.md`. Без указания на что именно.
- Если Сюзерен ответит на отзыв ответчика, потом на возражения на отзыв — два файла `Ответ позиция.md`? Коллизия.

**Правка:** формат — `{дата} Ответ на {документ} позиция.md` (например, «Ответ на отзыв ответчика»).

### 5.5. TTL / устаревание позиции

Дело развивается — позиция устаревает. Если с момента `build-position` прошло 2 месяца и в индекс добавились документы → `prepare-hearing` использует устаревшую позицию.

Нет флага «устарела», нет детектора («новые документы после даты позиции»). Это продолжение проблемы §5.3 из [reviews/legal-review.md](legal-review.md).

### 5.6. Нестандартные требования

- Признание права.
- Обязание совершить действие.
- Расторжение договора.
- Понуждение к заключению договора.

Шаблон «требования → суммы» не работает. В граничных случаях упомянуто, но без раскрытия.

### 5.7. Изменение роли в процессе

Сценарий: клиент подал встречный иск → стал одновременно истцом по встречному требованию и ответчиком по основному. В `case.yaml` `our_client.party_id` — одна роль. SKILL.md рассматривает истец/ответчик как взаимоисключающие.

## 6. Реальные проблемы на практике

### 6.1. Юрист читает позицию как аналитику, а не алгоритмически

`{дата} Начальная позиция.md` — объёмный документ (в реальности 10-30 страниц). Юрист читает его целиком. Но структура тяжёлая: 7 разделов, таблицы, audit-формат.

Для **чтения** нужен резюмирующий раздел в начале (TL;DR на пол-страницы): требования, перспективы, ключевые риски, 3 главные рекомендации. Сейчас этого нет.

### 6.2. Цепочка позиций затрудняет историю

После 10 итераций в корне: `2026-03-01 Начальная.md`, `2026-03-15 Обновленная.md`, `2026-04-01 Обновленная.md`, `2026-04-10 Ответ.md`, `2026-04-20 Обновленная.md`. Как быстро понять «что поменялось за последний месяц»? Нет:
- Хронологического index'а.
- Выделения diff'ов.

**Решение:** `.vassal/positions-timeline.md` — короткий лог с датами и 1-строчным описанием изменения.

### 6.3. Модель даст процент уверенности — юрист забудет, что это не статистика

Даже если юрист понимает, что `80%` — экспертная оценка модели, не вероятность. Клиенту он это не объяснит. Это **риск злоупотребления артефактом**.

### 6.4. Нет «что делать дальше» раздела

Стратегия усиления есть (п.g). Но нет явного «ACTION ITEMS: (1) запросить у клиента акт сверки до 20.04; (2) заявить ходатайство об экспертизе…». Юрист потом сам выписывает из текста — неэффективно.

### 6.5. Большое дело — позиция размывается

При деле с 50 требованиями (например, пакет исков по взысканию задолженности с портфеля контрагентов) — таблицы становятся огромными, audit-формат каждого вывода — нечитаемым. Скилл не масштабируется.

## 7. Рекомендации (приоритизированно)

### P0 (критично)

1. **Починить расхождение путей**: в SKILL.md везде корень дела, не `.vassal/positions/`. Удалить создание `.vassal/positions/` из init-case (§4.1).
2. **Постусловия + валидация** (§2.4).
3. **Убрать числовые проценты вероятностей** — оставить словесную шкалу (§4.9).
4. **Разрешить противоречие `Начальная / initial`**: выбрать один словарь (§4.8).
5. **Формат имени для «Ответ» позиций** — с указанием на что отвечает (§5.4).

### P1 (важно)

6. **Двухуровневое сохранение** (как в legal-review): машинно-читаемые файлы в `.vassal/position/` + сводный в корне (§4.4).
7. **Развести зоны с prepare-hearing**: стратегический vs тактический прогноз (§4.10).
8. **Согласовать `case.yaml` обновления** со схемой (§4.3).
9. **Зафиксировать конвенцию имён продуктов плагина** в conventions.md (§4.2).
10. **TTL / детект устаревания** позиции (§5.5).
11. **Расчёт сумм через Python** или с явным показом формул (§4.11).
12. **Идемпотентность: два запуска в один день** (§4.6).
13. **TL;DR-раздел в начале позиции** (§6.1).

### P2 (UX)

14. **`.vassal/positions-timeline.md`** с хронологией изменений (§6.2).
15. **ACTION ITEMS раздел** (§6.4).

## 8. Резюме

Build-position — **стратегический центр** плагина между фактической аналитикой (`legal-review`) и генерацией процдокументов (`prepare-hearing`). По замыслу — хороший скилл с разумной декомпозицией на 7 разделов.

По исполнению — **самый противоречивый**:
- **Баг идемпотентности** (расхождение путей внутри одного SKILL.md) → обновления позиции фактически не работают.
- **Процентные оценки вероятностей** — опасная практика, которая может попасть в консультацию клиенту.
- **Отсутствие машинно-читаемого выхода** — в отличие от legal-review, вся позиция в одном «человечном» файле, `prepare-hearing` вынужден её парсить.
- **Словарь типов не согласован** (русский vs английский).
- **Метаданные `supersedes` не используются**.

Три правки закрывают критическую массу:
1. Единый путь к позициям (корень дела везде).
2. Двухуровневое сохранение (`.vassal/position/` + корень).
3. Уход от числовых процентов к словесным шкалам с обоснованием.
