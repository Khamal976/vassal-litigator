# Ревизия скилла `draft-judgment`

**Дата:** 2026-04-20
**Файлы:** [skills/draft-judgment/SKILL.md](../skills/draft-judgment/SKILL.md), [commands/draft-judgment.md](../commands/draft-judgment.md)
**Связанные:** [ARCHITECTURE.md §8.10, §10](../ARCHITECTURE.md), [skills/appeal/SKILL.md](../skills/appeal/SKILL.md), [skills/analyze-hearing/SKILL.md](../skills/analyze-hearing/SKILL.md)
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** составление проекта судебного решения по делу до его вынесения. При наличии 3-5 решений конкретного судьи по аналогичным делам — адаптирует стиль, структуру и формулировки. Метод опирается на «цифровой профиль судьи» (Мошкин С.А., runetlex.academy).

**Позиция в workflow:** скилл фазы 4 «Завершение и обжалование» по [ARCHITECTURE.md](../ARCHITECTURE.md:1000-1006), **но** на схеме он стоит **до** `appeal` и **после** `analyze-hearing`. Это внутренне противоречиво: если решение уже вынесено — зачем проект решения? А если не вынесено — почему скилл «завершения»?

Судя по содержанию SKILL.md (использование проекта «глазами суда», адаптация стиля, прогноз исхода) — скилл **префактумный**: готовится до вынесения решения. Но конкретный триггер («за сколько до заседания», «перед прениями», «в любой момент после сформированной позиции») — не зафиксирован. Это не процедурный скилл, а **аналитико-прогнозный**.

**Парные / последующие скиллы:**
- Вход: `build-position` (позиция), `prepare-hearing` (аргументы на заседании), `analyze-hearing` (профиль судьи из прошедших заседаний по этому делу), `legal-review` (квалификация, нормы).
- Выход: **отсутствует явная связь с `appeal`.** Если проект предсказывал отказ, а суд удовлетворил — или наоборот, — нет механизма сравнения «что предсказали vs что вышло». См. §4.8.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Стороны, суд, судья, предмет |
| `.vassal/index.yaml` | да | Реестр документов |
| `{корень}/*позиция.md` | да | Позиции — основа описательной части |
| `.vassal/hearings/*.md` | нет | Контекст прошедших заседаний |
| `.vassal/hearings/judge-profile.md` | нет | Профиль судьи из `analyze-hearing` (по этому делу) |
| `.vassal/hearings/judge-style.md` | нет | Стиль из предыдущих запусков `draft-judgment` — **но правил синхронизации нет** (§4.3) |
| `.vassal/mirrors/*.md` (ключевых) | да | Зеркала доказательств обеих сторон |
| `.vassal/analysis/*.md` | нет | Аналитика от `legal-review` |
| **Решения судьи по аналогичным делам** | нет | 3-5 .pdf/.docx для анализа стиля — источник не формализован (§5.1) |
| Вводная Сюзерена | да | В чью пользу, исход, особые указания |

### 2.2. Выход

| Файл | Путь |
|---|---|
| Проект решения (markdown) | `{корень}/{ГГГГ-ММ-ДД} Проект решения суда.md` |
| Проект решения (docx, опционально) | `{корень}/{ГГГГ-ММ-ДД} Клиент Проект решения суда.docx` |
| Профиль стиля судьи | `.vassal/hearings/judge-style.md` (если был анализ стиля) |
| Запись в истории | `.vassal/history.md` |

**Чего НЕ на выходе (но должно быть — см. §4.7, §4.8):**
- Структурированной оценки «какая часть иска вероятно будет удовлетворена» — это самый ценный артефакт для клиента.
- Артефакта сравнения с реальным решением (для обжалования).
- Записи в `index.yaml` (проект — внутренний артефакт, не официальный документ; но если оформляется в .docx — логично, что он в реестре проходит как `type: прогноз` или аналогично).
- Записи в timeline `case.yaml`.

### 2.3. Предусловия — есть в SKILL.md

- Дело инициализировано, есть `case.yaml`.
- Есть хотя бы одна позиция в корне.
- Желательно: отчёты о заседаниях, профиль судьи.
- Опционально: 3-5 решений судьи для анализа стиля.

### 2.4. Постусловия — ❌ нет

Должны быть:
- Файл `{дата} Проект решения суда.md` создан, содержит все 4 классические части.
- В резолютивной части — ответ по каждому заявленному требованию (не потеряно ни одного).
- Если был анализ стиля — `judge-style.md` создан и содержит атрибуцию по каким решениям (даты, номера дел).
- Если .docx — оба файла (md и docx) существуют, содержимое совпадает по структуре.
- `history.md` обновлён.

## 4. Противоречия и баги

### 4.1. Двойственность ролей без методологического разведения

[SKILL.md:38](../skills/draft-judgment/SKILL.md) (п.1 входных данных): «В чью пользу составляем проект? (нашего клиента или нейтральный прогноз)».
[SKILL.md:144](../skills/draft-judgment/SKILL.md) граничные случаи: «проект не в нашу пользу (прогноз проигрыша) → составить объективно, указать где позиция клиента слабая».

Это **две принципиально разные задачи**:
1. **«В нашу пользу»** — проект, который можно передать суду / использовать как заготовку для судьи. Аргументация односторонняя, выборка практики — в нашу сторону. Использование: усилить резолютивную линию.
2. **«Нейтральный прогноз»** — симуляция того, как вероятно решит суд. Взвешенная аргументация, учёт позиции оппонента, оценка рисков. Использование: предупредить клиента, подготовиться к апелляции заранее.

В SKILL.md они **слиты в один алгоритм**. Фаза 3 одинаковая для обоих. Opus-субагент получает одну и ту же инструкцию. Разница — только в том, что Сюзерен «попросил». На практике это значит:
- При «в нашу пользу» модель может скатиться в нейтральность (потому что её тренируют быть объективной).
- При «нейтрально» — скатиться в линию клиента (потому что весь загруженный контекст — от клиента).

**Правка:** развести на **две ветки** в SKILL.md с разными системными инструкциями Opus-субагенту:
- Ветка «проект для судьи» — системный промпт «ты помощник судьи, составляющий проект решения в пользу стороны X, учитывая её позицию как ведущую».
- Ветка «нейтральный прогноз» — системный промпт «ты коллегия, оценивающая баланс позиций сторон, выработай вероятный исход».

Без этого оба режима работают одинаково посредственно.

### 4.2. **Главная архитектурная проблема**: генерация .docx напрямую

[SKILL.md:125](../skills/draft-judgment/SKILL.md): «Если да — сгенерируй .docx напрямую (**НЕ через arbitrum-docx**)».

Одновременно [ARCHITECTURE.md:927](../ARCHITECTURE.md): в разделе «Скиллы, использующие arbitrum-docx» **явно указан `draft-judgment`**. Документация и SKILL.md противоречат.

Проверка остальных скиллов: `appeal`, `cassation`, `prepare-hearing`, `analyze-hearing`, `legal-review`, `build-position` — **все** используют `arbitrum-docx` (см. grep). `draft-judgment` — единственный, который идёт прямым путём.

**Последствия прямой генерации:**

| Плюс | Минус |
|---|---|
| Нет внешней зависимости → при отсутствии `arbitrum-docx` скилл работает | Не подхватывает изменения шаблона `arbitrum-docx` |
| Упрощает работу, если судебный акт специфичен (шапка другая, чем у процедокументов) | Нарушает принцип единой точки оформления |
| Быстрее по токенам (не читает SKILL.md внешнего плагина) | 4 скилла читают чужой SKILL.md, 1 — нет → концептуальная неконсистентность |
| | Параметры оформления (Times New Roman 12, 1.5 интервал) — **дублированы** в SKILL.md. Если в `arbitrum-docx` правила изменятся — `draft-judgment` их не подхватит |
| | Имя файла: `ГГГГ-ММ-ДД Клиент Проект решения суда.docx` — **в корне дела** (не в процессуальной папке), что отличается от других скиллов и противоречит [conventions.md §«Именование процессуальных папок»](../shared/conventions.md) |

**Содержательный аргумент «за» прямую генерацию:** проект решения — не процессуальный документ стороны. У него нет просительной части, нет подписи представителя, нет шапки «в арбитражный суд от...». Шаблон `arbitrum-docx` (заточенный под ходатайства/жалобы/отзывы) ему не подходит. Это **специфичный формат судебного акта**.

**Правильное архитектурное решение:** расширить `arbitrum-docx` отдельным типом шаблона «судебный акт» (подобно тому как там уже есть «Тип 1: процессуальный с шапкой» и «Тип 2: аналитический без шапки»). Получится «Тип 3: судебный акт» — с именем суда, составом суда, без подписи представителя. Тогда все 5 скиллов используют `arbitrum-docx` единообразно.

Пока этого нет, в SKILL.md **обязательно зафиксировать** причину отклонения: «draft-judgment генерирует .docx напрямую, потому что формат судебного акта не покрывается шаблонами arbitrum-docx. Если arbitrum-docx получит тип «судебный акт» — перейти на него».

### 4.3. `judge-profile.md` vs `judge-style.md` — два файла, неясная граница

Два скилла пишут в `.vassal/hearings/`:
- `analyze-hearing` — создаёт/дополняет `judge-profile.md` (см. [SKILL.md:130](../skills/analyze-hearing/SKILL.md:130), [§«Формат профиля судьи»](../skills/analyze-hearing/SKILL.md:184)). Содержание: общий стиль, предпочтения в аргументации, паттерны вопросов, триггеры, подход к доказательствам.
- `draft-judgment` — создаёт `judge-style.md` (см. [SKILL.md:74](../skills/draft-judgment/SKILL.md:74)). Содержание: структурные особенности, стилистические особенности, подход к применению норм.

**Пересечения:**
- «Подход к применению норм» (style) ⇄ «Предпочтения в аргументации» (profile).
- «Стилистические особенности, формальный/разговорный тон» (style) ⇄ «Общий стиль, формальный/неформальный» (profile).

**Различия:**
- `profile.md` — **наблюдение за живым судьёй на заседании** (из транскрипции).
- `style.md` — **анализ письменных актов судьи** по аналогичным делам.

Концептуально это **разные срезы одной сущности**. Но SKILL.md ни одного скилла не ссылается на другой:
- `draft-judgment` читает `judge-profile.md` (п.5 фазы 1), но не использует его при анализе стиля в фазе 2.
- `analyze-hearing` ничего не знает про `judge-style.md`.

**Последствия:**
1. Два файла описывают одного судью с разных углов и **не сводятся**. Нет «объединённого портрета».
2. При следующем запуске `draft-judgment` по другому делу с тем же судьёй — `judge-style.md` специфичен **для одного дела** (он в `.vassal/` этого дела). Переиспользования нет.
3. `judge-profile.md` тоже дело-специфичен — по архитектуре плагина каждое дело изолировано.

**Правки:**
1. **Объединить** в один файл `.vassal/hearings/judge.md` с двумя секциями «По заседаниям» и «По письменным актам». Оба скилла дополняют свои секции.
2. Зафиксировать в SKILL.md `draft-judgment`: «При наличии `judge-profile.md` — использовать его не только для контекста, но и как входную информацию при генерации (судья предпочитает такой-то тон → учесть в мотивировочной части)».
3. **Глобальный профиль судьи** между делами — отдельная проблема (см. §6.3).

### 4.4. Опциональность образцов решений — деградация без чётких границ

[SKILL.md:59](../skills/draft-judgment/SKILL.md): фаза 2 начинается с «Если Сюзерен загрузил решения судьи по аналогичным делам».

Что если их нет? По [граничным случаям:143](../skills/draft-judgment/SKILL.md:143): «составить в стандартном стиле арбитражных решений».

Проблемы:
1. **«Стандартный стиль арбитражных решений» — не определён.** Нет эталонного шаблона. Модель импровизирует. В реальности разные судьи — разные стили (Мосгорсуд vs районный суд vs арбитраж округа).
2. **Нет шкалы деградации.** Качество проекта при 0 / 1 / 3 / 5 образцах — разное, но SKILL.md не предупреждает Сюзерена: «при 1 образце стиль уловить не получится, выдам нейтральный».
3. **Нет триггера «собери образцы».** Скилл мог бы сам предложить: «у меня нет образцов решений судьи. Хочешь, найду через kad.arbitr.ru?». Но функционала нет, и в SKILL.md предложения нет.
4. **Нет валидации релевантности.** Если Сюзерен загрузил 5 решений этого судьи — но все по разным категориям (договорные + банкротство + земельные) — стиль будет размытый. Скилл не проверяет категорийную близость к нашему делу.

**Правка:** в phase 2 явно проверять:
- Количество образцов (порог: 3 минимум для анализа стиля).
- Категория каждого (близкая к нашей?).
- При <3 или при несовпадении категории — предупредить: «не могу выделить достоверный профиль, буду работать в стандартном стиле, вот что это значит: ...».

### 4.5. Метод Мошкина требует **публикации** решений

Метод предполагает, что у судьи есть публичные решения. Но:
- Не все решения публикуются (особенно в СОЮ).
- Новые судьи (менее 1-2 лет стажа) имеют мало актов.
- Судебные составы меняются — «профиль» судьи, собранный на 3-летних актах, уже может быть неактуален.

SKILL.md не учитывает эти реалии. Сюзерен получит пресуппозицию «дай 3-5 решений», не сможет — и скилл откатится на неопределённый fallback.

### 4.6. Повторный запуск в одну дату — коллизия имени

[SKILL.md:120](../skills/draft-judgment/SKILL.md): `{ГГГГ-ММ-ДД} Проект решения суда.md`.

Если Сюзерен запускает скилл в одну дату дважды (первая версия — «в нашу пользу», вторая — «нейтральный прогноз») — файлы совпадут. Что происходит:
- Перезапись?
- Суффикс `(2)`?
- Добавление «времени»?

Не описано. Та же проблема, что в [build-position §4.6](build-position.md).

**Предлагаемое имя:** `{ГГГГ-ММ-ДД} Проект решения суда ({режим}).md`, где `{режим}` = `в пользу клиента` | `нейтральный`. Это и коллизию решает, и показывает пользователю важное различие (см. §4.1).

### 4.7. Резолютивная часть — нет чек-листа

Это — **самая важная часть для юриста**, потому что:
- Определяет исполнимость.
- Основа для расчёта госпошлины, судебных расходов, сумм.
- При обжаловании — главный объект анализа.

[SKILL.md:99-103](../skills/draft-judgment/SKILL.md):
> **Резолютивная часть:**
> - Решение по каждому требованию (удовлетворить/отказать, суммы)
> - Распределение судебных расходов
> - Порядок обжалования

Это слишком общо. **Нет чек-листа обязательных элементов.** В реальном судебном акте резолютивная часть содержит:

1. **Требование №N — формулировка решения:**
   - Взыскать с кого (точное наименование должника с ИНН/ОГРН).
   - В пользу кого (наименование взыскателя).
   - Какую сумму (основной долг + проценты + неустойка + пошлина — с разложением).
   - **С какого момента начисляются проценты** (если заявлены «по день фактической оплаты» — формула с указанием базы).
   - При нескольких должниках: солидарно / субсидиарно / в долях.
2. **Момент исполнения:**
   - Немедленно / после вступления в силу.
   - Если исполнение в натуре — срок (в течение N дней).
3. **Распределение госпошлины:**
   - Точная сумма.
   - Пропорциональность при частичном удовлетворении.
   - Зачёт/возврат из бюджета, если пошлина была уплачена излишне.
4. **Судебные расходы:**
   - Представитель, эксперты, переводчик.
5. **Обеспечительные меры:**
   - Сохранить до вступления в силу / отменить с даты вступления.
6. **Обращение к исполнению:**
   - Исполнительный лист — выдать по заявлению / направить почтой.
7. **Порядок и срок обжалования:**
   - 1 месяц в апелляционный суд X через суд первой инстанции.

Без чек-листа — модель пропустит какой-то элемент. По аналогии с [prepare-hearing §5.7](prepare-hearing.md) (чек-лист просительной части) — нужно иметь зеркальный **чек-лист резолютивной части**.

### 4.8. Нет связи с `appeal` — «предсказание vs реальность»

Сценарий:
1. `draft-judgment` составил нейтральный прогноз: «по требованию о взыскании неустойки — отказ, по основному долгу — удовлетворить частично 80%».
2. Суд вынес решение: основной долг удовлетворён полностью, неустойка в 30%.
3. `appeal` запускается на это решение.

Полезный артефакт: **сравнение прогноза и факта**. Где разошлись? Какие наши аргументы суд принял/отклонил неожиданно? Это **готовая основа для апелляции**:
- Где отклонение = ошибка суда в нашу невыгоду → основания для обжалования.
- Где отклонение = ошибка прогноза → слабость нашей аналитики, учесть на будущее.

Ни `draft-judgment`, ни `appeal` об этом не знают:
- `draft-judgment` не ссылается на `appeal`.
- `appeal` в фазе 1 загружает позиции, заседания, доказательства — но **не `{дата} Проект решения суда.md`**.

**Правка:**
1. В SKILL.md `draft-judgment` фаза 4 (preview) сохранять в frontmatter проекта структурированный блок «ожидаемый исход» (по требованиям).
2. В SKILL.md `appeal` фаза 1 — читать последний проект решения, если он есть, и включать блок «расхождение с прогнозом» в preview фазы 3.

Это превращает `draft-judgment` из разового упражнения в **часть петли обратной связи**.

### 4.9. Доставка «перспектив дела» клиенту организована слабо

По сути, `draft-judgment` даёт ответ на самый частый вопрос клиента: **«Какие у нас шансы?»**. Структурированно:
- Какая часть иска удовлетворена → сумма, которую клиент получит (или заплатит).
- На какие риски суд укажет в мотивировке.

Но в SKILL.md фаза 4 (preview) показывает Сюзерену «резолютивную часть кратко + 2-3 мотива» — и всё. **Нет формата «отчёт для клиента»** (как в `analyze-hearing`, где есть `{дата}-client-brief.md`).

Получается: дорогой Opus-анализ → материал доходит **только** до юриста → юрист сам пересказывает клиенту в упрощённой форме. Часть сигнала теряется.

**Правка:** параллельно с проектом решения сохранять `{дата}-forecast-client-brief.md` или предлагать Сюзерену экспортировать «Прогноз решения для клиента.docx» — адаптированную версию без юридического жаргона, с чёткими тезисами:
- «Вероятно, суд взыщет X рублей».
- «Риски: A, B, C».
- «Основания для апелляции, если такое решение нас не устроит: ...».

### 4.10. Граничные случаи — поверхностное покрытие

[SKILL.md:141-146](../skills/draft-judgment/SKILL.md) перечисляет 4 случая. Из них:
- «нет решений судьи → стандартный стиль» — см. §4.4, «стандартный» не определён.
- «проект не в нашу пользу» — см. §4.1, не разведено методологически.
- «дело с несколькими требованиями → отдельный вывод по каждому» — ок, но без чек-листа (§4.7).
- «частичное удовлетворение → пропорциональное распределение расходов» — правильно, но не сказано как считать.

**Что не покрыто:**
- **Решение о прекращении производства** (утверждение мирового, отказ от иска, неподведомственность) — отдельная структура, нет мотивировочной в обычном виде.
- **Оставление без рассмотрения** (несоблюдение досудебного порядка, подан не уполномоченным лицом) — отдельный формат.
- **Заочное решение** (ответчик не явился) — особенности мотивировки.
- **Решение по делу об оспаривании ненормативного акта** (госорган как ответчик) — иная структура.
- **Решения банкротных обособленных споров** — в `case.yaml` категория «банкротство» есть, но SKILL.md её не учитывает. Это зеркало проблемы из [legal-review §5.5](legal-review.md).
- **Нестандартные иски** (признание права, обязание совершить действие, расторжение договора) — в резолютивной часто специфика (не суммы, а действия и сроки).

### 4.11. Имя .docx — «Клиент» для проекта решения неуместно

[SKILL.md:133](../skills/draft-judgment/SKILL.md): `ГГГГ-ММ-ДД Клиент Проект решения суда.docx`.

По [conventions.md:24](../shared/conventions.md:24) «Отправитель» в имени — это чей документ. Но проект решения **не исходит от клиента**. Это наше внутреннее аналитическое произведение, которое симулирует решение суда. Если буквально — «отправителем» надо ставить имя судьи или суда, но это тоже неверно (суд не отправлял).

**Правильнее:**
- `{дата} Проект решения суда.docx` (без отправителя, как у `Предварительный анализ документов` в `legal-review`).
- Или `{дата} Представитель Проект решения.docx` (по аналогии с вариантом, обсуждавшимся в [legal-review §4.4](legal-review.md)).

Это снова связано с проблемой именования продуктов плагина, зафиксированной в других ревизиях — в `shared/conventions.md` нужен отдельный раздел «продукты плагина».

### 4.12. Конфликт команды и скилла по образцам решений

[commands/draft-judgment.md:11](../commands/draft-judgment.md:11): «Если `$ARGUMENTS` содержит путь к решениям судьи — загрузи для анализа стиля».

SKILL.md не знает про этот путь. В фазе 2 скилл говорит «если Сюзерен загрузил» — т.е. предполагается, что файлы уже физически где-то лежат. Где? Во «Входящих документах/»? Прямо указанный путь из `$ARGUMENTS`?

Если команда и скилл рассогласованы, произойдёт то же, что с intake (см. [intake §4.1](intake.md)): непредсказуемое поведение.

**Правка:** зафиксировать в SKILL.md, что источник образцов — **во «Входящих документах/» в отдельной папке `judge-samples/`** (или параметризованно через `$ARGUMENTS`). После анализа — переместить в `.vassal/raw/judge-samples-{дата}/` и в `На удаление/`.

### 4.13. `judge-style.md` накапливается **только внутри одного дела**

Архитектурно — это правильно (`.vassal/` изолирован по делам). Но **по сути неправильно для метода Мошкина**: цифровой профиль судьи должен накапливаться **между делами**, потому что:
- Один и тот же судья ведёт десятки дел за год.
- Чем больше образцов решений (по любым делам) — тем точнее профиль.

Сейчас юрист, запуская `draft-judgment` по второму делу того же судьи, **начинает накопление заново**. Вся работа по первому делу — в другом `.vassal/`.

**Правка:** см. §6.3 — глобальное хранилище профилей судей (`~/.vassal-global/judges/{судья}.md`).

## 5. Непокрытые edge-cases

### 5.1. Образцы решений в разных форматах

Типично: решения скачиваются с [kad.arbitr.ru](https://kad.arbitr.ru) как .pdf (сканы с подписями), с sudact.ru — как html/pdf с разметкой. Скилл должен извлекать текст. В SKILL.md нет упоминания:
- Кто извлекает текст (Haiku через `extract_text.py`? Read tool визуально?).
- Что с OCR-артефактами в судебных решениях (важные детали — номера статей, даты, суммы — критичны; ошибка в OCR = ошибка в стиле).
- Что с анонимизацией (решения в публичном доступе часто с заменёнными именами). Профиль стиля от этого не страдает, но надо это понимать.

### 5.2. Документ в корне дела vs в процессуальной папке

[SKILL.md:118-122, 135](../skills/draft-judgment/SKILL.md) — .md и .docx в корне дела.

По [conventions.md:36-43](../shared/conventions.md:36) процессуальные документы — в процессуальных папках `{дата} Сторона - действие`. Но:
- Проект решения — **не процессуальный документ** стороны. Он никуда не подаётся.
- Формально аналогичен `Начальная позиция.md` — аналитический продукт → корень дела. Это **правильно.**

Но тогда коррекция: в SKILL.md надо зафиксировать «этот документ **аналитический**, не процессуальный, поэтому в корне без процпапки». Иначе у юриста когнитивный диссонанс — `.docx` в корне, а не рядом с «ходатайствами и отзывами».

### 5.3. TTL / устаревание проекта

Сценарий: подготовили проект 2026-03-15. К заседанию 2026-05-10 в дело поступили 3 новых документа от оппонента, изменилась позиция. Проект решения **устарел**.

Нет флага «устарел», нет детектора «новые документы после даты проекта». Если Сюзерен по невнимательности покажет клиенту старый прогноз — будет ошибка.

Та же проблема, что в [legal-review §5.3](legal-review.md), [build-position §5.5](build-position.md) — системная для плагина.

### 5.4. Переполнение контекста — большое дело

Проект решения требует **полного контекста**: все позиции + все зеркала ключевых + отчёты о заседаниях + образцы решений судьи. Для дела на 100+ документов + 3-4 прошедших заседаний + 5 образцов решений судьи по 20 страниц — 100k+ токенов.

Нет лимитов, нет предупреждений. Это проблема, общая для всех Opus-скиллов плагина, но особенно остро здесь, потому что образцы решений — дополнительный источник, которого нет у других скиллов.

### 5.5. Итеративная корректировка — дорогая

[SKILL.md:114](../skills/draft-judgment/SKILL.md): «Если Сюзерен корректирует — вернись к фазе 3».

Фаза 3 = полная регенерация проекта Opus-субагентом. Для мелкой правки «усиль мотив по неустойке» — перегенерация **всего** решения. Это дорого и медленно.

Нужен инкрементальный режим: «перегенерировать только мотивировочную часть по требованию №2, остальное оставить».

### 5.6. Встречный иск / несколько истцов

Если в деле встречный иск — проект решения должен содержать решения по **обеим группам требований**. SKILL.md предполагает линейную структуру «истец → ответчик». Для встречки — две резолютивные части, или одна с разделами? Нет правила.

### 5.7. Нет режима «быстрый прогноз»

Иногда нужен просто вопрос «каковы шансы в процентах, по какой сумме?» без полного проекта решения (пример: юрист на коротком брифе перед встречей с клиентом). Скилл предлагает только **полный режим** — 6 фаз, Opus, контекст дела целиком.

Turbo-режим (по аналогии с [prepare-hearing §4.9, §6.3](prepare-hearing.md)): краткий прогноз (1 экран) + оценка вероятностей по требованиям, без генерации полного текста решения.

### 5.8. Неучёт стадии процесса

Стадия, когда делают проект решения:
- **До предварительного** — слишком рано, доказательства неполные, оппонент ещё не отыгрался.
- **После последнего до прений** — оптимально.
- **Перед прениями** — ещё лучше.
- **После прений, до объявления** — роскошь (проект нужен был **до** выработки аргументов прений).

SKILL.md не спрашивает про стадию, режим работы один. Между тем, на ранней стадии прогноз заведомо шаткий, об этом надо Сюзерена предупреждать.

### 5.9. Делегирование Opus — без системного промпта для субагента

[SKILL.md:78](../skills/draft-judgment/SKILL.md): «Делегируй Opus-субагенту (model: "opus")».

Но без формализованного системного промпта:
- В ветке «в нашу пользу» — «ты помощник судьи, склоняющегося к позиции стороны X».
- В ветке «нейтральный прогноз» — «ты коллегия, оценивающая баланс».

Сейчас — просто «Opus». Модель берёт свой дефолтный режим, который может быть смещён. Та же проблема, что в [prepare-hearing §4.5](prepare-hearing.md): Opus-субагент работает без разведения ролей.

## 6. Реальные проблемы на практике

### 6.1. Стоимость генерации

Полный проект решения по делу средней сложности = большое контекстное окно + Opus. Для юриста, который пользуется плагином в Cowork-сессии, это заметные токены. Скилл не показывает предварительную оценку стоимости и не предлагает turbo-режим.

### 6.2. Метод Мошкина — практичность vs риски

Плюсы метода «цифрового профиля судьи»:
- Теоретически точнее.
- Помогает юристу «настроиться» на судью.

Риски, не упомянутые в SKILL.md:
- **Вторичность.** Если судья читает проект стороны, написанный «в его стиле», — это может быть воспринято как манипуляция.
- **Публичность решений.** Не все судьи публикуют своё творчество. В СОЮ — 30-40% публикаций по разным категориям.
- **Автоматизация судейского творчества.** Есть этическая зона — насколько корректно «имитировать стиль судьи». Хотя практика «проектов решений» в регионах — реальна, она на грани.

SKILL.md про эти риски молчит, но хорошо бы в самом начале (перед фазой 1) дать Сюзерену один абзац «ограничения метода», чтобы он мог осознанно решить.

### 6.3. Профиль судьи между делами — самая большая потеря ROI

Разработали профиль стиля судьи Ивановой в деле А33-1/2026. В деле А33-2/2026 — **заново**. В деле А33-3/2026 — **снова с нуля**. Каждый раз — дорогой Opus + ручная загрузка 3-5 решений.

В Cowork-архитектуре плагина есть `.vassal/` дела. Но нет **общего хранилища на уровне пользователя**, где жил бы профиль судьи (и, например, типовые аргументы оппонента — другой сквозной ресурс).

**Предлагаемое решение:** `~/.vassal-global/judges/{ФИО}-{суд}.md`. При первом столкновении с судьёй — скилл проверяет наличие. При обновлении в одном деле — сохраняет обратно в глобальное хранилище. Это выходит за рамки одной ревизии, но для `draft-judgment` — **главный источник реальной экономии**.

### 6.4. Финальная правка в Word — разрыв синхронизации

Как и в [prepare-hearing §6.5](prepare-hearing.md): юрист откроет .docx, поправит формулировки, сохранит. При следующем запуске скилл возьмёт **.md-версию** (неактуальную). Никакой синхронизации.

### 6.5. «Готовый проект для судьи» — мифический use case

Практика «подбрасывания судье проекта решения» — региональная и неформальная. В крупных арбитражах (АС Москвы, АС СПб) — не работает. В некоторых регионах — работает, но через конкретные договорённости.

`draft-judgment` маркетинговый образ — «проект для судьи» — создаёт впечатление, что плагин генерирует документ, который реально будет принят. В реальности это в 90% случаев **аналитический инструмент для стороны**, а не «заготовка для суда».

SKILL.md это не разделяет. Стоило бы в «Зачем нужен проект решения» честно сказать: два use case — (a) **аналитика** (прогноз для клиента + подготовка к апелляции); (b) **заготовка для судьи** (региональная практика, не везде применима).

## 7. Рекомендации (приоритизированно)

### P0 (критично)

1. **Развести две ветки «в нашу пользу» vs «нейтральный прогноз»** методологически: разные системные промпты Opus-субагенту, разные имена файлов (§4.1, §4.6).
2. **Чек-лист резолютивной части** — зеркально чек-листу просительной части из [prepare-hearing](prepare-hearing.md) (§4.7). Без него главная часть акта хрупка.
3. **Разрешить противоречие с arbitrum-docx:** зафиксировать в SKILL.md обоснование прямой генерации + исправить [ARCHITECTURE.md §10](../ARCHITECTURE.md:927) (убрать draft-judgment из списка, либо начать его переводить на arbitrum-docx через «Тип 3: судебный акт») (§4.2).
4. **Постусловия + валидация «все требования из позиции → в резолютивной»** (§2.4).
5. **Именование файла без «Клиент»** — это не документ клиента (§4.11).

### P1 (важно)

6. **Единый профиль судьи** `judge.md` вместо `judge-profile.md` / `judge-style.md` с двумя секциями (§4.3).
7. **Граница деградации при нехватке образцов** (<3, нерелевантные категории) — явное предупреждение Сюзерену (§4.4).
8. **Связь с `appeal`**: frontmatter-блок «ожидаемый исход» в проекте + чтение этого блока в `appeal` (§4.8).
9. **Формат отчёта для клиента** — `{дата} Прогноз решения клиент.md` по аналогии с `analyze-hearing/client-brief.md` (§4.9).
10. **Покрытие нестандартных решений**: прекращение, оставление без рассмотрения, заочное, банкротные, иски о признании/обязании (§4.10).
11. **Системный промпт для Opus-субагента** с разведением ролей (§5.9).
12. **Источник образцов**: формализовать путь (Входящие/judge-samples/), синхронизация команды и SKILL.md (§4.12).

### P2 (UX и качество)

13. **Turbo-режим** — краткий прогноз без полного текста (§5.7).
14. **Инкрементальные итерации** — регенерация по блокам, не целиком (§5.5).
15. **TTL проекта** — детект новых документов после даты проекта (§5.3).
16. **Стадия процесса** — в preview отмечать ограниченность ранних прогнозов (§5.8).
17. **Ограничения метода в начале SKILL.md** — публичность решений, этика, региональная специфика (§6.2).
18. **Разведение двух use case** в описании скилла: аналитический инструмент vs заготовка для судьи (§6.5).
19. **Синхронизация .docx ↔ .md** при повторных запусках (§6.4).
20. **Лимит контекста** для больших дел (§5.4).

### P3 (архитектурно-сквозные)

21. **Глобальное хранилище профилей судей** `~/.vassal-global/judges/` — переиспользование между делами (§6.3, §4.13). Самый крупный источник потенциальной экономии.
22. **`arbitrum-docx` Тип 3: судебный акт** — расширение внешнего плагина, чтобы draft-judgment вернулся в общий контракт (§4.2).
23. **Встречный иск в резолютивной** — отдельные разделы (§5.6).
24. **Обновление timeline в case.yaml** при создании проекта — как событие «подготовлен прогноз» (§2.2).

## 8. Резюме

Draft-judgment — **концептуально самый амбициозный** скилл плагина: он один пытается делать прогноз (не аналитику прошлого, не оформление настоящего, а моделирование будущего исхода). Опора на метод Мошкина С.А. — сильная сторона, редкая для LLM-скиллов.

Исполнение — **раздвоенное**:
- **Методологически** — не разведены две принципиально разные задачи (проект «в нашу пользу» vs «нейтральный прогноз»). Оба режима в одном алгоритме — оба получатся посредственно.
- **Архитектурно** — скилл выпадает из общего контракта плагина: **единственный** не использует `arbitrum-docx` (при том, что в ARCHITECTURE.md он указан как использующий). Параметры оформления дублированы. Причина — обоснованная (судебный акт не лезет в шаблоны для процдокументов), но решение неправильное (надо расширить `arbitrum-docx`, а не обходить).
- **Связи с другими скиллами** — слабые. `judge-profile.md` и `judge-style.md` не объединены. Связи с `appeal` (сравнение прогноза и факта) нет. Доставка прогноза клиенту не организована. Глобальный профиль судьи между делами не накапливается — основной ROI метода Мошкина теряется.

**Главная проблема:** скилл делает **дорогую прогнозную работу, которая исчезает после одного использования**. В одном деле — каждый запуск заново разбирается с профилем судьи; между делами — тот же судья анализируется с нуля; после вынесения реального решения — никто не сверяет прогноз с фактом. Это **антипаттерн «эфемерной дорогой аналитики»**, описанный в [prepare-hearing §4.1](prepare-hearing.md), но в `draft-judgment` он острее, потому что накопление «цифрового профиля судьи» — **единственное, что оправдывает метод Мошкина в принципе**.

Три правки закрывают критическую массу:
1. **Разведение «в нашу пользу» / «нейтральный прогноз»** — разные промпты, разные имена файлов.
2. **Чек-лист резолютивной части** — зеркало к просительной в prepare-hearing.
3. **Единый профиль судьи, накопительный между делами** (`~/.vassal-global/judges/` + объединение `profile`+`style`).

Остальные 20+ пунктов — поступательные улучшения. Но первые три — **обязательны**, иначе скилл красивый, но не даёт реального выигрыша на вторых и третьих запусках.
