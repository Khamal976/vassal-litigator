# Ревизия скилла `appeal`

**Дата:** 2026-04-20
**Файлы:** [skills/appeal/SKILL.md](../skills/appeal/SKILL.md), [commands/appeal.md](../commands/appeal.md)
**Связанные:** [skills/cassation/SKILL.md](../skills/cassation/SKILL.md), [skills/prepare-hearing/SKILL.md](../skills/prepare-hearing/SKILL.md), [skills/draft-judgment/SKILL.md](../skills/draft-judgment/SKILL.md), [skills/analyze-hearing/SKILL.md](../skills/analyze-hearing/SKILL.md), [references/deadlines-reference.md](../skills/legal-review/references/deadlines-reference.md), внешний плагин `arbitrum-docx`
**Версия плагина:** 0.4.0

## 1. Назначение и роль в workflow

**Декларация:** анализ решения суда первой инстанции, систематический поиск оснований для обжалования по ст. 270 АПК / ст. 330 ГПК, ранжирование оснований, генерация текста апелляционной жалобы в `.docx` через `arbitrum-docx`.

**Позиция в workflow:** первый скилл фазы «Обжалование», после цикла `prepare-hearing` / `analyze-hearing` и вынесения решения. Входит в триаду инстанционного обжалования: `appeal` → `cassation` → (надзор, не покрыт). Сильнейший единичный потребитель Opus-токенов — объём аналитики сопоставим с `build-position`, плюс текст жалобы.

**Парные / последующие скиллы:**
- Вход: позиции (`build-position`), аналитические блоки (`legal-review`/`.vassal/analysis/`), отчёты о заседаниях (`analyze-hearing`), проект решения (`draft-judgment`) — на практике скилл использует далеко не всё (см. §4.2-4.3).
- Выход: `cassation` (если апелляция отказала), `prepare-hearing` (если есть заседание по апелляции) — оба соседства НЕ формализованы в SKILL.md.

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| `.vassal/case.yaml` | да | Стороны, суд, хронология |
| `.vassal/index.yaml` | да | Реестр документов |
| Решение суда первой инстанции (md-зеркало) | **да** | Главный объект анализа |
| `{корень}/*позиция.md` (все позиции) | да | Сверка с тем, что мы заявляли |
| `.vassal/hearings/*.md` | да | Что говорил судья, что зафиксировано |
| `.vassal/mirrors/*.md` (ключевых доказательств) | да | База для оснований «недоказанность», «несоответствие» |
| Наши и оппонента процдокументы | да | База для оснований «проигнорировал довод» |
| Вводная от Сюзерена | нет | Приоритетные основания, срок, стратегия |
| `arbitrum-docx` SKILL.md | да (фаза 5) | Оформление `.docx` |

**НЕ читается, хотя должно бы** (см. §4.2-4.3):
- `.vassal/analysis/*.md` — продукты `legal-review` (сроки, квалификация, подсудность) — нигде не упомянуты в фазе 1.
- Проект решения из `draft-judgment` — прямой сигнал «что модель ожидала, чего не дождалась» — не упомянут.

### 2.2. Выход

| Файл | Путь |
|---|---|
| Апелляционная жалоба (md) | `{корень}/{ГГГГ-ММ-ДД} Апелляционная жалоба.md` |
| Апелляционная жалоба (docx) | `{корень}/{ГГГГ-ММ-ДД} Клиент Апелляционная жалоба.docx` через `arbitrum-docx` |
| Запись в истории | `.vassal/history.md` |
| Обновление индекса | `.vassal/index.yaml` |
| Обновление `case.yaml` | событие «подготовка апелляционной жалобы», возможно статус |

**Что НЕ сохраняется** (эфемерная работа Opus — та же проблема, что у `prepare-hearing`):
- Ранжированный список оснований с оценкой силы каждого.
- Стресс-тест оснований (фаза 3).
- Контраргументы оппонента по каждому основанию.
- «Что модель отбросила и почему» — при следующем запуске (например, после возражений оппонента) всё переделываем заново.

### 2.3. Предусловия — есть в SKILL.md

- `.vassal/case.yaml` существует.
- Решение суда первой инстанции получено (md-зеркало).
- Желательно: полный контекст дела (позиции, отчёты, доказательства).

### 2.4. Постусловия — ❌ нет

Должны быть:
- `.md` и `.docx` файлы созданы и соответствуют друг другу.
- Решение помечено в индексе как обжалуемое (флаг на уровне записи решения, либо связка в bundle).
- Каждое основание в тексте жалобы имеет ссылку как минимум на одну норму и один фрагмент решения или материалов дела.
- Просительная часть содержит обязательные элементы (см. §5.8).
- В `case.yaml` зафиксированы дата изготовления решения, дата истечения срока подачи, дата подачи.

## 4. Противоречия и баги

### 4.1. Отсутствует водораздел «апелляция vs кассация»

`cassation` [SKILL.md:17-22](../skills/cassation/SKILL.md) и [таблица SKILL.md:171-181](../skills/cassation/SKILL.md) явно противопоставляет кассацию апелляции: «кассация проверяет только право, не переоценивает доказательства». **В `appeal` — ни одной фразы о том, что апелляция, в отличие от кассации, пересматривает дело по существу и МОЖЕТ переоценивать факты и доказательства.**

Это асимметрия: если в проекте появится первый вопрос «ты делаешь апелляцию или кассацию?», читающий только `appeal` не поймёт отличия. Если Сюзерен перепутает триггер (скажет «кассация» после решения первой инстанции), `cassation` объяснит, что он не туда обратился. Обратного fail-safe в `appeal` нет — ничто не проверяет, что обжалуется именно решение первой инстанции, а не постановление апелляции.

**Следствие для стиля жалобы:** `cassation` задаёт стиль («акцент на нарушениях норм, ссылки на ВС РФ, не переоценка доказательств»). У `appeal` стилевого ориентира нет. В реальности модель будет импровизировать между «как в кассации» и «как в отзыве на иск».

**Правка:** зеркальный блок «Ключевая специфика апелляции» в начале SKILL.md — минимум 3 тезиса (переоценка доказательств возможна, факты и право — полный пересмотр, стиль аргументации на уровне конкретных доказательств). И таблица отличий, симметричная [cassation SKILL.md:173-180](../skills/cassation/SKILL.md).

### 4.2. `draft-judgment` не используется, хотя это готовый детектор оснований

Если до решения был запущен `draft-judgment` (проект решения, который модель ожидала), а затем суд вынес иное — расхождение между проектом и реальным решением — **это прямой путь к ранжированному списку оснований**:
- Что суд установил иначе, чем мы ожидали → кандидат на «несоответствие выводов обстоятельствам».
- Какие нормы применил иначе → кандидат на «неправильное применение».
- Какие наши доводы не рассмотрел → кандидат на «неполное выяснение».

SKILL.md `appeal` ни слова про `draft-judgment`. Фаза 1 загружает «все позиции», «отчёты о заседаниях», «ключевые доказательства» — но не проект решения. Даже если он есть в корне дела (`{дата} Проект решения суда.md`), явного указания «загрузи и сопоставь» нет.

**Правка:** в фазу 1 — явный шаг:
> Если в корне дела есть файл `* Проект решения суда.md` (предшествующий запуск `draft-judgment`) — загрузи и сопоставь с вынесенным решением. Расхождения — кандидаты в основания обжалования.

### 4.3. `analyze-hearing` читается номинально, без извлечения сигналов

В фазе 1 п.5: «Загрузи отчёты о заседаниях из `.vassal/hearings/`». Но `analyze-hearing` создаёт три разных типа отчётов:
- `{дата}-internal.md` — вопросы судьи, слабости оппонента.
- `{дата}-judge-profile.md` — паттерны судьи.
- `{дата}-opponent-analysis.md` — уклончивые ответы, расхождения.

Эти отчёты содержат **готовые кандидаты в основания апелляции:**
- Судья задал вопрос → оппонент уклонился → суд всё равно принял довод оппонента = «недоказанность» + «несоответствие выводов».
- Судья не исследовал обстоятельство, которое мы просили → «неполное выяснение».
- Приобщение документов под возражение, которое не отражено в мотивировочной части → потенциальное процессуальное.

В `appeal` SKILL.md нет инструкции **какие сигналы извлекать** из отчётов о заседаниях. Модель может прочитать, а может пропустить. На практике — скорее «посмотрит в общих чертах».

**Правка:** в фазе 2 п.9 — явный обходной цикл по отчётам с перечнем сигналов.

### 4.4. Срок 1 месяц — упомянут, но не активен

[SKILL.md:29](../skills/appeal/SKILL.md): «Срок подачи -- 1 месяц со дня изготовления решения в полном объёме (ст. 259 АПК)».

[SKILL.md:88-92](../skills/appeal/SKILL.md) (фаза 2 п.11): «Проверка сроков: дата изготовления, крайний срок, если пропущен — основания для восстановления».

Но **дата изготовления решения нигде не берётся из структурированных данных**. Её должны извлечь вручную из текста решения (в резолютивной части обычно «решение изготовлено в полном объёме ДД.ММ.ГГГГ»). В `case.yaml` нет поля `judgment_issued_date`. В `index.yaml` у записи решения нет специального поля.

На практике:
- Модель увидит «решение датировано 20.03.2026» и примет это за дату изготовления, хотя резолютивная могла быть оглашена раньше, а мотивировочная изготовлена позже.
- Если даты изготовления в тексте решения нет → импровизация.

**Ни одного предупреждения пользователю вида «осталось 5 дней».** Ни триггера «скоро срок — упростить анализ, сразу к тексту» (как должно бы быть по аналогии с §4.9 prepare-hearing-ревизии).

**Правки:**
1. В схему `case.yaml` — добавить:
   ```yaml
   appeal:
     judgment_issued_date: null   # Дата изготовления решения в полном объёме
     deadline: null               # Расчётный срок подачи (judgment_issued_date + 1 месяц)
     filed_date: null             # Когда подали жалобу
   ```
2. В фазе 1 — шаг «запроси у Сюзерена дату изготовления решения; если он её не знает — попроси указать страницу решения с этой датой; извлеки и зафиксируй в `case.yaml`».
3. В фазе 2 — адаптивное правило: если до deadline осталось < 5 дней → turbo-режим: минимум стресс-теста, сразу к тексту с явным баннером Сюзерену.

### 4.5. Дополнительные доказательства (ст. 268 АПК) не покрыты

АПК РФ ст. 268 ч. 2: «Дополнительные доказательства принимаются арбитражным судом апелляционной инстанции, если лицо, участвующее в деле, обосновало невозможность их представления в суд первой инстанции по причинам, не зависящим от него, и суд признает эти причины уважительными». В `appeal` SKILL.md — **ни слова**.

Реальный сценарий: у Сюзерена есть новый документ (например, экспертиза, которую не успели получить в первой инстанции), он хочет приобщить его к жалобе. Скилл должен:
- Проверить, что документ не был доступен ранее (или представление было невозможно).
- Сформулировать ходатайство о приобщении с обоснованием невозможности по ст. 268.
- Отразить это в просительной части жалобы.

Всего этого нет. На практике — либо Сюзерен сам знает и пишет, либо документ будет отклонён апелляцией.

**Правка:** отдельный подраздел «Дополнительные доказательства» с мини-алгоритмом и шаблоном обоснования невозможности.

### 4.6. Связка с `prepare-hearing` после подачи не формализована

После подачи жалобы апелляционный суд назначит заседание. К нему нужен полноценный `prepare-hearing` (письменные пояснения, ходатайства, возможно новые доказательства с обоснованием по ст. 268).

SKILL.md `appeal` останавливается на Фазе 6 «Обновление состояния». Нет инструкции «следующий шаг — перед заседанием апелляции запустить `prepare-hearing` с этим документом как входом».

Одновременно `prepare-hearing` не рассчитан на работу по уже поданной апелляционной жалобе как основному документу — он ориентирован на первую инстанцию (таблица типов документов [prepare-hearing SKILL.md:175-183](../skills/prepare-hearing/SKILL.md) — отзыв, ходатайство, возражение, пояснения, дополнение к позиции). «Письменные пояснения к апелляционной жалобе» формально попадают в «пояснения», но явного указания нет.

**Правка:**
1. В `appeal` Фаза 6 добавить: «Следующий шаг: за 5-7 дней до заседания в апелляции — `prepare-hearing` с типом «пояснения/дополнения к апелляционной жалобе».»
2. В `prepare-hearing` — распознавать контекст апелляции (например, последний поданный документ — апелляционная жалоба) и предлагать соответствующие типы.

### 4.7. Ранжирование оснований без критериев

[SKILL.md:84-86](../skills/appeal/SKILL.md): «Ранжирование оснований: по силе (от наиболее перспективного к наименее); безусловные основания — приоритет».

«По силе» — что это? Возможные интерпретации:
- Вероятность удовлетворения апелляцией.
- Размер влияния на резолютивную часть (отменяется всё решение или только часть).
- Опора на доказательства (сильные доказательства → сильное основание).
- Опора на норму права (прямое нарушение vs расширительное толкование).

Нет чек-листа оценки. Модель выдаст свой ранг, но критерии не показаны — значит, Сюзерен не может спорить с ранжированием на своём языке.

**Правка:** ввести явную многофакторную оценку для каждого основания в фазе 2:
```
Для каждого основания оцени:
- Юридическая обоснованность: прямое vs интерпретационное нарушение нормы.
- Доказательственная поддержка: есть ли в деле документы, подтверждающие довод.
- Судебная практика: есть ли практика в пользу нашей позиции.
- Влияние на резолютивную: отмена целиком / в части / изменение мотивировки.
- Риск отклонения: насколько сильно апелляция типично отклоняет такое основание.
Итоговая сила = функция всех 5 факторов + явное пояснение.
```

И: числовые проценты — запретить (та же проблема, что §4.9 build-position-ревизии). Только словесные шкалы с обоснованием.

### 4.8. Стресс-тест в одной сессии — не red team

[SKILL.md:93-98](../skills/appeal/SKILL.md) Фаза 3: «Для каждого основания: как оппонент будет возражать; насколько устойчиво при проверке; есть ли практика».

Делается тем же Opus, что строил основания. Та же проблема, что в `prepare-hearing` (§4.5 ревизии prepare-hearing): модель склонна к consistency, смягчает собственный red team.

**Правка:** явный субагент «юрист оппонента, защищающий решение суда» с системным промптом «Разбей эти основания апелляции. Используй мотивировочную часть решения как арсенал».

### 4.9. `arbitrum-docx` без fallback (сквозная проблема)

[SKILL.md:148-153](../skills/appeal/SKILL.md) Фаза 5: оформление через `arbitrum-docx`, Тип 1.

Проблемы идентичны §4.3 ревизии `prepare-hearing`:
1. Feature detection отсутствует.
2. Если плагин не установлен — скилл упадёт/импровизирует.
3. Версионирование контракта не оговорено.
4. Минимум нужен fallback «сохранить `.md`, предупредить Сюзерена».

Дополнительно — специфика `appeal`: срок подачи жёсткий (1 месяц). Если `arbitrum-docx` упал в последний день — критическая ситуация. Fallback здесь **важнее**, чем для prepare-hearing.

### 4.10. `case.yaml` — какие поля обновляются?

[SKILL.md:159-161](../skills/appeal/SKILL.md) Фаза 6 п.21: «Добавь событие, обнови статус дела (если применимо)».

`case.yaml` v2 имеет `status: active | suspended | completed | archived`. «Апелляция подана» — это какое состояние? Вариант `pending_appeal` в схеме **отсутствует**. Либо скилл импровизирует (пишет `active` — тогда статус неинформативен), либо вводит своё значение (нарушает схему).

Также нет полей для дат апелляции (см. §4.4) и для результата апелляции.

**Правка:** расширить схему `case.yaml`:
```yaml
status: active | suspended | pending_appeal | pending_cassation | completed | archived
appeal:
  filed_date: null
  hearing_date: null
  result: null     # upheld | overturned | modified | null
  ruling_date: null
```

### 4.11. Конфликт команды и скилла

[commands/appeal.md:7-9](../commands/appeal.md):
- Если `$ARGUMENTS` содержит основания — как приоритетные.
- Если `$ARGUMENTS` содержит путь к решению — загрузить.

SKILL.md: «Уточни… решение загружено? основания? срок?». При наличии аргументов — не переспрашивать, но команда не говорит, как парсить «основания vs путь». Если Сюзерен напишет `/vassal-litigator:appeal C:\решения\от-20-марта.pdf неправильно применили ст. 395` — путь и основание в одной строке. Парсинг — на модель.

### 4.12. `update-index` не триггерится

Сценарий: Сюзерен получил решение из суда, положил в корень дела мимо `intake`. Запускает `appeal`. SKILL.md не говорит «проверь, что решение есть в индексе, иначе запусти `update-index`». Фаза 1 п.3: «Загрузи решение — md-зеркало». Если зеркала нет — модель импровизирует (попробует прочитать PDF, создать зеркало на лету), что не соответствует остальному плагину.

## 5. Непокрытые edge-cases

### 5.1. Решение частично в нашу пользу — жалоба на часть

Граничный случай упомянут: «Решение полностью в нашу пользу, но Сюзерен хочет обжаловать часть → жалоба на часть». Но **нет инструкции**:
- В просительной части явно обозначить «отменить в части… в остальной части решение оставить без изменения».
- Связать обжалуемую часть с конкретными пунктами резолютивной части.
- Риск запуска апелляционного пересмотра «в полном объёме» (противник тоже подаёт апелляцию на нашу часть).

### 5.2. Встречная апелляция оппонента

Упомянуто: «Оппонент тоже подал апелляцию → учесть при составлении, подготовить возражения». Но:
- Нет отдельного скилла / режима для **отзыва на апелляционную жалобу оппонента**. Это отдельный документ с иной структурой.
- Нет инструкции, когда подавать отзыв (срок — до рассмотрения в апелляции, регулируется ст. 262 АПК).
- Нет связки с `add-opponent`, который мог бы обработать поступившую жалобу оппонента.

### 5.3. Госпошлина — реальная сумма не рассчитывается

[SKILL.md:134](../skills/appeal/SKILL.md) только в приложениях: «Документ об уплате госпошлины». Но:
- Госпошлина за апелляционную жалобу — 50% от пошлины за иск имущественного характера (ст. 333.21 НК РФ) или фиксированная для неимущественных.
- Скилл не рассчитывает сумму пошлины.
- Не формирует платёжное поручение / реквизиты.
- Не предупреждает «пошлина должна быть оплачена до подачи жалобы».

### 5.4. Частичная оплата пошлины / льготы

Если клиент — льготник (ст. 333.37 НК РФ — прокуратура, госорган и пр.) или заявляет о рассрочке / отсрочке — нужно ходатайство. Не покрыто.

### 5.5. Апелляция на определение (а не на решение)

Крупный пробел. В АПК апелляция подаётся на:
- **Решения** — ст. 259 АПК, срок 1 месяц.
- **Определения**, которые подлежат обжалованию отдельно — общий срок 1 месяц по ч. 3 ст. 188 АПК. Специальные сроки: **15 дней** для определений из ч. 3 ст. 188 (по обеспечительным мерам и др. — ч. 3 ст. 272 АПК). Определение об утверждении мирового — отдельная история: обжалуется **в кассацию в течение месяца** (ст. 141 ч. 11 АПК), минуя апелляцию.

SKILL.md `appeal` полностью ориентирован на решения. Слово «определение» не упоминается. Если Сюзерен запустит скилл с `$ARGUMENTS` «определение об отказе в обеспечении», скилл построит стандартную жалобу на решение, со стандартной структурой и сроком 1 месяц — это **фактическая ошибка** (для определений по обеспечительным мерам — 15 дней, ч. 3 ст. 272 АПК).

> **Уточнение по «утверждению мирового»:** определение об утверждении мирового соглашения **в апелляционном порядке не обжалуется** — оно подаётся в кассационный суд округа в срок 1 месяц со дня вынесения (ст. 141 ч. 11 АПК). То есть это вообще не зона `appeal`, а зона `cassation`. Скилл должен это распознавать и предлагать переадресацию.

**Правка:** в Фазу 1 — явный вопрос «обжалуется решение или определение?», с маршрутизацией на разные алгоритмы. Для определения — сокращённая структура (обычно без фактических обстоятельств, акцент на процессуальных нарушениях), другие сроки по категории.

### 5.6. Упрощённое производство — срок 15 дней

[deadlines-reference.md:66](../skills/legal-review/references/deadlines-reference.md): «Упрощённое производство — апелляция 15 дней (ст. 229 АПК)».

В `appeal` SKILL.md — только срок 1 месяц. Если дело рассматривалось в упрощённом порядке, скилл не скорректирует срок → высокий риск пропуска.

**Правка:** в Фазу 1 — вопрос «какая процедура была в первой инстанции?» или читать из `case.yaml` флаг `case.simplified_procedure`. Затем выбирать правильный срок.

### 5.7. СОЮ-апелляция (ГПК) — структурно иная

SKILL.md упоминает ст. 330 ГПК в чек-листе оснований, но весь остальной алгоритм (сроки, структура приложений) — под АПК. Для дел в СОЮ:
- Срок — 1 месяц (ст. 321 ГПК), но начало течения отличается.
- Апелляция на решения мировых судей — свои правила (ст. 320 ГПК).
- Апелляционная жалоба направляется через суд первой инстанции (а не прямо в апелляцию) — ст. 321 ч. 2 ГПК.

Дифференциации нет.

### 5.8. Просительная часть — нет чек-листа

[SKILL.md:128-132](../skills/appeal/SKILL.md): «Просительная часть: отменить (полностью/в части), принять новый акт / направить на новое рассмотрение, распределение расходов».

Не хватает:
- Конкретная формулировка «отменить» для каждого пункта резолютивной — требование отменить «абзацы 2-4 резолютивной части» vs «решение целиком».
- «Принять новый акт» — точная формулировка нового решения (повторяем наши исходные просительные + скорректированные с учётом того, что произошло).
- «Направить на новое рассмотрение» — обоснование, почему апелляция сама не может принять новый акт (редкие основания: полностью не исследованы факты; неисследованные доводы требуют сбора доказательств).
- Распределение пошлины в случае отмены / изменения — требование возместить пошлину первой и апелляционной инстанции.
- Обеспечительные меры: при удовлетворённом иске оппонента — ходатайство о приостановлении исполнения (ст. 265.1 АПК) — не упомянуто.
- Проценты «по день фактической оплаты» — перенос из первой инстанции.

### 5.9. Возражения на решение в протоколе заседания / замечания на протокол

Если в протоколе первого заседания — расхождения с тем, что произошло, и `analyze-hearing` их зафиксировал (фаза 4 замечания на протокол — [analyze-hearing SKILL.md:92-103](../skills/analyze-hearing/SKILL.md)) — это дополнительный сигнал для процессуальных оснований апелляции. Не используется.

### 5.10. Контекст больше 100k — переполнение

При большом деле (решение на 40 страниц + 3-4 позиции + 5 отчётов о заседаниях + десятки доказательств) — легко 120k+ токенов. Нет политики усечения (брать саммари, брать только цитируемые в решении доказательства, и пр.).

### 5.11. Нет возможности запуска «анализ оснований без написания жалобы»

Реальный сценарий: Сюзерен хочет сначала понять, стоит ли обжаловать (оценка перспектив), и только потом решать, писать ли жалобу. Сейчас скилл — монолитный pipeline. Альтернатива — остановиться после фазы 3 с результатом «анализ перспектив апелляции». Не поддерживается явно.

**Правка:** режим `--analyze-only`: фазы 1-3, результат — `{дата} Анализ оснований для апелляции.md` без текста жалобы.

### 5.12. Ссылки на доказательства в жалобе — без «t. X, л.д. YY»

В апелляционных жалобах юристы указывают том и лист дела (т. 2, л.д. 145). В плагине — нет структуры для томов/листов дела. Но есть doc-id в индексе. Как соотнести?

Это сквозная проблема (не только `appeal`), но именно в жалобе отсутствие ссылок на листы дела — заметный недостаток.

## 6. Реальные проблемы на практике

### 6.1. Типовой сценарий «проиграли — быстро апелляцию»

Юрист получает решение (часто за 1-2 недели до конца срока), запускает `appeal` в спешке. Скилл ожидает «полный контекст дела» — а на практике:
- Позиций может не быть (дело вёл другой юрист, клиент пришёл с решением).
- Отчётов о заседаниях нет (никто не делал `analyze-hearing`).
- В `case.yaml` половина полей пустая.

Сейчас скилл формально отработает, но качество анализа сильно упадёт — и без предупреждения Сюзерену.

**Правка:** в фазе 1 — «покрытие контекста»: сколько позиций, отчётов, зеркал доказательств доступно. Если < N — явное предупреждение «анализ будет поверхностным, качество жалобы ограничено».

### 6.2. Решение — PDF-скан на 40 страниц

Если решение выложено на картинках (фото экрана или скан-пдф), `intake` мог дать md-зеркало плохого качества. Мотивировочная часть в решении суда — ключевой объект для анализа `appeal`. Если зеркало низкого качества, вся аналитика — на песке. Скилл не проверяет `quality` / `needs_manual_review` для записи решения.

**Правка:** в фазе 1 — специальная проверка качества md-зеркала именно для обжалуемого решения. Если `quality: low` или `needs_manual_review: true` — остановка, предложение перечитать/перераспознать.

### 6.3. Сюзерен пишет жалобу вручную, скилл добавляет оснований

Часто юрист уже написал проект жалобы, хочет проверить, не упустил ли основания. Текущий pipeline не поддерживает «инкрементальный» режим — он создаёт жалобу с нуля. Сюзерену придётся сравнивать свой текст с чужим.

**Правка:** режим `--critique`: Сюзерен вставляет свой проект, скилл проверяет по чек-листу 5 групп оснований, предлагает пропущенные.

### 6.4. Клиент требует «написать пожёстче»

Апелляция часто содержит эмоциональные обороты (в рамках процессуального стиля) — «суд проигнорировал», «выводы очевидно противоречат», «недопустимо толкование». Скилл не учитывает стилистическую составляющую — генерирует нейтральный юридический текст. В жалобу потом приходится руками «закручивать».

### 6.5. Ручная правка `.docx` → рассинхрон

Та же проблема, что у `prepare-hearing` (§6.5): юрист дорабатывает `.docx` в Word, `.md` отстаёт. При следующем действии со скиллом — `.md` устаревшее.

### 6.6. «Стратегия обжалования» vs «текст жалобы»

В реальности процесс — двухступенчатый:
1. Анализ: имеет ли смысл обжаловать, на каких основаниях, каковы перспективы.
2. Если решили обжаловать — пишем текст.

SKILL.md сливает оба шага. См. §5.11.

## 8. Резюме

Appeal — **технически самый крупный потребитель Opus** в плагине (анализ решения + систематический поиск оснований + стресс-тест + текст жалобы), но методологически **вторая по проработке инстанционная ветка** после `cassation`.

Сильные стороны:
- Явный чек-лист 5 групп оснований (a-e) — редкий для плагина структурированный промпт.
- Безусловные основания — приоритет (корректно юридически).
- Систематический пересмотр как философия — заявлен.

Главные слабости — **три категории асимметрии**:

1. **С `cassation`:** там есть блок «ключевая специфика» и таблица отличий → `appeal` не объясняет, что он — про пересмотр по факту и праву. Модель будет путать стили.
2. **С реальным АПК:** не покрыта апелляция на определение (другие сроки — 15 дней по обеспечению), упрощённое производство (15 дней вместо месяца), доп. доказательства по ст. 268, СОЮ-специфика, обеспечительные меры, госпошлина. Это **фактические пробелы**, способные привести к пропуску срока или отказу по формальному основанию.
3. **С соседними скиллами плагина:** `draft-judgment` (готовый детектор расхождений), `analyze-hearing` (вопросы судьи как кандидаты в основания), `.vassal/analysis/*.md` (сроки, квалификация) — **не используются**, хотя именно апелляция — где они дают максимальный эффект.

Плюс сквозные болезни плагина: эфемерность Opus-работы (как в prepare-hearing), отсутствие постусловий (как везде), fallback на `arbitrum-docx` (4 скилла), `update-index` не триггерится.

Минимальный набор правок для закрытия критической массы:
1. Блок «ключевая специфика апелляции» + таблица отличий с кассацией.
2. Развилка решение/определение/упрощённое + правильные сроки.
3. Поля срока в `case.yaml` + предупреждение «осталось N дней».
4. Fallback для `arbitrum-docx`.
5. Включение `draft-judgment` и сигналов из `analyze-hearing` в фазу 1.

Первые три — блокирующие (ошибка в сроке — неисправимо). Четвёртая и пятая — делают из скилла действительно интегрированную часть плагина, а не изолированный генератор.
