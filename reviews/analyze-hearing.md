# Ревизия скилла `analyze-hearing`

**Дата:** 2026-04-20
**Файлы:** [skills/analyze-hearing/SKILL.md](../skills/analyze-hearing/SKILL.md), [commands/analyze-hearing.md](../commands/analyze-hearing.md)
**Связанные:** [ARCHITECTURE.md §8.9](../ARCHITECTURE.md), [reviews/prepare-hearing.md](prepare-hearing.md), внешний плагин `arbitrum-docx`
**Версия плагина:** 0.4.0

---

## 1. Назначение и роль в workflow

**Декларация:** обработка транскрипции судебного заседания — структурирование речи (Sonnet) и глубокий анализ (Opus) с формированием трёх отчётов (внутренний, для клиента, профиль судьи + анализ оппонента), опциональной сверкой с протоколом и генерацией замечаний на протокол, обновлением `case.yaml`. Опционально — оформление отчёта клиенту в .docx.

**Позиция в workflow:** парный скилл к `prepare-hearing` — работает *постфактум* (после заседания), тогда как `prepare-hearing` — *до*. Ключевой источник накопленных данных о заседаниях для всех последующих скиллов:
- `prepare-hearing` читает `.vassal/hearings/*` при подготовке к следующему заседанию.
- `appeal` / `cassation` читают отчёты для обоснования доводов.
- `draft-judgment` читает `judge-profile.md`.
- `build-position` косвенно использует (через новые данные в `case.yaml: timeline`).

**Уникальность:** **единственный скилл**, формирующий «устный» профиль оппонента и судьи — то, чего нельзя извлечь из письменных документов. Это value-add, которого нет у конкурентов.

**Методологическая опора:** алгоритм Мошкина С.А. (runetlex.academy). Центральный тезис скилла — «транскрипция важнее протокола» (протоколы теряют ~70% информации, транскрипция имеет прямое процессуальное значение через замечания на протокол).

## 2. Контракт скилла

### 2.1. Вход

| Источник | Обязательный | Использование |
|---|---|---|
| Транскрипция (текст, файл, вставка в чат) | да | Основной вход, три разных источника — см. §4.1 |
| `.vassal/case.yaml` | да | Стороны, контекст, предыдущая дата заседания |
| `{корень}/*позиция.md` | да | Последняя позиция — для контекста «что планировали» |
| Протокол суда (опц.) | нет | Для сверки в фазе 4 |
| `.vassal/hearings/judge-profile.md` (опц.) | нет | Предыдущий профиль судьи для наращивания |
| `.vassal/hearings/{дата}-prep.md` (если ввести — см. §4.6) | нет | План подготовки «что мы хотели» для сопоставления |
| `arbitrum-docx` SKILL.md | да (в фазе 7) | Оформление .docx |

### 2.2. Выход

| Файл | Путь |
|---|---|
| Внутренний отчёт | `.vassal/hearings/{ГГГГ-ММ-ДД}-internal.md` |
| Отчёт клиенту (md) | `.vassal/hearings/{ГГГГ-ММ-ДД}-client-brief.md` |
| Профиль судьи | `.vassal/hearings/{ГГГГ-ММ-ДД}-judge-profile.md` — см. §4.4 про путь |
| Анализ оппонента | `.vassal/hearings/{ГГГГ-ММ-ДД}-opponent-analysis.md` — **или** `.vassal/analysis/opponent-{дата}.md` (см. §4.3) |
| Оригинал транскрипции | `.vassal/raw/hearing-{ГГГГ-ММ-ДД}/transcript.txt` |
| Замечания на протокол (опц.) | `{корень}/{дата} Клиент Замечания на протокол.docx` |
| Отчёт клиенту (.docx, опц.) | `{корень}/{дата} Клиент Отчёт о судебном заседании.docx` |
| Обновление карточки дела | `.vassal/case.yaml: timeline[]`, `next_hearing`, `next_hearing_source: transcript` |
| Обновление истории | `.vassal/history.md` |
| Обновление индекса | `.vassal/index.yaml` — если приобщены документы |

**Эфемерные артефакты** (создаются, не сохраняются):
- Сверка с протоколом (фаза 4) — если замечания не оформляются в .docx, результат сверки нигде не остаётся.
- Результаты анализа приобщённых документов (в заседании) — попадают в `index.yaml` только как записи, без md-зеркал.

### 2.3. Предусловия — есть в SKILL.md

- Дело инициализировано (`case.yaml` существует).
- Транскрипция предоставлена.

Но **не проверены**:
- Что транскрипция — текст (не аудио/видео). Скилл только в описании говорит «не с аудио напрямую» — но нет входной валидации.
- Что дата заседания в транскрипции ≥ `case.yaml: next_hearing` (реалистичность).
- Что заседание — по *этому* делу (номер дела в транскрипции совпадает с `case.yaml: case.number`).

### 2.4. Постусловия — ❌ нет

Должны быть:
- Каждый из 4 отчётов существует (или явно помечен «пропущен»).
- `case.yaml` остался валидным YAML.
- Новый `next_hearing` (если обозначен) не раньше даты текущего заседания.
- Все даты в обновлённом timeline — не позже текущего дня.
- `history.md` содержит запись со ссылкой на отчёты.
- Если приобщены документы — они в `index.yaml` с `source` и `filing_status`.
- Все пометки `[?]` из транскрипции либо сняты после подтверждения Сюзерена, либо явно сохранены в отчётах (а не «размазаны» по case.yaml).

## 4. Противоречия и баги

### 4.1. Неоднозначность источника транскрипции

[SKILL.md:40-46](../skills/analyze-hearing/SKILL.md) перечисляет три источника:
- Файл во «Входящих документах/»
- Текст вставлен в чат
- «Файл загружен напрямую»

[commands/analyze-hearing.md:10-12](../commands/analyze-hearing.md) добавляет четвёртый — через `$ARGUMENTS`:
- `$ARGUMENTS` содержит текст транскрипции
- `$ARGUMENTS` содержит путь к файлу

Проблема — **нет единого триггера приоритета**. Если в `$ARGUMENTS` строка «разбери» + во «Входящих документах/» лежит файл «transcript.pdf» + параллельно Сюзерен вставил кусок текста в чат — что выбирать?

Кроме того:
1. **PDF-транскрипция** (из сервиса автотранскрипции) — SKILL.md не упоминает OCR pipeline, предполагает готовый текст. Но на практике многие сервисы отдают PDF с текстовым слоем.
2. **Разница между «текст в чате» и «файл с текстом»** — в первом случае текст уже в контексте модели, во втором нужно `Read`. Скилл не различает, а это влияет на pipeline (фаза 1 шаг 1: «сохрани в `.vassal/raw/hearing-{дата}/transcript.txt`» — но если текст уже в чате, нужно сначала Write файл на диск).
3. **«Файл загружен напрямую»** (п. 1 вариант 3) — что это означает в Claude Code? В отличие от Claude.ai, здесь нет «загрузки в диалог» — только чтение с диска. Формулировка унаследована из другой среды.

**Правка:** явная таблица приоритетов источников с дефолтом. И отдельно — обработка PDF/docx-транскрипций через `scripts/extract_text.py` (как в `intake`).

### 4.2. [СНЯТО — ошибка ревизии]

Изначально здесь была заявлена «фактическая ошибка»: будто бы SKILL.md указал срок замечаний в АПК как 5 дней, а реально якобы 3.

**Это галлюцинация ревизионного агента.** Действующая редакция АПК РФ ст. 155 ч. 7 — «**пятидневный срок** после подписания соответствующего протокола». SKILL.md написан корректно. Пункт снят полностью; нумерация остальных подразделов сохранена для целостности ссылок.

> **Урок:** правовые утверждения в ревизиях, не подтверждённые прямой цитатой из НПА, требуют верификации. См. примечание в §8.

### 4.3. Коллизия путей с `add-opponent`

Проблема выявлена в [WORKFLOW-REVIEW.md §6](../WORKFLOW-REVIEW.md):
- `add-opponent` пишет в `.vassal/analysis/opponent-{дата}.md` (экспресс-анализ процессуальных документов оппонента).
- `analyze-hearing` в текущей версии пишет в `.vassal/hearings/{дата}-opponent-analysis.md` (анализ устной позиции оппонента).

**На первый взгляд — разные пути.** Но ARCHITECTURE.md и описание скилла непоследовательно ссылаются:
- [SKILL.md:131](../skills/analyze-hearing/SKILL.md): «`{ГГГГ-ММ-ДД}-opponent-analysis.md`» — в `.vassal/hearings/`.
- Но `prepare-hearing` читает аналитику оппонента из `.vassal/analysis/opponent-*.md` ([prepare-hearing/SKILL.md:57](../skills/prepare-hearing/SKILL.md)).

Значит, анализ оппонента из `analyze-hearing` **не попадает в контекст `prepare-hearing` при следующей итерации**! `prepare-hearing` ищет в `.vassal/analysis/opponent-*.md`, а там ничего от заседания нет — только от `add-opponent`.

Фактически результат работы Opus по устной позиции оппонента **теряется** между подготовками — его никто не читает.

**Правки (один из вариантов):**
1. Переименовать файл в `analyze-hearing`: `.vassal/analysis/opponent-hearing-{дата}.md`.
2. В `prepare-hearing` расширить glob: `.vassal/analysis/opponent-*.md` + `.vassal/hearings/*-opponent-analysis.md`.
3. Лучший вариант — единая схема именования с префиксом-источником:
   - `.vassal/analysis/opponent-filing-{дата}.md` (от `add-opponent`)
   - `.vassal/analysis/opponent-hearing-{дата}.md` (от `analyze-hearing`)

### 4.4. `judge-profile.md` — дата в имени vs «один профиль на дело»

[SKILL.md:130](../skills/analyze-hearing/SKILL.md): «`{ГГГГ-ММ-ДД}-judge-profile.md`» (с датой в префиксе).
[SKILL.md:184](../skills/analyze-hearing/SKILL.md): «Если файл `.vassal/hearings/judge-profile.md` уже существует — дополнить новыми данными» (без даты).

**Противоречие внутри SKILL.md:**
- Если файлы с датой — **каждое заседание создаёт новый профиль**, накопления нет.
- Если файл без даты (`judge-profile.md`) — накопление есть, но не стыкуется с конвенцией именования всех остальных файлов в hearings/.

Дополнительное подтверждение проблемы:
- `draft-judgment` ищет `.vassal/hearings/judge-profile.md` ([draft-judgment/SKILL.md:53](../skills/draft-judgment/SKILL.md)) — **без даты**.
- `draft-judgment` сам пишет в `.vassal/hearings/judge-style.md` ([draft-judgment/SKILL.md:74](../skills/draft-judgment/SKILL.md)) — ещё один кандидат!

Итого: три варианта имени (`{дата}-judge-profile.md`, `judge-profile.md`, `judge-style.md`) — расходятся между скиллами.

**Правка:** единый файл `judge-profile.md` (без даты) с аппенд-логикой или версионированием внутри файла. Дата — в поле `last_updated` frontmatter'а (уже есть в шаблоне [SKILL.md:192](../skills/analyze-hearing/SKILL.md)).

### 4.5. Профиль судьи — только внутри одного дела

Это **главное архитектурное ограничение** по ценности.

Текущая схема: `judge-profile.md` живёт в `.vassal/hearings/` конкретного дела. Если у юриста 5 дел у одного и того же судьи Иванова — **пять отдельных профилей**, которые не знают друг о друге.

А ведь именно здесь уникальная ценность плагина:
- После 20 заседаний у судьи Иванова в разных делах — накапливается объёмная картина его стиля.
- Новое дело стартует «с чистого листа», повторно собирая то, что уже известно.

Это именно то, что предлагается в [NOTION-INTEGRATION-PROPOSAL.md §2.3](../NOTION-INTEGRATION-PROPOSAL.md) (база `Judges`). Но даже без Notion — можно сделать `~/.vassal-shared/judges/{ФИО-суд}.md` как общий пул.

**Правка (минимальная):** ввести опциональный внешний путь в frontmatter:
```yaml
shared_profile: "~/.vassal-shared/judges/Иванов-И-И-АС-Красноярского-края.md"
```
Если указан — скилл читает и пишет в общий файл вдобавок к локальному.

**Правка (идеал):** интеграция с Notion `Judges` (отдельный тикет).

### 4.6. Нет связки с `prep.md` от `prepare-hearing`

В [reviews/prepare-hearing.md §4.1](prepare-hearing.md) предложено ввести `.vassal/hearings/{дата}-prep.md` — артефакт подготовки к заседанию (red team, стресс-тест, план).

`analyze-hearing` должен бы **открывать `prep.md`** по той же дате и делать самое ценное сопоставление:

| Что планировали | Что случилось | Расхождение |
|---|---|---|
| Заявить ходатайство об экспертизе | Заявили, отказано | Готовить повторное с доп. обоснованием |
| Судья примет акт сверки | Не приобщил, отложил | Проанализировать причину отказа |
| Оппонент надавит на срок давности | Не надавил | Оппонент пропустил — зафиксировать |

Сейчас этой связки нет. Каждый скилл работает в своей временной точке независимо, хотя они парные и были задуманы как цикл.

**Правка:** в фазе 1 п. 3 (после чтения позиции) — читать `.vassal/hearings/{дата}-prep.md` если существует. В внутреннем отчёте — отдельный раздел «План vs факт» с таблицей.

Это связывает [reviews/prepare-hearing.md §4.2](prepare-hearing.md) и замыкает цикл префактум ↔ постфактум.

### 4.7. Фаза 6 пишет `{дата}-opponent-analysis.md` без frontmatter

В фазе 6 п. 10 [SKILL.md:127-131](../skills/analyze-hearing/SKILL.md) перечислены 4 файла, но **формат frontmatter описан только для internal.md и judge-profile.md** ([SKILL.md:159-168, 187-194](../skills/analyze-hearing/SKILL.md)).

`client-brief.md` и `opponent-analysis.md` — без frontmatter. Но именно frontmatter читают другие скиллы для навигации (`type`, `date`, `next_hearing`). Без него последующий скилл вынужден парсить тело файла.

**Правка:** дополнить форматы для всех 4 файлов. Минимум — `type: hearing-report`, `subtype: {internal|client-brief|judge-profile|opponent-analysis}`, `date`, `hearing_date`, `case_number`.

### 4.8. Схема `case.yaml: timeline` недоописана

[SKILL.md:116](../skills/analyze-hearing/SKILL.md): «Новые события в timeline (с пометкой `source: transcript`)».

[case-schema.yaml:70-74](../shared/case-schema.yaml) описывает timeline так:
```yaml
timeline:
  - date: null
    event: "Описание события"
    source: user     # user | transcript | agent
```

Проблемы:
1. **Что именно считается событием?** Заседание целиком (1 запись) или каждое процессуальное действие в нём (приобщение акта, отказ в ходатайстве, отложение)? SKILL.md не уточняет.
2. **Ссылка на отчёт.** У события нет поля `ref` со ссылкой на `hearings/{дата}-internal.md` — значит, по timeline нельзя быстро перейти к детальному отчёту.
3. **Ссылка на приобщённые документы.** Если суд приобщил акт — `timeline` зафиксирует факт приобщения, но не свяжет с `doc-NNN`.

**Правка схемы:**
```yaml
timeline:
  - date: "2026-04-20"
    event: "Судебное заседание: отложено до 15.05, приобщён акт сверки (doc-042)"
    source: transcript
    ref: ".vassal/hearings/2026-04-20-internal.md"
    related_docs: ["doc-042"]
```

### 4.9. `next_hearing_source` не унифицирован с общими источниками

[case-schema.yaml:18](../shared/case-schema.yaml): `next_hearing_source: null # user | transcript | kad`.
[case-schema.yaml:73](../shared/case-schema.yaml): `source: user # user | transcript | agent`.

Разные словари: у даты заседания — `user | transcript | kad`, у события — `user | transcript | agent`. Нет `kad` в timeline, нет `agent` в hearing source.

`analyze-hearing` должен писать `next_hearing_source: transcript` — это правильно, но нужно сразу держать в голове разнобой при расширении.

**Правка:** унифицировать в conventions.md словарь `source`:
- `user` — Сюзерен задал вручную.
- `transcript` — из `analyze-hearing`.
- `document` — из `intake`/`add-opponent` (документ из дела).
- `kad` — из интеграции с КАД.Арбитр (когда появится).
- `agent` — выведено эвристически скиллом.

### 4.10. Отчёт для клиента в .docx, но замечания на протокол — процессуальный документ с шапкой

[SKILL.md:139-146](../skills/analyze-hearing/SKILL.md):
- Отчёт клиенту — тип 2 (аналитический, без шапки) — в **корень дела**.
- Замечания на протокол — тип 1 (процессуальный, с шапкой) — тоже в **корень дела**.

По [conventions.md:36-43](../shared/conventions.md) (раздел «Именование процессуальных папок»): процессуальные документы идут в папку `{ГГГГ-ММ-ДД} {Сторона} - {действие}/` (пример в строке 12 — `2026-04-01 Истец - исковое заявление/`). Замечания на протокол — процессуальный документ. Значит, должен быть в папке `{дата} Клиент - замечания на протокол/`, не в корень.

Та же проблема, что и в [reviews/prepare-hearing.md §5.4](prepare-hearing.md): SKILL.md игнорирует конвенцию о процессуальных папках.

**Правка:** замечания на протокол — в процессуальную папку. Отчёт клиенту (аналитический) — в корень или в подпапку `.vassal/exports/` (это не процессуальный документ и не для суда).

### 4.11. Оперативный режим без сохранения — дорогой Opus в вакууме

[SKILL.md:148-154](../skills/analyze-hearing/SKILL.md): «Если мы на перерыве — пропустить структурирование, сразу к анализу, **без сохранения файлов**».

Это разумная идея, но в текущем виде — деньги Opus работают впустую. После заседания Сюзерен уже нашёл применение выводам (устно сказал им в зале), но сама аналитика не попадает в постфактумный отчёт.

**Правка:** в конце оперативного режима — короткий промпт «Сохранить эти заметки в `.vassal/hearings/{дата}-operative.md`? Они будут использованы при полном анализе после заседания».

### 4.12. `arbitrum-docx` без fallback

Та же проблема, что и в [reviews/prepare-hearing.md §4.3](prepare-hearing.md) и [WORKFLOW-REVIEW.md §4](../WORKFLOW-REVIEW.md).

Два разных момента в скилле требуют `arbitrum-docx`:
- Отчёт клиенту (.docx)
- Замечания на протокол (.docx)

Если плагин недоступен — оба пропускаются без явного уведомления. Для замечаний на протокол это критично — срок ограничен (5 дней по АПК), поступление в суд блокируется.

**Правка (как в prepare-hearing):** feature detection + fallback на `.md` в соответствующей папке с явным предупреждением.

## 5. Непокрытые edge-cases

### 5.1. Транскрипция отсутствует (заседание без записи)

В граничных случаях [SKILL.md:222](../skills/analyze-hearing/SKILL.md): «Сюзерен дал только протокол (без транскрипции) — работай с тем что есть, но предупреди, что протокол содержит ~30% информации, и рекомендуй расшифровать аудиозапись».

Но есть третий случай — **ни транскрипции, ни протокола**. Например:
- Заседание было, но аудиозапись не велась.
- Сюзерен присутствовал, делал пометки на листе.
- Клиент спрашивает «ну, как прошло?».

Скилл не описывает интерактивный опрос Сюзерена: «Что произошло? Какие решения? Даты? Приобщённое?». Без такого режима нельзя зафиксировать заседание в `case.yaml` в принципе — а он должен быть, т.к. это основной источник хронологии.

**Правка:** добавить режим «интервью» — Сюзерен отвечает на чек-лист вопросов, скилл строит отчёт по ответам.

### 5.2. Множественные заседания в один день

Сценарий: утром предварительное, вечером основное. Или два дела у одного клиента в один день у одного судьи.

Текущее именование: `{ГГГГ-ММ-ДД}-internal.md` — **коллизия имени**.

**Правка:** при повторном заседании в ту же дату — суффикс времени начала: `2026-04-20T10-30-internal.md`, `2026-04-20T15-00-internal.md`. Или — суффикс номера: `2026-04-20-1-internal.md`, `2026-04-20-2-internal.md`. Явно в SKILL.md.

### 5.3. Одно заседание — две части (несколько транскрипций)

Иногда сервис транскрипции разбивает длинное заседание (5+ часов) на файлы по часу. Скилл получит 5 транскрипций одной даты.

В граничных случаях [SKILL.md:220](../skills/analyze-hearing/SKILL.md) упомянуто обратное: «Несколько заседаний в одной транскрипции → разбей на отдельные отчёты». Но **обратного случая** (одно заседание — несколько транскрипций) нет.

**Правка:** если в `raw/hearing-{дата}/` несколько файлов с одной датой — склеить в один `transcript.txt` (с маркерами границ) перед обработкой.

### 5.4. Транскрипция очень плохая — какой порог отказа?

[SKILL.md:218](../skills/analyze-hearing/SKILL.md): «Транскрипция очень плохого качества → структурируй что можно, сомнительное помечай `[?]`, предупреди Сюзерена».

Но если транскрипция на 80% нечитаема (скажем, автораспознавание упало на акценте), формальный запуск Opus'а на мусорном входе — это деньги в пустоту, плюс риск галлюцинации («Opus додумал на основе плохих намёков»).

**Правка:** Sonnet-фаза должна возвращать сигнал «quality: low» (доля `[?]` > N%). При low — прерывать pipeline до Opus и спрашивать Сюзерена: «Качество транскрипции низкое (X% сомнительных фрагментов). Продолжать глубокий анализ, или сначала улучшить транскрипцию?».

### 5.5. Повторный запуск на ту же транскрипцию

Сюзерен не помнит, что уже запускал. Или хочет «переанализировать с новой вводной».

Сейчас — молчаливая перезапись файлов в `.vassal/hearings/`. Нет diff'а, нет предупреждения. Аналог проблемы 4.2 из [reviews/legal-review.md](legal-review.md).

**Правка:** при повторном запуске — предложить:
1. Перезаписать (с diff'ом, как в legal-review).
2. Создать версионированный отчёт (`2026-04-20-internal-v2.md`).
3. Отменить.

### 5.6. Превью обновлений `case.yaml` — что если модель перепутала дату?

[SKILL.md:111-117](../skills/analyze-hearing/SKILL.md): preview даёт список предлагаемых изменений. Но:

**Реальный сценарий:** в транскрипции судья говорит «откладываю на пятнадцатое мая». Модель интерпретирует как `2026-05-15`, но судья имел в виду `2026-05-05` (сказал неразборчиво, модель галлюцинировала). В preview пишется `next_hearing: 2026-05-15` — Сюзерен устал после заседания, подтверждает «ок». **Пропущено заседание.**

Preview это не предотвращает. Нужны дополнительные защиты:
1. **Цитата-обоснование** рядом с каждой датой: «`next_hearing: 2026-05-15` — цитата из транскрипции: «[… откладываю на пятнадцатое мая …]»». Чтобы Сюзерен мог сравнить.
2. **Расчёт разумности даты:** отложение через 3 дня — подозрительно, через 10-30 дней — нормально, через 6 месяцев — сверхподозрительно.
3. **Маркер `[?]` обязателен** на любой дате из транскрипции, если не было чёткой формулировки в протоколе.
4. **Двойная проверка:** если дата была сказана в транскрипции и есть расписание КАД.Арбитр — сверить (сейчас интеграции нет, но через веб-поиск можно).

### 5.7. Решение по существу — переход к `appeal`

[SKILL.md:221](../skills/analyze-hearing/SKILL.md): «Заседание окончилось вынесением решения → особое внимание резолютивной части, предложи analyze для обжалования».

Опечатка? Должно быть «предложи `appeal`» (или минимум «предложи скилл для обжалования»). Слово «analyze» здесь двусмысленно — это название класса скиллов, а не конкретный скилл.

Кроме того — резолютивная часть в заседании ≠ мотивированное решение. Мотивированное решение изготавливается позже (до 5 дней по АПК, до 5 дней по ГПК). Скилл не различает: есть только резолютивная → срок на апелляцию считается от мотивированного.

**Правка:** если в транскрипции оглашена резолютивная часть, зафиксировать «ожидается мотивированное решение» в timeline и отправить Сюзерена к `add-opponent` / `intake` когда мотивированное поступит.

### 5.8. Приобщённые в заседании документы — как попадают в index.yaml?

[SKILL.md:133](../skills/analyze-hearing/SKILL.md): «Если в заседании приобщены новые документы — добавь записи в `index.yaml` с пометкой `source: court` или соответствующей стороны».

Но:
1. **Файла документа нет.** В заседании приобщили бумагу, Сюзерен получил копию определения о приобщении. Физически PDF/скан появится позже (неделя-две до получения из суда). Что записать в `index.yaml` сейчас?
2. **Нет md-зеркала.** Добавление в `index.yaml` без зеркала ломает инвариант [mirror-template.md](../shared/mirror-template.md).
3. **filing_status.** Если приобщили **наш** документ — нужно проставить `filing_status: filed` в существующей записи. Если чужой — создать placeholder.

**Правка:** ввести статус документа `pending` / `announced` (документ объявлен в заседании, физически ещё нет) + `awaiting: true` в схеме. `analyze-hearing` создаёт placeholder. Последующий `intake` / `add-opponent` — заполняет физический файл и меняет статус.

### 5.9. Аудиозапись как исходник (не транскрипция)

Description явно говорит: «Скилл работает с текстовыми транскрипциями (не с аудио напрямую)».

Но реалистичный сценарий:
- Сюзерен кладёт `.mp3` в «Входящие документы/».
- Запускает `analyze-hearing`.

Что произойдёт? Модель попытается прочитать бинарник. Упадёт молчаливо или выдаст бессмыслицу.

**Правка:** явная проверка расширения. Если .mp3/.m4a/.wav/.ogg — сообщить: «Плагин не транскрибирует аудио. Рекомендуемые сервисы: [список]. После получения .txt — запусти скилл повторно».

### 5.10. Нет заседания по техническим причинам

Связано с [WORKFLOW-REVIEW.md §8](../WORKFLOW-REVIEW.md). Скилл никак не обрабатывает сценарий «судья ушёл на больничный, заседание не состоялось». А это частая ситуация.

**Правка:** новая команда/подрежим `/analyze-hearing --no-session "перенос на 15 мая, судья на больничном"` — просто фиксирует факт в timeline + обновляет `next_hearing`.

### 5.11. Сравнение транскрипции и протокола — глубина анализа

[SKILL.md:92-103](../skills/analyze-hearing/SKILL.md) фаза 4 опциональна. Критерий «существенности» расхождений не формализован.

Юридически важные расхождения (по Мошкину):
- Искажение процессуальных ходатайств (заявили, а в протоколе нет).
- Искажение ответов сторон на вопросы суда.
- Искажение правовой квалификации.
- Пропуск приобщения документов.

Не-существенные: формулировки вводных реплик, словесные огрехи.

Сейчас: порог «существенных» — на усмотрение модели. Разные дела → разный фильтр.

**Правка:** чек-лист «что считать существенным» из 5-7 пунктов, с примерами.

### 5.12. Анонимизация для клиентской версии

Client-brief может содержать чувствительные детали (позиция судьи «явно не в нашу пользу»), которые юристу неуместно показывать клиенту в таком виде. Сейчас различие только в общем уровне детализации.

**Правка:** клиентская версия должна фильтровать:
- Оценочные суждения о судье.
- Внутренние тактические замечания.
- «Слабости нашей позиции» (хотя доктринально спорно — часть юристов считает, что клиент должен знать).

Это политическое решение, но должно быть настраиваемо в SKILL.md (или флагом в `case.yaml`).

### 5.13. Срок на замечания уже вышел

SKILL.md предлагает замечания, если Сюзерен даёт протокол для сверки. Но не проверяется — **прошло ли уже 5 дней** (АПК ст. 155 ч. 7) или **5 дней** (ГПК ст. 231). Если заседание было 2 недели назад, замечания подать нельзя.

**Правка:** в фазе 4 проверять разницу между текущей датой и датой подписания протокола. Если > срока — предупредить «срок пропущен», предложить **восстановление срока** через ходатайство (с обоснованием).

## 6. Реальные проблемы на практике

### 6.1. Стоимость Opus-анализа транскрипции большого заседания

3-часовое заседание — ~40-60k токенов транскрипции. Opus-анализ с цитированием = минимум 2x на вход, 20-30k на выход. Один запуск — **несколько долларов**. Не описано в UI — юрист видит счёт постфактум.

Для скилла, который запускается после каждого заседания (и в многокруговых делах — десятки раз), это материальные деньги.

**Правка:** показывать в preview предполагаемую стоимость анализа: «Opus-анализ: ~X тыс. токенов, стоимость ~$Y».

### 6.2. «Профиль судьи по одному заседанию» — фальшивая уверенность

После первого заседания скилл сгенерирует `judge-profile.md` с разделами «стиль», «предпочтения», «триггеры». Но одного заседания недостаточно для надёжного профиля — это **выборка из 1**.

Юрист получит «профиль», будет применять его к стратегии — а это перегонка шума в сигнал.

**Правка:**
1. В шапке профиля — `hearings_analyzed: N` (уже есть). Под N добавить confidence: `low` при N<3, `medium` при N=3-5, `high` при N>5.
2. Явный disclaimer в профиле: «Основано на 1 заседании — требует подтверждения».
3. В SKILL.md [строка 130](../skills/analyze-hearing/SKILL.md): «создавать `judge-profile.md` только если накоплено достаточно данных» — но «достаточно» не определено. Ввести порог: не создавать профиль при N<2; вместо этого — `judge-observations.md` (наблюдения, не профиль).

### 6.3. Post-hearing cognitive load на юриста

После заседания юрист устал. Preview с 20 предложенными изменениями `case.yaml` + «посмотрите на 2-3 пункта с `[?]`» + «подтвердить ли замечания на протокол?» + «оформить ли .docx?» — слишком много решений за раз.

**Правка:** разбить preview на 3 коротких шага:
1. Фактология: даты, следующее заседание, приобщённые документы.
2. Качество: что помечено `[?]`, что подтвердить.
3. Follow-up: замечания, отчёт клиенту, оформление.

Каждый — на отдельную сессию подтверждения.

### 6.4. Юрист правит отчёт клиенту руками — ничего не подтягивается

Аналог проблемы из [reviews/prepare-hearing.md §6.5](prepare-hearing.md): Сюзерен открывает .docx, правит, отправляет клиенту. Правки не попадают обратно в `.md`. При повторном запуске скилл видит «старую» версию.

### 6.5. «Транскрипция важнее протокола» — но Сюзерен сам транскрипцию не делает

Методологически скилл опирается на транскрипцию. Но:
- Откуда она? Сюзерен платит отдельному сервису? Сам расшифровывает аудиозапись? Клиент делает?
- Стоимость транскрипции судебного заседания (3 часа) — порядка 1-3 тыс. рублей через коммерческий сервис.
- Не все суды дают доступ к аудиозаписи сразу — некоторые выдают через месяц.

Скилл игнорирует эту «логистическую» проблему. На практике многие заседания проходят без транскрипции → скилл работает в режиме «только протокол» (30% информации) → метод Мошкина не реализуется.

**Правка:** в SKILL.md или в отдельном документе — ссылки на рекомендуемые сервисы транскрипции, правила получения аудиозаписи из суда (ст. 155 АПК).

### 6.6. Слишком много файлов в hearings/ после 10 заседаний

Предварительное, основное, отложение → 30-40 файлов за дело. Навигация сложная.

Структура сейчас плоская: `.vassal/hearings/{дата}-internal.md`, `{дата}-client-brief.md`, и т.д.

**Правка:** папка на заседание — `.vassal/hearings/{дата}/internal.md`, `{дата}/client-brief.md`, `{дата}/opponent-analysis.md`. `judge-profile.md` — на уровень выше (`.vassal/hearings/judge-profile.md`, накопительный).

### 6.7. Опечатка в метаданных внутреннего отчёта

[SKILL.md:162](../skills/analyze-hearing/SKILL.md):
```yaml
next_hearing: ГГГГ-ММ-ДД или null
```

В YAML «или null» — строка, а не схема. Нужно `next_hearing: null  # ГГГГ-ММ-ДД если назначено`. Мелочь, но при копировании шаблона юристу — риск синтаксической ошибки.

## 7. Рекомендации (приоритизированно)

### P0 (критично)

1. **Разрешить коллизию `opponent-*.md`** с `add-opponent` (§4.3) — минимум префикс-разведение, иначе анализ устной позиции оппонента теряется.
2. **Унифицировать путь к `judge-profile.md`** (§4.4) — убрать дату из имени, сделать накопительным.
3. **Постусловия + валидация** `case.yaml`, YAML-валидность, наличие всех 4 отчётов (§2.4).
4. **Preview дат с цитатами-обоснованиями** (§5.6) — предотвращение галлюцинаций «5 мая / 15 мая».
5. **Fallback для `arbitrum-docx`** (§4.12) — особенно для замечаний на протокол (срок!).

### P1 (важно)

7. **Связка с `{дата}-prep.md`** — раздел «План vs факт» во внутреннем отчёте (§4.6). Замыкает цикл prepare → analyze.
8. **Кросс-дело профиль судьи** (§4.5) — минимум опция `shared_profile: path`, в идеале — Notion-интеграция.
9. **Расширение схемы `case.yaml: timeline`** (§4.8): поля `ref`, `related_docs`.
10. **Единый словарь `source`** в conventions.md (§4.9).
11. **Frontmatter во всех 4 отчётах** (§4.7).
12. **Политика повторного запуска** (§5.5).
13. **Порог качества транскрипции перед Opus** (§5.4) — не запускать глубокий анализ на мусоре.
14. **Обработка документов, объявленных в заседании** (§5.8) — статус `pending/announced` в index.yaml.
15. **Проверка срока замечаний на протокол** (§5.13) — предупреждение о пропуске.
16. **Размещение замечаний в процессуальную папку, не в корень** (§4.10).
17. **Оперативный режим — опциональное сохранение** (§4.11).
18. **Интерактивный режим «без транскрипции»** (§5.1).

## 8. Резюме

Analyze-hearing — **концептуально самый амбициозный** скилл плагина. Редкий пример в legal-tech, где система осознанно работает с *устной* составляющей процесса (паттерны речи судьи, уклончивость оппонента), а не только с документами. Три отчёта, разделение моделей (Sonnet + Opus), правильное недоверие к транскрипции (`[?]`, preview) — всё это хорошая инженерия.

Но в деталях — системные проблемы:

1. **Главная архитектурная проблема — изолированность ценности.** Два уникальных value-add скилла (профиль судьи + анализ устной позиции оппонента) фактически **теряются между циклами**:
   - Профиль судьи живёт в одном деле. У того же судьи в другом деле — пустой профиль. Кросс-дело памяти нет. Это именно то, что предлагает закрыть Notion-слой (§4.5, NOTION-INTEGRATION-PROPOSAL §2.3).
   - Анализ оппонента по заседанию пишется в путь, где его не ищет `prepare-hearing` (коллизия с `add-opponent`, §4.3). Значит, при подготовке к следующему заседанию — заново (или не учитывается вовсе).

3. **Разрыв цикла с `prepare-hearing`.** Два парных скилла (до/после заседания) не разговаривают. Нет раздела «план vs факт», хотя именно эта функция — главный обучающий механизм (§4.6). Без неё аналитика prepare-hearing существует в вакууме, результаты заседания — в вакууме, и связи нет.

4. **Preview дат без цитат** (§5.6) — единственная защита от галлюцинаций модели на данных, которые нельзя воспроизвести (транскрипция одноразовая). Сюзерен устал после заседания и подтверждает «ок» не глядя. Нужна принудительная сверка с цитатой.

Три правки, закрывающие критическую массу:
1. **Разведение `opponent-*.md` путей + раздел «План vs факт» в internal.md** (§4.3, §4.6).
2. **Preview дат с обязательной цитатой-обоснованием + маркер `[?]`** (§5.6).
3. **Накопительный `judge-profile.md` без даты в имени** (§4.4) — иначе value-add Мошкина не работает кросс-дело.
